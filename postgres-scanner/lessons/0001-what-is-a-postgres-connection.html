The PostgreSQL Connection: Why Every Connection Has a Cost

You are scanning 3,000 PostgreSQL servers concurrently. Before every query, your scanner opens a connection. To keep your scanner safe — and to keep production DBAs trusting you — you need to understand exactly what happens the moment you call pgx.Connect().

The Core Insight
Every PostgreSQL connection is a real OS process on the server you are connecting to. Not a thread. Not a file descriptor. A full, forked process. This is the most important fact about PostgreSQL connection management.

1. PostgreSQL's Process Model

Most databases use threads for connections. PostgreSQL is different: it uses processes. The architecture looks like this:

sequenceDiagram participant SC as Your scanner (Go) participant PM as postmaster participant B1 as backend #1 participant B2 as backend #2 participant B3 as backend #3 SC->>PM: goroutine 1 — TCP connect PM->>B1: fork() → new OS process B1-->>SC: ready (~2–5 MB RAM) SC->>PM: goroutine 2 — TCP connect PM->>B2: fork() → new OS process B2-->>SC: ready (~2–5 MB RAM) SC->>PM: goroutine 3 — TCP connect PM->>B3: fork() → new OS process B3-->>SC: ready (~2–5 MB RAM)
Each TCP connect from your scanner forks a real OS process on the server

The postmaster is the parent process. When your scanner connects, postmaster calls fork() to create a new child process — the backend process — dedicated to your connection. That process lives until your connection closes.

Why does PostgreSQL use processes instead of threads?
Isolation. If one backend crashes (e.g. due to a corrupt query), it cannot corrupt memory shared by other connections. The postmaster simply cleans up the dead process. Threads in the same address space don't have this protection. See: Analyzing the Limits of Connection Scalability in Postgres (Citus Data)

2. What Each Connection Costs the Server

Opening a connection is not free. It costs the server you are connecting to:

ResourceCost per connectionNotes
RAM ~2–5 MB (idle) More if a query uses work_mem (default 4 MB) for sorts/hashes. An AWS study measured ~1.5 MB idle, ~14.5 MB under load per connection. [source]
CPU (context switches) Small but real Every SQL round-trip requires a kernel context switch to the backend process. At 1,000+ connections this becomes measurable overhead.
Shared data structures Slot in procarray & lock table PostgreSQL's lock table and process array are fixed-size. Every connection — even idle — holds a slot. This is why 10,000 connections hurts even with plenty of RAM.
Authentication time ~1–10 ms TCP handshake + TLS + password hash. For your scanner, multiplied by 3,000 servers.
The max_connections Hard Limit
Every PostgreSQL server has a max_connections setting (default: 100, production servers often set to 200–500). If your scanner opens a connection and the server is already at its limit, you get: 
FATAL: sorry, too many clients already
Worse: your scanner could be the connection that pushes a production server over its limit, blocking legitimate application traffic.

3. The Scanner Is Different from a Web App

A typical web application uses a connection pool: it opens 10–50 connections at startup and reuses them for the lifetime of the application. This amortizes the connection cost over thousands of requests.

Your scanner is fundamentally different:

Target servers 1 (or a few) 3,000+
Connection lifetime Hours/days (pooled) Seconds (open, check, close)
Queries per connection Thousands 40–50 (then done)

one ephemeral connection per server

For your scanner, the right pattern is: connect → run all checks → close immediately. Each server gets exactly one connection for the duration of its health check.

4. The Five Rules of a Polite Scanner

You are a guest on production servers you do not own. These five rules ensure you leave no trace and cause no harm.

Rule 1: Always set connect_timeout

Without a timeout, your goroutine will hang indefinitely if a server is down or firewalled. 3,000 servers means you will encounter many dead hosts.

Rule 2: Always set statement_timeout

Some catalog queries (pg_stat_user_tables, bloat queries) can run for seconds or minutes on a loaded server with millions of tables. Without this, your scanner becomes the problem. Set it to 5,000 ms (5 seconds) as a starting point.

Rule 3: Always set lock_timeout

A few monitoring queries acquire brief row-level locks on catalog tables. If another transaction holds that lock, your query waits. Set lock_timeout to 1,000 ms to fail fast instead of blocking.

Rule 4: Always set application_name

Every DBA watches pg_stat_activity. If they see an unknown connection running queries, they will kill it. Set application_name to something identifiable like healthchecker-v1. This is the equivalent of introducing yourself before sitting at someone's desk.

Rule 5: Use the pg_monitor role

Ask DBAs to grant pg_monitor to your scanner user. It gives read access to all pg_stat_* views without superuser privileges. Never run health checks as a superuser — it wastes reserved connection slots and is a larger blast radius if something goes wrong.

5. Putting It Together in Go (pgx v5)

Here is the minimal safe connection pattern for your scanner:

import (
    "context"
    "fmt"
    "time"
    "github.com/jackc/pgx/v5"
)

func checkServer(host, dbname, user, password string) error {
    // Build DSN with connect_timeout in the URL
    dsn := fmt.Sprintf(
        "postgres://%s:%s@%s/%s?connect_timeout=5&sslmode=prefer",
        user, password, host, dbname,
    )

    config, err := pgx.ParseConfig(dsn)
    if err != nil { return err }

    // Safety parameters — set before any query runs
    config.RuntimeParams["application_name"]                  = "healthchecker-v1"
    config.RuntimeParams["statement_timeout"]                   = "5000"  // 5s, in ms
    config.RuntimeParams["lock_timeout"]                        = "1000"  // 1s, in ms
    config.RuntimeParams["idle_in_transaction_session_timeout"]  = "10000" // 10s, in ms

    // Context carries a hard outer deadline
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()

    conn, err := pgx.ConnectConfig(ctx, config)
    if err != nil { return fmt.Errorf("connect: %w", err) }
    defer conn.Close(context.Background()) // always close!

    // Run your 40-50 health check queries here...
    return runChecks(ctx, conn)
}
Why RuntimeParams?
Setting statement_timeout via RuntimeParams in pgx v5 sends a SET command as part of the connection startup message — before any query runs. This is safer than sending SET statement_timeout = ... as a separate query, because it is atomic with the connection establishment.

Concurrency: The Worker Pool Pattern

You have 3,000 servers to scan. You should not spawn 3,000 goroutines all opening connections at once. Instead, use a semaphore (buffered channel) to cap the number of simultaneous connections: 

const maxConcurrent = 200 // tune this: start at 100, increase if safe

func scanAll(servers []Server) {
    sem := make(chan struct{}, maxConcurrent)
    var wg sync.WaitGroup

    for _, server := range servers {
        wg.Add(1)
        sem <- struct{}{}        // acquire a slot (blocks when maxConcurrent reached)

        go func(s Server) {
            defer wg.Done()
            defer func() { <-sem }() // release slot when done
            if err := checkServer(s.Host, s.DBName, s.User, s.Password); err != nil {
                // log err, mark server as failed
            }
        }(server)
    }

    wg.Wait()
}
Why a semaphore and not just a worker pool?
Both work. A semaphore is simpler when the work items (servers) are already in a slice. A channel-based worker pool is better when items arrive as a stream. You are already using the semaphore pattern — this is just giving it a name and explaining why it is correct.

Summary

  • Every PostgreSQL connection = a forked OS process (~2–5 MB RAM, holds a lock-table slot)
  • max_connections is a hard limit; exceeding it produces a fatal error
  • Your scanner uses ephemeral connections: open → check → close immediately
  • Five safety rules: connect_timeout, statement_timeout, lock_timeout, application_name, pg_monitor role
  • Use a semaphore (buffered channel) to cap concurrent connections from your scanner
