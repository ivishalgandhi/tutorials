Error Handling & Timeouts

A scanner that hits 3,000 servers will encounter every class of failure PostgreSQL can produce: servers that are down, servers that reject authentication, queries that time out, networks that drop packets. The question is not whether errors will happen — they will — but whether one failing server can corrupt, stall, or crash the goroutine scanning the next server. It cannot. This lesson establishes the patterns that make that guarantee.

The Core Rule
Per-server isolation: every error is contained to the goroutine that encountered it. A connection refusal on server 412 must not affect server 413. A panic in a goroutine must not propagate to the scan coordinator. errors.As() classifies the error. A deferred recover captures panics. The coordinator only sees a typed result.

1. The Four Error Classes

Every error your scanner receives falls into one of four classes. Each class has a different cause, a different Go type, and a different correct response:

Class Go type Example causes Scanner response
Network error *net.OpError Connection refused, host unreachable, TCP timeout Skip server, log as unreachable
Server error *pgconn.PgError Auth failure, missing permission, query cancelled Classify by SQLSTATE code (see below)
Context error context.DeadlineExceeded connect_timeout fired in Go before TCP handshake Skip server, log as connect_timeout
Statement timeout *pgconn.PgError SQLSTATE 57014 Query exceeded statement_timeout Skip category, log as statement_timeout, continue other checks
SQLSTATE codes
Every PostgreSQL error has a 5-character SQLSTATE code. The first two characters are the class. For example: 28 = invalid authorisation, 57 = operator intervention, 3D = invalid catalog name. The full table is at postgresql.org/docs/current/errcodes-appendix.html. Key codes for a scanner:

28000 / 28P01 — invalid authorisation (wrong password / pg_hba rejected)
3D000 — invalid catalog name (database does not exist)
57014 — query_canceled (statement_timeout or lock_timeout fired)
55P03 — lock_not_available (lock_timeout fired)
42501 — insufficient_privilege (user lacks SELECT on view)

2. Classifying Errors with errors.As()

pgx wraps errors. Never compare with ==; always use errors.As() or errors.Is() to unwrap: 

import (
    "context"
    "errors"
    "net"
    "github.com/jackc/pgx/v5/pgconn"
)

type ErrClass string

const (
    ErrOK               ErrClass = "ok"
    ErrUnreachable      ErrClass = "unreachable"
    ErrAuthFailure      ErrClass = "auth_failure"
    ErrDBNotFound       ErrClass = "db_not_found"
    ErrPermission       ErrClass = "permission_denied"
    ErrStatementTimeout ErrClass = "statement_timeout"
    ErrConnectTimeout   ErrClass = "connect_timeout"
    ErrUnknown          ErrClass = "unknown"
)

func classifyErr(err error) ErrClass {
    if err == nil { return ErrOK }

    // Server-reported error: inspect SQLSTATE code
    var pgErr *pgconn.PgError
    if errors.As(err, &pgErr) {
        switch pgErr.Code {
        case "28000", "28P01": return ErrAuthFailure
        case "3D000":           return ErrDBNotFound
        case "42501":           return ErrPermission
        case "57014", "55P03": return ErrStatementTimeout
        }
        return ErrUnknown
    }

    // Go context deadline: connect_timeout fired client-side
    if errors.Is(err, context.DeadlineExceeded) {
        return ErrConnectTimeout
    }

    // Network error: server unreachable
    var netErr *net.OpError
    if errors.As(err, &netErr) {
        return ErrUnreachable
    }

    return ErrUnknown
}
errors.As vs errors.Is
Use errors.As(err, &target) when you want to inspect a specific error type (like *pgconn.PgError to read the SQLSTATE code). Use errors.Is(err, sentinel) when you want to match a specific sentinel value (like context.DeadlineExceeded). Both unwrap error chains — they work even if pgx has wrapped the original error several layers deep.

3. Per-Server Isolation

Each goroutine must return a typed result — never panic, never call log.Fatal, never propagate raw errors up through a channel. The deferred recover() is the last line of defence: 

type ScanResult struct {
    Server   Server
    Checks   []CheckResult
    ErrClass ErrClass
    ErrMsg   string
}

func checkServer(ctx context.Context, s Server) (result ScanResult) {
    result.Server = s

    // Catch any panic — a bug in one server's handler must not crash the scanner
    defer func() {
        if r := recover(); r != nil {
            result.ErrClass = ErrUnknown
            result.ErrMsg = fmt.Sprintf("panic: %v", r)
        }
    }()

    conn, err := connectSafe(ctx, s)
    if err != nil {
        result.ErrClass = classifyErr(err)
        result.ErrMsg = err.Error()
        return // named return — deferred recover still runs
    }
    defer conn.Close(ctx)

    for _, path := range []string{
        "checks/activity.toml",
        "checks/replication.toml",
        "checks/vacuum.toml",
    } {
        checks, err := runCategory(ctx, conn, path)
        if err != nil {
            // Category failed (e.g. statement_timeout) — record and continue
            result.Checks = append(result.Checks, CheckResult{
                Name:     path,
                ErrClass: classifyErr(err),
                ErrMsg:   err.Error(),
            })
            continue
        }
        result.Checks = append(result.Checks, checks...)
    }
    return
}
Named returns + defer
The function returns a named result ScanResult. When an early return fires, the deferred recover() still executes and can modify result before the value is actually returned. This is the idiomatic Go pattern for per-goroutine panic isolation.

4. Permanent vs Transient Errors

Two errors look similar but require completely different responses:

ErrorClassRetry?Why
Auth failure (28P01) Permanent No Wrong credentials will not self-heal. Retrying creates noise in pg_log.
Statement timeout (57014) Permanent for this scan No If the health check itself is slow, the server is the problem to report, not retry.
Connection refused Transient (maybe) Once, after 2s Server may be restarting. One retry is reasonable; more creates thundering herd risk.
Context cancelled Permanent No The scan coordinator cancelled the context intentionally. Stop immediately.
Never retry auth failures
An authentication failure retried 3,000 times across 3,000 servers per scan run will fill pg_log with noise, trigger fail2ban rules, and potentially lock out the pg_monitor user on servers with auth_delay configured. Auth failures are a configuration problem — fix the credentials, don't retry.

5. The Error Budget Concept

A single error is expected. A pattern of errors is a signal. After each scan run, compute the error rate per class: 

type ScanSummary struct {
    Total        int
    Succeeded    int
    ByErrClass   map[ErrClass]int
}

func summarise(results []ScanResult) ScanSummary {
    s := ScanSummary{ByErrClass: make(map[ErrClass]int)}
    for _, r := range results {
        s.Total++
        if r.ErrClass == ErrOK {
            s.Succeeded++
        } else {
            s.ByErrClass[r.ErrClass]++
        }
    }
    return s
}

// Thresholds to alert on:
// unreachable     > 5%   → network issue or inventory stale
// auth_failure    > 1%   → credential rotation problem
// stmt_timeout    > 2%   → check query too expensive for these servers

6. The Decision Tree

flowchart TD A([error returned]) --> B{errors.As pgconn.PgError?} B -- yes --> C{SQLSTATE code} C -- 28000/28P01 --> D[auth_failure\nskip server\nno retry] C -- 3D000 --> E[db_not_found\nskip server] C -- 42501 --> F[permission_denied\ncheck pg_monitor role] C -- 57014/55P03 --> G[statement_timeout\nskip category\ncontinue others] C -- other --> H[unknown pg error\nlog code + message] B -- no --> I{errors.Is DeadlineExceeded?} I -- yes --> J[connect_timeout\nskip server] I -- no --> K{errors.As net.OpError?} K -- yes --> L[unreachable\nskip server] K -- no --> M[unknown\nlog raw error]
Every error path terminates with a classification — never a bare error string sent to the coordinator

Summary

  • Every error falls into one of four classes: network, server (PgError), context deadline, or statement timeout
  • Use errors.As(err, &pgErr) to get the SQLSTATE code; use errors.Is for sentinel values like context.DeadlineExceeded
  • Each goroutine returns a typed ScanResult — never panics, never calls log.Fatal
  • A deferred recover() catches panics and writes them into the result — named returns make this clean
  • Category-level errors (statement timeout on one check) should not abort the other categories for the same server
  • After a scan, compute error rates by class — a spike in auth_failure or unreachable is a systemic signal
  • Never retry auth failures. One retry for connection refused is acceptable; more creates a thundering herd
