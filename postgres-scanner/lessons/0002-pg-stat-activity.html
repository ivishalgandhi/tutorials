pg_stat_activity: The View Your Scanner Reads Most

Your scanner connects to a server, runs 40–50 checks, and disconnects. The most important of those checks query a single view: pg_stat_activity. It is PostgreSQL's live window into every backend process on the server — what each connection is doing right now.

The Core Insight
pg_stat_activity has one row per backend process. The state column tells you whether each connection is doing work, sitting idle, or — the dangerous case — sitting idle inside an open transaction. From three queries against this view, your scanner can tell if a server is healthy, saturated, or quietly accumulating problems.

1. What pg_stat_activity Contains

Each row is a backend process (see Lesson 1). The columns your scanner cares about:

ColumnTypeWhat it tells you
pidintegerOS process ID of this backend
usenamenameThe database user for this connection
application_nametextThe client's self-reported name (from RuntimeParams — this is why we set it in Lesson 1)
statetextWhat this backend is doing right now — the most important column
backend_typetextWhether this is a client connection, autovacuum worker, WAL sender, etc.
query_starttimestamptzWhen the current query started (NULL if idle)
xact_starttimestamptzWhen the current transaction started (NULL if no active transaction)
wait_event_typetextWhy the backend is waiting (Lock, IO, Client, etc.) — NULL if not waiting
querytextThe most recent SQL statement (truncated at track_activity_query_size bytes, default 1024)
Permission note
Without the pg_monitor role, non-superusers see all rows but the query column is blanked out for other users' sessions. With pg_monitor, your scanner sees everything. This is one of the key reasons to request that role. [docs]

2. The state Column — What It Really Means

state is the health signal. Every scanner check against pg_stat_activity filters on this column.

state valueMeaningDanger level
active Executing a query right now Normal — this is what healthy connections look like
idle Connected, no query, no open transaction Benign — but many idle connections waste process slots
idle in transaction Inside a BEGIN block but not executing a query Dangerous — holds locks, blocks autovacuum, can escalate to connection exhaustion
idle in transaction (aborted) Transaction failed but was never rolled back Dangerous — same risks as above; typically a bug in application code
fastpath function call Executing a fast-path function Rare, benign for your purposes
NULL Background worker (autovacuum, WAL sender, etc.) Filter these out with backend_type = 'client backend'
Why "idle in transaction" is the silent killer
A session in idle in transaction is sitting inside a BEGIN block doing nothing. While it waits, it holds a transaction snapshot — PostgreSQL cannot clean up (vacuum) any row that was visible when this transaction started. On a busy table, this causes dead-tuple accumulation and bloat within minutes. It also holds any row locks it acquired, blocking other writers.

This is why idle_in_transaction_session_timeout (set in Lesson 1) is critical: it automatically kills these sessions before they cause damage. [source]

3. Three Queries Your Scanner Runs

From pg_stat_activity alone, your scanner extracts three signals. All three run on the same connection, one after another.

Query A — Connection Saturation

How full is this server? A server at 90%+ of max_connections is one spike away from rejecting connections entirely. 

-- How saturated is this server's connection limit?
SELECT
  count(*) AS total,
  count(*) FILTER (WHERE state = 'active')              AS active,
  count(*) FILTER (WHERE state = 'idle')                AS idle,
  count(*) FILTER (WHERE state = 'idle in transaction') AS idle_in_txn,
  (SELECT setting::int
   FROM pg_settings
   WHERE name = 'max_connections')                       AS max_connections
FROM pg_stat_activity
WHERE backend_type = 'client backend';
Why backend_type = 'client backend'?
pg_stat_activity includes rows for PostgreSQL's own internal processes: autovacuum workers, WAL senders, checkpointers, background workers. Filtering to 'client backend' gives you only actual client connections — the ones that count against max_connections.

Thresholds for connection saturation (total / max_connections):

Used %SignalWhat to do
< 70%HealthyNo action needed
70–90%WarningAlert the DBA team; investigate connection pooler config
> 90%CriticalImminent risk of connection storm; escalate immediately

Any non-zero idle_in_txn count older than 60 seconds is independently alertable.

Query B — Long-Running Queries

A query running for more than 5 minutes is almost certainly stuck, blocking, or a runaway. Your scanner flags these as signals to investigate. 

-- Queries running longer than 5 minutes right now
SELECT
  pid,
  usename,
  application_name,
  now() - query_start                AS query_age,
  wait_event_type,
  wait_event,
  left(query, 120)                   AS query_snippet
FROM pg_stat_activity
WHERE state = 'active'
  AND query_start < now() - interval '5 minutes'
  AND backend_type = 'client backend'
ORDER BY query_start ASC;

Thresholds for query age:

AgeSignalNotes
< 1 minNormalMost OLTP queries should be milliseconds; some analytics are expected
1–5 minWatchCould be a slow report; check wait_event_type
> 5 minAlertLikely blocking or a runaway; note wait_event = 'Lock' specifically

Query C — Idle-in-Transaction Age

Even a single idle-in-transaction session older than 60 seconds can hold locks and block autovacuum on busy tables. This query surfaces the worst offenders. 

-- Sessions stuck inside a transaction doing nothing
SELECT
  pid,
  usename,
  application_name,
  now() - xact_start   AS txn_age,
  now() - state_change AS idle_for,
  left(query, 80)       AS last_query
FROM pg_stat_activity
WHERE state IN ('idle in transaction', 'idle in transaction (aborted)')
  AND xact_start < now() - interval '30 seconds'
ORDER BY xact_start ASC;

4. How Your TOML-Driven Scanner Runs These

Your scanner has a two-phase design. Understanding this unlocks why connections.toml is special compared to every other file in checks/.

sequenceDiagram participant CLI as Go CLI participant CT as connections.toml participant IDB as Inventory DB participant S1 as server 1 participant S2 as server 2 participant SN as server 3000 Note over CLI,CT: Phase 1 — Inventory lookup CLI->>CT: read inventory DSN + query CLI->>IDB: connect CLI->>IDB: SELECT host, port, dbname... IDB-->>CLI: [server1 … server3000] CLI->>IDB: close Note over CLI,SN: Phase 2 — Concurrent health checks (semaphore-limited) par CLI->>S1: connect + run checks/*.toml S1-->>CLI: results CLI->>S1: close and CLI->>S2: connect + run checks/*.toml S2-->>CLI: results CLI->>S2: close and CLI->>SN: connect + run checks/*.toml SN-->>CLI: results CLI->>SN: close end
Phase 1 is sequential — one inventory query. Phase 2 is concurrent — up to N servers at once (semaphore).

connections.toml is the bootstrap file — it holds the inventory database connection and the query that produces the server list. Every other TOML under checks/ is a health-check category run against each server in that list. 

# connections.toml — inventory DB config + server discovery

[inventory]
dsn   = "postgres://scanner:secret@inventory-db:5432/infra"
query = """
  SELECT host, port, dbname, username, password
  FROM servers
  WHERE active = true
  ORDER BY host
"""

# checks/activity.toml — health checks run against each scanned server

[[checks]]
name        = "connection_saturation"
description = "Connection usage vs max_connections"
sql         = """
  SELECT
    count(*)                                               AS total,
    count(*) FILTER (WHERE state = 'active')              AS active,
    count(*) FILTER (WHERE state = 'idle in transaction') AS idle_in_txn,
    (SELECT setting::int FROM pg_settings
     WHERE name = 'max_connections')                      AS max_connections
  FROM pg_stat_activity
  WHERE backend_type = 'client backend'
"""

[[checks]]
name        = "long_running_queries"
description = "Queries active for more than 5 minutes"
sql         = """
  SELECT count(*) FROM pg_stat_activity
  WHERE state = 'active'
    AND query_start < now() - interval '5 minutes'
    AND backend_type = 'client backend'
"""

[[checks]]
name        = "stuck_transactions"
description = "Sessions idle in transaction for more than 30 seconds"
sql         = """
  SELECT count(*) FROM pg_stat_activity
  WHERE state IN ('idle in transaction', 'idle in transaction (aborted)')
    AND xact_start < now() - interval '30 seconds'
"""

At runtime: Phase 1 reads the inventory, Phase 2 fans out. Because the result shape varies per query, rows are scanned into []any dynamically: 

// ── Types ────────────────────────────────────────────────────

type Server struct {
    Host, Port, DBName, User, Password string
}

type InventoryConfig struct {
    DSN   string `toml:"dsn"`
    Query string `toml:"query"`
}

type ConnectionsFile struct {
    Inventory InventoryConfig `toml:"inventory"`
}

type Check struct {
    Name        string `toml:"name"`
    Description string `toml:"description"`
    SQL         string `toml:"sql"`
}

type CategoryFile struct {
    Checks []Check `toml:"checks"`
}

type CheckResult struct {
    Name    string
    Columns []string
    Rows    [][]any
    Err     error
}

// ── Phase 1: load server list from inventory DB ───────────────

func loadServers(ctx context.Context) ([]Server, error) {
    var cfg ConnectionsFile
    if _, err := toml.DecodeFile("connections.toml", &cfg); err != nil {
        return nil, err
    }

    conn, err := pgx.Connect(ctx, cfg.Inventory.DSN)
    if err != nil { return nil, fmt.Errorf("inventory connect: %w", err) }
    defer conn.Close(ctx)

    rows, err := conn.Query(ctx, cfg.Inventory.Query)
    if err != nil { return nil, err }
    defer rows.Close()

    var servers []Server
    for rows.Next() {
        var s Server
        if err := rows.Scan(&s.Host, &s.Port, &s.DBName, &s.User, &s.Password); err != nil {
            return nil, err
        }
        servers = append(servers, s)
    }
    return servers, rows.Err()
}

// ── Phase 2: run one category file against an open connection ──

func runCategory(ctx context.Context, conn *pgx.Conn, path string) ([]CheckResult, error) {
    var cat CategoryFile
    if _, err := toml.DecodeFile(path, &cat); err != nil {
        return nil, fmt.Errorf("load %s: %w", path, err)
    }

    results := make([]CheckResult, 0, len(cat.Checks))
    for _, chk := range cat.Checks {
        res := CheckResult{Name: chk.Name}

        rows, err := conn.Query(ctx, chk.SQL)
        if err != nil {
            res.Err = fmt.Errorf("%s: %w", chk.Name, err)
            results = append(results, res)
            continue
        }
        for _, fd := range rows.FieldDescriptions() {
            res.Columns = append(res.Columns, fd.Name)
        }
        for rows.Next() {
            vals, _ := rows.Values() // []any — pgx maps PG types to Go automatically
            res.Rows = append(res.Rows, vals)
        }
        rows.Close()
        res.Err = rows.Err()
        results = append(results, res)
    }
    return results, nil
}
Why rows.Values() instead of Scan()
Scan() requires knowing the column types at compile time. Since your queries live in TOML files and can change without recompiling, rows.Values() is the right tool — it returns each row as []any, letting you iterate columns dynamically. pgx maps PostgreSQL types to sensible Go types automatically: int8 → int64, text → string, interval → pgtype.Interval. The inventory query uses Scan() because you do know the shape — it always returns the same server columns.

5. How to Read the Output

Each CheckResult has a name, column list, and rows. Printing them for the connections category looks like this — and the same loop works for every category: 

for _, r := range results {
    if r.Err != nil {
        fmt.Printf("[ERROR] %s: %v\n", r.Name, r.Err)
        continue
    }
    fmt.Printf("\n── %s ──\n", r.Name)
    fmt.Println(strings.Join(r.Columns, " | "))
    for _, row := range r.Rows {
        parts := make([]string, len(row))
        for i, v := range row {
            parts[i] = fmt.Sprintf("%v", v)
        }
        fmt.Println(strings.Join(parts, " | "))
    }
}

Summary

  • pg_stat_activity has one row per backend process; filter to backend_type = 'client backend' to count only real connections
  • The state column is the key signal: active is healthy, idle in transaction is dangerous
  • Three queries cover the main failure modes: connection saturation, long-running queries, and stuck transactions
  • Run all three on the same connection you already opened — no extra connections needed
  • Alert thresholds: >90% saturation = critical; >70% = warning; any stuck txn >30s = warning
