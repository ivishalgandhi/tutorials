pgx v5 — The Right Driver for a Scanner

The previous lessons built a working scanner using Go's standard database/sql package. It connects, queries, and processes results — but at 3,000 concurrent targets the seams show: rows.Scan() demands compile-time column knowledge, DSN string construction is brittle, and a failed connection returns a plain error that tells you nothing about why it failed. This lesson migrates the scanner to pgx v5 — a PostgreSQL-native Go driver that solves all three problems.

1. Why database/sql Is Not the Right Tool for a Scanner

database/sql was designed for long-lived applications with stable schemas and persistent connection pools. A scanner has neither: it opens a single connection per target, runs TOML-driven queries whose columns change between deployments, and must classify connection failures precisely. Four specific mismatches make it the wrong tool:

  • Pool-first design. sql.Open() creates a connection pool, not a single connection. Pool internals manage reconnection logic a scanner never needs and cannot control cleanly.
  • rows.Scan() requires compile-time knowledge. You must declare destination variables whose types match the query columns exactly — impossible when a TOML file can add or remove columns between deployments.
  • No clean GUC path. Setting statement_timeout or application_name requires appending query parameters to the DSN string or sending SET commands after connecting — both fragile at scale.
  • Opaque error types. database/sql returns a plain error. You cannot tell whether the server refused the connection, rejected the password, or hit a timeout without parsing the error message string — which changes between PostgreSQL versions.
The Scan() Problem
Every TOML query you add is a compile-time contract. Change a column and the scanner breaks: 
// database/sql — must know columns at compile time
var count   int
var maxConn int
rows.Scan(&count, &maxConn) // breaks if query adds or removes a column
A TOML-driven scanner cannot predict its columns at compile time. database/sql has no answer to this.

2. pgx v5 — What It Gives the Scanner

pgx v5 is a pure-Go PostgreSQL driver that speaks the wire protocol directly — no cgo, no C library. For a scanner, four features matter:

Featurepgx v5 APIWhat it solves
Dynamic columns rows.Values() → []any Reads any result set without declaring destination variables
Typed server errors *pgconn.PgError Server errors carry .Code, .Message, .Detail, .Hint
Struct-based config pgx.ConnectConfig() Set host, port, user, password, timeout as typed fields — no DSN string to construct
GUC injection config.RuntimeParams Sends SET commands before the first query — guaranteed GUC delivery

3. ConnectConfig — The Scanner's Connect Function

The pgx connect pattern replaces every DSN string with a typed struct. All connection parameters — including the GUCs that protect the target server from runaway queries — are set before the connection is made: 

import (
    "github.com/jackc/pgx/v5"
    "github.com/jackc/pgx/v5/pgconn"
)

func connectSafe(ctx context.Context, s Server) (*pgx.Conn, error) {
    config, err := pgx.ParseConfig("") // start from an empty config
    if err != nil {
        return nil, err
    }
    config.Host           = s.Host
    config.Port           = uint16(s.Port)
    config.User           = "pg_monitor"
    config.Password       = s.Password
    config.Database       = "postgres"
    config.ConnectTimeout = 5 * time.Second
    config.RuntimeParams  = map[string]string{
        "statement_timeout":                  "5000", // ms
        "lock_timeout":                       "1000", // ms
        "application_name":                   "health-scanner",
        "idle_in_transaction_session_timeout": "5000",
    }
    return pgx.ConnectConfig(ctx, config)
}

pgx.ParseConfig("") returns a default pgx.Config struct rather than parsing a DSN. Setting fields explicitly — instead of string-encoding them into a DSN — means no quoting bugs, no URL encoding issues, and no risk of a password containing @ or / breaking the parser. The function returns a live *pgx.Conn (a single connection, not a pool) ready to query.

What RuntimeParams Does
Each key-value pair in config.RuntimeParams becomes a SET key = value command sent to PostgreSQL immediately after the connection is established, as part of the startup handshake. The server enforces these GUCs for the lifetime of that connection. statement_timeout = 5000 means every query on that connection will be cancelled by the server if it runs longer than 5 seconds — protecting 3,000 target servers from a scanner query that accidentally touches a 100-million-row table.

4. rows.Values() — The Scanner's Core Pattern

The function below replaces every type-specific rows.Scan() call. It runs any SQL query, reads any columns, and returns the result as a slice of maps — ready to be serialised to JSON or evaluated by the alerting layer: 

func runQuery(ctx context.Context, conn *pgx.Conn, sql string) ([]map[string]any, error) {
    rows, err := conn.Query(ctx, sql)
    if err != nil {
        return nil, err
    }
    defer rows.Close()

    cols := rows.FieldDescriptions() // []pgconn.FieldDescription — one entry per SELECT column

    var results []map[string]any
    for rows.Next() {
        vals, err := rows.Values()
        if err != nil {
            return nil, err
        }
        row := make(map[string]any, len(cols))
        for i, col := range cols {
            row[string(col.Name)] = vals[i]
        }
        results = append(results, row)
    }
    return results, rows.Err()
}
The Core Insight
rows.Values() is the reason to use pgx. It decouples your scanner from column definitions — any TOML query, any server, any PostgreSQL version. The i-th entry in the []any slice corresponds to the i-th pgconn.FieldDescription returned by rows.FieldDescriptions(), so you always know which value maps to which column name without declaring a single destination variable.

5. Error Classification with pgx

When a connect or query call fails, the scanner needs to know why. A retry makes sense for a transient network blip but not for a wrong password. pgx surfaces server-side failures as a typed *pgconn.PgError value that carries the full SQLSTATE code: 

import "errors"
import "github.com/jackc/pgx/v5/pgconn"

func classifyErr(err error) string {
    if err == nil {
        return "ok"
    }
    var pgErr *pgconn.PgError
    if errors.As(err, &pgErr) {
        switch pgErr.Code {
        case "28000", "28P01": return "auth_failure"    // invalid_authorization_specification
        case "3D000":           return "db_not_found"    // invalid_catalog_name
        case "57014":           return "query_cancelled" // query_canceled — statement_timeout fired
        case "55P03":           return "lock_timeout"
        }
        return "pg_error:" + pgErr.Code
    }
    if errors.Is(err, context.DeadlineExceeded) {
        return "connect_timeout"
    }
    return "network_error"
}
SQLSTATE Codes
Every PostgreSQL error carries a 5-character SQLSTATE code. The first two characters identify the class: 28 = authorisation failure, 3D = invalid catalog name, 57 = operator intervention, 55 = object not in prerequisite state. The full table of codes is in the PostgreSQL documentation appendix. Note that SQLSTATE 57014 is officially named query_canceled (US spelling, single-l) — this is what fires when statement_timeout expires. The PostgreSQL server cancels the query; pgx surfaces it as a *pgconn.PgError with .Code == "57014".

errors.As(err, &pgErr) unwraps the error chain looking for a *pgconn.PgError. If found, the server returned a structured protocol-level error and the SQLSTATE code is available. If not, the failure happened at the network layer — TCP refused, TLS handshake failed, context deadline exceeded — and no SQLSTATE exists. The two branches require completely different scanner responses.

6. Migration Path from database/sql

Migrating an existing scanner is a mechanical 5-step process:

  1. Add the dependency. 
    go get github.com/jackc/pgx/v5
  2. Replace the connect call. Swap sql.Open("postgres", dsn) with the connectSafe function from Section 3. Delete the DSN string.
  3. Replace the query call. Swap db.QueryContext(ctx, sql) with conn.Query(ctx, sql). The method signatures are similar; the return type changes from *sql.Rows to pgx.Rows.
  4. Replace rows.Scan(). Delete all destination variable declarations. Replace rows.Scan(&a, &b, &c) with the rows.Values() + map construction pattern from Section 4.
  5. Clean up the import side-effect. Remove _ "github.com/lib/pq" or any other blank import used to register a database/sql driver. pgx does not use the driver-registration pattern — import it directly and call its functions.

The resulting scanner is simpler: fewer types, no DSN string construction, and connection failures that carry enough information to route into the right error bucket without string parsing.

Summary

  • pgx.ParseConfig("") plus struct fields replaces DSN string construction entirely; config.RuntimeParams delivers GUCs to the server via the startup handshake

  • Migration is mechanical: sql.Open → pgx.ConnectConfig, db.QueryContext → conn.Query, rows.Scan → rows.Values()
