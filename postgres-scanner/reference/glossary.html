Glossary

Core vocabulary for PostgreSQL connection management and health checking. Every term here appears in the lessons.

Connection Model

backend process connection model
The OS process PostgreSQL forks for every client connection. Each backend handles exactly one connection. It persists until the connection closes.
Scanner implication: opening a connection on a production server spawns a new OS process. Always close connections promptly.
max_connections GUC / configuration
Server-wide limit on simultaneous client connections. Default: 100. Includes superuser_reserved_connections (default 3). Exceeding it produces: FATAL: sorry, too many clients already.
Scanner implication: your scanner uses 1 connection per server while checking it. Never hold it open longer than needed.
connection pool pattern
A cache of open database connections reused across multiple requests. Avoids the cost of TCP handshake + authentication on every query. Used in long-running services (web servers, APIs).
Scanner implication: for a health-check scanner, you typically do NOT pool — you open one connection, run checks, close it. A pool makes sense if you run checks repeatedly against the same server.
pgxpool Go library
The connection pool implementation in the pgx Go driver. Manages a set of *pgx.Conn connections. Safe for concurrent use by many goroutines.
goroutine Go concurrency
Go's lightweight concurrent execution unit. Cheaper than an OS thread (~2KB stack vs ~2MB). The Go runtime multiplexes many goroutines onto a small number of OS threads.
Scanner implication: you can spawn thousands of goroutines — but you must rate-limit how many actually hold open database connections at once.
semaphore (buffered channel) Go concurrency
A counting semaphore limits how many goroutines can proceed past a point simultaneously. In Go, implemented as a buffered channel: sem := make(chan struct{}, N). Sending blocks when the buffer is full; receiving frees a slot.
Scanner implication: your concurrency limiter is a semaphore. Set N to something that won't overload your scanner host's outbound connections.
worker pool pattern
A fixed set of goroutines that pull work items from a shared channel. Unlike spawning one goroutine per item, this caps memory and concurrency precisely.

Timeouts & Safety Parameters

connect_timeout GUC / DSN parameter
Maximum time in seconds to wait for a connection to be established. Set in the connection string: connect_timeout=5. A value of 0 means unlimited.
Scanner implication: always set this. A dead or firewall-blocked server will hang forever without it.
statement_timeout GUC / runtime parameter
Maximum time a single SQL statement may run before being cancelled with an error. Set in milliseconds: SET statement_timeout = '5000'. Applies per-session when set as a RuntimeParam in pgx.
Scanner implication: the single most important safety knob. Without it, a slow catalog query can block for minutes on a loaded server.
lock_timeout GUC / runtime parameter
Maximum time to wait to acquire a lock. If the lock cannot be acquired within this time, the statement fails. Set in milliseconds.
Scanner implication: some catalog queries acquire brief locks. Set this to 1000ms to avoid your scanner being blocked by table-level locks on production.
idle_in_transaction_session_timeout GUC / runtime parameter
Terminates sessions that sit idle inside an open transaction longer than the specified time. Prevents abandoned transactions from holding locks indefinitely.
Scanner implication: set this as a safety net in case a bug causes your scanner to open a transaction and crash without closing it.
application_name GUC / runtime parameter
A string identifying the client application. Visible in pg_stat_activity.application_name. Has no functional effect — purely for observability.
Scanner implication: always set this (e.g. healthchecker-v1). It lets DBAs see your scanner in pg_stat_activity and know it is not a rogue connection.

Privileges & Roles

pg_monitor built-in role (PG 10+)
A predefined role that grants read access to all monitoring views (pg_stat_*, pg_locks, etc.) without superuser. The minimal-privilege choice for health-check tools.
Scanner implication: ask DBAs to grant pg_monitor to your scanner user instead of superuser. Never run health checks as a superuser if you can avoid it.
superuser_reserved_connections GUC
Number of connections reserved exclusively for superusers. Default: 3. Even if max_connections is exhausted, superusers can still connect to fix the problem.
Scanner implication: your scanner should not use superuser credentials — or it could eat into these reserved slots.

Catalog Views

pg_stat_activity catalog view
One row per backend process. Columns include: state, query, wait_event_type, application_name, backend_start, query_start. The primary view for connection health checks.
pg_settings catalog view
One row per configuration parameter (GUC). Columns: name, setting, unit, context. Use SHOW parameter_name as a shorthand for single lookups.
pg_stat_replication catalog view
One row per active streaming replication connection. Columns: sent_lsn, replay_lsn, write_lag, flush_lag, replay_lag. Only populated on primaries.
pg_stat_user_tables catalog view
One row per user table. Tracks autovacuum, dead tuples, sequential scans, index scans, etc. Key source for vacuum health checks.
