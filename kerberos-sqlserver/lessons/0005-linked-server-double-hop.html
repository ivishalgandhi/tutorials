Linked Server Double-Hop and Delegation

A user runs a query on SQL Server A. That query pulls data from a linked server, SQL Server B. The user's credential must travel from the client to A, then from A to B. This is the double-hop. Kerberos handles it through delegation; NTLM cannot. When it fails, the linked server logs in as ANONYMOUS LOGON and the query fails with login denied.

1. The Double-Hop in One Picture

sequenceDiagram participant C as Client participant A as SQL Server A participant B as SQL Server B participant K as AD KDC C->>K: Give me a ticket for MSSQLSvc/sqlA.corp.local K-->>C: Service ticket for A C->>A: Query + ticket Note over A: Linked server needs data from B A->>K: Delegate this ticket to MSSQLSvc/sqlB.corp.local Note over K: Only allowed if A is trusted for delegation K-->>A: Forwarded ticket for B A->>B: Connect + forwarded ticket B->>B: Verify ticket, check user is authorized B-->>A: Result set

Figure 1. Double-hop flow. SQL Server A must be trusted to forward the user's ticket to SQL Server B.

Key Concept

Delegation is a permission on the middle tier service account, not on the end user or the destination server. It says: "SQL Server A is allowed to reuse the user's Kerberos ticket to talk to SQL Server B."

2. Constrained Delegation: The Modern Default

There are two practical ways to configure delegation today:

  • Kerberos constrained delegation (KCD) — classic, configured in Active Directory Users and Computers on the middle-tier account. You list the exact SPNs the account may delegate to.
  • Resource-based constrained delegation (RBCD) — configured on the destination account. The destination says: "I trust SQL Server A's account to delegate users to me." This does not require Domain Admin privileges on the middle-tier account object.
# Example SPNs to allow SQL Server A's gMSA to delegate to
MSSQLSvc/sqlB.corp.local:1433
MSSQLSvc/sqlB.corp.local

# With RBCD, on sqlB's account or gMSA you set msDS-AllowedToActOnBehalfOfOtherIdentity
# to include CORP\sqlA$
Trap

Unconstrained delegation allows the middle tier to forward tickets to any service. Avoid it. Use constrained delegation or resource-based constrained delegation, and list only the SPNs your linked servers actually need.

4. PostgreSQL Has No Double-Hop, But the Rule Still Applies

PostgreSQL does not host linked servers the way SQL Server does. A PostgreSQL query cannot natively forward a user's identity to a second PostgreSQL server. However, a SQL Server linked server that points to PostgreSQL via the PostgreSQL ODBC driver can double-hop: client → SQL Server A → PostgreSQL. The same delegation rules apply to SQL Server A's service account.

PostgreSQL Note

For SQL Server → PostgreSQL double-hop, the PostgreSQL side only needs a valid keytab and pg_hba.conf entry for the SQL Server's Windows principal. The SQL Server side needs the SPN for PostgreSQL registered or delegated. Most teams instead run the linked server under a fixed remote credential to avoid Kerberos delegation to PostgreSQL.

5. Fixing the Intermittent Case

The most frustrating double-hop bug is intermittent ANONYMOUS LOGON. It often happens after a Kerberos ticket expires and the middle tier tries to reuse a stale service ticket. The fix is usually:

  1. Confirm all SPNs in the chain are registered and unique.
  2. Purge tickets on the middle tier with klist purge.
  3. Confirm constrained delegation covers every SPN form the client may request.
  4. Restart the middle-tier SQL Server service only if the service account itself changed.
