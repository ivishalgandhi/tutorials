gMSA and SPN Registration

Lesson 1 showed why Kerberos lives or dies by the service identity. This lesson turns that into operations: choosing a Group Managed Service Account, registering SPNs, granting the right AD permissions, and verifying everything before restarting SQL Server.

1. Why a gMSA Beats a Domain User Account

A Group Managed Service Account is an AD account that multiple servers can use, with automatic password rotation and simplified SPN management. For SQL Server it is the Microsoft-recommended replacement for manually maintained domain user service accounts.

CharacteristicDomain usergMSA
Password managementManual or custom scriptAD rotates automatically
Usable on multiple serversRisky shared credentialDesigned for it

Info

gMSA requires Windows Server 2012 R2 or later for the domain functional level and a KDS root key. Failover clusters support gMSA for SQL Server 2016 and later. See Microsoft Learn: Manage Group Managed Service Accounts.

2. Create and Install a gMSA

Run these on a domain-joined machine with the AD PowerShell module:

# 1. Create a security group for the servers that will use the gMSA
New-ADGroup -Name "SQL Server Hosts" -GroupScope Global -Path "OU=Groups,DC=corp,DC=local"

# 2. Add the computer accounts of the SQL Server nodes
Add-ADGroupMember -Identity "SQL Server Hosts" -Members sql01$, sql02$

# 3. Create the gMSA (KDS root key must already exist)
New-ADServiceAccount `
  -Name sqlsvc `
  -DNSHostName sqlsvc.corp.local `
  -PrincipalsAllowedToRetrieveManagedPassword "SQL Server Hosts"

# 4. On each SQL Server node, install the gMSA
Install-ADServiceAccount -Identity sqlsvc

# 5. Verify
Test-ADServiceAccount -Identity sqlsvc

When you change the SQL Server service account in SQL Server Configuration Manager, leave the password blank. The host retrieves it automatically from AD.

3. Grant Permission to Register SPNs

By default, SQL Server tries to register its SPN at startup. With a gMSA you can let it self-register, but the service account needs Validated write to service principal name, plus Read servicePrincipalName and Write servicePrincipalName on its own AD object.

# Delegated via PowerShell — run as Domain Admin or account operator with rights
$gMSA = Get-ADServiceAccount -Identity sqlsvc
$acl = Get-Acl -Path "AD:\$($gMSA.DistinguishedName)"

$sid = [System.Security.Principal.SecurityIdentifier] (New-Object System.Security.Principal.NTAccount "CORP\sqlsvc$").Translate([System.Security.Principal.SecurityIdentifier])

$rule = New-Object System.DirectoryServices.ActiveDirectoryAccessRule `
  $sid, "WriteProperty", "Allow", [GUID] "f3a64788-5306-11d1-a9c5-0000f80367c1", "Self"
# The GUID is the schemaIDGuid for the servicePrincipalName attribute.

$acl.AddAccessRule($rule)
Set-Acl -Path "AD:\$($gMSA.DistinguishedName)" -AclObject $acl
Practical Tip

In many organizations, AD admins prefer to register SPNs themselves and disable self-registration. Either path works, but the gMSA still wins on password management. If AD manages SPNs, keep the register/unregister permissions restricted and document the ticket path for SPN changes.

4. Register SPNs Manually with setspn

Whether SQL Server self-registers or AD does it, the SPNs must exist and must be correct. Use setspn -S to add only if the SPN is not already present elsewhere.

# Default instance, TCP
setspn -S MSSQLSvc/sql01.corp.local:1433 CORP\sqlsvc$
setspn -S MSSQLSvc/sql01.corp.local CORP\sqlsvc$

# Named instance on static port 51433
setspn -S MSSQLSvc/sql02.corp.local:51433 CORP\sqlsvc$
setspn -S MSSQLSvc/sql02.corp.local:INSTANCE CORP\sqlsvc$

# Always On availability group listener
setspn -S MSSQLSvc/aglistener.corp.local:1433 CORP\sqlsvc$
setspn -S MSSQLSvc/aglistener.corp.local CORP\sqlsvc$
Critical

SPNs must be unique across the entire AD forest. setspn -X finds duplicates. A duplicate SPN causes intermittent authentication failures or forces NTLM fallback depending on which account the KDC returns first.

5. Verify Everything

After SPN registration, verify from the SQL Server host:

# List SPNs on the gMSA
setspn -L CORP\sqlsvc$

# Check for duplicates in the current domain
setspn -X

# Query a specific SPN across the entire forest (useful for multi-domain environments)
setspn -F -Q MSSQLSvc/sql01.corp.local:1433

# Restart SQL Server and watch the error log for the registration message
# "The SQL Server Network Interface library successfully registered ..."

5.1 Discovering SPNs across domains

In a single-domain forest, setspn -X and setspn -Q look at your own domain by default. In a multi-domain forest you often need to target a specific domain. Use the -T switch with the DNS name of the domain you want to query:

# List every SPN registered in a specific trusted domain
setspn -T eu.corp.local -Q */*

# Look for duplicate SPNs in a specific domain
setspn -T eu.corp.local -X

# Find every MSSQLSvc SPN in a specific domain
setspn -T eu.corp.local -Q MSSQLSvc/*
Info

The -T switch is not emphasized in current Microsoft Learn docs, but it is widely used in the field to scope SPN discovery to one domain instead of running a heavy forest-wide query. For a full forest search, combine -F with -Q or -X. In very large forests, setspn -X -F can time out; targeting one domain with -T is often faster and gentler on the Global Catalog.

Then test a remote connection and check the DMV:

-- From a remote client using Windows Authentication
SELECT net_transport, auth_scheme
FROM sys.dm_exec_connections
WHERE session_id = @@SPID;

6. PostgreSQL Keytab Counterpart

SQL Server stores its service key in the LSA secret associated with the gMSA. PostgreSQL has no LSA; it uses a keytab file. The operational parallel is:

SQL ServerPostgreSQL
SPN on gMSA: MSSQLSvc/sql01.corp.local:1433Principal in keytab: postgres/sql01.corp.local@CORP.LOCAL
Service account password managed by ADKeytab file protected by filesystem permissions
Verify with setspn -L and DMVVerify with klist -k and PostgreSQL log

7. Do You Need to Restart SQL Server for AD Changes?

Usually no. SPN changes are read by the KDC immediately. A running SQL Server instance does not cache its own SPN list. However, if you change the service account that SQL Server runs under, a service restart is required. Changing only the SPNs or delegation settings does not require a restart.

SQL Server Impact

Existing client connections are not dropped by an SPN change. Only new connections will use the new Kerberos path. That means you can fix an SPN during business hours without restarting SQL Server, as long as you verify from a fresh remote connection.
