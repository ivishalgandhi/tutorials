PostgreSQL Keytab Setup and Verification

SQL Server stores its service key in the account secret managed by AD. PostgreSQL on Linux has no LSA, so it keeps its service key in a keytab file. This lesson walks through creating the principal, exporting the keytab, wiring it into PostgreSQL, and proving that GSSAPI authentication is working.

1. The Keytab Is the PostgreSQL SPN

For SQL Server, the KDC looks up an SPN registered in AD. For PostgreSQL, the server itself looks up the principal in its keytab. The client and the server must agree on the same principal name, and that name must match the FQDN the client used to connect.

2. Create the AD Principal and Keytab

On a domain-joined Windows machine with domain-admin rights, create a user or gMSA mapped to the PostgreSQL service principal, then export the keytab.

# Create a dedicated AD user for the PostgreSQL host (one per server is cleanest)
New-ADUser -Name pgdb01 -SamAccountName pgdb01 -UserPrincipalName pgdb01@corp.local `
  -AccountPassword (Read-Host -AsSecureString "Password") -Enabled $true

# Export a keytab for the postgres principal using ktpass
ktpass /princ postgres/pg01.corp.local@CORP.LOCAL `
       /mapuser CORP\pgdb01 `
       /crypto AES256-SHA1 `
       /ptype KRB5_NT_SRV_HST `
       /out c:\postgres.keytab `
       -SetPass +rndPass
Trap

The -SetPass +rndPass option randomizes the AD password and writes the current key into the keytab. If you reset the AD password later without re-exporting the keytab, authentication breaks. Treat the keytab and the AD account password as one object.

After moving the keytab to the PostgreSQL server, inspect it:

# List principals in the keytab
klist -k /etc/postgresql/16/main/postgres.keytab

# Expected output includes a line like:
#    1  postgres/pg01.corp.local@CORP.LOCAL

3. Configure the PostgreSQL Server

PostgreSQL needs two things: the path to the keytab and the right pg_hba.conf line.

Set the environment variable before the postmaster starts. On systemd systems, use a drop-in file:

# /etc/systemd/system/postgresql@16-main.service.d/kerberos.conf
[Service]
Environment=KRB5_KTNAME=/etc/postgresql/16/main/postgres.keytab

Reload systemd and restart PostgreSQL so it reads the variable:

sudo systemctl daemon-reload
sudo systemctl restart postgresql@16-main

Then add the pg_hba.conf entry. Use hostssl because gss authenticates but does not encrypt:

# /etc/postgresql/16/main/pg_hba.conf
hostssl  appdb  +app_users  10.0.1.0/24  gss  map=domain2pg

If Active Directory usernames differ from PostgreSQL role names, add a map in pg_ident.conf:

# /etc/postgresql/16/main/pg_ident.conf
# mapname    system-username      database-username
domain2pg    /^(.*)@CORP\.LOCAL$   \1

Reload the configuration:

SELECT pg_reload_conf();
PostgreSQL Note

The keytab file must be readable by the postgres OS user and no one else. A common mistake is copying it with root:root permissions. Use chown postgres:postgres and chmod 600.

4. Verify the Connection

From a domain-joined client with a Kerberos ticket, connect by FQDN:

# Acquire a ticket first if you do not have one
kinit alice@CORP.LOCAL

# Connect by FQDN; psql will use GSSAPI automatically
psql "host=pg01.corp.local dbname=appdb user=alice sslmode=require"

Confirm on the server side from the PostgreSQL log:

# A successful GSSAPI connection logs something like:
LOG:  connection authorized: user=alice database=appdb

You can also request the ticket before connecting to prove the principal is resolvable:

# Ask the KDC for a ticket for the PostgreSQL principal
kvno postgres/pg01.corp.local@CORP.LOCAL

If kvno succeeds but psql fails, the problem is on the PostgreSQL server side: keytab path, permissions, pg_hba.conf, or principal name mismatch.

Key Concept

The principal in the keytab must exactly match what the client requests. If the client connects to pg01 while the keytab contains postgres/pg01.corp.local@CORP.LOCAL, GSSAPI will fail. Standardize on FQDN everywhere.
