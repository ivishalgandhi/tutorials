Named Instances, AG Listeners, and the 1200 SPN Limit

One default instance is easy. Real environments have named instances, Always On listeners, failover clusters, and many services sharing one gMSA. This lesson shows the exact SPNs each requires and the AD limit that caps how far you can push a single account.

1. Named Instances Need Static Ports

SQL Server named instances default to dynamic ports. Dynamic ports break Kerberos because the SPN contains the port number; if the port changes at startup, the SPN no longer matches. Before you register SPNs for a named instance, pin it to a static port in SQL Server Configuration Manager.

# Correct SPNs for a named instance APPDB on sql02.corp.local, static port 51433
setspn -S MSSQLSvc/sql02.corp.local:51433 CORP\sqlsvc$
setspn -S MSSQLSvc/sql02.corp.local:APPDB  CORP\sqlsvc$

# Note: the instance-name SPN is used for named-pipe/shared-memory paths,

# while the port SPN is used for TCP connections.
Trap

If you register an SPN with a port and then the instance later uses dynamic ports, Kerberos will fail after the next restart that picks a different port. Set the static port before you register the SPN, and document it in your build sheet.

2. Always On Availability Group Listeners

Clients connect to the listener FQDN, not to individual replica hostnames. The SPN must therefore be registered on the listener FQDN and mapped to the account running SQL Server on every replica. All replicas in an AG typically use the same gMSA.

# SPNs for the AG listener AgListener.corp.local, default port 1433
setspn -S MSSQLSvc/AgListener.corp.local:1433 CORP\sqlsvc$
setspn -S MSSQLSvc/AgListener.corp.local      CORP\sqlsvc$

# Replicas themselves also need their own host SPNs if anyone connects directly
setspn -S MSSQLSvc/sql01.corp.local:1433        CORP\sqlsvc$
setspn -S MSSQLSvc/sql02.corp.local:1433        CORP\sqlsvc$
Auto-registration does not cover listeners

Even when the SQL Server service account (or gMSA) has permission to register its own SPNs, SQL Server only auto-registers SPNs for the local instance's host name and port. The AG listener SPN must be registered manually with setspn by someone with rights to modify the service account in AD. Microsoft explicitly documents this for Always On listeners.

2.1 Who can register the listener SPN?

The SPN is written on the shared SQL Server service account, not on the listener's Virtual Computer Object. You need permission to modify servicePrincipalName on that account. The three practical paths are:

PathRequired privilege
Domain AdminFull control over AD objects.
Account OperatorCan create/modify accounts and run setspn.
Delegated (preferred)On the specific service account: Validated write to service principal name, plus Read servicePrincipalName and Write servicePrincipalName.
Practical Tip

For production, ask your AD team to delegate the three SPN permissions on the gMSA instead of handing out Domain Admin. The SQL Server service account itself does not need listener-SPN self-registration rights — SQL Server never auto-registers listener SPNs anyway.

Key Concept

During a failover, the listener DNS record points to the new primary. The SPN does not move; the account running SQL Server on the new primary already owns it. No SPN change is required at failover time if all replicas share the gMSA and the listener SPNs are already registered.

4. The ~1200 SPN Practical Limit

The servicePrincipalName attribute in Active Directory is multivalued and non-linked. It does not use Linked Value Replication. The practical limit is about 1200 entries per AD object, and the exact ceiling shrinks as individual SPNs grow longer.

Risk

If you put every SQL Server instance, listener, SSRS web site, and SSAS instance in one gMSA, you will eventually hit the limit. Symptoms include SPN registration failures, LDAP error 0x00000b (LDAP_ADMIN_LIMIT_EXCEEDED), and replication issues for that object.

Mitigation strategies:

  • Use separate gMSAs per environment or per application tier.
  • Prefer port-based SPNs over host-based duplicates when clients can be standardized.
  • Audit SPNs regularly: setspn -L and setspn -X.
  • Delete obsolete SPNs when servers are decommissioned.

5. PostgreSQL Keytab Parallel

PostgreSQL does not register SPNs in AD. Instead, the DBA or AD team creates the principal in AD and exports it into the PostgreSQL keytab. Each PostgreSQL server has its own keytab; there is no single AD object accumulating hundreds of SPN entries.

# On Windows, export a keytab for the postgres principal
ktpass /princ postgres/sql01.corp.local@CORP.LOCAL `
       /mapuser CORP\pgdb01$ `
       /crypto AES256-SHA1 `
       /ptype KRB5_NT_SRV_HST `
       /out c:\postgres.keytab `
       -SetPass +rndPass

# On Linux, ktutil can also import the keytab and verify the principal
klist -k /etc/postgresql/16/main/postgres.keytab
Info

Because PostgreSQL uses per-host keytabs, the 1200-SPN limit is not a PostgreSQL problem directly. However, the same AD object can still accumulate SPNs if it hosts other Windows services. Keep the keytab principal distinct from other service SPNs.
