What Kerberos Means for SQL Server and PostgreSQL

Your organization is deprecating NTLMv2. For the databases you run, that means every Windows-integrated or AD-backed connection must now negotiate Kerberos instead of silently falling back to NTLM. This lesson explains the mechanics in one picture, and shows you how to verify that Kerberos is actually being used on both SQL Server and PostgreSQL.

1. The Kerberos Ticket Flow

Kerberos is not a password exchange. It is a ticket exchange. A client proves its identity once to Active Directory, receives a ticket-granting ticket, and then uses that ticket to request service-specific tickets from the Key Distribution Center. The database service proves itself by decrypting the service ticket with a secret only it possesses.

sequenceDiagram participant C as Client participant K as AD Key Distribution Center participant S as SQL Server / PostgreSQL C->>K: AS-REQ: authenticate me (password or smart card) K-->>C: AS-REP: Ticket-Granting Ticket (TGT) Note over C: User opens connection to sql01.corp.local C->>K: TGS-REQ: service ticket for MSSQLSvc/sql01.corp.local K-->>C: TGS-REP: service ticket C->>S: Connect + present service ticket S->>S: Decrypt ticket with service account key S-->>C: Connection accepted

Figure 1. Kerberos authentication flow. The user's password never crosses the network after the initial TGT request.

Key Concept

For the KDC to issue a service ticket, it must know which AD account owns the service. That mapping is the Service Principal Name (SPN). Without the right SPN, the ticket cannot be issued, Kerberos fails, and the client falls back to NTLM.

2. How SQL Server Uses Kerberos

SQL Server's Database Engine service runs under an account. If that account has the correct SPN registered, remote clients can request and present a Kerberos service ticket. If the SPN is missing or wrong, Windows integrated connections fall back to NTLM.

Verify the authentication scheme from a remote machine:

-- Run this while connected with Windows Authentication
SELECT net_transport, auth_scheme
FROM sys.dm_exec_connections
WHERE session_id = @@SPID;
auth_scheme Meaning Action
KERBEROS Remote Kerberos ticket exchange succeeded. Desired state.
NTLM Kerberos failed; NTLM challenge-response is being used. Check SPN, DNS name, and service account permissions.
Windows / Negotiate variants SSPI negotiated a package; inspect further with DMV. Use auth_scheme to see the final choice.
sequenceDiagram participant C as Client participant K as AD KDC participant S as SQL Server C->>S: Connect using 192.168.1.50 C->>K: Request service ticket for MSSQLSvc/192.168.1.50 K-->>C: No such SPN found Note over C: SSPI falls back to NTLM C->>S: NTLM challenge-response S->>S: Verify credential against local SAM/LSA S-->>C: Connection accepted (auth_scheme = NTLM)

Figure 2. When the client uses an IP address, the KDC cannot find an SPN, Kerberos fails, and the client falls back to NTLM challenge-response with the SQL Server host.

SQL Server Note

Local connections from the server itself often show NTLM even when everything is correct, because Windows prefers NTLM for local loopback. Always test from a remote client.

3. How PostgreSQL Uses Kerberos

PostgreSQL does not have native Windows Authentication. Instead it uses GSSAPI (the cross-platform Kerberos API) via the gss authentication method, or SSPI on Windows. On Linux, the PostgreSQL server needs a keytab file that contains the long-term key for the database's service principal. The keytab lets PostgreSQL prove it owns the postgres/sql01.corp.local@CORP.LOCAL principal. No keytab means no Kerberos authentication.

On the PostgreSQL server, the keytab path is usually set via the KRB5_KTNAME environment variable before the postmaster starts:

# /etc/postgresql/16/main/environment or a systemd drop-in
KRB5_KTNAME=/etc/postgresql/16/main/postgres.keytab

# Create the keytab on a domain-joined machine with ktutil or ktpass
ktutil
ktutil:  read_kt /tmp/primary.keytab
ktutil:  read_kt /tmp/secondary.keytab
ktutil:  write_kt /etc/postgresql/16/main/postgres.keytab
ktutil:  quit

The matching pg_hba.conf entry is one line:

# GSSAPI / Kerberos over TLS for the application subnet
hostssl  appdb  +app_users  10.0.1.0/24  gss
# Include a username map if AD usernames differ from PostgreSQL role names
hostssl  appdb  all         10.0.1.0/24  gss  map=domain2pg

Verify a PostgreSQL Kerberos connection by checking the log. A successful GSSAPI connection logs an authentication message with the principal name. You can also query pg_stat_activity for the active user; the connection method itself is not exposed there, so logs and packet captures are the authoritative source.

Common Trap

PostgreSQL's gss method does not encrypt the connection. It only authenticates it. Pair it with hostssl (or use GSSAPI encryption) to protect data in transit. Authentication without encryption is like checking ID at the door and then shouting secrets in the hallway.

4. The SPN Is the Whole Game

For SQL Server, the SPN tells AD: "MSSQLSvc/sql01.corp.local:1433 is owned by CORP\gMSA$." For PostgreSQL, the equivalent principal in the keytab is usually postgres/sql01.corp.local@CORP.LOCAL. The name the client uses in its connection string must match the SPN or principal registered in AD.

Requirement SQL Server PostgreSQL
Service identity registered in AD SPN on service account Principal in PostgreSQL keytab
Client connection name FQDN or registered host name FQDN matching the keytab principal
IP address allowed? No No
Fallback if Kerberos fails NTLM (if enabled) Connection refused, or next pg_hba rule

5. Three Requirements for Database Kerberos

  1. Same AD forest or trusted forest. The client, the database server, and the KDC must all trust the same domain hierarchy.
  2. A valid service identity. SPN for SQL Server; keytab principal for PostgreSQL.
  3. Client connects by the registered name. FQDN recommended. IP addresses and localhost do not work.
NTLM Fallback Is Now a Failure Mode

Before the migration, NTLM fallback was harmless redundancy. After the migration, it is a policy violation. Every connection that silently falls back to NTLM is a connection that must be fixed: SPN, DNS name, or service account.

6. Try It

Your First Action

Pick one SQL Server instance and one PostgreSQL instance. From a remote client, connect by FQDN. On SQL Server run the DMV query above. On PostgreSQL check the log for GSSAPI success or failure. Write down: (1) the connection name used, (2) whether Kerberos succeeded, and (3) the first error or fallback indicator you see.
