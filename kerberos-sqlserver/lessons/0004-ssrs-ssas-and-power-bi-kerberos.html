SSRS, SSAS, and Power BI Kerberos

The Database Engine is only part of the estate. Reporting Services, Analysis Services, and Power BI also authenticate through Windows integrated security, and each needs the right SPN or gateway configuration. This lesson covers the SPN shapes and service settings that keep them on Kerberos instead of NTLM.

1. Reporting Services Needs an HTTP SPN

SQL Server Reporting Services runs as a web application. Clients connect to a URL, so the service class is HTTP, not MSSQLSvc. The SPN belongs on the account running the Report Server service — a gMSA, a domain user, or the computer account if it runs as NETWORK SERVICE.

# SSRS on the default web site, FQDN reports.corp.local, default port 80
setspn -S HTTP/reports.corp.local CORP\ssrs$

# If you also accept connections by NetBIOS short name, add it sparingly
setspn -S HTTP/reports CORP\ssrs$

The Report Server rsreportserver.config controls which authentication types are advertised:

<AuthenticationTypes>
  <RSWindowsNegotiate/>
  <RSWindowsKerberos/>
  <RSWindowsNTLM/>
</AuthenticationTypes>

RSWindowsNegotiateSSPI picks Kerberos if an SPN exists, otherwise NTLM.Default. Allows graceful fallback during migration.
RSWindowsKerberosKerberos only.After migration, to block silent NTLM fallback.
RSWindowsNTLMNTLM only.Remove after migration.
Trap

If the HTTP SPN is missing or registered on the wrong account, browsers will silently fall back to NTLM and SSRS will work — until your organization disables NTLMv2. Verify the scheme by inspecting the Authorization header in a network trace or by temporarily forcing RSWindowsKerberos and confirming access still works.

2. Analysis Services Uses MSOLAPSvc.3

Analysis Services has its own service class. The SPN format is MSOLAPSvc.3/FQDN[:instance]. The .3 corresponds to the SSAS version lineage; do not shorten it to MSOLAPSvc alone.

# SSAS default instance on ssas01.corp.local
setspn -S MSOLAPSvc.3/ssas01.corp.local CORP\ssas$

# SSAS named instance TABULAR on the same host
setspn -S MSOLAPSvc.3/ssas01.corp.local:TABULAR CORP\ssas$

Clients such as Excel or Power BI Desktop connect to ssas01.corp.local or ssas01.corp.local\TABULAR. The same FQDN rule applies: IP addresses and short names will fall back or fail.

Info

If SSAS runs on a failover cluster, register the SPN on the shared service account or gMSA, not on individual computer objects. The virtual network name needs its own SPN just like an Always On listener.

3. Power BI and the Double-Hop Problem

Power BI has two common Kerberos touch points:

  • Power BI Desktop connecting directly to SSAS or SQL Server — this is a single hop from the analyst's machine to the data source. Just ensure the data source SPN is correct.
  • Power BI Report Server or the on-premises data gateway refreshing reports or answering DirectQuery requests — this is often a double hop: user → gateway/service → data source. It needs delegation.

For the gateway scenario, the service account that runs the gateway or Power BI Report Server must be trusted for constrained delegation to the target SPNs:

# Delegation targets the gateway service account must be allowed to present tickets for
MSSQLSvc/sql01.corp.local:1433
MSOLAPSvc.3/ssas01.corp.local
Key Concept

Constrained delegation says: "The gateway service account is allowed to forward a user's Kerberos ticket to these specific downstream services." Without it, the second hop falls back to NTLM or fails entirely when NTLM is disabled.

4. A Common Verification Pattern

For every service, the verification chain is the same:

  1. Confirm the SPN is registered on the account running the service.
  2. Confirm the client connects by FQDN.
  3. Confirm no duplicate SPN exists with setspn -X.
  4. Temporarily force Kerberos-only in the service config; confirm access still works.
  5. Audit connections for NTLM fallback before removing the fallback option.
