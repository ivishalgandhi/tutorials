Detecting Silent NTLM Fallback and Change Impact

After the migration, NTLM fallback is no longer a safety net — it is a silent failure. This lesson shows how to find fallback connections before they break, and how to predict whether an AD, SPN, keytab, or service change will disrupt applications.

1. Why Silent NTLM Fallback Is Now a Bug

Before the migration, clients that fell back to NTLM still worked. After NTLMv2 is disabled, those same clients will fail outright. The only safe path is to discover and fix every fallback connection while NTLM is still enabled.

Risk

Do not wait for users to report login failures. Audit first, fix the SPNs and connection names, then disable NTLM fallback as a final step.

2. Detect NTLM on SQL Server

The fastest signal is the Database Engine's own DMV. Query it from every production instance and record the result:

-- All integrated-security connections and their scheme
SELECT
    @@SERVERNAME AS server_name,
    c.session_id,
    s.login_name,
    c.client_net_address,
    c.auth_scheme,
    c.net_transport,
    DB_NAME(s.database_id) AS database_name
FROM sys.dm_exec_connections c
JOIN sys.dm_exec_sessions s ON c.session_id = s.session_id
WHERE c.auth_scheme IN ('KERBEROS', 'NTLM')
  AND s.login_name NOT LIKE 'NT AUTHORITY%'
ORDER BY c.auth_scheme, s.login_name;

Treat every auth_scheme = NTLM row as a ticket. The ticket needs one of three fixes:

  • Register the missing SPN.
  • Change the application connection string from IP/short name to FQDN.
  • Fix delegation if it is a double-hop scenario.

3. Detect Non-Kerberos Connections on PostgreSQL

PostgreSQL does not expose auth_scheme in a DMV, but the log does. Enable log_connections temporarily during the audit window and grep for the authentication method:

# /etc/postgresql/16/main/postgresql.conf
log_connections = on

# After reload, look for GSSAPI vs password or md5 entries
grep -i "connection authorized\|authentication failed\|GSSAPI" /var/log/postgresql/postgresql-16-main.log

For a programmatic check, inspect pg_stat_activity for clients that should be using Kerberos but are not, and cross-reference with your pg_hba.conf lines. If a subnet is allowed both gss and scram-sha-256, password connections can still slip through.

PostgreSQL Note

The cleanest audit is to put Kerberos-only subnets in a separate pg_hba.conf rule and remove password methods for those sources. Then any password connection attempt is immediately visible as a fallback.

5. Communication Plan

Most application teams do not know they connect by IP address or short name. Before you disable NTLM, send them a checklist:

  1. Connection strings must use the database server's FQDN.
  2. No IP addresses or localhost aliases.
  3. Report any linked server, SSRS subscription, or Power BI refresh that uses pass-through authentication.
  4. Test in the pre-production environment with NTLM disabled first.
Practical Tip

Run the DMV and log audit weekly during the migration. When the NTLM count hits zero and stays there for two full business cycles, schedule the NTLM disable window.

6. Try It

Your Action

Pick your busiest SQL Server instance and run the DMV query above. Export the results to a spreadsheet. Sort by auth_scheme. For every NTLM row, identify whether the fix is an SPN, a connection-string change, or a delegation change. Do the same for one PostgreSQL instance by turning on log_connections for one hour and reviewing the log.
