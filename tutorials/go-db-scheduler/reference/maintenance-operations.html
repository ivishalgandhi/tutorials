Maintenance Operations Reference

The authoritative T-SQL and PostgreSQL commands run by the scheduler executors. Sourced from Microsoft Learn and PostgreSQL Documentation.

SQL SERVER T-SQL Operations

Full Database Backup

Operation type: full_backup · Requires maintenance window
BACKUP DATABASE [$(db)]
  TO DISK = N'$(path)\$(db)_$(date).bak'
  WITH
    COMPRESSION,           -- reduces backup size 50–80%
    CHECKSUM,              -- validates pages during backup
    STATS = 10,            -- progress every 10%
    NAME = N'$(db) Full Backup $(date)',
    DESCRIPTION = N'Scheduled by db-scheduler';
OptionWhy it matters
COMPRESSIONReduces backup file size and I/O; default off pre-2008 Enterprise
CHECKSUMDetects page corruption during backup; required for later restore verification
STATS = 10Emits progress rows parseable by the executor for long-running backups

Transaction Log Backup

Operation type: log_backup · Every 15–60 min on FULL recovery model
BACKUP LOG [$(db)]
  TO DISK = N'$(path)\$(db)_$(date).trn'
  WITH COMPRESSION, CHECKSUM, STATS = 10;
SKIP CONDITION

Only execute on databases in FULL or BULK_LOGGED recovery model. A database in SIMPLE recovery model does not have a transaction log to back up — attempting this raises error 4214.

Integrity Check (DBCC CHECKDB)

Operation type: integrity_check · Requires maintenance window · Dynamic timeout
-- Full check (weekly or monthly, large databases):
DBCC CHECKDB ([$(db)])
  WITH NO_INFOMSGS, ALL_ERRORMSGS;

-- Physical-only check (nightly, large databases):
DBCC CHECKDB ([$(db)])
  WITH PHYSICAL_ONLY, NO_INFOMSGS;

-- Estimate only — no actual I/O (pre-flight check):
DBCC CHECKDB ([$(db)])
  WITH ESTIMATEONLY;
ModeDurationWhat it checks
defaultHours on TB databasesLogical + physical consistency, all indexes
PHYSICAL_ONLY~10× fasterPhysical page structure only — misses logical errors
ESTIMATEONLYSecondsReturns estimated temp space needed, no actual checks

Index Rebuild

Operation type: index_rebuild · Requires maintenance window
-- Rebuild all indexes on all tables (online, Enterprise only):
ALTER INDEX ALL ON [$(schema)].[$(table)]
  REBUILD WITH (ONLINE = ON, SORT_IN_TEMPDB = ON);

-- Single index (for targeted rebuild):
ALTER INDEX [$(idx)] ON [$(schema)].[$(table)]
  REBUILD WITH (ONLINE = ON);

-- Reorganize (lighter, no table lock, fragmentation 10–30%):
ALTER INDEX ALL ON [$(schema)].[$(table)]
  REORGANIZE;
WHEN TO USE EACH

REBUILD: fragmentation > 30% or heap tables. Takes exclusive lock (Standard) or online lock (Enterprise, ONLINE=ON).
REORGANIZE: fragmentation 10–30%. Always online. Suitable during production hours.

Update Statistics

Operation type: update_stats · Lightweight, safe during production
-- Full scan of all statistics on all tables:
EXEC sp_updatestats;   -- skips unchanged tables

-- Full scan of a specific table (accurate, slow):
UPDATE STATISTICS [$(schema)].[$(table)] WITH FULLSCAN;

-- Sampled (faster, less accurate):
UPDATE STATISTICS [$(schema)].[$(table)] WITH SAMPLE 30 PERCENT;
POSTGRESQL SQL Operations

VACUUM

Operation type: vacuum · Safe during production · Skips locked tables
-- Routine vacuum + stats update (does NOT reclaim disk space):
VACUUM ANALYZE;            -- entire database
VACUUM ANALYZE $(table);   -- single table

-- Full vacuum — reclaims disk space, acquires exclusive lock:
VACUUM FULL ANALYZE $(table);  -- requires maintenance window

-- Skip locked tables (safe for background scheduling):
VACUUM (SKIP_LOCKED) $(table);
CommandReclaims disk?Lock?Use when
VACUUM ANALYZENo (marks dead rows)ShareUpdateExclusiveLockRoutine — nightly on high-churn tables
VACUUM FULLYes (rewrites table)AccessExclusiveLockTable bloat > 30% — maintenance window only
VACUUM (SKIP_LOCKED)NoNone if lockedProduction hours — skip busy tables silently

REINDEX

Operation type: index_rebuild · Requires maintenance window (CONCURRENTLY is safer)
-- Rebuild all indexes in a database (locks each table in turn):
REINDEX DATABASE $(db);

-- Rebuild all indexes on one table:
REINDEX TABLE $(schema).$(table);

-- Rebuild concurrently — no exclusive lock (PostgreSQL 12+):
REINDEX TABLE CONCURRENTLY $(schema).$(table);

-- Single index concurrently:
REINDEX INDEX CONCURRENTLY $(schema).$(idx);
GOTCHA — REINDEX DATABASE

REINDEX DATABASE cannot use CONCURRENTLY. For production fleets, iterate over tables and call REINDEX TABLE CONCURRENTLY per table instead. The executor should fetch the table list first and loop.

ANALYZE

Operation type: update_stats · Very lightweight · Safe during production
-- Update statistics for all tables in current database:
ANALYZE VERBOSE;

-- Single table:
ANALYZE $(schema).$(table);

-- Specific columns (targeted after a bulk insert):
ANALYZE $(schema).$(table) (col1, col2);

Logical Backup (pg_dump)

Operation type: full_backup · Shell command, not SQL · Dynamic timeout by size
// Go executor runs pg_dump as an os/exec command:
cmd := exec.CommandContext(ctx,
    "pg_dump",
    "--host", server.Hostname,
    "--port", strconv.Itoa(server.Port),
    "--username", creds.User,
    "--dbname", db.Name,
    "--format=custom",         // compressed, parallel-restore capable
    "--compress=6",            // gzip level 6
    "--file", backupPath,
    "--verbose",               // enables progress parsing
)
cmd.Env = append(os.Environ(), "PGPASSWORD="+creds.Password)
WHY NOT EXECUTE pg_dump OVER SQL?

Unlike SQL Server's BACKUP DATABASE, PostgreSQL does not have a built-in SQL backup command. pg_dump is a client-side binary that reads data over a normal connection. The executor must shell out — os/exec.CommandContext — so the dynamic context timeout also kills the shell process.
