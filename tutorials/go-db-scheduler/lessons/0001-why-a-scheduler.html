Why a Database Fleet Needs a Scheduler

Running maintenance on a single database is trivial — a cron job and a shell script will do. But what happens when that single database becomes a fleet of 10,000 databases spread across 500 servers, running a mix of SQL Server and PostgreSQL, across multiple regions and tiers? The naive approach — one cron entry per job per database — collapses under its own weight. This lesson explains exactly why, and introduces the policy-driven architecture that replaces it.

The Scale Problem

Before we can understand why a scheduler is necessary, we need to appreciate the target scale. These are the operational parameters the system is designed to handle:

Dimension Minimum Target Notes
Servers in fleet 500 1,000 Mix of SQL Server and PostgreSQL instances
Databases 10,000 50,000 Across all servers, all tiers
Jobs dispatched per day — 1,000,000+ Includes lightweight health checks and log backups
Concurrent running jobs 50 200 Throttled per server by MaxConcurrentOps
Job duration range 5 seconds 12 hours Health check vs. DBCC CHECKDB on a 10 TB database
Database size range 10 MB 10 TB Drives dynamic timeout calculation

The scale numbers alone reveal the problem. With 50,000 databases, a naive cron-based approach requires 50,000 cron entries — just for full backups. Add differential backups, log backups, integrity checks, index maintenance, and statistics updates, and you are managing hundreds of thousands of static job definitions. And cron, a tool designed for a single host in the 1970s, breaks in the following specific ways at this scale:

  • Time drift and no coordination. When the same job is scheduled on multiple dispatcher nodes, each fires independently. You get duplicate executions and no way to ensure exactly-once semantics.
  • No conflict detection. Cron does not know that a full backup and an integrity check on the same database should never run simultaneously. It will happily fire both at midnight.
  • No window awareness. Cron expressions fire at a time; they have no concept of "only run this during the 1 AM–5 AM maintenance window on weekdays, accounting for DST, in America/Chicago."
  • Thundering herd. If 5,000 databases all have 0 1 * * * as their backup schedule, all 5,000 backup jobs start at 01:00:00. Disk I/O, network bandwidth, and backup storage are instantly saturated.
  • Static definitions can't survive fleet churn. Databases are provisioned and deprovisioned daily. A static job list is perpetually stale.

The Hermes Inspiration

Modern AI agent platforms — and production-grade scheduling systems like Hermes — solved an analogous problem: how do you schedule work across a dynamically changing set of resources without hardcoding a job per resource? The answer is policies.

In a policy-driven model, you do not define what should happen to each individual resource. Instead, you define rules that apply to classes of resources, and the system generates work dynamically by evaluating those rules against live inventory. Consider the difference:

  • Job-driven (old way): "Back up database Northwind on server sql-prod-01 every day at 1 AM."
    Repeated 50,000 times. For every database. Manually maintained.
  • Policy-driven (new way): "Back up all ONLINE, non-read-only, production-tier SQL Server databases whose last full backup is older than 24 hours — every night between 1 AM and 5 AM, staggered across the window."
    Defined once. Applies automatically to every matching database, including ones added tomorrow.

Hermes takes this further: when a new agent (resource) registers, existing policies immediately apply to it. No human intervention. No ticket. The scheduler discovers new work from inventory refresh, not from manual job creation. This is the model our database scheduler adopts wholesale.

Policy-Driven vs. Job-Driven Design

The practical consequences of the design choice are large. Here is a head-to-head comparison across the concerns that matter most at fleet scale:

Concern Static — 50,000 job rows Policy-driven — 20 policies
New database added to fleet Manual — an administrator must add a new row for each job type Auto-discovered on next inventory refresh; all matching policies apply immediately
Schedule change for a job type Update 1,000 rows (one per database) Update 1 policy row
Conflict detection Not possible — static jobs have no runtime awareness of each other Derived from OperationType conflict matrix at dispatch time
Storage footprint 50,000 rows that must be kept in sync with fleet state forever 20 policy rows + live inventory (already maintained)
Thundering herd Likely — all jobs with the same cron expression fire at the same second Stagger algorithm distributes jobs across the maintenance window per server
Environment-specific overrides Requires duplicating job rows per environment Policy TargetSelector filters by tier, region, tags
Audit and observability Hard — 50,000 rows have overlapping history Clean — policy execution traces in a single structured log stream
KEY INSIGHT

A policy is a template. Jobs are computed dynamically from policies × inventory. You store the recipe, not the meal. When inventory changes, the meal changes automatically — the recipe stays the same.

Each layer has a distinct responsibility:

  • Inventory Layer. The live registry of servers and databases, refreshed on a configurable interval (default 15 minutes). Backed by a local store (SQLite or Postgres) that mirrors discovered fleet state.
  • Policy Generation Layer. On each scheduling tick, GenerateAll() evaluates every policy's TargetSelector against live inventory and produces a set of ScheduledJob candidates. No candidates are persisted — they are computed fresh.
  • Scheduling Layer. The cron scheduler fires on policy-defined intervals. The stagger algorithm offsets each database's job start time within the maintenance window to prevent simultaneous execution. Jobs are pushed into a bounded channel queue.
  • Execution Layer. A worker pool of goroutines drains the queue. Each goroutine dispatches to the appropriate engine executor (SQL Server or PostgreSQL) based on the server's Engine field.
  • Coordination Layer. Leader election ensures a single dispatcher node in multi-instance deployments. Distributed locks provide defense-in-depth for split-brain scenarios — even if two nodes briefly believe they are leader, only one can acquire the per-job lock.
  • Observability Layer. Every job state transition emits a Prometheus counter or gauge. Grafana dashboards surface queue depth, job latency histograms, failure rates, and per-server concurrency in real time.

The Ten Design Principles

Every architectural decision in this course flows from ten explicit principles. Learning them now will help you understand why the code is structured the way it is — rather than memorising patterns without context.

1. Policy-driven, not job-driven

Maintenance rules are expressed as policies over classes of databases, not as individual job definitions. The policy generator computes what needs to run by joining policies with live inventory. Adding a database to the fleet automatically enrolls it in all applicable policies — zero manual steps.

2. Exactly-once execution

In a horizontally-scaled deployment, multiple scheduler instances may run simultaneously. Leader election (via etcd or Redis) designates a single active dispatcher. A distributed lock per job is acquired before execution begins, providing defense-in-depth: even in a brief split-brain scenario, only one instance can execute a given job at a time.

3. Server-aware concurrency

Each DBServer carries a MaxConcurrentOps field. The worker pool tracks in-flight jobs per server and refuses to dispatch a new job if the server is already at its limit. This prevents a burst of generated jobs from overwhelming a server's I/O subsystem regardless of worker pool size.

4. Conflict detection

The conflict matrix defines which operation pairs cannot run simultaneously on the same database (or the same server for I/O-heavy operations). Before dispatching a job, the orchestrator calls HasConflict(). If a conflict is detected, the job is recorded with StatusConflict and retried on the next tick rather than failing.

5. Maintenance window awareness

Heavy operations — full backups, integrity checks, index rebuilds — carry a RequiresWindow: true flag on their policy. The scheduler will not dispatch these jobs outside a defined maintenance window. Windows are stored with IANA timezone names so DST transitions are handled correctly automatically.

6. Staggered scheduling

The stagger algorithm distributes jobs across a maintenance window proportionally. Given a 4-hour window and 120 databases on a server, each database receives a unique offset between 0 and 239 minutes. No two databases on the same server fire simultaneously for the same operation type. The thundering herd is eliminated structurally.

7. Dynamic timeouts

Job timeouts scale with database size. A 100 MB database gets a 30-second base timeout for a backup; a 5 TB database might get a 6-hour timeout for an integrity check. The timeout formula is configurable per operation type: base + (sizeGB × perGBMultiplier), capped at MaxTimeout. A job that exceeds its dynamic timeout is cancelled and marked StatusTimeout — not retried, but alertable.

8. Environment agnostic

The scheduler runs identically on a single developer laptop (SQLite, in-process, no etcd), a bare-metal data centre (Postgres store, Redis for locks), and Kubernetes (etcd leader election, persistent volumes). Environment differences are injected through a Config struct — the core logic never checks the deployment environment directly.

9. Auto-discovery

The inventory refresh loop runs on a configurable interval. When a new database appears, it is detected on the next refresh and immediately becomes eligible for policy matching. When a database goes offline or is deprovisioned, the scheduler stops generating jobs for it without any manual intervention.

10. Graceful shutdown

On receipt of SIGTERM or SIGINT, the scheduler stops accepting new jobs, waits for in-flight jobs to complete (up to a configurable drain timeout, default 5 minutes), then exits cleanly. Long-running jobs — like a 6-hour CHECKDB — are allowed to finish. The process will not exit mid-backup.
