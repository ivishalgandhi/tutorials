Data Models — Servers, Databases, and Operations

The scheduler's power comes from rich metadata. Every decision — which jobs to generate, which servers are overloaded, which operations conflict, how long a timeout should be — is derived from three core types: DBServer, Database, and OperationType. This lesson walks through those types and the conflict matrix that sits at the heart of the scheduler.

DBServer — The Fleet Registry

Every physical or virtual database server in the fleet is represented by a DBServer struct. It carries everything the scheduler needs to know about the host: what engine it runs, how many concurrent operations it can tolerate, when its maintenance windows are, and which region and tier it belongs to. 

type EngineType string

const (
    EngineSQLServer  EngineType = "sqlserver"
    EnginePostgreSQL EngineType = "postgresql"
)

type ServerTier string

const (
    TierProduction  ServerTier = "production"
    TierStaging     ServerTier = "staging"
    TierDevelopment ServerTier = "development"
    TierDR          ServerTier = "dr"
)

type DBServer struct {
    ID                 string              `json:"id"`
    Hostname           string              `json:"hostname"`
    Port               int                 `json:"port"`
    Engine             EngineType          `json:"engine"`
    Version            string              `json:"version"`
    Tier               ServerTier          `json:"tier"`
    Region             string              `json:"region"`
    MaxConcurrentOps   int                 `json:"max_concurrent_ops"`
    ConnectionString   string              `json:"-"`
    Tags               map[string]string    `json:"tags"`
    MaintenanceWindows []MaintenanceWindow  `json:"maintenance_windows"`
    Enabled            bool                `json:"enabled"`
}

MaxConcurrentOps is the single most important throttle in the system. It prevents the scheduler from running 20 simultaneous backups on a server with 4 spindles. A busy SQL Server production instance might have MaxConcurrentOps=3; a dedicated backup server might allow 8. The worker pool tracks in-flight jobs per server ID and will not dispatch a new job when the count is at capacity.

Notice that ConnectionString is tagged json:"-". It is intentionally excluded from all JSON serialisation — credentials must never appear in API responses, logs, or Prometheus labels.

GOTCHA

MaxConcurrentOps must be set conservatively. For OLTP SQL Server boxes, start at 2–3. Index rebuilds and integrity checks are I/O-intensive — running three simultaneously on the same spindles will saturate them. If you are unsure, start at 1 for production servers and raise it after observing real I/O utilisation during maintenance windows.

Database — The Maintenance Subject

The Database struct represents a single logical database on a server. It carries the metadata that the policy generator, conflict checker, and executor all need to make decisions. 

type Database struct {
    ID             string          `json:"id"`
    Name           string          `json:"name"`
    ServerID       string          `json:"server_id"`
    Server         *DBServer       `json:"server,omitempty"`
    SizeBytes      int64           `json:"size_bytes"`
    RecoveryModel  string          `json:"recovery_model"` // FULL, SIMPLE, BULK_LOGGED
    State          string          `json:"state"`           // ONLINE, OFFLINE, RESTORING
    IsReadOnly     bool            `json:"is_read_only"`
    LastFullBackup *time.Time      `json:"last_full_backup"`
    LastIntegrity  *time.Time      `json:"last_integrity_check"`
    LastVacuum     *time.Time      `json:"last_vacuum"`
    Priority       int             `json:"priority"`        // 1=critical, 5=low
    ExcludedOps    []OperationType `json:"excluded_ops"`
    Enabled        bool            `json:"enabled"`
}

RecoveryModel is crucial for log backup policies. Log backups only make sense for databases in FULL recovery model. The policy generator checks this field — a SIMPLE recovery model database is automatically excluded from log backup policies by the TargetSelector. No job is generated, so there is no runtime failure to handle.

LastFullBackup, LastIntegrity, LastVacuum are pointer fields — they are nil when the operation has never run on this database. The helper methods below use this to make scheduling decisions: 

// NeedsBackup returns true if the database has never been backed up,
// or if the last full backup is older than maxAge.
func (d *Database) NeedsBackup(maxAge time.Duration) bool {
    if d.LastFullBackup == nil {
        return true // never backed up
    }
    return time.Since(*d.LastFullBackup) > maxAge
}
// IsSystemDB returns true for built-in system databases on both
// SQL Server and PostgreSQL. These are excluded from most policies.
func (d *Database) IsSystemDB() bool {
    systemDBs := map[string]bool{
        // SQL Server system databases
        "master":    true,
        "msdb":      true,
        "model":     true,
        "tempdb":    true,
        // PostgreSQL system databases
        "postgres":  true,
        "template0": true,
        "template1": true,
    }
    return systemDBs[strings.ToLower(d.Name)]
}
SYSTEM DATABASES

System databases are excluded from most policies by default. tempdb cannot be backed up at all. master and msdb need special handling — they are small but critical, and must be backed up after any schema-level change. Set IncludeSystemDBs: true only on policies designed specifically for them, not on general maintenance policies.

OperationType — What Jobs Actually Do

OperationType is a typed string constant that identifies the kind of work a job performs. The full catalogue of supported operations spans both engines and several maintenance categories:

Category Operation Description
SQL Server Backup full_backup BACKUP DATABASE ... TO DISK WITH COMPRESSION, CHECKSUM
SQL Server Backup differential_backup Incremental backup since last full; restores faster than repeated log replays
SQL Server Backup log_backup T-log backup (FULL recovery model only); enables point-in-time restore
SQL Server Maintenance index_rebuild ALTER INDEX REBUILD — for fragmentation > 30%; rewrites index pages entirely
SQL Server Maintenance index_reorganize ALTER INDEX REORGANIZE — for 10–30% fragmentation; online, less locking
SQL Server Maintenance update_statistics UPDATE STATISTICS with sample or full scan; keeps query plans optimal
SQL Server Maintenance integrity_check DBCC CHECKDB — verifies every data and index page; detects corruption
PostgreSQL Backup pg_basebackup Physical cluster backup via pg_basebackup; streams the entire data directory
PostgreSQL Backup pg_dump Logical backup — schema plus data in SQL or custom format; database-level granularity
PostgreSQL Maintenance vacuum_analyze VACUUM (VERBOSE, ANALYZE) — reclaims dead tuple space; updates planner stats
PostgreSQL Maintenance vacuum_full VACUUM FULL — reclaims table bloat; takes exclusive lock on each table
PostgreSQL Maintenance reindex_concurrently REINDEX CONCURRENTLY (PostgreSQL 14+) — rebuilds indexes without blocking reads
Cross-platform health_check Quick connectivity and state check; verifies the database is reachable and ONLINE
Cross-platform cleanup_old_backups Removes backup files older than retention_days from backup storage

Operation Weights — Understanding Resource Impact

Every operation carries four weight scores on a scale of 1–10 for IO, CPU, Lock contention, and Network. The scheduler uses these scores to reason about server load at a high level, though the primary throttle is always MaxConcurrentOps. Weight scores help operators understand why a given combination of operations is dangerous — and they drive future enhancements like weighted concurrency limits.

Operation IO CPU Lock Network Notes
integrity_check 10 8 5 1 Reads every data page — maximally I/O-intensive
full_backup 9 5 3 8 High I/O plus high network to backup target
index_rebuild 9 7 8 1 Rewrites all index pages; holds schema locks
vacuum_full 10 6 10 1 Exclusive table lock; rewrites the entire heap
pg_basebackup 9 4 2 8 Streams the entire PostgreSQL cluster directory
vacuum_analyze 4 3 2 1 Safe to run during business hours for most workloads
log_backup 3 2 1 4 Very lightweight; designed to run every 15–30 minutes
health_check 1 1 0 1 Near-zero impact; run as frequently as needed

The Conflict Matrix — Who Blocks Whom

KEY INSIGHT

The conflict matrix is the heart of safe scheduling. It answers: "Can I start operation X on database D while operation Y is already running on D?" If HasConflict(X, Y) returns true, the new job is deferred with StatusConflict and retried on the next scheduling tick. No data is put at risk, and no jobs are silently dropped. 

var OperationConflicts = map[OperationType][]OperationType{
    // Full backup conflicts: heavy I/O operations that can corrupt or
    // interfere with the backup stream.
    OpFullBackup:     {OpDiffBackup, OpIntegrityCheck, OpIndexRebuild, OpVacuumFull},

    // DBCC CHECKDB reads every page — any concurrent write-heavy
    // operation causes false positives or excessive I/O contention.
    OpIntegrityCheck: {OpFullBackup, OpDiffBackup, OpIndexRebuild, OpVacuumFull},

    // Index rebuild holds schema locks; backup and checkdb can't
    // proceed cleanly while pages are being rewritten.
    OpIndexRebuild:   {OpFullBackup, OpIntegrityCheck, OpVacuumFull, OpReindex},

    // VACUUM FULL holds exclusive locks on every table it touches.
    OpVacuumFull:     {OpFullBackup, OpIndexRebuild, OpReindex, OpPgBaseBackup},

    // REINDEX conflicts with VACUUM FULL and index_rebuild (both
    // compete for the same index structures).
    OpReindex:        {OpVacuumFull, OpIndexRebuild},

    // Log backup is designed to be lightweight and non-blocking.
    OpLogBackup:      {}, // never conflicts

    // VACUUM ANALYZE only conflicts with VACUUM FULL (both want
    // to process the visibility map and free space map).
    OpVacuumAnalyze:  {OpVacuumFull},

    // REINDEX CONCURRENTLY never takes exclusive locks — it's safe
    // to run alongside any other operation.
    OpReindexConcur:  {}, // concurrent — never conflicts
}

// HasConflict returns true if op1 and op2 cannot run concurrently
// on the same database. The check is directional (op1 → op2).
func HasConflict(op1, op2 OperationType) bool {
    for _, conflict := range OperationConflicts[op1] {
        if conflict == op2 {
            return true
        }
    }
    return false
}

Server-Level Conflicts

Database-level conflict detection handles the case where two operations target the same database. But some operations are so resource-intensive that they conflict at the server level — even when they target different databases on the same host.

DBCC CHECKDB is the canonical example. It reads every data and index page in a database, saturating the I/O subsystem. Running two CHECKDB operations simultaneously on the same SQL Server — even on different databases — will overwhelm the server's spindles and cause both to run three to four times slower. The same is true for VACUUM FULL on PostgreSQL. 

// ServerLevelConflicts defines operations that cannot run concurrently
// on the same SERVER, regardless of which database they target.
var ServerLevelConflicts = map[OperationType][]OperationType{
    // Only one DBCC CHECKDB may run per SQL Server at a time.
    OpIntegrityCheck: {OpIntegrityCheck},

    // Only one VACUUM FULL may run per PostgreSQL server at a time.
    OpVacuumFull: {OpVacuumFull},
}

The ServerConcurrencyController (built in Lesson 4) checks both the database-level conflict matrix and this server-level conflict map before dispatching any job. A new job is blocked if either check fails.

COMMON MISTAKE

Forgetting server-level conflicts is a frequent production mistake. If you have 30 databases on a SQL Server and CHECKDB is scheduled for all of them, the scheduler will correctly refuse to run more than one at a time because OpIntegrityCheck is in its own conflict list. Without this check, 30 simultaneous CHECKDB operations would completely saturate the server's I/O — even though each one targets a different database and passes the database-level conflict check.

Maintenance Windows

Heavy operations must be confined to maintenance windows — time periods when the server experiences minimal user-facing load. The MaintenanceWindow struct stores a window definition in a timezone-aware, DST-correct way. 

type MaintenanceWindow struct {
    Name       string          `json:"name"`
    DaysOfWeek []time.Weekday  `json:"days_of_week"`
    StartHour  int             `json:"start_hour"`
    StartMin   int             `json:"start_min"`
    EndHour    int             `json:"end_hour"`
    EndMin     int             `json:"end_min"`
    Timezone   string          `json:"timezone"` // IANA e.g. "America/New_York"
    Enabled    bool            `json:"enabled"`
}

// IsActive returns true if 'now' falls within this window.
// Handles midnight-crossing windows and DST transitions correctly.
func (w *MaintenanceWindow) IsActive(now time.Time) bool {
    loc, err := time.LoadLocation(w.Timezone)
    if err != nil {
        return false
    }
    local := now.In(loc)
    weekday := local.Weekday()

    inDay := false
    for _, d := range w.DaysOfWeek {
        if d == weekday {
            inDay = true
            break
        }
    }
    if !inDay {
        return false
    }

    h, m, _ := local.Clock()
    current := h*60 + m
    start   := w.StartHour*60 + w.StartMin
    end     := w.EndHour*60 + w.EndMin

    if end < start {
        // window crosses midnight: e.g. 23:00–02:00
        return current >= start || current < end
    }
    return current >= start && current < end
}

The midnight-crossing check (if end < start) handles windows like "23:00–02:00" that span two calendar days. Without it, a naive range check would always return false for any time in a cross-midnight window.

TIMEZONE HANDLING

Time zone handling is critical and often gets wrong. A window defined as "01:00–05:00 America/New_York" will correctly shift by one hour during DST transitions because time.LoadLocation plus .In() perform the full IANA timezone conversion at call time. Never store UTC offsets (like -05:00) — they are fixed and will be wrong for half the year in regions that observe DST. Always store IANA timezone names like "America/New_York" or "Europe/London".
