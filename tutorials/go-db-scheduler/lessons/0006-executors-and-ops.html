Executors, Observability, and Production Deployment

The orchestrator from Lesson 5 knows when and whether to run a job, but it delegates the actual work to an Executor. This lesson implements both executors — SQL Server (T-SQL) and PostgreSQL (SQL + shell) — then wires up Prometheus metrics so you can observe what the scheduler is doing in production across 5,000 servers.

REFERENCE SHEET

The SQL and T-SQL commands in this lesson are catalogued in the Maintenance Operations reference sheet. Bookmark it — you will return to it when writing executor code.

The Executor Interface

You saw this interface in Lesson 5. Let's understand why it is designed this way before implementing it. 

type Executor interface {
    Execute(ctx context.Context, job *ScheduledJob) (Result, error)
}

type Result struct {
    Duration    time.Duration
    BytesWritten int64          // for backups
    Message     string          // last STATS row or completion message
}

One method, one contract. This is a deliberately small interface. In Go, small interfaces are easier to mock in tests, easier to satisfy, and easier to understand. The orchestrator never knows whether it is talking to SQL Server or PostgreSQL — it only calls Execute. The engine-specific logic is completely isolated inside each executor struct.

Why a single method?What you gain
One operation per job The interface matches the domain model exactly — one ScheduledJob → one Execute call
Context for timeout/cancel Dynamic timeouts (Lesson 3) arrive via the context — the executor just honours them with ctx
Uniform return type The orchestrator can record metrics and log results without engine-specific branching

SQL Server Executor

SQL Server operations run as T-SQL statements via a database/sql connection. The ExecContext method respects the context timeout, so a 12-hour CHECKDB is cancelled automatically if the context is cancelled. 

type SQLServerExecutor struct {
    db *sql.DB   // connection pool to this server
}

func (e *SQLServerExecutor) Execute(ctx context.Context, job *ScheduledJob) (Result, error) {
    start := time.Now()

    switch job.Operation {
    case OpFullBackup:
        return e.fullBackup(ctx, job, start)
    case OpLogBackup:
        return e.logBackup(ctx, job, start)
    case OpIntegrityCheck:
        return e.checkDB(ctx, job, start)
    case OpIndexRebuild:
        return e.rebuildIndexes(ctx, job, start)
    case OpUpdateStats:
        return e.updateStats(ctx, job, start)
    default:
        return Result{}, fmt.Errorf("unsupported operation: %s", job.Operation)
    }
}

Full Backup (T-SQL)

The T-SQL BACKUP DATABASE statement is executed as a string query. STATS = 10 causes SQL Server to emit a row every 10% of progress, which the executor can read to report intermediate progress. Source: Microsoft Learn — BACKUP. 

func (e *SQLServerExecutor) fullBackup(ctx context.Context, job *ScheduledJob, start time.Time) (Result, error) {
    backupPath := backupFilePath(job.DB, "full", start)

    // ExecContext runs the T-SQL and respects the context deadline.
    // If ctx is cancelled (shutdown or timeout), the query is killed.
    _, err := e.db.ExecContext(ctx, `
        BACKUP DATABASE [@db]
        TO DISK = @path
        WITH COMPRESSION, CHECKSUM, STATS = 10,
             NAME = @name,
             DESCRIPTION = 'Scheduled by db-scheduler'
    `,
        sql.Named("db",   job.DB.Name),
        sql.Named("path", backupPath),
        sql.Named("name", fmt.Sprintf("%s Full %s", job.DB.Name, start.Format("2006-01-02"))),
    )
    if err != nil {
        return Result{}, wrapError(err, "full_backup")
    }
    return Result{Duration: time.Since(start), Message: "backup complete"}, nil
}

Integrity Check (T-SQL)

DBCC CHECKDB is SQL Server's integrity check command. For large databases use PHYSICAL_ONLY for nightly runs; the full check weekly. Source: Microsoft Learn — DBCC CHECKDB. 

func (e *SQLServerExecutor) checkDB(ctx context.Context, job *ScheduledJob, start time.Time) (Result, error) {
    // Choose mode from policy config.
    mode := "NO_INFOMSGS"
    if v, ok := job.Policy.Config["physical_only"]; ok && v == true {
        mode = "PHYSICAL_ONLY, NO_INFOMSGS"
    }
    query := fmt.Sprintf("DBCC CHECKDB ([%s]) WITH %s, ALL_ERRORMSGS", job.DB.Name, mode)

    _, err := e.db.ExecContext(ctx, query)
    return Result{Duration: time.Since(start)}, err
}

PostgreSQL Executor

PostgreSQL operations split into two categories: those that run as SQL statements (VACUUM, REINDEX, ANALYZE) and those that run as shell commands (pg_dump). Both are handled by the same executor struct. 

type PostgreSQLExecutor struct {
    db    *sql.DB         // for SQL operations
    creds DBCredentials   // for shell commands (pg_dump)
}

func (e *PostgreSQLExecutor) Execute(ctx context.Context, job *ScheduledJob) (Result, error) {
    start := time.Now()
    switch job.Operation {
    case OpFullBackup:
        return e.pgDump(ctx, job, start)
    case OpVacuum:
        return e.vacuum(ctx, job, start)
    case OpIndexRebuild:
        return e.reindex(ctx, job, start)
    case OpUpdateStats:
        return e.analyze(ctx, job, start)
    default:
        return Result{}, fmt.Errorf("unsupported operation: %s", job.Operation)
    }
}

VACUUM ANALYZE (SQL)

VACUUM ANALYZE reclaims dead tuple space and updates query planner statistics. Use SKIP_LOCKED during production hours — it silently skips tables that are heavily in use rather than blocking. Source: PostgreSQL Docs — VACUUM. 

func (e *PostgreSQLExecutor) vacuum(ctx context.Context, job *ScheduledJob, start time.Time) (Result, error) {
    // VACUUM cannot run inside a transaction, so ExecContext is sufficient.
    // SKIP_LOCKED makes it safe to schedule during production hours.
    _, err := e.db.ExecContext(ctx,
        fmt.Sprintf("VACUUM (ANALYZE, SKIP_LOCKED, VERBOSE)"),
    )
    return Result{Duration: time.Since(start)}, err
}

pg_dump (Shell Command)

PostgreSQL has no SQL-level backup command. pg_dump is a client binary that writes to a file. The Go executor runs it via exec.CommandContext — the context timeout also kills the shell process. 

func (e *PostgreSQLExecutor) pgDump(ctx context.Context, job *ScheduledJob, start time.Time) (Result, error) {
    backupPath := backupFilePath(job.DB, "pgdump", start)

    // exec.CommandContext: if ctx is cancelled, the OS kills the process.
    cmd := exec.CommandContext(ctx,
        "pg_dump",
        "--host",     job.Server.Hostname,
        "--port",     strconv.Itoa(job.Server.Port),
        "--username", e.creds.User,
        "--dbname",   job.DB.Name,
        "--format=custom",   // compressed, parallel-restore ready
        "--compress=6",       // gzip level 6
        "--file",     backupPath,
    )
    // Pass password via environment variable — never via command line (visible in ps)
    cmd.Env = append(os.Environ(), "PGPASSWORD="+e.creds.Password)

    output, err := cmd.CombinedOutput()
    return Result{Duration: time.Since(start), Message: string(output)}, err
}
GO BEGINNER — exec.CommandContext

exec.CommandContext(ctx, "pg_dump", ...) creates a shell command that is bound to the context. If the context is cancelled (timeout or shutdown), Go sends SIGKILL to the shell process automatically. This is the same dynamic timeout mechanism used by SQL Server's ExecContext — the executor does not need separate timeout logic.

Observability — Prometheus Metrics

Prometheus works by scraping your application on a /metrics endpoint. Your Go binary exports numbers; Prometheus pulls them every 15 seconds and stores them as time series. Grafana visualises those time series.

The Three Metric Types You Need

TypeBehaviourUse forExample
Counter Only increases, resets on restart Total jobs completed, total errors scheduler_jobs_total{op,status}
Gauge Goes up and down freely Jobs currently in-flight, queue depth scheduler_jobs_in_flight{server}
Histogram Counts observations in configurable buckets Job duration distribution (p50, p95, p99) scheduler_job_duration_seconds{op}
COUNTER VS GAUGE — THE COMMON MISTAKE

Use a Counter for anything that only ever goes up: jobs completed, errors seen, bytes written. Use a Gauge for anything that goes both up and down: jobs currently running, queue length, memory used. Using a Counter for "jobs in flight" is a common mistake — it would never decrease, so you could never tell when the queue drains.

Declaring Metrics with promauto

promauto is the idiomatic way to declare Prometheus metrics in Go — it registers them automatically at startup so you cannot forget to call prometheus.MustRegister. Source: Prometheus — Instrumenting a Go Application. 

import (
    "github.com/prometheus/client_golang/prometheus"
    "github.com/prometheus/client_golang/prometheus/promauto"
)

type schedulerMetrics struct {
    // Counter: jobs_total incremented on every job completion.
    // Labels: op (e.g. "full_backup"), status ("success", "failed", "conflict", "timeout")
    jobsTotal *prometheus.CounterVec

    // Gauge: jobs currently running on a given server.
    jobsInFlight *prometheus.GaugeVec

    // Histogram: duration of completed jobs in seconds.
    // Buckets cover the full range: 5s to 12h.
    jobDuration *prometheus.HistogramVec

    // Gauge: length of the job channel queue.
    queueDepth prometheus.Gauge
}

func newMetrics() *schedulerMetrics {
    return &schedulerMetrics{
        jobsTotal: promauto.NewCounterVec(prometheus.CounterOpts{
            Name: "scheduler_jobs_total",
            Help: "Total jobs completed, by operation type and final status.",
        }, []string{"operation", "status"}),

        jobsInFlight: promauto.NewGaugeVec(prometheus.GaugeOpts{
            Name: "scheduler_jobs_in_flight",
            Help: "Jobs currently executing, by server hostname.",
        }, []string{"server"}),

        jobDuration: promauto.NewHistogramVec(prometheus.HistogramOpts{
            Name:    "scheduler_job_duration_seconds",
            Help:    "Duration of completed jobs in seconds.",
            Buckets: []float64{5, 30, 120, 600, 1800, 3600, 7200, 21600, 43200},
            // buckets: 5s, 30s, 2m, 10m, 30m, 1h, 2h, 6h, 12h
        }, []string{"operation"}),

        queueDepth: promauto.NewGauge(prometheus.GaugeOpts{
            Name: "scheduler_queue_depth",
            Help: "Current number of jobs waiting in the job channel.",
        }),
    }
}

Recording Metrics in the Dispatch Loop

func (o *Orchestrator) dispatch(ctx context.Context, job *ScheduledJob) {
    // ... (window check, conflict check, semaphore acquire) ...

    // Gauge: increment when job starts
    o.metrics.jobsInFlight.With(prometheus.Labels{"server": job.Server.Hostname}).Inc()
    defer o.metrics.jobsInFlight.With(prometheus.Labels{"server": job.Server.Hostname}).Dec()

    start := time.Now()
    result, err := o.executor(job.Server.Engine).Execute(jobCtx, job)

    // Counter: record final status
    status := "success"
    if err != nil {
        status = classifyError(err)   // "failed", "timeout", "canceled"
    }
    o.metrics.jobsTotal.With(prometheus.Labels{
        "operation": string(job.Operation),
        "status":    status,
    }).Inc()

    // Histogram: observe actual duration
    o.metrics.jobDuration.With(prometheus.Labels{
        "operation": string(job.Operation),
    }).Observe(result.Duration.Seconds())

    _ = result   // store result via o.store.RecordResult(job, result, err)
}

Before shipping to your 5,000-server fleet, verify each of these:
