Replication Lag & Vacuum Health

Lesson 2 covered connection health — the right now state of the server. This lesson adds two more signals every scanner needs: replication lag (is this server current with its primary?) and vacuum health (is dead-tuple accumulation creeping toward a performance cliff?). Together, these three areas — connections, replication, vacuum — cover the majority of production PostgreSQL incidents.

The Core Problem
Your scanner connects to 3,000 servers without knowing in advance which are primaries and which are replicas. The replication checks are completely different depending on the role. The solution: write one query that works on any server and returns meaningful values for both roles.

1. Primary vs Replica — Why It Matters

PostgreSQL streaming replication has two roles:

Rolepg_is_in_recovery()Where replication data lives
Primary false pg_stat_replication — one row per connected replica, with lag measurements
Replica true pg_last_xact_replay_timestamp() — timestamp of last WAL record applied

The naive approach — two separate queries, one for each role — requires your scanner to know the server's role before running the check. Instead, use one query that handles both, returning NULL for columns that don't apply:

flowchart LR A([connect to server]) --> B{pg_is_in_recovery?} B -- false\nprimary --> C["pg_stat_replication\n→ replica_count, max_lag"] B -- true\nreplica --> D["pg_last_xact_replay_timestamp\n→ lag_seconds"] C --> E([report results]) D --> E
One query handles both roles — primary columns are NULL on replicas, replica columns are NULL on primaries

2. The Replication Query (Works on Any Server)

-- Works on both primary and replica. NULL columns are expected.
SELECT
  pg_is_in_recovery()                           AS is_replica,

  -- Replica-side: seconds behind primary
  CASE
    WHEN NOT pg_is_in_recovery()    THEN NULL
    WHEN pg_last_wal_receive_lsn() =
         pg_last_wal_replay_lsn()   THEN 0    -- fully caught up
    ELSE extract(epoch FROM
           now() - pg_last_xact_replay_timestamp())::int
  END                                           AS replica_lag_seconds,

  -- Primary-side: how many replicas are streaming
  (SELECT count(*)
   FROM pg_stat_replication)                   AS streaming_replicas,

  -- Primary-side: worst replay lag across all replicas
  (SELECT max(extract(epoch FROM replay_lag))::int
   FROM pg_stat_replication)                   AS max_replica_lag_seconds;

Sample output for each role:

Server roleis_replicareplica_lag_secondsstreaming_replicasmax_replica_lag_seconds
Primary with 2 replicas false NULL 2 4
Replica, 12s behind true 12 0 NULL
Standalone (no replication) false NULL 0 NULL
The LSN equality check
pg_last_wal_receive_lsn() = pg_last_wal_replay_lsn() means the replica has received and fully applied all WAL from the primary — lag is genuinely zero. Without this check, pg_last_xact_replay_timestamp() grows stale on an idle primary even when the replica is perfectly current, giving a false positive. [source]

Thresholds for replica lag:

LagSignalNotes
< 10sHealthyNormal streaming replication under load
10s – 60sWarningNetwork pressure or replica I/O bound; investigate
> 60sCriticalReplica is falling behind; risk of data loss on failover

3. Vacuum Health — The Slow Accumulating Risk

Every UPDATE and DELETE in PostgreSQL leaves behind dead tuples — the old row versions that MVCC must keep until no active transaction can see them. Autovacuum reclaims them on a schedule, but it can fall behind. When it does, tables bloat, query plans degrade, and in extreme cases the server approaches transaction ID wraparound — a hard shutdown PostgreSQL enforces to protect data integrity.

Why this matters for a scanner
Vacuum problems are silent for weeks, then sudden. A table at 30% dead tuples looks fine until a query plan switches from an index scan to a sequential scan and latency spikes 10×. Your scanner is often the first tool to catch this — before the application team notices. 
-- Top 10 tables with the most dead tuple accumulation
SELECT
  schemaname,
  relname                                       AS table_name,
  n_dead_tup,
  n_live_tup,
  round(100.0 * n_dead_tup /
    NULLIF(n_live_tup + n_dead_tup, 0), 1)     AS dead_pct,
  last_autovacuum,
  last_autoanalyze,
  now() - last_autovacuum                       AS since_last_vacuum
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
ORDER BY n_dead_tup DESC
LIMIT 10;

Thresholds for dead tuple percentage:

dead_pctSignalNotes
< 5%HealthyAutovacuum is keeping up
5–20%WarningAutovacuum may be throttled or losing ground on high-churn tables
> 20%CriticalSignificant bloat; query plan degradation likely; manual VACUUM needed
What NULLIF(n_live_tup + n_dead_tup, 0) protects against
Empty tables have zero live and dead tuples. Dividing by zero would crash the query. NULLIF(expr, 0) returns NULL instead of zero, making the division return NULL safely for empty tables.

4. Your TOML Files for These Checks

# checks/replication.toml

[[checks]]
name        = "replication_status"
description = "Role detection, replica lag, and primary replica count"
sql         = """
  SELECT
    pg_is_in_recovery() AS is_replica,
    CASE
      WHEN NOT pg_is_in_recovery() THEN NULL
      WHEN pg_last_wal_receive_lsn() = pg_last_wal_replay_lsn() THEN 0
      ELSE extract(epoch FROM now() - pg_last_xact_replay_timestamp())::int
    END AS replica_lag_seconds,
    (SELECT count(*) FROM pg_stat_replication)                    AS streaming_replicas,
    (SELECT max(extract(epoch FROM replay_lag))::int
     FROM pg_stat_replication)                                    AS max_replica_lag_seconds
"""
# checks/vacuum.toml

[[checks]]
name        = "vacuum_health"
description = "Tables with high dead tuple accumulation"
sql         = """
  SELECT schemaname, relname, n_dead_tup, n_live_tup,
    round(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 1) AS dead_pct,
    last_autovacuum,
    now() - last_autovacuum AS since_last_vacuum
  FROM pg_stat_user_tables
  WHERE n_dead_tup > 10000
  ORDER BY n_dead_tup DESC
  LIMIT 10
"""
Zero rows is a healthy result
The vacuum query returns zero rows when no table has more than 10,000 dead tuples. Your runner should treat an empty result as a pass, not an error. This is the same pattern as the long-running queries and stuck-transactions checks from Lesson 2.

5. The Full Three-Category Scanner

With all three check files in place, your scanner runs the same runCategory function from Lesson 2 three times per server — one connection, three files: 

func checkServer(ctx context.Context, s Server) ([]CheckResult, error) {
    conn, err := connectSafe(ctx, s) // from Lesson 1: RuntimeParams + timeouts
    if err != nil { return nil, err }
    defer conn.Close(ctx)

    categories := []string{
        "checks/activity.toml",
        "checks/replication.toml",
        "checks/vacuum.toml",
    }

    var all []CheckResult
    for _, path := range categories {
        results, err := runCategory(ctx, conn, path)
        if err != nil {
            // category failed (e.g. statement_timeout hit) — record and continue
            all = append(all, CheckResult{Name: path, Err: err})
            continue
        }
        all = append(all, results...)
    }
    return all, nil
}

Summary

  • pg_is_in_recovery() tells you whether a server is a primary or replica — your scanner must check this before interpreting replication data
  • Write one replication query that returns meaningful columns for both roles, using NULL where a column doesn't apply
  • The LSN equality check prevents false-positive lag on idle primaries
  • Dead tuples accumulate silently — pg_stat_user_tables.n_dead_tup and dead_pct are the key signals
  • Zero rows from the vacuum query is a pass; many rows from the replication query is also fine (just replicas reporting lag)
  • Three TOML files, one connection, one runCategory loop — the full scanner pattern is complete
