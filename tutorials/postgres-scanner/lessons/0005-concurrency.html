Concurrency at Scale: Scanning 3,000 Servers Safely

Lesson 4 covered what to do when individual servers fail — timeouts, connection errors, and bad credentials. This lesson covers the other axis of scale: running your scanner against thousands of servers concurrently without overwhelming your own machine or the targets. The solution is controlled concurrency — goroutines, channels, and context cancellation working in concert.

Core Insight
Concurrency is not parallelism. Your scanner is I/O bound — goroutines spend most of their time waiting for network responses. 50–100 concurrent goroutines can drive thousands of servers because they are mostly idle. A pool of 50 goroutines checking 3,000 servers is not 50× slower than checking them all at once; it is safer and nearly as fast.

1. Why Naive Goroutine Fan-out Fails at Scale

The first instinct: launch one goroutine per server. For 10 servers this works fine. For 3,000 it causes cascading failures. 

// DANGEROUS — launches 3,000 goroutines simultaneously
for _, s := range servers {
    go checkServer(ctx, s)
}
Three things break at once
PostgreSQL max_connections. Most servers allow 100–500 simultaneous connections. Sending 3,000 connection attempts in the same instant means thousands of those attempts are refused with "too many connections" — which logs noise or triggers alerts on the very servers you are supposed to monitor silently.

OS file descriptor limits. Each TCP connection consumes a file descriptor. The default ulimit -n on many Linux systems is 1,024. Launching 3,000 goroutines can exhaust this instantly, crashing your scanner process with EMFILE: too many open files.

Memory. Each goroutine carries stack space plus a live TCP connection object. At roughly 5 MB per goroutine (stack + TCP send/receive buffers + TLS state + pgx connection struct), 3,000 concurrent goroutines consume approximately 3,000 × 5 MB = 15 GB — likely more RAM than your scanner host has available.

2. The Semaphore Pattern (Buffered Channel)

The idiomatic Go solution uses a buffered channel as a counting semaphore. Sending to the channel acquires a slot; receiving from it releases one. 

sem := make(chan struct{}, 50) // max 50 concurrent

var wg sync.WaitGroup
for _, s := range servers {
    wg.Add(1)
    sem <- struct{}{} // acquire slot — blocks when channel is full
    go func(srv Server) {
        defer wg.Done()
        defer func() { <-sem }() // release slot when goroutine exits
        checkServer(ctx, srv)
    }(s)
}
wg.Wait()

How it works: the buffered channel has capacity 50. When 50 goroutines are already running, the 51st sem <- struct{}{} blocks the loop — no new goroutine is launched until a running one exits and executes <-sem in its deferred release. At any moment, at most 50 goroutines are alive. The loop automatically resumes as slots free up.

3. The Worker Pool Pattern (Channel of Work)

Instead of throttling goroutine creation, pre-create a fixed number of goroutines and feed them work through a channel. 

work := make(chan Server, len(servers))
for _, s := range servers {
    work <- s
}
close(work) // signal: no more items will ever be sent

var wg sync.WaitGroup
for i := 0; i < 50; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()
        for s := range work { // range exits when channel is closed and empty
            checkServer(ctx, s)
        }
    }()
}
wg.Wait()

How it works: all 3,000 server entries are loaded into a buffered channel upfront, then the channel is closed. Exactly 50 goroutines are launched. Each loops over the channel with range — pulling the next server, checking it, pulling the next. When the channel is both closed and empty, range exits naturally and the goroutine calls wg.Done().

4. Pattern Trade-offs

Pattern Goroutine count Best for Scanner verdict
Semaphore Up to N per run I/O-bound, uneven latency Good fit
Worker pool Exactly N always Steady throughput, CPU+I/O mix Also good
Naive fan-out 1 per server Never at scale Dangerous

For a scanner, both the semaphore and worker pool are valid. The semaphore is simpler to retrofit into existing code that already iterates over servers. The worker pool gives slightly more predictable memory usage because goroutine count is fixed from the start. Choose either; avoid the naive pattern entirely.

5. Sizing the Pool

How do you pick the right N? Use Little's Law, which relates throughput, concurrency, and latency: 

concurrency ≈ target_rate × avg_latency

For your scanner: 3,000 servers in 60 seconds = 50 servers/sec. If each server takes ~2 seconds on average (5-second statement_timeout, most finish in 1–3 s): 

N = 50 servers/sec × 2 s/server = 100 concurrent goroutines

Start at 50–100 and measure actual throughput. Increase N only if you are falling short of the time budget and your infrastructure supports it.

The max_connections ceiling
Your scanner opens one connection per server — but so does every application process running on that server. If a target has max_connections = 100, keeping your scanner's concurrency at 10–20 leaves the vast majority of available connections for real application traffic. A practical rule: stay at 10–20% of the lowest max_connections you expect to see on targets. Matching or exceeding the limit competes directly with live workloads.

6. Context Propagation and Cancellation

Controlling goroutine count is necessary but not sufficient. You also need a way to stop all in-flight work — for example, when a user presses Ctrl+C, or when a whole-scan deadline expires. Go's context package handles this with a two-level timeout. 

ctx, cancel := context.WithTimeout(context.Background(), 90*time.Second) // whole scan timeout
defer cancel()

// Per-server timeout layered on top
func checkServer(parentCtx context.Context, s Server) error {
    ctx, cancel := context.WithTimeout(parentCtx, 10*time.Second)
    defer cancel()
    conn, err := pgx.ConnectConfig(ctx, config)
    // ...
}
Two-level timeout
The outer context (90 s) is created once for the whole scan and passed into every goroutine as parentCtx. The inner context (10 s) is created fresh inside checkServer as a child of the outer. When the outer context is cancelled — because 90 seconds elapsed or because cancel() was called explicitly — every child context is cancelled immediately, regardless of where its per-server timer stands. A single cancel() call from main tears down all in-flight connections at once. No goroutine leaks.

The pattern composes cleanly with both concurrency designs. Pass the outer ctx into each goroutine in the semaphore loop or the worker pool. Every blocking call — pgx.ConnectConfig, query execution, even the sem <- struct{}{} acquire — respects context cancellation when you use select with ctx.Done() for the acquire step in latency-sensitive code.

Summary

  • Launching one goroutine per server at 3,000 targets exhausts max_connections, OS file descriptors, and memory — never do this at scale
  • The semaphore pattern uses a buffered channel: send acquires a slot and blocks the loop when full; receive in a deferred function releases the slot
  • The worker pool pattern uses a fixed number of goroutines that range over a closed work channel until it is empty
  • Both patterns cap concurrency at N; the semaphore is simpler to retrofit, the worker pool gives a fixed goroutine count from the start
  • Pool size formula: N ≈ target_rate × avg_latency; for 3,000 servers in 60 s at 2 s average, N ≈ 100; start at 50–100 and measure
  • Stay at 10–20% of the lowest max_connections on target servers to leave headroom for real application traffic
  • Use a two-level context: 90 s outer for the whole scan, 10 s inner per server; cancelling the outer tears down all in-flight work immediately with no goroutine leaks
