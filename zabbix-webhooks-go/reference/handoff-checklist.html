Handoff Checklist: Questions for Your Zabbix Team

When another team manages Zabbix, you cannot guess the network, security, or operational boundaries. Use this checklist in your kickoff meeting. The answers shape how your Go receiver is exposed, authenticated, and scaled.

1. Endpoint and networking

  • What URL should we give you?
    Decide whether the receiver runs on a public host, a corporate DMZ, or an internal network segment that the Zabbix server can reach.
  • Can the Zabbix server reach our IP and port?
    Firewalls, NAT, and cloud security groups often block outbound HTTPS from the Zabbix server.
  • Do you require a proxy?
    If so, the webhook must expose an HTTPProxy parameter and call req.setProxy() in the JavaScript.
  • Is TLS required, and what certificate trust is in place?
    Self-signed certs, internal CAs, or mTLS requirements change how you run the receiver.

2. Authentication

  • What shared secret or token should the webhook send?
    A bearer token is the simplest shared secret. Agree on rotation, storage, and who can see it.
  • Do you require IP allow-listing, mTLS, or HMAC signatures?
    These are stronger than a bearer token but add complexity on both sides.
  • How do we rotate the secret without dropping alerts?
    Support two valid tokens during rotation, or accept the secret via a header you can change in the Zabbix media type.

3. Events and scope

  • Which triggers will send to our webhook?
    Ask for the action name, conditions, and a list of host groups or templates involved.
  • Will we receive recovery events?
    Recovery events share the same {EVENT.ID} as the problem, so your receiver must update the existing row rather than insert a new one.
  • Will we receive updates (acknowledgments, manual close, severity changes)?
    These need separate handling in the receiver and a different event correlation strategy.
  • What is the expected event volume?
    Use this to size the receiver, database, and idempotency retention window.

4. Payload and macros

  • Which Zabbix macros will be included?
    At minimum confirm {EVENT.ID}, {EVENT.VALUE}, {EVENT.CLOCK}, {EVENT.SEVERITY}, {HOST.NAME}, {TRIGGER.NAME}, and {EVENT.TAGSJSON}.
  • Can we agree on a fixed JSON schema?
    A schema contract prevents silent breakage when either side changes.
  • What is the maximum payload size?
    Long messages or many tags can exceed your receiver's body limit or Zabbix's macro expansion limit.

5. Retry and timeout behavior

  • What is the webhook Timeout setting?
    Zabbix allows 1–60s; default is 30s. Your receiver must respond faster than this.
  • How many attempts and what interval are configured?
    Defaults are 3 attempts with 10s intervals. This affects your idempotency window and alert latency.
  • What status codes will Zabbix treat as success vs. failure?
    Zabbix retries on any non-2xx. A 400 for a malformed payload should not be retried, but Zabbix will retry it anyway; reject bad payloads only for truly unrecoverable cases.

6. Testing and go-live

  • Can we test against a staging Zabbix action before production?
    The media type test button has a 3-second hard-coded timeout, so real trigger testing is more reliable.
  • Who will create the media type and action?
    Often the Zabbix team must do this because it requires Admin privileges.
  • Who owns the webhook on call?
    Clarify which team responds when deliveries fail or the receiver is down.

7. Observability

  • Where do failed webhook attempts show up?
    Zabbix logs them in Reports → Action log and in the server log.
  • Can we get alerts if the webhook fails repeatedly?
    Ask the Zabbix team to set up an internal escalation when the media type error rate rises.
  • What should we expose from our receiver?
    A /health endpoint and metrics for delivery count, duplicate count, and error rate help both teams.
One-page summary

After the meeting, write down: the endpoint URL, the auth method, the agreed JSON schema, the macro list, the timeout/attempt settings, and the on-call rotation. Email it to both teams. This prevents "it worked in my test" surprises later.
