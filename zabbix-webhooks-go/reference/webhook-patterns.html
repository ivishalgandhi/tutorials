Webhook Patterns Quick Reference

Compressed facts for designing, configuring, and operating a Zabbix → Go webhook receiver.

Glossary

TermMeaning in this context
Webhook An HTTP callback: Zabbix sends a POST request to your Go server when an alert fires or recovers.
Media type Zabbix's delivery channel. Type Webhook runs a JavaScript snippet that builds and sends the HTTP request.
Action Zabbix rule that links triggers/events to operations, including "send message via webhook media type."
Event A problem, recovery, update, or other Zabbix occurrence. Webhooks fire after an alert is created.
Idempotency Running the same event twice produces the same final state as running it once.
Idempotency key A stable identifier for a logical event. Zabbix provides {EVENT.ID} for this purpose.
At-least-once delivery Zabbix retries on non-2xx or timeout, so the same event may arrive more than once.
Retry storm Returning 5xx or timing out while the event is already partially processed, causing repeated deliveries.

Webhooks vs. Polling

When you need...UseWhy
Near-real-time reaction to alerts Webhook Polling adds latency equal to the poll interval.
Audit trail of every event Webhook You receive the event once from Zabbix and store it yourself.
Historical analysis across many hosts Polling / API Zabbix history is the source of truth; pulling lets you query arbitrary time ranges.
Low operational surface area Polling No inbound listener, TLS, or firewall rules required.
Correlate alerts with external systems Webhook Push the event into your own database, then join it with other data.
Simple "is this alert still open?" checks Polling Zabbix already maintains problem state; don't rebuild it if you only need snapshots.

Idempotency primitives

Claim an event before processing so retries are harmless.

PostgreSQL

INSERT INTO events (event_id, received_at, status, payload)
VALUES ($1, NOW(), 'pending', $2)
ON CONFLICT (event_id) DO NOTHING
RETURNING event_id;

MySQL / MariaDB

INSERT INTO events (event_id, received_at, status, payload)
VALUES (?, NOW(), 'pending', ?)
ON DUPLICATE KEY UPDATE event_id = event_id;

In Go, check RowsAffected() or use the xmax = 0 trick to know whether you won the race and should process the event.

Operation checklist

  • Return 200 OK only after the event has been durably recorded.
  • Return 200 OK for duplicates too; otherwise Zabbix will retry.
  • Validate the payload shape and reject malformed requests with 400.
  • Verify a shared secret or bearer token before parsing.
  • Do heavy work asynchronously; keep the HTTP handler fast.
  • Store raw payloads for at least a few days for debugging and replay.
  • Trim the idempotency table after a retention window (e.g., 30 days).
