1. The One Frame

This lesson is about the second half of Domain 1: when one agent is not enough, you split the work across a coordinator and subagents. The exam tests whether you can tell the difference between a coordinator problem (bad decomposition) and a subagent problem (bad execution).

The user-facing symptom almost always looks like the subagent: a missing topic, a shallow answer, a duplicate section. But if the subagent executed successfully and produced coherent output, the fault is upstream — in the coordinator's brief.

2. Hub-and-Spoke: The Only Topology

The canonical multi-agent architecture is a hub-and-spoke: the coordinator sits in the center, subagents radiate outward, and every message between subagents routes through the hub.

flowchart LR C["Coordinator"] S1["Subagent 1
(search)"] S2["Subagent 2
(analysis)"] S3["Subagent 3
(synthesis)"] C <-->|Task| S1 C <-->|Task| S2 C <-->|Task| S3 NOTE["No direct subagent-to-subagent links"] NOTE -.-> S1 NOTE -.-> S2 NOTE -.-> S3
All inter-subagent communication goes through the coordinator. Direct agent-to-agent links break observability and error control.

Why no direct agent-to-agent links? Because the coordinator must aggregate, validate, and recover. If subagent A talks directly to subagent B, the coordinator loses observability, can't handle failures, and can't decide whether the intermediate result is good enough.

Distractor
Any exam answer that says "have subagent A pass results directly to subagent B" is wrong. The hub-and-spoke topology is the exam's default correct architecture.

3. Subagent Context Isolation

A subagent is a fresh agent instance. It has its own conversation history, its own system prompt, its own tool whitelist, and no memory of the coordinator's conversation.

sequenceDiagram participant C as Coordinator participant S as Subagent C->>S: Task prompt: "Analyze this document.
Document: [...]
Output format: [...]" Note over S: Own context window
Own system prompt
Own tools S-->>C: Final result only Note over C: Parent never sees intermediate tool calls
Only the subagent's final result returns to the coordinator. Intermediate tool calls stay inside the subagent.

What a subagent receives:

  • Its own system prompt (defined in the coordinator's agent roster)
  • The invocation prompt from the coordinator (the Task tool input)
  • Tool definitions for tools explicitly listed for that subagent

What a subagent does NOT receive:

  • Coordinator's conversation history
  • Coordinator's system prompt
  • Coordinator's tool results
  • Skills (unless explicitly listed in the subagent definition)
Mnemonic
Subagent state: nothing carries over. The prompt is the universe.

This is why "forward the full conversation transcript to the human reviewer" is a wrong answer. The full transcript is noise; the subagent needs only the structured inputs and output contract defined by the coordinator.

4. The Task Tool — Without It, Nothing Delegates

In the Claude Agent SDK, the coordinator spawns subagents via the built-in Task tool. If the coordinator's allowedTools list does not include "Task", the coordinator will never delegate — it will try to do everything inline. 

from claude_agent_sdk import AgentDefinition

coordinator = AgentDefinition(
    name="research-coordinator",
    description="Plans research and delegates to specialized subagents.",
    system_prompt="You are a research coordinator...",
    allowed_tools=["Task", "verify_fact"],  # ← Task must be present
    agents={
        "web-searcher": AgentDefinition(
            description="Searches the web for current sources on a topic.",
            prompt="You are a web research specialist...",
            tools=["WebSearch", "WebFetch"],
            model="sonnet",
        ),
        "analyst": AgentDefinition(
            description="Analyzes and summarizes documents.",
            prompt="You are a document analysis specialist...",
            tools=["ReadFile"],
            model="sonnet",
        ),
    }
)
Common exam bug
"Won't delegate" almost always means the coordinator's allowedTools is missing "Task". The model wants to delegate, but the harness doesn't let it.

Parallel spawning

A coordinator can call multiple Task tools in a single response. The SDK executes them concurrently.

sequenceDiagram participant C as Coordinator participant S1 as Subagent A participant S2 as Subagent B participant S3 as Subagent C C->>S1: Task: "Research X" C->>S2: Task: "Research Y" C->>S3: Task: "Research Z" Note over S1,S3: All three run in parallel S1-->>C: Result A S2-->>C: Result B S3-->>C: Result C
One coordinator response can spawn multiple subagents. The SDK runs them in parallel automatically.
Constraint
Subagents cannot spawn their own subagents. The orchestration tree is one level deep.

5. Task Decomposition Patterns

The coordinator's first job is to decide how to split the work. The exam tests four patterns:

PatternWhenExam signal
Fixed sequential pipeline Stable structure; each step consumes the prior step's output "Every X follows the same flow"
Dynamic decomposition Varying requests; typed subagents (search / doc / synth) "Same roles, scope varies per query"
Orchestrator–worker Uniform workers differentiated only by prompt-supplied scope "N workers, each a different sub-topic"
Evaluator–optimiser Explicit checkable quality criteria; first pass often inadequate "Iterate until coverage/format passes"

Evaluator–optimiser trap

The evaluator–optimiser pattern only works when the evaluator can check quality objectively. If the criteria are subjective — "make the report more compelling" — the loop spins indefinitely because the evaluator can always argue the output isn't good enough yet.

Wrong answer pattern
"Iterate until the synthesis agent is satisfied with quality" — wrong when "satisfied" is not objectively defined.

6. Failure Tracing — Blame the Right Layer

This table is the most important exam tool in this lesson. When a multi-agent system produces bad output, the symptom tells you where to look:

SymptomRoot causeFix
Missing topics in output Coordinator's decomposition too narrow Assign subagents to cover all required dimensions
Duplicated work / redundant sources Coordinator's scope partitioning sloppy Define non-overlapping scopes per subagent
Shallow on every topic Subagent prompts or tool budgets Improve subagent instructions or give more tool budget
Missing source attribution Context-passing format (prose instead of claim→source) Pass structured data with claim → source mappings
Coordinator "won't delegate" allowedTools missing "Task" Add "Task" to coordinator's tool list
Rule
If subagents complete successfully and the output is missing dimensions, blame the coordinator's decomposition.
If subagents complete successfully and produce overlapping work, blame the coordinator's partitioning.
If subagents fail or produce poor quality, blame the subagent prompts or tools.

7. Error Propagation — Structured Context

When a subagent fails, the coordinator needs enough information to decide what to do next: retry, route around, or escalate. A generic "search unavailable" message is useless.

Good structured error context:

  • Failure type (timeout, 429, 404, malformed input)
  • The attempted query / task
  • Partial results, if any
  • Alternative approaches the coordinator could try

Bad answers:

  • "Catch the error and return an empty result set marked as successful" — masks failure as success
  • "Return a generic 'search unavailable' status" — strips the coordinator of context
  • "Forward the full conversation transcript to a human reviewer" — unstructured noise

8. Principle of Least Privilege

Give each subagent only the tools it needs. The synthesis agent from the official sample questions needs to verify simple facts — dates, names, statistics — but not run full web searches.

Correct design: Give the synthesis agent a scoped verify_fact tool for the simple 85% case, and keep complex verifications routed through the coordinator.

Wrong design: Give the synthesis agent access to all web search tools. This violates separation of concerns and lets the synthesis agent drift into research territory.

Exam signal
"85% of verifications are simple, 15% require deeper investigation" → scoped tool for the 85%, coordinator handles the 15%.

9. Multi-Agent Tradeoffs

Anthropic's own multi-agent research system beat single-agent Claude Opus 4 by 90.2% on internal research evals — but at roughly 15× the token cost.

Use multi-agent whenDon't use multi-agent when
Heavy parallelization is possible Agents must tightly share the same context
Information exceeds a single context window Many dependencies between subagents
Task value justifies the token cost Few truly parallelizable subtasks (most coding tasks)
Proportionate design
Single agent beats multi-agent when a single agent suffices. Don't over-engineer.

10. Apply It — Official Sample Questions

Official Sample Question 7 — Multi-Agent Research Coverage

After running the system on the topic "impact of AI on creative industries," you observe that each subagent completes successfully: the web search agent finds relevant articles, the document analysis agent summarizes papers correctly, and the synthesis agent produces coherent output. However, the final reports cover only visual arts, completely missing music, writing, and film production. When you examine the coordinator's logs, you see it decomposed the topic into three subtasks: "AI in digital art creation," "AI in graphic design," and "AI in photography." What is the most likely root cause?

A) The synthesis agent lacks instructions for identifying coverage gaps in the findings it receives from other agents.
B) The coordinator agent's task decomposition is too narrow, resulting in subagent assignments that don't cover all relevant domains of the topic.
C) The web search agent's queries are not comprehensive enough and need to be expanded to cover more creative industry sectors.
D) The document analysis agent is filtering out sources related to non-visual creative industries due to overly restrictive relevance criteria.

Why B is correct
The prompt says all three subagents complete successfully and produce coherent output. The coordinator's logs show the decomposition was visual-arts-only. The subagents did exactly what they were told. Missing dimensions = coordinator's decomposition.
Official Sample Question 8 — Subagent Failure Propagation

The web search subagent times out while researching a complex topic. You need to design how this failure information flows back to the coordinator agent. Which error propagation approach best enables intelligent recovery?

A) Return structured error context to the coordinator including the failure type, the attempted query, any partial results, and potential alternative approaches.
B) Implement automatic retry logic with exponential backoff within the subagent, returning a generic "search unavailable" status only after all retries are exhausted.
C) Catch the timeout within the subagent and return an empty result set marked as successful.
D) Propagate the timeout exception directly to a top-level handler that terminates the entire research workflow.

Why A is correct
The coordinator needs structured context to decide: retry with a different query, proceed with partial results, or escalate. B hides context. C masks failure as success. D gives up on the entire workflow when one subagent hiccups.
Official Sample Question 9 — Least Privilege for Verification

During testing, you observe that the synthesis agent frequently needs to verify specific claims while combining findings. Currently, when verification is needed, the synthesis agent returns control to the coordinator, which invokes the web search agent, then re-invokes synthesis with results. This adds 2-3 round trips per task and increases latency by 40%. Your evaluation shows that 85% of these verifications are simple fact-checks (dates, names, statistics) while 15% require deeper investigation. What's the most effective approach to reduce overhead while maintaining system reliability?

A) Give the synthesis agent a scoped verify_fact tool for simple lookups, while complex verifications continue delegating to the web search agent through the coordinator.
B) Have the synthesis agent accumulate all verification needs and return them as a batch to the coordinator at the end of its pass, which then sends them all to the web search agent at once.
C) Give the synthesis agent access to all web search tools so it can handle any verification need directly without round-trips through the coordinator.
D) Have the web search agent proactively cache extra context around each source during initial research, anticipating what the synthesis agent might need to verify.

Why A is correct
Option A is the principle of least privilege applied to a latency problem: give the synthesis agent only the small tool it needs for the common case, keep the coordinator in the loop for complex cases. B creates blocking dependencies. C over-provisions and breaks separation of concerns. D is speculative caching.

Practice — Wrong Answer Elimination

These questions follow the official exam format. For each one, think about why the wrong answers are wrong.

The synthesis agent needs a tool to detect coverage gaps across subagent outputs.
The coordinator's task decomposition is too narrow and omitted key energy sectors.
The search agent should use broader queries to surface non-renewable alternatives.
The analysis agent is filtering out non-solar/wind sources before synthesis.
The subagent descriptions are too vague for the coordinator to choose one.
The coordinator's system prompt does not explicitly instruct it to delegate.
The coordinator's allowedTools list is missing 'Task', so it cannot invoke subagents.
The subagents are defined with model='inherit' and inherit a model that is too expensive.
Return structured error context (failure type, query, partial results, alternatives) to the coordinator.
Add automatic retry with exponential backoff inside the subagent and return empty if it still fails.
Forward the full subagent conversation transcript to the human reviewer for manual intervention.
Pre-cache the entire internal API in the coordinator so the subagent never needs to call it.
The writer subagent needs more few-shot examples of high-quality reports.
The evaluator's criteria are not objectively checkable, so the loop has no stable termination condition.
The coordinator should directly assign the writer subagent instead of using an evaluator loop.
The loop should use a higher-tier model for the evaluator to reach convergence faster.
