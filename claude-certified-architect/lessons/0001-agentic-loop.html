The Agentic Loop & The One Frame That Unlocks This Exam

Domain 1 — Agentic Architecture & Orchestration — carries 27% of the exam, the single heaviest domain. But more importantly: its concepts appear as distractors in D2–D5 questions too. If D1 is weak, every other domain suffers.

This lesson covers two things: the mechanics of the agentic loop (stop_reason), and the one conceptual frame — from an actual exam passer — that resolves most questions across all five domains.

The One Frame
Prompts are probabilistic. Hooks are deterministic.
When a single failure causes financial loss, a security breach, or a compliance violation: use hooks. Always. No matter how specific or well-worded the prompt is. This distinction is tested relentlessly across all 5 domains.
Not on this exam — don't waste prep time on: Constitutional AI · RLHF · fine-tuning · embedding models · computer use · vision API · streaming/SSE · OAuth/key rotation · rate limits · quotas · pricing · prompt caching implementation details · cloud provider setup · token counting

1. Three Rules That Resolve Most Questions

These rules, distilled from the official exam guide, cut through the noise on nearly every D1 scenario:

#RuleWhat it eliminates
1 Deterministic beats probabilistic when stakes are real. Financial · security · compliance · safety → hooks or programmatic gates. Style · tone · formatting → prompts. Any answer proposing "a stronger system prompt" for a financial-risk scenario
2 Trace failures to their origin. Scope problems (missing topics, duplicate coverage) = upstream coordinator decomposition. Quality problems = subagent prompts or tools. Blaming the subagent when the coordinator gave it a narrow brief
3 Proportionate fixes first. Single agent beats multi-agent when sufficient. Prompts beat hooks when stakes are low. Don't over-engineer. Any answer adding a routing classifier or specialised subagent when the root cause is a bad tool description

A) PreToolUse/PostToolUse hook ← correct
B) Stronger system prompt instruction

If the scenario involves financial loss, security, or compliance — A is the answer, every time.

3. The Agentic Loop — Know This Cold

Every agentic system wraps the Messages API in a loop. The loop terminates on one machine-readable signal, not on text content. Here is the canonical structure: 

messages = [{"role": "user", "content": user_input}]

while True:
    response = client.messages.create(model=MODEL, tools=tools, messages=messages)

    if response.stop_reason == "end_turn":
        break                         # ← only legitimate exit

    if response.stop_reason == "tool_use":
        tool_results = execute_tools(response.content)
        messages.append({"role": "assistant", "content": response.content})
        messages.append({"role": "user",      "content": tool_results})
        # loop continues ↑
stop_reasonMeaningAction
"tool_use"Model wants to call tools — loop continuesExecute all tool_use blocks, append results, call API again
"end_turn"Model finishedStop the loop. Return result to user.
"max_tokens"Response truncatedHandle as error — response is incomplete
"stop_sequence"Custom stop sequence hitDeliberate termination — inspect context
flowchart TD A([User request]) --> B[Call Messages API] B --> C{stop_reason?} C -->|tool_use| D[Extract ALL tool_use blocks] D --> E[Execute tools] E --> F[Append tool_results as user turn] F --> B C -->|end_turn| G([Return final text ✓]) C -->|max_tokens| H([Error: truncated ✗]) style A fill:#e8f0fe,stroke:#1a56db style G fill:#ecfdf5,stroke:#059669 style H fill:#fff0ee,stroke:#c0392b style C fill:#f3f2ee,stroke:#555
The only legitimate loop exit is stop_reason: "end_turn". Never text matching.

Three anti-patterns — always wrong answers in the exam

  1. Parsing natural language (if "I'm done" in response.text) to terminate the loop
  2. Using an iteration cap as the primary stopping mechanism (caps are safety backstops only)
  3. Checking content[0].type == "text" — tool_use responses can begin with a text block

4. Hooks: PreToolUse vs PostToolUse

Hooks are the deterministic enforcement layer. Two hooks dominate the exam. The keyword in the scenario tells you which one:

HookFiresUse casesExam keyword triggers
PreToolUse Before tool executes Gates, authorisation, prerequisite checks, financial limits, capturing prior state hash for audit "prior state" · "before X happens" · "block if" · "prerequisite" · "audit"
PostToolUse After tool, before model sees result Normalise formats, redact PII, truncate oversized results, enrich with metadata "heterogeneous formats" · "normalise" · "redact" · "model confused by varying outputs"
Important distinction
Hooks don't rank tools. Hooks gate, block, or transform. To make Claude prefer one tool over another, use prompts and tool descriptions. To make Claude unable to bypass a rule, use a hook.

5. Apply It: Official Sample Question

Official Sample Question — Scenario: Customer Support Agent

Production data shows that in 12% of cases, your agent skips get_customer entirely and calls lookup_order using only the customer's stated name, occasionally leading to misidentified accounts and incorrect refunds. What change would most effectively address this reliability issue?

A) Add a programmatic prerequisite that blocks lookup_order and process_refund calls until get_customer has returned a verified customer ID.
B) Enhance the system prompt to state that customer verification via get_customer is mandatory before any order operations.
C) Add few-shot examples showing the agent always calling get_customer first, even when customers volunteer order details.
D) Implement a routing classifier that analyses each request and enables only the subset of tools appropriate for that request type.

Why A is correct — the reasoning pattern
A is a programmatic gate → 100% deterministic. The scenario says "incorrect refunds" — financial harm. Rule 1 applies immediately.

Why B and C are wrong: prompts and few-shot examples are ~95–97% reliable. That 3–5% failure rate means some customers still get incorrect refunds. When single failures cause financial harm, probabilistic compliance is not acceptable.

Why D is wrong: it addresses tool availability, not tool ordering. The agent can still skip get_customer within the enabled set.

6. Distractor-Recognition Cribsheet

Most exam questions have one correct answer and three distractors that sound plausible. Learning why the wrong answers are wrong matters more than just reading the right answer.

Reject these phrases when stakes are financial / security / compliance:

  • "Add a stronger system prompt instruction…"
  • "Include few-shot examples demonstrating the correct sequence…"
  • "Route to a specialised subagent whose prompt emphasises…"
  • "Forward the full conversation transcript to the human reviewer"
  • "Catch errors and return an empty string silently"
  • "Retry on every error type with the same delay"
  • "Have subagent A pass results directly to subagent B"

Lean toward these in the correct answer:

  • "PreToolUse hook that blocks…" / "PostToolUse hook that normalises…"
  • "Idempotency key generated on first attempt, passed on every retry"
  • "Pass structured data with claim → source mappings"
  • "Classify errors and handle each category appropriately"
  • "Coordinator's decomposition is too narrow" (for missing-topic scenarios)
  • "Add Task to coordinator's allowedTools" (for "won't delegate" bugs)
  • "Bounded timeout with fallback message" (for latency hangs)

Practice — Wrong Answer Elimination

These questions follow the official exam format. For each one, think about why the wrong answers are wrong — that's the skill the exam tests.

The iteration cap is not set, causing runaway loops when Claude produces verbose output.
The loop uses natural language matching instead of stop_reason, so any phrasing variation causes it to miss the termination signal.
The agent is not appending tool results correctly, causing the model to repeat tool calls indefinitely.
The Messages API does not guarantee consistent stop_reason values across model versions.
Add a PreToolUse hook that blocks process_refund unless get_customer has previously returned a verified customer ID in this session.
Rewrite the system prompt to make the verification requirement more prominent, and add two more targeted few-shot examples.
Deploy a routing classifier trained on historical tickets to determine which requests need identity verification before reaching the agent.
Add a specialised verification subagent whose entire system prompt focuses on enforcing identity checks.
The synthesis agent lacks instructions for detecting coverage gaps across the findings it receives.
The web search agent's queries need to be broadened to surface non-technology sources automatically.
The coordinator's task decomposition is too narrow — it assigned only technology-related subtasks and omitted other relevant dimensions of the topic.
The document analysis agent is filtering out non-technical sources before passing results to synthesis.
PreToolUse hook that rewrites the tool call parameters to request a specific timestamp format from each API.
PreToolUse hook that validates timestamp format in the model's tool call before execution.
Add explicit timestamp format requirements to each tool's description and a few-shot example showing correct reconciliation.
PostToolUse hook that normalises all timestamps to ISO 8601 before the model receives the tool result.
