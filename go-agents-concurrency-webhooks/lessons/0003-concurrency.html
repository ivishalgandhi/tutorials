Concurrency Control

A webhook receiver is useless if it melts under load. A coding agent that spawns one goroutine per task will eventually exhaust memory, file descriptors, or the API rate limit of the service it calls. In this lesson you will learn four tools that keep concurrent Go programs predictable: worker pools, semaphores, rate limiting, and backpressure.

The Concurrency Spectrum in Real Agents

Your blog post places real agents on a spectrum. Pi uses child processes for strong isolation; Keen Code uses goroutines and channels for lightweight throughput; OpenHands uses Python asyncio plus a durable event log for long-running sessions. The choice depends on what you are optimizing for.

Runtime Concurrency primitive Isolation Overhead Best fit
Pi Child processes + JSONL/IPC Strong High per agent Security-critical, multi-tenant agents
Keen Code Goroutines + streaming channels Cooperative Very low per agent High-throughput, many sub-agents
OpenHands Asyncio + durable event log Same process Low per coroutine Long-running sessions, ML-heavy workflows

This lesson focuses on the Keen Code end of the spectrum: Go primitives that let you model each sub-agent, tool call, or wait-state as a cheap unit of concurrency. Even if you later add durable logs or process isolation, the goroutine-and-channel model is the right starting point for the runtime.

Worker Pools: Fixed, Long-Lived Goroutines

A worker pool is a fixed set of goroutines that wait on a jobs channel. When work arrives, the first idle worker picks it up. When there is no more work, the dispatcher closes the jobs channel, each worker exits its for job := range jobs loop, and a sync.WaitGroup waits for every worker to finish before the program moves on.

This pattern is ideal when the work is homogeneous: every job is the same shape, and the cost of keeping workers alive is small compared to the cost of spawning and destroying them. It is the default pattern for a webhook dispatcher that processes many events of the same type. 

package main

import (
    "fmt"
    "sync"
    "time"
)

// Job is the unit of work. In a real agent this might be a webhook payload.
type Job struct {
    ID   int
    Name string
}

// Result reports what happened. A real agent might return an error or action output.
type Result struct {
    ID    int
    Value string
}

// worker reads jobs until the channel is closed, then signals Done.
func worker(id int, jobs <-chan Job, results chan<- Result, wg *sync.WaitGroup) {
    defer wg.Done()
    for job := range jobs {
        // Simulate work. In production this would call a tool or API.
        time.Sleep(50 * time.Millisecond)
        results <- Result{ID: job.ID, Value: fmt.Sprintf("worker %d processed %s", id, job.Name)}
    }
}

func main() {
    const numWorkers = 3
    const numJobs = 10

    jobs := make(chan Job, numJobs)
    results := make(chan Result, numJobs)
    var wg sync.WaitGroup

    // Start a fixed pool of workers.
    for i := 1; i <= numWorkers; i++ {
        wg.Add(1)
        go worker(i, jobs, results, &wg)
    }

    // Send jobs. The workers will pick them up concurrently.
    for i := 1; i <= numJobs; i++ {
        jobs <- Job{ID: i, Name: fmt.Sprintf("job-%d", i)}
    }
    close(jobs) // signal that no more work is coming

    // Collect results in a separate goroutine so sending never blocks.
    go func() {
        wg.Wait()
        close(results)
    }()

    for result := range results {
        fmt.Println(result.Value)
    }
}
KEY INSIGHT

Closing the jobs channel is the standard way to tell workers to stop. A for range over a channel exits automatically when the channel is closed and drained, so the worker returns without extra control logic.

Semaphores: A Bounded Channel as a Token Bucket

A semaphore is a channel that holds N tokens. Before a goroutine starts work, it takes a token. When the work is done, it returns the token. The buffered channel acts as a concurrency limiter: if N goroutines are already running, the next one blocks until a token is released.

In Go, a semaphore is usually just make(chan struct{}, N). A worker pool starts the goroutines up front; a semaphore starts them on demand but keeps the number of active ones bounded. 

package main

import (
    "fmt"
    "sync"
    "time"
)

// processOne simulates a task. In a real agent this might call an LLM API.
func processOne(id int, sem chan struct{}, wg *sync.WaitGroup) {
    defer wg.Done()

    sem <- struct{}{}        // acquire
    defer func() { <-sem }() // release

    fmt.Printf("task %d starting (active: %d)\n", id, len(sem))
    time.Sleep(100 * time.Millisecond)
    fmt.Printf("task %d done\n", id)
}

func main() {
    const maxConcurrent = 2
    const totalTasks = 6

    sem := make(chan struct{}, maxConcurrent)
    var wg sync.WaitGroup

    for i := 1; i <= totalTasks; i++ {
        wg.Add(1)
        go processOne(i, sem, &wg)
    }

    wg.Wait()
    fmt.Println("all tasks finished")
}
COMMON MISTAKE

Forgetting to release the token creates a slow leak. Always pair the acquire with a defer that releases the token, even if the function returns early because of an error.

When to Choose a Worker Pool vs a Semaphore

Both patterns cap concurrency, but they differ in ergonomics and lifecycle. The table below maps each pattern to the agent shapes you will build in the rest of the course.

Concern Worker Pool Semaphore
Goroutine lifecycle Fixed N goroutines created once New goroutine per task
Best for Homogeneous, continuous streams Heterogeneous, bursty tasks
Stop signal Close the jobs channel Cancel context or drain queue
When the agent starts Workers already warm Workers start on demand

Use a worker pool for the dispatcher in Lesson 4: it owns a long-running queue and wants predictable memory. Use a semaphore when a handler needs to call three different APIs concurrently but must not open more than a handful of connections at once.

Rate Limiting: Token Bucket with time.Ticker

Capping the number of concurrent tasks is not enough. If each task calls an external API, you can still overwhelm that API with a burst of requests. Rate limiting controls how many requests are allowed over time.

The simplest standard-library token bucket uses a time.Ticker that refills a channel. The worker must receive from the channel before it is allowed to do work. The channel size is the bucket size; the ticker interval is the refill rate. 

package main

import (
    "fmt"
    "time"
)

func main() {
    // Allow 2 requests per second, with a burst of 2.
    limiter := time.NewTicker(500 * time.Millisecond)
    defer limiter.Stop()

    for i := 1; i <= 5; i++ {
        <-limiter.C // wait for the next token
        fmt.Printf("request %d at %s\n", i, time.Now().Format("15:04:05.000"))
    }
}
WHY THIS MATTERS

A rate limiter is part of being a good API citizen. If your agent calls GitHub, OpenAI, or an internal service, exceeding the rate limit will get you throttled or banned. The standard library gives you the primitives to build one in a few lines.

Backpressure: Say No Before You Drown

Backpressure is the opposite of buffering blindly. When the queue is full, the receiver refuses new work instead of accepting it into an ever-growing backlog. For an HTTP webhook receiver, the right response is usually 503 Service Unavailable with a Retry-After header, so the provider can retry later. 

package main

import (
    "fmt"
    "net/http"
    "time"
)

const maxQueueSize = 5

var jobs = make(chan string, maxQueueSize)

func webhookHandler(w http.ResponseWriter, r *http.Request) {
    // Non-blocking send: if the queue is full, reject the request.
    select {
    case jobs <- r.URL.Path:
        w.WriteHeader(http.StatusAccepted)
        fmt.Fprintln(w, "accepted")
    default:
        w.Header().Set("Retry-After", "5")
        http.Error(w, "queue full", http.StatusServiceUnavailable)
    }
}

func main() {
    // One worker processes the queue.
    go func() {
        for job := range jobs {
            fmt.Println("processing", job)
            time.Sleep(200 * time.Millisecond)
        }
    }()

    http.HandleFunc("/webhook", webhookHandler)
    http.ListenAndServe(":8080", nil)
}
KEY INSIGHT

A select with a default case makes a channel send non-blocking. If the queue is full, the handler immediately returns 503. This protects the worker pool from unbounded backlog.
