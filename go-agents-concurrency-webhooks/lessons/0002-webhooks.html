Receiving Webhooks in Go

In the agent loop, the sense step is how the agent learns that the world changed. For a coding agent, that signal is usually a webhook from GitHub, a CI system, or a task dispatcher. In this lesson you will build a production-style webhook receiver in Go using only the standard library: net/http, crypto/hmac, sync, and io.LimitReader.

The goal is to receive a payload, prove it came from the right sender, acknowledge it quickly, and process it safely without blocking the HTTP response. The five patterns we need are: a net/http handler, HMAC-SHA256 signature verification, body size limits, an asynchronous acknowledgement, and idempotency with replay protection.

Webhooks Are the Agent's Inbox

A webhook is just an HTTP POST sent by a provider when something happens. The receiver's job is to accept the request, trust it only if it can be verified, and hand the event off to the agent loop. Because the provider is watching the response, the receiver must answer quickly. A slow handler causes retries, duplicates, and alert noise.

KEY INSIGHT

The HTTP handler should only do three fast things: read the body, verify the signature, and return HTTP 200. Expensive work belongs in a goroutine or a worker queue so the connection is released.

A Minimal net/http Handler

Go's net/http package gives us an HTTP server without any external framework. You register a function with http.HandleFunc and start listening with http.ListenAndServe. Here is the smallest possible receiver. 

package main

import (
    "fmt"
    "net/http"
)

func main() {
    http.HandleFunc("/webhook", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprint(w, "ok")
    })
    http.ListenAndServe(":8080", nil)
}

This is not a REST framework, a router, or a middleware chain. It is just a function that receives an http.ResponseWriter and an *http.Request. That is the right place to start because every production feature we add will live inside this function.

Limit the Body Size with io.LimitReader

Webhook payloads should be small, but attackers may send enormous bodies. Reading an unbounded body into memory can crash the process. The standard-library pattern is to wrap the request body with io.LimitReader. The reader returns io.EOF once the cap is reached, so we also check the final size. 

package main

import (
    "fmt"
    "io"
    "net/http"
)

const maxBodyBytes = 1 << 20

func handler(w http.ResponseWriter, r *http.Request) {
    body, err := io.ReadAll(io.LimitReader(r.Body, maxBodyBytes+1))
    if err != nil {
        http.Error(w, "failed to read body", http.StatusBadRequest)
        return
    }
    if len(body) > maxBodyBytes {
        http.Error(w, "payload too large", http.StatusRequestEntityTooLarge)
        return
    }
    fmt.Fprintf(w, "received %d bytes", len(body))
}
COMMON MISTAKE

Using io.ReadAll(r.Body) without a limit lets a single request consume unlimited memory. Always cap the body before reading it, even when you trust the provider.

Verify HMAC-SHA256 with Constant-Time Comparison

HMAC is a shared-secret signature. The provider and the receiver both know a secret. The provider sends the body plus a header with the signature; the receiver recomputes the signature and checks that the values match. Because the signature is a secret, the comparison must be constant-time: it must take the same amount of time regardless of how many bytes match. Otherwise an attacker can measure response times to guess the signature one byte at a time.

Go provides this in crypto/hmac and crypto/subtle through hmac.Equal. Never compare signatures with ==. 

package main

import (
    "crypto/hmac"
    "crypto/sha256"
    "encoding/hex"
    "io"
    "net/http"
    "strings"
)

const secret = "your-webhook-secret"

func verify(body []byte, signature string) bool {
    sig := strings.TrimPrefix(signature, "sha256=")
    got, err := hex.DecodeString(sig)
    if err != nil {
        return false
    }
    want := hmac.New(sha256.New, []byte(secret))
    want.Write(body)
    return hmac.Equal(got, want.Sum(nil))
}

func handler(w http.ResponseWriter, r *http.Request) {
    body, _ := io.ReadAll(r.Body)
    if !verify(body, r.Header.Get("X-Signature")) {
        http.Error(w, "invalid signature", http.StatusUnauthorized)
        return
    }
    w.WriteHeader(http.StatusOK)
}

The signature is computed over the raw bytes of the body. That is why we read the body once and store it in a slice. If you parse JSON first, you must still keep the original bytes for the HMAC check.

SECURITY NOTE

A comparison using == stops at the first different byte. That timing difference leaks information. hmac.Equal compares every byte in constant time, so an attacker learns nothing.

Acknowledge Asynchronously

A provider expects a quick HTTP 200. If the receiver does expensive work before responding, the provider may time out and retry, creating duplicate events. The fix is to accept the event into a queue, return 200, and process the event in a background goroutine. 

package main

import (
    "fmt"
    "io"
    "net/http"
    "time"
)

type event struct {
    id      string
    payload []byte
}

func main() {
    events := make(chan event, 100)

    go worker(events)

    http.HandleFunc("/webhook", func(w http.ResponseWriter, r *http.Request) {
        payload, _ := io.ReadAll(r.Body)
        events <- event{id: "evt-123", payload: payload}
        w.WriteHeader(http.StatusOK)
    })

    http.ListenAndServe(":8080", nil)
}

func worker(events <-chan event) {
    for e := range events {
        time.Sleep(100 * time.Millisecond)
        fmt.Printf("processed %s\n", e.id)
    }
}

The buffered channel acts as a small queue. The goroutine drains the channel and does the work. If the worker is slow, the buffer fills up and the handler blocks until there is room. That is a simple form of backpressure, which we will expand in the next lesson.

Idempotency and Replay Protection

Even with async acknowledgement, providers may retry. The same event may arrive twice. The receiver must remember which event IDs it has already processed and skip duplicates. An in-memory map protected by a sync.Mutex is the simplest standard-library solution. 

package main

import (
    "net/http"
    "strconv"
    "sync"
    "time"
)

const replayWindow = 5 * time.Minute

type seenIDs struct {
    mu  sync.Mutex
    ids map[string]struct{}
}

func newSeen() *seenIDs {
    return &seenIDs{ids: make(map[string]struct{})}
}

func (s *seenIDs) add(id string) bool {
    s.mu.Lock()
    defer s.mu.Unlock()
    if _, ok := s.ids[id]; ok {
        return false
    }
    s.ids[id] = struct{}{}
    return true
}

func validTimestamp(ts string) bool {
    if ts == "" {
        return true
    }
    t, err := strconv.ParseInt(ts, 10, 64)
    if err != nil {
        return false
    }
    return time.Since(time.Unix(t, 0)).Abs() <= replayWindow
}

Replay protection adds a time window. The provider sends a timestamp header; the receiver rejects events that are too old or too far in the future. This limits how long a captured request can be replayed by an attacker. The example above accepts a five-minute window.

WHY THIS MATTERS

In production, idempotency is usually stored in a database. The same map-and-mutex pattern scales to that: check the database before processing, and commit the ID in the same transaction as the work.

Production Webhook Design for Agent Runtimes

Your blog post calls webhooks the lingua franca of the event-driven backbone. The inbound webhook is how the agent learns that the world changed, so the receiver has to be reliable before it is clever. The table below maps the five design decisions from the post to the Go code you have already seen.

Concern Why it matters Go primitive used
Authentication A public endpoint that triggers agent work is a tempting target. crypto/hmac + hmac.Equal
Idempotency GitHub, PagerDuty, and others retry; duplicate events must be no-ops. map + sync.Mutex
Retries and backpressure Blindly accepting more work when overloaded crashes the runtime. select with default returning 503
Bounded payload Attackers can exhaust memory with huge POST bodies. io.LimitReader
Async acknowledgement Providers time out and retry if the handler keeps the connection open. Buffered channel + background goroutine

A Complete Webhook Receiver

Here is a single, runnable program that combines all the pieces. It reads a limited body, checks the timestamp and HMAC signature, acknowledges the request, and hands the event to a worker that deduplicates by event ID. 

package main

import (
    "crypto/hmac"
    "crypto/sha256"
    "encoding/hex"
    "fmt"
    "io"
    "net/http"
    "os"
    "strconv"
    "strings"
    "sync"
    "time"
)

const maxBodyBytes = 1 << 20
const replayWindow = 5 * time.Minute

type event struct {
    id      string
    payload []byte
}

func main() {
    secret := os.Getenv("WEBHOOK_SECRET")
    if secret == "" {
        secret = "dev-secret"
    }

    seen := newSeen()
    events := make(chan event, 100)

    go worker(events, seen)

    http.HandleFunc("/webhook", func(w http.ResponseWriter, r *http.Request) {
        webhookHandler(w, r, secret, events)
    })

    fmt.Println("listening on :8080")
    http.ListenAndServe(":8080", nil)
}

func webhookHandler(w http.ResponseWriter, r *http.Request, secret string, events chan<- event) {
    if r.Method != http.MethodPost {
        http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
        return
    }

    ts := r.Header.Get("X-Event-Timestamp")
    if !validTimestamp(ts) {
        http.Error(w, "timestamp outside allowed window", http.StatusBadRequest)
        return
    }

    sig := r.Header.Get("X-Signature")
    if sig == "" {
        http.Error(w, "missing signature", http.StatusBadRequest)
        return
    }

    body, err := io.ReadAll(io.LimitReader(r.Body, maxBodyBytes+1))
    if err != nil {
        http.Error(w, "failed to read body", http.StatusInternalServerError)
        return
    }
    r.Body.Close()

    if len(body) > maxBodyBytes {
        http.Error(w, "payload too large", http.StatusRequestEntityTooLarge)
        return
    }

    if !verifySignature(body, secret, sig) {
        http.Error(w, "invalid signature", http.StatusUnauthorized)
        return
    }

    eventID := r.Header.Get("X-Event-ID")
    if eventID == "" {
        eventID = hex.EncodeToString(hmacHash(append([]byte(ts), body...), secret))[:16]
    }

    w.WriteHeader(http.StatusOK)
    w.Write([]byte("ok"))

    events <- event{id: eventID, payload: body}
}

func validTimestamp(ts string) bool {
    if ts == "" {
        return true
    }
    t, err := strconv.ParseInt(ts, 10, 64)
    if err != nil {
        return false
    }
    return time.Since(time.Unix(t, 0)).Abs() <= replayWindow
}

func verifySignature(body []byte, secret, sig string) bool {
    sig = strings.TrimPrefix(sig, "sha256=")
    got, err := hex.DecodeString(sig)
    if err != nil {
        return false
    }
    return hmac.Equal(got, hmacHash(body, secret))
}

func hmacHash(data []byte, secret string) []byte {
    h := hmac.New(sha256.New, []byte(secret))
    h.Write(data)
    return h.Sum(nil)
}

func worker(events <-chan event, seen *seenIDs) {
    for e := range events {
        if seen.add(e.id) {
            fmt.Printf("processed event %s: %d bytes\n", e.id, len(e.payload))
        } else {
            fmt.Printf("skipped duplicate event %s\n", e.id)
        }
    }
}

type seenIDs struct {
    mu  sync.Mutex
    ids map[string]struct{}
}

func newSeen() *seenIDs {
    return &seenIDs{ids: make(map[string]struct{})}
}

func (s *seenIDs) add(id string) bool {
    s.mu.Lock()
    defer s.mu.Unlock()
    if _, ok := s.ids[id]; ok {
        return false
    }
    s.ids[id] = struct{}{}
    return true
}

You can run this with WEBHOOK_SECRET=test-secret go run main.go. Send it a request with X-Signature, X-Event-ID, and X-Event-Timestamp headers, and the handler will verify, deduplicate, and log the event.
