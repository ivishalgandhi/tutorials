The Dispatcher

The dispatcher is the bridge between the HTTP handler and the worker pool. It reads events from a channel and hands each one to a worker. The worker then runs the agent loop for that single event.

The dispatcher in this course is just a for event := range events loop plus a sync.WaitGroup. It is deliberately small: the concurrency control lives in the worker pool, and the cancellation control lives in the context. This separation makes the program easy to reason about. 

func dispatcher(ctx context.Context, events <-chan Event, workers int) {
    var wg sync.WaitGroup
    for i := 0; i < workers; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            for e := range events {
                runAgent(ctx, id, e)
            }
        }(i)
    }
    wg.Wait()
}
KEY INSIGHT

The dispatcher does not decide how to process an event; it only routes events to workers. The worker does not decide how to limit concurrency; it only consumes from the shared channel. Each layer has one job.

A Complete Program

The program below ties everything together. It starts an HTTP server on :8080, accepts POST events at /webhook, enqueues them, and processes them with three workers. When you press Ctrl+C it stops accepting new connections, drains the queue, and exits. 

package main

import (
    "context"
    "fmt"
    "log/slog"
    "net/http"
    "os"
    "os/signal"
    "sync"
    "sync/atomic"
    "time"
)

// Event is the payload that arrives from the webhook.
type Event struct {
    ID      string
    Action  string
    Payload []byte
}

func main() {
    logger := slog.New(slog.NewTextHandler(os.Stdout, nil))

    ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt)
    defer stop()

    const (
        queueSize   = 100
        numWorkers  = 3
        httpAddr    = ":8080"
    )

    events := make(chan Event, queueSize)
    var processed atomic.Uint64

    var wg sync.WaitGroup
    for i := 0; i < numWorkers; i++ {
        wg.Add(1)
        go worker(ctx, i, events, &wg, logger, &processed)
    }

    mux := http.NewServeMux()
    mux.HandleFunc("/webhook", func(w http.ResponseWriter, r *http.Request) {
        webhookHandler(w, r, events, logger)
    })
    mux.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintln(w, "ok")
    })

    srv := &http.Server{
        Addr:    httpAddr,
        Handler: mux,
    }

    go func() {
        logger.Info("server listening", "addr", httpAddr)
        if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            logger.Error("server error", "err", err)
        }
    }()

    <-ctx.Done()
    logger.Info("shutdown signal received")

    shutdownCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()
    if err := srv.Shutdown(shutdownCtx); err != nil {
        logger.Error("shutdown error", "err", err)
    }

    close(events)
    wg.Wait()

    logger.Info("server stopped", "processed", processed.Load())
}

func webhookHandler(w http.ResponseWriter, r *http.Request, events chan<- Event, logger *slog.Logger) {
    if r.Method != http.MethodPost {
        http.Error(w, "method not allowed", http.StatusMethodNotAllowed)
        return
    }

    eventID := r.Header.Get("X-Event-ID")
    if eventID == "" {
        eventID = fmt.Sprintf("evt-%d", time.Now().UnixNano())
    }

    select {
    case events <- Event{ID: eventID, Action: r.URL.Query().Get("action"), Payload: []byte("{}")}:
        logger.Info("event accepted", "id", eventID)
        w.WriteHeader(http.StatusAccepted)
    default:
        logger.Warn("queue full", "id", eventID)
        w.Header().Set("Retry-After", "5")
        http.Error(w, "queue full", http.StatusServiceUnavailable)
    }
}

func worker(ctx context.Context, id int, events <-chan Event, wg *sync.WaitGroup, logger *slog.Logger, processed *atomic.Uint64) {
    defer wg.Done()
    for e := range events {
        logger.Info("worker started", "worker", id, "event", e.ID)
        runAgent(ctx, e, logger)
        processed.Add(1)
        logger.Info("worker finished", "worker", id, "event", e.ID)
    }
}

func runAgent(ctx context.Context, e Event, logger *slog.Logger) {
    if err := ctx.Err(); err != nil {
        logger.Warn("agent cancelled before start", "event", e.ID, "err", err)
        return
    }

    action := decide(e)
    if action == "" {
        logger.Info("no action", "event", e.ID)
        return
    }

    logger.Info("acting", "event", e.ID, "action", action)
    time.Sleep(50 * time.Millisecond) // simulate work
    logger.Info("action complete", "event", e.ID, "action", action)
}

func decide(e Event) string {
    if e.Action == "" {
        return "noop"
    }
    return e.Action
}
WHY THIS MATTERS

This is the smallest complete agent runtime. It has no external dependencies, no framework, and no distributed coordination. Yet it already demonstrates the four hard parts of production agent systems: event ingestion, bounded concurrency, cancellation propagation, and graceful shutdown.

The Blog Post's Hybrid Architecture, in Miniature

Your blog post concludes that the winning production architecture is a combination of three layers: a systems-language runtime for orchestration, durable event logs for state, and webhooks for real-world integration. The program above is a miniature of that stack:

Blog-post layer What it does Where it lives in this code
Systems-language runtime Lightweight concurrency and single-binary deployment Go, goroutines, and the worker pool
Durable event log Survives restarts and enables replay The bounded events channel (production would persist to a log or database)
Webhook integration Tells the agent the world changed The /webhook handler that enqueues events

A real system would replace the in-memory channel with a durable queue (Postgres, Redis, or a log) and add a database-backed event log before dispatching. The Go code that wraps the queue stays the same shape.

Graceful Shutdown

Graceful shutdown has two phases. First, stop accepting new HTTP requests so the load balancer or provider stops sending work. Second, close the event channel and wait for the workers to finish. The code above does exactly this:

  1. signal.NotifyContext creates a context that is cancelled on SIGINT or SIGTERM.
  2. srv.Shutdown closes the listener and waits for in-flight HTTP requests to finish.
  3. close(events) signals that no more events will arrive.
  4. wg.Wait() blocks until every worker has drained its last event and returned.
COMMON MISTAKE

Calling srv.Shutdown only waits for HTTP requests. It does not wait for background workers. Always pair it with a sync.WaitGroup that tracks your goroutines, or you may exit mid-task.

Observability: Logging and Metrics

For a learning project, structured logging is enough. The standard library log/slog gives you key-value pairs without pulling in a third-party logger. The example uses an atomic counter to track how many events were processed. In production you would export that counter to Prometheus, but the shape is the same. 

var processed atomic.Uint64

// In the worker:
processed.Add(1)

// At shutdown:
logger.Info("server stopped", "processed", processed.Load())
