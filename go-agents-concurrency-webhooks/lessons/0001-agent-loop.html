The Agent Loop in Go

Your blog post argues that coding agents teach us something important about concurrency and webhooks. Before we can talk about webhooks or worker pools, we need the simplest piece: the agent loop. In this lesson you will build a tiny but real loop in Go — sense → decide → act — and learn the three Go tools that make it work: goroutines, channels, and context.Context.

The post calls this the nervous system: the code that wraps the model so a 30-second thought process does not freeze the whole environment. The loop is the first cell of that nervous system.

The Three Jobs of an Agent Loop

An agent is just a program that keeps acting on the world until it is done. The loop has three responsibilities:

  • Sense. Gather input. For a coding agent this might be a prompt, a file change, or a webhook payload.
  • Decide. Pick the next action. This can be a hard-coded rule, a state machine, or an LLM call.
  • Act. Execute the action and observe the result.

The loop repeats until a terminal condition is met: the task is complete, an error is unrecoverable, or the operator tells it to stop. This is the same shape whether the agent is a cron job, a GitHub bot, or a coding assistant.

KEY INSIGHT

The agent loop is control flow, not artificial intelligence. The "decide" step can be a simple if statement today and an LLM tomorrow. The loop shape stays the same.

A Minimal Loop in Go

Here is the smallest idiomatic version. It does not use an LLM; it just watches a mock "inbox" and prints actions. Notice the four Go habits that appear in almost every production loop:

  • ctx context.Context is the first parameter of every function that does I/O.
  • ctx.Err() is checked before doing work so cancellation is respected.
  • A for loop with a clear terminal condition.
  • Errors are returned, not swallowed.
package main

import (
    "context"
    "fmt"
    "time"
)

// Sense returns the next event, or an error if something goes wrong.
func Sense(ctx context.Context) (string, error) {
    if err := ctx.Err(); err != nil {
        return "", err
    }
    time.Sleep(100 * time.Millisecond)
    return "build", nil
}

// Decide maps a sensed event into a concrete action.
func Decide(event string) string {
    switch event {
    case "build":
        return "run go build ./..."
    case "test":
        return "run go test ./..."
    default:
        return "noop"
    }
}

// Act performs the action. In a real agent this would call a tool or API.
func Act(ctx context.Context, action string) error {
    if err := ctx.Err(); err != nil {
        return err
    }
    fmt.Printf("executing: %s\n", action)
    time.Sleep(50 * time.Millisecond)
    return nil
}

func RunAgent(ctx context.Context, maxIterations int) error {
    for i := 0; i < maxIterations; i++ {
        event, err := Sense(ctx)
        if err != nil {
            return fmt.Errorf("sense: %w", err)
        }

        action := Decide(event)
        if action == "noop" {
            fmt.Println("nothing to do")
            continue
        }

        if err := Act(ctx, action); err != nil {
            return fmt.Errorf("act: %w", err)
        }

        if action == "run go build ./..." {
            fmt.Println("build succeeded; stopping")
            return nil
        }
    }
    return fmt.Errorf("reached max iterations (%d)", maxIterations)
}

func main() {
    ctx := context.Background()
    if err := RunAgent(ctx, 10); err != nil {
        fmt.Println("agent error:", err)
    }
}

This program is not concurrent yet, but it is already cancelable. If we replace context.Background() with a context that has a deadline, every call to ctx.Err() will return the cancellation reason and the loop stops cleanly.

Why Go: Goroutines

A Go goroutine is an independently scheduled function. You start one by writing go f(). It is cheaper than an OS thread, so it is normal to have thousands of them. The scheduler multiplexes them onto a small number of OS threads.

For an agent, goroutines are the building block of parallel work: a webhook handler can start a goroutine to process the event while returning HTTP 200 immediately. A worker pool can run many agents at once. A background poller can sense events concurrently with the main loop. 

func main() {
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()

    // Start a background sensor that writes events into a channel.
    events := make(chan string)
    go Sensor(ctx, events)

    // The main loop reads from the channel.
    for event := range events {
        action := Decide(event)
        if err := Act(ctx, action); err != nil {
            fmt.Println("act failed:", err)
            break
        }
    }
}

func Sensor(ctx context.Context, out chan<- string) {
    defer close(out)
    for {
        if err := ctx.Err(); err != nil {
            return
        }
        time.Sleep(100 * time.Millisecond)
        out <- "build"
    }
}
COMMON MISTAKE

A goroutine that is never told to stop leaks. Always pair a goroutine with a cancellation mechanism: a context.Context, a done channel, or a closed job channel.

Channels Are the Glue

A channel is a typed conduit for values between goroutines. The arrow operator <- sends or receives, and close(ch) signals that no more values will come. Channels are how goroutines coordinate without sharing memory.

Two channel patterns show up constantly in agent code:

  • Buffered channel. make(chan string, 10) holds up to 10 values before send blocks. Use it as a queue or a semaphore.
  • Directional channel. chan<- string can only send; <-chan string can only receive. Use them in function signatures to make intent explicit.
// Generator pattern: a goroutine produces values, the main goroutine consumes them.
func Counter(ctx context.Context, out chan<- int) {
    defer close(out)
    for i := 1; ; i++ {
        select {
        case <-ctx.Done():
            return
        case out <- i:
            time.Sleep(100 * time.Millisecond)
        }
    }
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 250*time.Millisecond)
    defer cancel()

    nums := make(chan int)
    go Counter(ctx, nums)

    for n := range nums {
        fmt.Println(n)
    }
}

The select block above is the idiomatic way to combine work with cancellation. It either sends the next value or exits when the context is cancelled.

Stopping the Loop with Context

context.Context is the standard way to carry deadlines, cancellation signals, and request-scoped values across API boundaries in Go. Every function that does I/O should accept a context as its first argument.

The pattern is always the same:

  1. Create a derived context: ctx, cancel := context.WithCancel(parent) or WithTimeout.
  2. Defer cancel() to avoid leaking resources.
  3. Pass ctx to every function that does work.
  4. Check ctx.Err() or <-ctx.Done() inside long-running work.
func main() {
    // The agent must stop after 500 ms, or when Ctrl+C is pressed.
    ctx, stop := signal.NotifyContext(
        context.Background(),
        os.Interrupt,
    )
    defer stop()

    ctx, cancel := context.WithTimeout(ctx, 500*time.Millisecond)
    defer cancel()

    if err := RunAgent(ctx, 100); err != nil {
        fmt.Println("stopped:", err)
    }
}
WHY THIS MATTERS

In production, your agent will receive a SIGTERM when the container shuts down. Without context cancellation, the loop keeps running until the orchestrator kills it. With context, the loop stops cleanly, finishes the current action, and returns.
