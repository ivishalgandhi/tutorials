Go Concurrency Patterns

Compressed reference for the patterns used in the course. Each pattern is shown with a minimal, compilable snippet and a note on when to use it.

1. Generator

A goroutine produces values; a consumer reads them. Close the channel when done.

func generator(ctx context.Context) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for i := 0; ; i++ {
            select {
            case <-ctx.Done():
                return
            case out <- i:
            }
        }
    }()
    return out
}

2. Worker Pool

A fixed set of workers drains a shared job channel. Best for sustained throughput.

func worker(id int, jobs <-chan int, results chan<- int, wg *sync.WaitGroup) {
    defer wg.Done()
    for j := range jobs {
        results <- j * j
    }
}

func runPool(work []int, workers int) []int {
    jobs := make(chan int, len(work))
    results := make(chan int, len(work))
    var wg sync.WaitGroup

    for w := 0; w < workers; w++ {
        wg.Add(1)
        go worker(w, jobs, results, &wg)
    }

    for _, j := range work {
        jobs <- j
    }
    close(jobs)

    go func() { wg.Wait(); close(results) }()

    var out []int
    for r := range results {
        out = append(out, r)
    }
    return out
}

3. Semaphore

A bounded channel limits concurrent work. Best for adding a cap to existing code without rewriting it.

func fetchAll(ctx context.Context, urls []string, max int) error {
    sem := make(chan struct{}, max)
    var wg sync.WaitGroup
    for _, u := range urls {
        wg.Add(1)
        go func(url string) {
            defer wg.Done()
            sem <- struct{}{}
            defer func() { <-sem }()
            _ = fetch(ctx, url)
        }(u)
    }
    wg.Wait()
    return nil
}

4. Done Channel

Close a channel to signal cancellation to many goroutines at once.

done := make(chan struct{})
// ... launch goroutines that select on <-done
close(done) // every goroutine receives the signal

5. Context Propagation

Pass ctx as the first parameter. Check ctx.Err() before work.

func doWork(ctx context.Context) error {
    if err := ctx.Err(); err != nil {
        return err
    }
    // ... do work
    return nil
}

When to Use Which

Pattern Use when Avoid when
Worker pool You have a steady stream of similar jobs and want predictable memory use. Jobs are bursty and have very different durations; a semaphore may be simpler.
Semaphore You need to limit concurrency in one section of existing code with minimal changes. You need to observe or manage individual workers; use a pool instead.
Generator One goroutine produces a sequence that others consume. Multiple producers need coordination; consider a job queue instead.
Done channel You need a one-shot "stop everything" signal. You need deadlines or request-scoped cancellation; use context.
Context propagation Any function that does I/O or runs in a request lifecycle. Passing optional configuration; use a struct instead.
