The Orchestrator — Concurrency Control and the Scheduling Loop

Lessons 2–4 built the data models, policies, and coordination primitives. Now everything converges in the orchestrator: the component that takes a stream of generated ScheduledJob objects and dispatches them safely across 5,000 servers without overwhelming any single server, running conflicting operations on the same database, or losing work during a shutdown.

This lesson introduces three new Go patterns you have not seen before: the worker pool, the per-server semaphore, and graceful shutdown via signal.NotifyContext. Each one is a standard Go idiom worth keeping in your toolbox permanently.

Why a Worker Pool? (Not a Goroutine Per Job)

The simplest way to process jobs concurrently in Go is to spawn a goroutine per job: 

// ❌  NAIVE — do not use this
for _, job := range jobs {
    go execute(job)   // new goroutine for every single job
}

This looks fine for 10 jobs. With 50,000 databases × 6 operation types at a fleet tick, you can generate 300,000 goroutines in milliseconds. Each goroutine starts with a 2 KB stack that grows as needed. At 300,000 goroutines × 2 KB minimum = 600 MB just to start — before any work happens. And the goroutines don't finish quickly: a full backup takes up to 12 hours.

The solution is a worker pool: a fixed number of goroutines that all read from a single shared channel. You control concurrency by controlling the number of goroutines, not the number of jobs. 

// ✅  WORKER POOL — fixed N goroutines drain a shared channel

// The channel is the queue. Buffer size = how many jobs can wait.
// If full, the tick loop blocks (backpressure — intentional).
jobs := make(chan *ScheduledJob, 500)

var wg sync.WaitGroup   // tracks all in-flight workers

for i := 0; i < workerCount; i++ {
    wg.Add(1)
    go func() {
        defer wg.Done()   // runs when this goroutine returns
        for job := range jobs {   // blocks waiting for work
            // "range over channel" exits automatically
            // when close(jobs) is called (graceful shutdown)
            dispatch(ctx, job)
        }
    }()
}
GO BEGINNER — for range over a channel

for job := range jobs is exactly like looping over a slice, except the loop blocks waiting for the next value instead of immediately finishing. When close(jobs) is called, the loop exits cleanly. This is the standard Go pattern for "process work until told to stop."

GO BEGINNER — sync.WaitGroup

A WaitGroup is a counter. wg.Add(1) increments it before launching a goroutine. wg.Done() (called via defer) decrements it when the goroutine finishes. wg.Wait() blocks until the counter reaches zero — meaning all goroutines are done. This is how graceful shutdown waits for in-flight jobs to finish.

Per-Server Semaphore — Protecting MaxConcurrentOps

A global worker pool limits total parallelism, but it doesn't protect individual servers. If 200 workers all pick jobs that happen to land on sql-prod-01, that server gets 200 simultaneous backup processes — saturating its disk I/O completely regardless of MaxConcurrentOps.

The per-server semaphore is the fix. It ensures that at most MaxConcurrentOps jobs are running on any single server at any moment. The Go idiom for a semaphore is a buffered channel: 

// A buffered channel with capacity N is a semaphore with N permits.
// This is explicitly documented in Effective Go:
// https://go.dev/doc/effective_go#channels

type serverSems struct {
    mu   sync.Mutex
    sems map[string]chan struct{}   // one semaphore per serverID
}

// sem() returns (or lazily creates) the semaphore for a server.
func (ss *serverSems) sem(serverID string, maxOps int) chan struct{} {
    ss.mu.Lock()
    defer ss.mu.Unlock()
    if _, ok := ss.sems[serverID]; !ok {
        ss.sems[serverID] = make(chan struct{}, maxOps)
    }
    return ss.sems[serverID]
}
GO BEGINNER — buffered channel as semaphore

A buffered channel of capacity N can hold N values. A send (sem <- struct{}{}) succeeds immediately if there is room — it "acquires a permit." If the channel is full, the send blocks until someone else does a receive (<-sem), which "releases a permit." The empty struct struct{}{} uses zero bytes — it's just a token, not real data.

The dispatch function acquires the semaphore before starting a job and releases it when the job finishes, regardless of whether the job succeeded or failed: 

func (o *Orchestrator) dispatch(ctx context.Context, job *ScheduledJob) {
    // ── 1. Check maintenance window ──────────────────────────
    if job.Policy.Schedule.RequireWindow {
        if !job.Server.IsInWindow(time.Now()) {
            job.Status = StatusSkipped   // try again next tick
            return
        }
    }

    // ── 2. Check conflict ────────────────────────────────────
    if o.store.HasConflict(job.DB.ID, job.Operation) {
        job.Status = StatusConflict   // re-queued next tick, not failed
        return
    }

    // ── 3. Acquire per-server semaphore ──────────────────────
    serverSem := o.sems.sem(job.Server.ID, job.Server.MaxConcurrentOps)
    select {
    case serverSem <- struct{}{}:
        // acquired — proceed
    case <-ctx.Done():
        job.Status = StatusCanceled   // shutdown arrived while waiting
        return
    }
    defer func() { <-serverSem }()   // release on return, always

    // ── 4. Apply dynamic timeout ─────────────────────────────
    timeout := job.Policy.Timeout.ForDatabase(job.DB)
    jobCtx, cancel := context.WithTimeout(ctx, timeout)
    defer cancel()

    // ── 5. Mark running, execute, record result ──────────────
    o.store.MarkRunning(job)
    result, err := o.executor(job.Server.Engine).Execute(jobCtx, job)
    o.store.RecordResult(job, result, err)
}
GOTCHA — select with semaphore acquire

Use select (not a bare send) when acquiring the semaphore so that a shutdown signal can unblock the goroutine. A bare serverSem <- struct{}{} would block forever if the server is at capacity and shutdown arrives — the program could never exit.

The Tick Loop

The tick loop is the heartbeat of the scheduler. Every N seconds (default: 60), it evaluates all policies against live inventory, generates the current set of candidate jobs, and pushes them onto the job channel. The per-server semaphores and conflict detector in the workers handle the traffic safely. 

func (o *Orchestrator) run(ctx context.Context) {
    ticker := time.NewTicker(o.cfg.TickInterval)
    defer ticker.Stop()

    for {
        select {
        case <-ctx.Done():
            return   // shutdown — stop generating new jobs
        case <-ticker.C:
            // Generate jobs for the current moment
            candidates := o.generator.GenerateAll(ctx, time.Now())
            for _, job := range candidates {
                select {
                case o.jobs <- job:
                    // enqueued
                case <-ctx.Done():
                    return   // shutdown mid-generation
                }
            }
        }
    }
}
WHY THE CHANNEL IS BOUNDED

The job channel has a fixed buffer (500 slots). If workers cannot drain the queue fast enough, the o.jobs <- job send will block — applying backpressure to the generator. This is intentional: it prevents the scheduler from accumulating millions of stale jobs during a slow period. The buffer size should be roughly workerCount × max_job_duration_multiplier for your fleet.

Graceful Shutdown — The Full Picture

Kubernetes sends SIGTERM before it kills a pod. Your scheduler might be in the middle of a 6-hour CHECKDB job when this happens. Graceful shutdown is the mechanism that says: "stop accepting new work, but let everything in-flight finish."

sequenceDiagram participant OS as OS / Kubernetes participant Main as main() participant Tick as Tick loop participant Pool as Worker pool participant Job as In-flight job OS->>Main: SIGTERM Main->>Tick: ctx.Done() — stop generating Main->>Pool: close(jobs) — workers drain queue Pool->>Job: dispatching continues until queue empty Job-->>Pool: job completes (up to 5 min drain timeout) Pool-->>Main: wg.Wait() unblocks Main-->>OS: process exits cleanly
Figure 5.2 — SIGTERM triggers a two-phase shutdown: (1) stop new job generation, (2) drain all in-flight jobs before exit. In-progress jobs are never killed mid-execution. 
func main() {
    // 1. ctx is cancelled when SIGTERM or SIGINT arrives.
    //    stop() tells the OS "we handled it, no default behaviour."
    ctx, stop := signal.NotifyContext(
        context.Background(),
        syscall.SIGTERM, syscall.SIGINT,
    )
    defer stop()

    jobs := make(chan *ScheduledJob, 500)
    var wg sync.WaitGroup

    // 2. Start N worker goroutines in the background.
    for i := 0; i < workerCount; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for job := range jobs {
                o.dispatch(ctx, job)
            }
        }()
    }

    // 3. Run the tick loop until SIGTERM.
    o.run(ctx)   // blocks here until ctx is cancelled

    // 4. Stop new jobs: closing the channel signals all workers
    //    to finish their current job and then exit the range loop.
    close(jobs)

    // 5. Drain with a 5-minute deadline.
    //    IMPORTANT: use a fresh context — the original ctx is already cancelled.
    drainCtx, cancel := context.WithTimeout(context.Background(), 5*time.Minute)
    defer cancel()

    done := make(chan struct{})
    go func() {
        wg.Wait()
        close(done)
    }()

    select {
    case <-done:
        log.Println("all jobs completed — clean exit")
    case <-drainCtx.Done():
        log.Println("drain timeout — some jobs still running, forcing exit")
    }
}
KEY INSIGHT — FRESH CONTEXT FOR DRAIN

When SIGTERM arrives, ctx is cancelled. You cannot use it to create the drain timeout — context.WithTimeout(ctx, 5*time.Minute) would immediately return a context that is already done. Always create the drain context from context.Background(), a context that is never cancelled.

The Complete Orchestrator in One View

Putting it all together, here is the Orchestrator struct and its method set. This is the skeleton; Lesson 6 fills in the Execute method on each executor. 

type Orchestrator struct {
    cfg       Config
    generator *PolicyGenerator
    store     ExecutionStore    // interface — records job state
    executors map[EngineType]Executor  // SQLServer or PostgreSQL
    sems      *serverSems         // per-server semaphores
    jobs      chan *ScheduledJob   // the main work queue
}

// Executor interface — each engine implements this one method.
type Executor interface {
    Execute(ctx context.Context, job *ScheduledJob) (Result, error)
}

// ExecutionStore interface — abstracts Redis / SQLite / Postgres backing store.
type ExecutionStore interface {
    HasConflict(dbID string, op OperationType) bool
    MarkRunning(job *ScheduledJob)
    RecordResult(job *ScheduledJob, r Result, err error)
}

// executor() routes to the correct engine executor.
func (o *Orchestrator) executor(engine EngineType) Executor {
    if ex, ok := o.executors[engine]; ok {
        return ex
    }
    panic("unknown engine: " + string(engine))
}
