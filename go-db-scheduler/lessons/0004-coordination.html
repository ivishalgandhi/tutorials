Coordination — Exactly-Once Execution

Running 50 scheduler nodes in Kubernetes sounds robust — until two of them both decide to execute the same backup job simultaneously, and you discover that two processes writing to the same backup file have corrupted it. The coordination layer is what transforms a distributed group of nodes into a single logical scheduler. It has three components: leader election (deciding which node dispatches jobs), a distributed lock (defense-in-depth for split-brain scenarios), and an execution store (shared memory of what has run and what is currently running).

The Exactly-Once Problem

In a distributed system, exactly-once execution is hard. The naive approach — "only the leader dispatches jobs" — is fragile: a network partition can leave two nodes both believing they are leader (a split-brain scenario). The diagram below shows a split-brain event and how the distributed lock acts as a second line of defence:

sequenceDiagram participant N1 as Node 1 (was leader) participant R as Redis participant N2 as Node 2 (new leader) Note over N1,R: Network partition — N1 can't reach Redis Note over N1: N1 still thinks it's leader (stale) R->>N2: Lease expired — you are the new leader N2->>R: SET maint:job-abc:01:00 NX — acquired lock N2->>N2: Executing full_backup on Northwind N1->>R: SET maint:job-abc:01:00 NX — ALREADY HELD Note over N1: Lock acquisition fails — job skipped on N1 Note over N1,N2: Exactly-once preserved despite split-brain
Figure 4.1 — Split-brain scenario. Node 1 loses connectivity to Redis and can no longer renew its leader lease. Node 2 acquires leadership and the job lock. When Node 1's network recovers, its attempt to acquire the same lock fails — the job runs exactly once.
KEY INSIGHT — DEFENSE IN DEPTH

Leader election alone is insufficient — it relies on the leader stepping down gracefully when connectivity is lost. The distributed lock is defense-in-depth: even if a node incorrectly believes it is leader, it cannot execute a job that someone else is already running. You need both layers. Leader election handles the common case; the distributed lock handles the failure case.

Redis Leader Election — SET NX with Heartbeat

The Redis leader election implementation uses a single key with a short TTL. The node that successfully sets the key owns leadership. It renews (heartbeats) the TTL every 5 seconds. If it stops heartbeating — because the process died or lost network access — the TTL expires and another node takes over: 

// redisLeader holds a Redis key to signal leadership.
// TTL is 15s; heartbeat renews every 5s.
type redisLeader struct {
    client    *redis.Client
    key       string       // e.g. "dbmaint:leader"
    nodeID    string       // unique per instance (hostname or UUID)
    isLeader  atomic.Bool
    callbacks []func(bool)
    mu        sync.Mutex
}

const leaderTTL          = 15 * time.Second
const heartbeatInterval  = 5 * time.Second

func (l *redisLeader) Start(ctx context.Context) error {
    ticker := time.NewTicker(heartbeatInterval)
    defer ticker.Stop()
    for {
        select {
        case <-ctx.Done():
            l.stepDown(ctx)
            return nil
        case <-ticker.C:
            l.tryAcquireOrRenew(ctx)
        }
    }
}

func (l *redisLeader) tryAcquireOrRenew(ctx context.Context) {
    // Try to SET key to our nodeID with NX (only if not exists) and TTL
    set, err := l.client.SetNX(ctx, l.key, l.nodeID, leaderTTL).Result()
    if err != nil {
        // Can't reach Redis — step down; TTL will expire naturally
        if l.isLeader.Load() {
            l.setLeader(false)
        }
        return
    }
    if set {
        // We won the election
        l.setLeader(true)
        return
    }

    // Key exists — check if we already hold it
    owner, err := l.client.Get(ctx, l.key).Result()
    if err != nil || owner != l.nodeID {
        l.setLeader(false)
        return
    }

    // We hold it — renew the TTL
    l.client.Expire(ctx, l.key, leaderTTL)
    l.setLeader(true)
}
GOTCHA — TTL AND HEARTBEAT SIZING

The 15-second TTL means a leader that loses connectivity to Redis keeps dispatching for up to 15 seconds before another node takes over. The heartbeat interval (5 seconds) must be significantly shorter than the TTL to ensure the key is renewed well before expiry — a good rule of thumb is TTL ≥ 3× heartbeat interval. A shorter TTL means faster failover but more Redis load and more sensitivity to transient latency spikes. A longer TTL increases the split-brain window. Tune these for your network's P99 latency to Redis.

Redis Distributed Lock — Lua Scripts for Safety

A simple SET NX + DEL sequence is unsafe as a distributed lock. Consider: node A sets the key and starts a long job. The lock's TTL expires before the job finishes. Node B sets the same key. Node A's job finishes and calls DEL key — it has just deleted node B's lock. A third node C then acquires the lock and runs the job concurrently with node B. The Lua script approach prevents this by making the check-and-delete atomic: 

// Acquire sets the lock key to a random token with NX + TTL.
// Returns true on success; the token is stored for safe release.
func (l *redisLock) Acquire(ctx context.Context, key string, ttl time.Duration) (bool, error) {
    token := generateToken() // 128-bit random value
    ok, err := l.client.SetNX(ctx, l.prefix+key, token, ttl).Result()
    if err != nil {
        return false, err
    }
    if ok {
        l.mu.Lock()
        l.tokens[key] = token
        l.mu.Unlock()
    }
    return ok, nil
}

// releaseLua atomically checks our token then deletes — safe against
// a lock that expired and was re-acquired by someone else.
var releaseLua = redis.NewScript(`
  if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
  else
    return 0
  end
`)

func (l *redisLock) Release(ctx context.Context, key string) error {
    l.mu.Lock()
    token := l.tokens[key]
    delete(l.tokens, key)
    l.mu.Unlock()

    if token == "" {
        return fmt.Errorf("no token held for key %q", key)
    }
    _, err := releaseLua.Run(ctx, l.client, []string{l.prefix + key}, token).Int()
    return err
}

Long-running jobs also need to extend their lock TTL to prevent expiry mid-execution. The extend script follows the same atomic check-then-act pattern: 

// extendLua atomically checks our token then refreshes the TTL.
var extendLua = redis.NewScript(`
  if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("PEXPIRE", KEYS[1], ARGV[2])
  else
    return 0
  end
`)

A goroutine spawned alongside the job executor calls extendLua at ttl / 3 intervals. If the job goroutine exits (success, failure, or panic), the extension goroutine is cancelled via context, and the lock expires naturally or is explicitly released.

CRITICAL — NEVER USE GET + DEL WITHOUT LUA

Never use GET + DEL as two separate Redis commands without the Lua script. Between the GET check and the DEL, another goroutine or another process can expire the lock and a third party can acquire it. Your DEL would then release someone else's lock, breaking exactly-once. The Lua script runs atomically on the Redis server — nothing can interleave between the GET and the DEL inside the script.

Kubernetes Lease — Production Leader Election

For Kubernetes-native deployments, the coordination.k8s.io/v1 Lease object is a better choice than a Redis key: it uses the K8s API server (already present and highly available), integrates with RBAC, and is the same mechanism used by the Kubernetes controller manager itself. The client-go leader-election library wraps this behind a clean callback interface: 

// k8sLeader uses the Kubernetes coordination.k8s.io/v1 Lease object
// as a distributed lock for leader election.
func NewK8sLeader(client kubernetes.Interface, namespace, name, nodeID string) *k8sLeader {
    return &k8sLeader{
        client:    client,
        namespace: namespace,
        name:      name,
        nodeID:    nodeID,
    }
}

func (l *k8sLeader) Start(ctx context.Context) error {
    le, err := leaderelection.NewLeaderElector(leaderelection.LeaderElectionConfig{
        Lock: &resourcelock.LeaseLock{
            LeaseMeta: metav1.ObjectMeta{
                Name:      l.name,
                Namespace: l.namespace,
            },
            Client: l.client.CoordinationV1(),
            LockConfig: resourcelock.ResourceLockConfig{
                Identity: l.nodeID,
            },
        },
        LeaseDuration:   15 * time.Second,
        RenewDeadline:   10 * time.Second,
        RetryPeriod:     2 * time.Second,
        ReleaseOnCancel: true,
        Callbacks: leaderelection.LeaderCallbacks{
            OnStartedLeading: func(ctx context.Context) {
                l.setLeader(true)
                <-ctx.Done()
            },
            OnStoppedLeading: func() { l.setLeader(false) },
            OnNewLeader:      func(identity string) { l.leaderID.Store(&identity) },
        },
    })
    if err != nil {
        return err
    }
    le.Run(ctx)
    return nil
}

The three durations in LeaderElectionConfig are worth understanding precisely:

Field Value Meaning
LeaseDuration 15s How long a candidate waits before stealing an expired lease
RenewDeadline 10s How long the leader tries to renew before giving up and stepping down
RetryPeriod 2s How often a non-leader polls the API server to check for an available lease

A comparison of the two election backends helps choose the right one for your environment:

Aspect Redis Kubernetes Lease
Infrastructure required Redis server (additional dependency) K8s API server (already present in K8s deployments)
Failure detection mechanism TTL-based key expiry TTL-based Lease object expiry
Best fit Non-K8s deployments (bare metal, VMs) K8s-native deployments

Single-Node Fallback — Development and Small Deployments

Not every deployment needs distributed coordination. A single developer running the scheduler locally, or a small deployment on one server, needs no leader election infrastructure. The singleNodeLeader satisfies the LeaderElector interface without touching any external system: 

// singleNodeLeader is always the leader. Used for development,
// single-server deployments, and testing. No infrastructure required.
type singleNodeLeader struct {
    nodeID string
}

func (l *singleNodeLeader) Start(ctx context.Context) error {
    <-ctx.Done()
    return nil
}
func (l *singleNodeLeader) IsLeader() bool               { return true }
func (l *singleNodeLeader) LeaderID() string             { return l.nodeID }
func (l *singleNodeLeader) OnLeadershipChange(func(bool)) {}
ENVIRONMENT SWITCHING

For local development and testing, single_node mode avoids Redis and Kubernetes entirely. The scheduler still runs correctly — it just doesn't coordinate with other nodes. Switch to redis or kubernetes coordination by changing one config line: coordination.mode: redis. The core scheduling logic never checks the coordination mode directly — it only calls leader.IsLeader(), which is the same method on all three implementations.

The Execution Store — Shared Job Memory

The execution store is the cluster's shared memory of what jobs have run, are running, and have failed. Every scheduler node reads from and writes to it. The interface is small and focused: 

type ExecutionStore interface {
    RecordStart(ctx context.Context, record ExecutionRecord) error
    RecordComplete(ctx context.Context, id string, result ExecutionResult) error
    GetHistory(ctx context.Context, jobID string, limit int) ([]ExecutionRecord, error)
    GetLastExecution(ctx context.Context, jobID string) (*ExecutionRecord, error)
    GetRunning(ctx context.Context) ([]ExecutionRecord, error)
    GetFailed(ctx context.Context, since time.Time, limit int) ([]ExecutionRecord, error)
    Cleanup(ctx context.Context, olderThan time.Duration) (int64, error)
}

The Redis implementation uses a hash per record plus two sorted sets for indexing. All three writes happen in a single pipeline to keep them atomic: 

// RecordStart stores an execution record in Redis.
// Key:   exec:RECORD_ID → hash of all fields
// Index: execidx:job:JOB_ID → sorted set scored by start timestamp
// TTL:   30 days
func (s *redisStore) RecordStart(ctx context.Context, rec ExecutionRecord) error {
    pipe := s.client.Pipeline()

    key := "exec:" + rec.ID
    pipe.HSet(ctx, key, encodeRecord(rec))
    pipe.Expire(ctx, key, 30*24*time.Hour)

    // Index by job ID so GetHistory is fast
    pipe.ZAdd(ctx, "execidx:job:"+rec.JobID, redis.Z{
        Score:  float64(rec.StartedAt.Unix()),
        Member: rec.ID,
    })
    // Index by status so GetRunning and GetFailed are fast
    pipe.ZAdd(ctx, "execidx:status:"+string(rec.Status), redis.Z{
        Score:  float64(rec.StartedAt.Unix()),
        Member: rec.ID,
    })

    _, err := pipe.Exec(ctx)
    return err
}

The status index is what makes the GetRunning and GetFailed queries efficient. They are ZRANGEBYSCORE queries on execidx:status:running and execidx:status:failed respectively — O(log N + M) where M is the result count, regardless of total history size. When a record transitions from running to completed or failed, the orchestrator calls RecordComplete, which moves the member from the running sorted set to the completed/failed one.

STALE JOB DETECTION

GetRunning is used by the orchestrator to detect stale jobs — those that appear as StatusRunning in the store but whose executor goroutine has crashed without recording a result. The executor heartbeats its record's UpdatedAt field every 30 seconds. Any record with UpdatedAt older than StaleJobTimeout (default 5 minutes) is considered stale, moved to StatusFailed, and the lock is released so the job can be retried on the next tick.

Lock Key Design — Preventing Re-Run at the Same Minute

The orchestrator's scheduling loop ticks every 30 seconds (configurable). A job due at 1:47 AM would be seen on both the 1:47:00 tick and the 1:47:30 tick. The lock key includes a time bucket that is the same for both ticks, ensuring only one dispatch succeeds: 

// lockKey returns a lock key scoped to this job's scheduled minute.
// This prevents re-dispatching the same job multiple times within one minute
// while allowing it to fire again in the next scheduled window.
func lockKey(jobID string, scheduledAt time.Time) string {
    bucket := scheduledAt.Truncate(time.Minute).Format("2006-01-02T15:04")
    return fmt.Sprintf("maint:%s:%s", jobID, bucket)
}
// Example: "maint:a3b4c5d6e7f8a9b0:2024-01-15T01:47"

Concretely: 1:47:00.Truncate(Minute) = 1:47:00 and 1:47:30.Truncate(Minute) = 1:47:00. Both ticks produce the same lock key. The first tick's SetNX succeeds; the second tick's fails. The job runs once.

The lock TTL for the job-dispatch lock is set to the job's configured timeout plus a buffer (typically 10 minutes). This ensures the lock is released even if the executor fails without calling Release — but not so quickly that a legitimately running long job loses its lock to a competing node.

KEY INSIGHT — THE TIME BUCKET

The time bucket is what prevents the same job firing twice when the orchestrator ticks every 30 seconds. If the scheduler checks at 1:47:00 and again at 1:47:30, both ticks see the same lock key "maint:JOBID:2024-01-15T01:47". The second tick's SetNX fails because the first tick holds the key — single execution guaranteed. The next day at 1:47 AM, the bucket is "...2024-01-16T01:47" — a completely new key — so the job fires again as intended.
