Policy Engine and Staggered Scheduling

A policy is a template that answers three questions: what operation should run, against which databases should it target, and according to what schedule? The policy engine sits between the static policy definitions and the live database inventory, generating a concrete job for every (policy, database) pair that matches. This lesson covers how policies are structured, how targets are filtered, how timeouts scale with database size, and how the stagger algorithm prevents 10,000 simultaneous backup jobs at 1:00 AM.

The MaintenancePolicy Type

Every scheduled operation is governed by a MaintenancePolicy. A single policy declaration covers an entire class of databases rather than naming each one individually. The two root structs are MaintenancePolicy and SchedulePolicy: 

type MaintenancePolicy struct {
    Name         string             `yaml:"name"`
    Description  string             `yaml:"description"`
    Operation    OperationType      `yaml:"operation"`
    Schedule     SchedulePolicy     `yaml:"schedule"`
    Targets      TargetSelector     `yaml:"targets"`
    Config       map[string]any     `yaml:"config"`
    Priority     int                `yaml:"priority"` // 1 = highest
    Timeout      TimeoutPolicy      `yaml:"timeout"`
    Retry        RetryPolicy        `yaml:"retry"`
    Notification NotificationPolicy `yaml:"notification"`
    Enabled      bool               `yaml:"enabled"`
}

type SchedulePolicy struct {
    CronExpr              string        `yaml:"cron_expr"`
    RequireWindow         bool          `yaml:"require_window"`
    StaggerMinutes        int           `yaml:"stagger_minutes"`
    SkipIfRunRecently     time.Duration `yaml:"skip_if_run_recently"`
    RunMissedOnWindowOpen bool          `yaml:"run_missed_on_window_open"`
}

Each field has a concrete purpose. Consider a full-backup policy for production SQL Server databases:

Field Example value Why it matters
Operation full_backup Selects the correct executor and conflict set at dispatch time
CronExpr "0 1 * * *" Base fire time before stagger offsets are applied
RequireWindow true Full backups only run inside the defined maintenance window
StaggerMinutes 120 Each database gets a unique offset hashed into [0, 120) minutes
SkipIfRunRecently 20h Prevents a second run if one succeeded within the last 20 hours
RunMissedOnWindowOpen true If the window opens and a scheduled run was missed, catch it up
Priority 1 Priority 1 (highest) preempts lower-priority jobs when the worker pool is full
Enabled true A fast circuit-breaker — set to false to pause without deleting the policy
READING THE CRON FIELD

CronExpr is a standard five-field cron expression (minute hour dom month dow). The stagger algorithm in StaggerByHash treats this as the base expression and produces a per-database offset. The base expression itself is never registered with the cron library directly — only the staggered derivatives are.

The TargetSelector — Filtering the Fleet

The TargetSelector is the policy's filter predicate. It encodes which servers and databases a policy applies to, using a combination of structural attributes and arbitrary tags: 

type TargetSelector struct {
    Engines          []EngineType        `yaml:"engines"`
    Tiers            []ServerTier        `yaml:"tiers"`
    Regions          []string            `yaml:"regions"`
    Tags             map[string]string  `yaml:"tags"`
    MinSizeBytes     int64              `yaml:"min_size_bytes"`
    MaxSizeBytes     int64              `yaml:"max_size_bytes"`
    ExcludeDBs       []string           `yaml:"exclude_dbs"`
    ExcludeServers   []string           `yaml:"exclude_servers"`
    IncludeSystemDBs bool               `yaml:"include_system_dbs"`
    RecoveryModels   []string           `yaml:"recovery_models"`
}

The Matches method is a pure function that applies all filters in sequence. The first failing check returns false immediately — short-circuit evaluation keeps it fast across millions of evaluations: 

func (ts TargetSelector) Matches(server *DBServer, db *Database) bool {
    // Engine filter
    if len(ts.Engines) > 0 && !contains(ts.Engines, server.Engine) {
        return false
    }
    // Tier filter
    if len(ts.Tiers) > 0 && !contains(ts.Tiers, server.Tier) {
        return false
    }
    // System DB exclusion
    if !ts.IncludeSystemDBs && db.IsSystemDB() {
        return false
    }
    // Recovery model filter (SQL Server log backups)
    if len(ts.RecoveryModels) > 0 && !contains(ts.RecoveryModels, db.RecoveryModel) {
        return false
    }
    // Size filter
    if ts.MinSizeBytes > 0 && db.SizeBytes < ts.MinSizeBytes {
        return false
    }
    if ts.MaxSizeBytes > 0 && db.SizeBytes > ts.MaxSizeBytes {
        return false
    }
    // Tag matching (all specified tags must be present)
    for k, v := range ts.Tags {
        if server.Tags[k] != v {
            return false
        }
    }
    // Explicit exclusions
    if contains(ts.ExcludeDBs, db.Name) {
        return false
    }
    return true
}
TAG MATCHING SEMANTICS

Tag matching uses AND logic: all specified tags must match. This lets you target subsets like {critical: true, environment: prod} without adding new tier values. A database is excluded if it matches any entry in ExcludeDBs. Tags are checked against the server's tag map, not the database's — servers carry environment and criticality metadata in this model.

Dynamic Timeout Calculation

A 30-minute timeout that is safe for a 10 GB database will kill a 500 GB database every time. A static timeout per policy cannot work across a fleet with 5-order-of-magnitude size variance. The TimeoutPolicy solves this with a linear formula capped at a maximum: 

type TimeoutPolicy struct {
    BaseTimeout  time.Duration `yaml:"base_timeout"`
    PerGBTimeout time.Duration `yaml:"per_gb_timeout"`
    MaxTimeout   time.Duration `yaml:"max_timeout"`
}

func (tp TimeoutPolicy) CalculateTimeout(dbSizeBytes int64) time.Duration {
    sizeGB := float64(dbSizeBytes) / (1024 * 1024 * 1024)
    timeout := tp.BaseTimeout + time.Duration(float64(tp.PerGBTimeout)*sizeGB)
    if tp.MaxTimeout > 0 && timeout > tp.MaxTimeout {
        return tp.MaxTimeout
    }
    return timeout
}

The following table works through the formula for a policy with base_timeout=10m, per_gb_timeout=3m, max_timeout=8h:

Database Size Base Per-GB contribution Calculated total After 8h cap
100 MB (0.1 GB) 10m 0.1 × 3m = 0.3m 10m 18s 10m 18s
10 GB 10m 10 × 3m = 30m 40m 40m
100 GB 10m 100 × 3m = 300m 5h 10m 5h 10m
1 TB (1,024 GB) 10m 1024 × 3m = 3,072m 51h 22m 8h (capped)
5 TB 10m 5,120 × 3m = 15,360m 256h 10m 8h (capped)
KEY INSIGHT

Dynamic timeouts are what make a single policy safe for databases ranging from 10 MB to 10 TB. Without per-GB scaling, a 30-minute timeout kills a 5 TB backup every time. Without a maximum cap, a misconfigured per-GB rate could set 50-hour timeouts that hold worker-pool slots indefinitely and mask genuinely stuck jobs. The cap is a safety valve — a job that hits it should alert.

The Thundering Herd Problem

Consider a fleet of 1,000 databases all governed by a full-backup policy with cron_expr: "0 1 * * *". At 01:00:00 the cron library fires. The policy generator evaluates all 1,000 databases as matching. All 1,000 jobs enter the worker-pool queue in the same scheduler tick.

The result is a thundering herd: backup storage receives 1,000 simultaneous write streams, SAN I/O queues saturate, network bandwidth on the backup path spikes to the physical limit, and the databases themselves see elevated I/O latency that leaks into production query performance — all at the same instant, every night. The stagger window does not increase total backup time per database — each database still runs its full backup. It staggers the start times so the fleet's aggregate I/O demand is smooth rather than spiked.

GOTCHA — STAGGER WINDOW VS. MAINTENANCE WINDOW

The stagger window must fit inside the maintenance window. A 120-minute stagger on a 90-minute maintenance window means databases hashed into [90, 119] minutes fire outside the window and are skipped. Always set stagger_minutes ≤ window duration − max expected job runtime.

The StaggerByHash Algorithm

The stagger algorithm must satisfy two properties: (1) each database gets a stable offset so the scheduler doesn't reschedule every database every time a new one is added, and (2) offsets are uniformly distributed across [0, staggerMinutes) so no sub-window is overloaded. FNV-32a hash of the database key achieves both: 

// StaggerByHash assigns a consistent cron offset to a database based
// on a hash of its key. The same key always gets the same offset,
// even as the total number of databases changes.
func StaggerByHash(baseCron string, staggerMinutes int, hashKey string) (string, error) {
    h := fnv.New32a()
    h.Write([]byte(hashKey))
    hash := h.Sum32()

    // Map hash to [0, staggerMinutes)
    offset := int(hash % uint32(staggerMinutes))

    return applyMinuteOffset(baseCron, offset)
}

// applyMinuteOffset shifts the minute field of a cron expression by offset minutes,
// handling hour overflow correctly.
func applyMinuteOffset(cronExpr string, offset int) (string, error) {
    parts := strings.Fields(cronExpr)
    if len(parts) != 5 {
        return "", fmt.Errorf("expected 5-field cron, got %d fields", len(parts))
    }

    baseMin, err := strconv.Atoi(parts[0])
    if err != nil {
        return "", fmt.Errorf("minute field %q is not a simple integer: %w", parts[0], err)
    }

    newMin  := (baseMin + offset) % 60
    overflow := (baseMin + offset) / 60

    parts[0] = strconv.Itoa(newMin)

    if overflow > 0 && parts[1] != "*" {
        baseHour, err := strconv.Atoi(parts[1])
        if err == nil {
            parts[1] = strconv.Itoa((baseHour + overflow) % 24)
        }
    }

    return strings.Join(parts, " "), nil
}

Worked example

Input: baseCron="0 1 * * *", staggerMinutes=120, hashKey="prod-db-server-01:Northwind"

  1. FNV-32a hash of "prod-db-server-01:Northwind" → some 32-bit integer, e.g. 3,827,045,193
  2. offset = 3,827,045,193 % 120 = 73
  3. applyMinuteOffset("0 1 * * *", 73): base minute 0 + 73 = 73; 73 < 60 is false, so newMin = 73 % 60 = 13, overflow = 1
  4. Hour field: 1 + 1 = 2
  5. Result: "13 2 * * *" → the Northwind database runs its backup at 2:13 AM
KEY INSIGHT — STABILITY

The hash is deterministic: the same database always gets the same slot regardless of what other databases exist in the fleet. This stability means the scheduler does not thrash when databases are added or removed. A new database gets its own consistent slot without shifting all the others. This is the crucial difference from round-robin or modular-index approaches, which renumber every slot when the total count changes.

Job Generation — GenerateAll

The output of the policy engine is a GeneratedJob per matching (policy, database) pair. All fields needed to dispatch and track the job are pre-computed here, so the orchestrator receives a self-contained value: 

type GeneratedJob struct {
    ID            string          // SHA256 prefix of "policyName:serverID:dbName"
    PolicyName    string
    Operation     OperationType
    Server        *DBServer
    Database      *Database
    CronExpr      string          // staggered from policy base
    Timeout       time.Duration   // calculated from DB size
    Priority      int             // policy.Priority * db.Priority
    Config        map[string]any  // policy config + db-specific overrides
    MaxRetries    int
    RetryDelay    time.Duration
    BackoffMult   float64
    ConflictsWith []OperationType
    RequireWindow bool
    SkipIfRecent  time.Duration
}

The deterministic ID lets any node in the cluster derive the same distributed-lock key for the same job: 

func generateJobID(policyName, serverID, dbName string) string {
    key := fmt.Sprintf("%s:%s:%s", policyName, serverID, dbName)
    sum := sha256.Sum256([]byte(key))
    return hex.EncodeToString(sum[:8]) // first 16 hex chars
}

GenerateAll iterates the cross product of policies and servers, calling TargetSelector.Matches on each database, then constructs the GeneratedJob:

  1. For each enabled policy in the policy store
  2. Query inventory for all servers and their databases
  3. For each server × database pair: call policy.Targets.Matches(server, db)
  4. On match: generate deterministic ID, call StaggerByHash for the cron expression, call CalculateTimeout with db.SizeBytes, merge policy.Config with any database-level config overrides
  5. Append to results slice
  6. Return slice sorted by (priority ASC, jobID ASC) for deterministic dispatch order
PERFORMANCE NOTE

Running GenerateAll against 1,000 servers × 50 databases × 20 policies = 1,000,000 (policy, database) evaluations. Most pairs don't match — TargetSelector quickly eliminates non-matching engines, tiers, and sizes in its early-exit checks. In practice this runs in under 200ms for 50,000 databases when inventory is cached in memory. The result is recomputed on each inventory refresh cycle, not on every tick.

A Real Policy Example — Production Full Backup

The following YAML is a complete, production-grade policy definition for full backups of production SQL Server databases. Reading it field-by-field illustrates how all the preceding concepts compose: 

name: sqlserver-full-backup-prod
description: Weekly full backup for all production SQL Server databases
operation: full_backup

schedule:
  cron_expr: "0 1 * * *"
  require_window: true
  stagger_minutes: 120
  skip_if_run_recently: 20h

targets:
  engines: [sqlserver]
  tiers: [production]
  exclude_dbs: [tempdb]

config:
  compression: true
  checksum: true
  verify: true
  backup_path: "/backups/sqlserver/{server}/{database}/"
  retention_days: 30

priority: 1

timeout:
  base_timeout: 10m
  per_gb_timeout: 3m
  max_timeout: 8h

retry:
  max_retries: 2
  initial_delay: 5m
  backoff_multiplier: 2.0

notification:
  on_failure: true
  on_timeout: true
  channels: [slack, email]

enabled: true

Walking through the decisions:

  • schedule.cron_expr "0 1 * * *" — base fire time at 1:00 AM; every database gets a staggered derivative of this.
  • require_window: true — full backups are heavy; they must not run outside the maintenance window even if a retry or catch-up fires at an unexpected time.
  • stagger_minutes: 120 — 2-hour spread. A fleet of 2,000 production databases dispatches roughly 16–17 databases per minute rather than 2,000 at once.
  • skip_if_run_recently: 20h — guards against double-execution on manual triggers or aggressive retry behaviour while still allowing a daily run every ~24 hours.
  • exclude_dbs: [tempdb] — tempdb cannot be backed up in SQL Server; including it would cause every job to fail.
  • config.checksum + verify — enables WITH CHECKSUM and a RESTORE VERIFYONLY pass post-backup. Slightly slower but detects corrupt backup files before they're needed in a disaster recovery scenario.
  • priority: 1 — highest priority. When the worker pool is saturated, full backup jobs preempt lower-priority index maintenance or statistics updates.
  • retry.backoff_multiplier: 2.0 — exponential backoff. First retry after 5 minutes, second after 10 minutes. Avoids hammering a server that is temporarily under heavy load.
