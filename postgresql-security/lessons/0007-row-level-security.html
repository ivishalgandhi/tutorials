Row-Level Security

Filtering Rows by Who Is Asking

1. What Row-Level Security Does

Standard GRANT/REVOKE controls whether a role can access a table at all. Row-Level Security (RLS) goes one layer deeper: it controls which rows within that table a role can see or modify. The filter is attached to the table itself and runs invisibly before any query result is returned — the user never sees the rows they are not permitted to see, and they cannot tell whether those rows exist.

Think of it this way: GRANT SELECT is the doorman checking if you can enter the building. RLS is a partition inside the building that silently routes you to only your floor. You can walk freely on your floor; the other floors simply do not appear to exist.

SQL Server Mapping

SQL Server has Row-Level Security (introduced in SQL Server 2016) using inline table-valued security predicate functions. You create a function that returns a filter expression, then bind it to a table using CREATE SECURITY POLICY with a FILTER PREDICATE (for SELECT) and an optional BLOCK PREDICATE (for INSERT/UPDATE/DELETE).

PostgreSQL's RLS is structurally similar but the implementation is different in three important ways: (1) the policy is attached directly to the table with CREATE POLICY, not via a separate security policy object; (2) the filter expression is inline in the policy, not in a separate function; (3) PostgreSQL's default when RLS is enabled but no policy exists is deny all rows — SQL Server defaults to allow all rows. That last difference has caused production outages.

2. Enabling RLS — The Deny-By-Default Behavior

-- Enable RLS on the orders table
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

-- At this point: superusers still see all rows
-- All other roles: ZERO rows visible, even with SELECT privilege on the table
Critical Difference from SQL Server

In SQL Server, enabling Row-Level Security and adding a FILTER PREDICATE is additive — all rows are visible until a predicate narrows them. In PostgreSQL, ALTER TABLE orders ENABLE ROW LEVEL SECURITY immediately hides all rows from non-superuser roles. There are zero visible rows until you create a policy that explicitly allows access. If you enable RLS on a production table before creating a policy, your application immediately returns empty result sets. Do not enable RLS on a live table without having the policies ready to deploy in the same transaction.

-- Safe deployment: wrap in a transaction
BEGIN;
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON orders
  AS PERMISSIVE FOR ALL TO app_role
  USING (tenant_id = current_setting('app.current_tenant', true)::int)
  WITH CHECK (tenant_id = current_setting('app.current_tenant', true)::int);
COMMIT;

3. CREATE POLICY Syntax — Every Clause Explained

CREATE POLICY policy_name
  ON table_name
  [AS PERMISSIVE | RESTRICTIVE]          -- default: PERMISSIVE
  [FOR ALL | SELECT | INSERT | UPDATE | DELETE]  -- default: ALL
  [TO role_name]                             -- default: PUBLIC (all roles)
  USING (filter_expression)                 -- applied to existing rows
  [WITH CHECK (check_expression)];          -- applied to rows being written

4. USING vs WITH CHECK — Reading Rows vs Writing Rows

This is the most conceptually important distinction in PostgreSQL RLS. The two clauses guard different operations:

  • USING is a filter on rows that already exist in the table. When a user runs SELECT, UPDATE, or DELETE, the USING expression is evaluated against each candidate row. Rows where the expression returns false are invisible — they do not appear in SELECT results, and UPDATE/DELETE silently skip them.
  • WITH CHECK is a constraint on rows the user is trying to write. It is evaluated against the new row values in INSERT and UPDATE operations. If the expression returns false, the write is rejected with an error.
-- Example: tenant user can only read their own rows AND only write rows for their tenant
CREATE POLICY tenant_isolation ON orders
  FOR ALL TO app_role
  USING      (tenant_id = current_setting('app.current_tenant', true)::int)
  WITH CHECK (tenant_id = current_setting('app.current_tenant', true)::int);

Consider what happens without WITH CHECK but with only USING:

  • Tenant 42 cannot see tenant 99's orders — the USING filter hides them.
  • But tenant 42 can insert a row with tenant_id = 99 — the insert is not filtered because USING only applies to existing rows.
  • Tenant 42 now has a row in the database attributed to tenant 99 that they cannot read back.
Common Error

If you specify only USING on a policy that covers FOR ALL or FOR INSERT, PostgreSQL uses the USING expression as the WITH CHECK expression automatically — but only for UPDATE, not for INSERT. For INSERT, if there is no WITH CHECK, the insert is not constrained by the policy. Always specify both clauses explicitly for multi-tenant insert protection.

5. Real-World Multi-Tenant Example

The canonical use case: a SaaS application where each tenant should see only their own data, even though all tenants share the same table.

-- Schema
CREATE TABLE app.orders (
  id         serial  PRIMARY KEY,
  tenant_id  int     NOT NULL,
  product    text    NOT NULL,
  quantity   int     NOT NULL,
  created_at timestamptz DEFAULT now()
);

-- Enable RLS (no rows visible until policy is created)
ALTER TABLE app.orders ENABLE ROW LEVEL SECURITY;

-- Policy: each tenant sees and writes only their own rows
CREATE POLICY tenant_isolation ON app.orders
  AS PERMISSIVE
  FOR ALL
  TO app_role
  USING      (tenant_id = current_setting('app.current_tenant', true)::int)
  WITH CHECK (tenant_id = current_setting('app.current_tenant', true)::int);

The application sets the tenant context at the start of each connection or transaction:

-- At session/transaction start, before any query:
SET app.current_tenant = '42';

-- Now all queries against orders automatically filter to tenant 42:
SELECT * FROM app.orders;         -- returns only tenant 42 rows
INSERT INTO app.orders (tenant_id, product, quantity)
  VALUES (42, 'widget', 5);          -- allowed
INSERT INTO app.orders (tenant_id, product, quantity)
  VALUES (99, 'widget', 5);          -- ERROR: fails WITH CHECK
current_setting second argument

current_setting('app.current_tenant', true) — the second argument true means "return NULL if the setting is not set" rather than raising an error. This is important for maintenance sessions (e.g., DBA queries, pg_dump) that do not set the tenant context. Without true, any query by a role without that setting would throw a runtime error rather than safely returning no rows.

6. The BYPASSRLS Attribute

Some roles legitimately need to see all rows regardless of RLS policies — backup tools, cross-tenant reporting roles, DBA maintenance scripts.

-- Superusers bypass RLS by default (no change needed)

-- Grant BYPASSRLS to a non-superuser role that needs all-row access
ALTER ROLE reporting_admin BYPASSRLS;

-- Revoke it when no longer needed
ALTER ROLE reporting_admin NOBYPASSRLS;

-- Check which roles have BYPASSRLS:
SELECT rolname, rolbypassrls
FROM   pg_roles
WHERE  rolbypassrls = true;
Security Note

BYPASSRLS is equivalent to superuser access for data visibility. A role with BYPASSRLS can read every row in every table that has RLS enabled, across all tenants. Treat it with the same caution as SUPERUSER. Audit it regularly. Do not grant it to service accounts or interactive user logins.

7. PERMISSIVE vs RESTRICTIVE Policies

When multiple policies apply to the same table and role, PostgreSQL combines them using logic that depends on the policy type:

graph TD Q["Query on protected table"] P["Pass any PERMISSIVE policy?"] R["Pass ALL RESTRICTIVE policies?"] A["Row is visible / write is allowed"] D1["Row is hidden / write is blocked\n(no permissive match)"] D2["Row is hidden / write is blocked\n(restrictive failed)"] Q --> P P -->|Yes| R P -->|No| D1 R -->|Yes| A R -->|No| D2

A row is accessible only when it passes at least one PERMISSIVE policy AND all RESTRICTIVE policies. Either failure alone blocks the row.

-- PERMISSIVE: tenant sees their own rows (OR logic across permissive policies)
CREATE POLICY tenant_access ON app.orders
  AS PERMISSIVE FOR ALL TO app_role
  USING (tenant_id = current_setting('app.current_tenant', true)::int);

-- RESTRICTIVE: never show soft-deleted rows, regardless of tenant (AND logic)
CREATE POLICY hide_deleted ON app.orders
  AS RESTRICTIVE FOR ALL TO app_role
  USING (deleted_at IS NULL);

-- Result: a row is visible only if tenant_id matches (permissive) AND deleted_at IS NULL (restrictive)
-- Even if there is a second permissive policy that grants wider access,
-- the RESTRICTIVE policy still blocks soft-deleted rows absolutely.

Use PERMISSIVE for the primary access rule (who owns this row). Use RESTRICTIVE for absolute constraints that must hold for all users regardless of other policies (e.g., never show deleted rows, never show rows from a suspended tenant).

9. Common Mistake 1 — Omitting WITH CHECK

Top Mistake — Coming from SQL Server

Writing a USING clause but no WITH CHECK clause for policies that cover INSERT. In SQL Server, the FILTER PREDICATE controls reads and the BLOCK PREDICATE controls writes — they are visually distinct in the DDL, making it obvious when you have only a read filter. In PostgreSQL, USING looks complete and self-sufficient. It is easy to think "I have a policy, inserts are covered."

They are not. A USING-only policy on a FOR ALL or FOR INSERT policy does not restrict INSERT. Tenant A can insert rows with tenant_id = tenant_B and those rows will silently be written to the database. Tenant A cannot read them back, but they are there, attributed to Tenant B.

Rule: for any multi-tenant table, always include both clauses:

USING      (tenant_id = current_setting('app.current_tenant', true)::int)
WITH CHECK (tenant_id = current_setting('app.current_tenant', true)::int)

10. Common Mistake 2 — Relying on RLS Without Revoking Direct Grants

Top Mistake — Coming from SQL Server

Treating RLS as a replacement for GRANT/REVOKE rather than a complement to it. RLS policies filter rows — they do not prevent a role from accessing the table at all. If a role has been granted SELECT on a table, RLS will filter which rows it sees. But if you forget to remove a broad grant from an earlier migration, the role will still reach the table (seeing only its filtered rows, or all rows if it has BYPASSRLS).

RLS and privilege grants work at different layers:

  • Privileges (GRANT/REVOKE): controls whether the role can touch the table
  • RLS policies: controls which rows the role can see or modify within that table

Both layers must be correct. A properly scoped RLS policy on a table that has been accidentally granted SELECT to PUBLIC is not truly secure — any authenticated role can see its permitted rows.
