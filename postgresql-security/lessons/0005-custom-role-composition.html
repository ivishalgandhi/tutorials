Custom Roles and Role Composition

Building a Role Hierarchy from Scratch

1. No Fixed Application Roles — You Build Them

In SQL Server, the server ships with fixed database roles (db_datareader, db_datawriter, db_owner, etc.) that you add users to. You can also create custom application roles using CREATE APPLICATION ROLE with a password, activated via sp_setapprole. The scaffolding is partly pre-built.

PostgreSQL ships with almost no fixed application roles. You build everything yourself from a single primitive: the role. A role is simultaneously the concept of a user, a group, and an application role. The login attribute determines whether it can authenticate; everything else is about what it can do.

SQL Server Mapping

SQL Server fixed roles (db_datareader, db_datawriter) have no PostgreSQL equivalents — you recreate their semantics yourself. CREATE APPLICATION ROLE maps loosely to a NOLOGIN role used as a permission group, but PostgreSQL roles are simpler: no activation password, no sp_setapprole ceremony. sp_addrolemember maps to GRANT role_name TO user_name.

The central pattern: create NOLOGIN roles as permission groups, then grant those groups to LOGIN roles (the actual users). The NOLOGIN role can never connect to a database directly — it exists only as a bundle of privileges.

2. The "Role as Group" Pattern

Here is the full lifecycle of the pattern, step by step:

-- Step 1: Create the permission group (cannot log in)
CREATE ROLE readonly_role NOLOGIN;

-- Step 2: Grant privileges to the group role
GRANT CONNECT ON DATABASE mydb TO readonly_role;
GRANT USAGE ON SCHEMA app TO readonly_role;
GRANT SELECT ON ALL TABLES IN SCHEMA app TO readonly_role;

-- Step 3: Create a real user (can log in)
CREATE ROLE alice WITH LOGIN PASSWORD 'change-me-in-vault';

-- Step 4: Add user to the group — alice inherits all SELECT on app
GRANT readonly_role TO alice;

After step 4, alice can connect and run SELECT on every table in the app schema. When you add a new table and re-run the GRANT SELECT ON ALL TABLES statement (or use ALTER DEFAULT PRIVILEGES — covered in Lesson 6), alice automatically gains access without any per-user grant.

Key Concept

GRANT SELECT ON ALL TABLES IN SCHEMA app TO readonly_role grants on all currently existing tables. New tables added later do not automatically inherit this grant. Use ALTER DEFAULT PRIVILEGES to cover future tables — this is one of the most-missed PostgreSQL behaviors for SQL Server DBAs.

3. Building a Reusable 3-Tier Hierarchy

Production schemas almost always need three permission tiers. The diagram shows how they stack:

graph BT admin_role["admin_role\n(DDL: CREATE TABLE, ALTER, DROP)"] readwrite_role["readwrite_role\n(SELECT + INSERT + UPDATE + DELETE)"] readonly_role["readonly_role\n(SELECT only)"] readwrite_role --> admin_role readonly_role --> readwrite_role

Arrows show role membership direction. admin_role inherits all privileges from readwrite_role, which inherits from readonly_role.

Here are the exact statements to build this hierarchy:

-- Tier 1: Read-only
CREATE ROLE readonly_role NOLOGIN;
GRANT CONNECT ON DATABASE mydb TO readonly_role;
GRANT USAGE ON SCHEMA app TO readonly_role;
GRANT SELECT ON ALL TABLES IN SCHEMA app TO readonly_role;

-- Tier 2: Read-write (inherits Tier 1 via role membership)
CREATE ROLE readwrite_role NOLOGIN;
GRANT readonly_role TO readwrite_role;
GRANT INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA app TO readwrite_role;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA app TO readwrite_role;

-- Tier 3: Admin (inherits Tier 2; adds DDL)
CREATE ROLE admin_role NOLOGIN;
GRANT readwrite_role TO admin_role;
-- DDL is governed by schema ownership or explicit CREATE privilege
GRANT CREATE ON SCHEMA app TO admin_role;

-- Assign real users to appropriate tiers
CREATE ROLE alice WITH LOGIN PASSWORD '...';
CREATE ROLE bob   WITH LOGIN PASSWORD '...';
CREATE ROLE carol WITH LOGIN PASSWORD '...';

GRANT readonly_role  TO alice;  -- alice: SELECT only
GRANT readwrite_role TO bob;    -- bob: SELECT + DML
GRANT admin_role     TO carol;  -- carol: SELECT + DML + DDL
Note on DDL in PostgreSQL

In SQL Server, db_ddladmin is a fixed role. In PostgreSQL, DDL access is controlled by schema ownership and the CREATE privilege on a schema. If a role owns a schema, it has full DDL on objects it creates within it. For objects it does not own, explicit GRANT ... ON TABLE ... TO role is required. admin_role receiving GRANT CREATE ON SCHEMA allows it to create new objects; altering or dropping objects it does not own still requires ownership transfer or superuser.

4. INHERIT vs NOINHERIT — Automatic vs Explicit Privilege Activation

INHERIT (the default) means that when a role is a member of another role, its privileges flow automatically. No action required — the user just connects and the permissions are active.

NOINHERIT breaks that automatic flow. The user still has the role membership, but the privileges do not activate until they explicitly run SET ROLE. This is analogous to sudo in Unix: you have the capability, but you must consciously invoke it.

-- Create a DBA login that does NOT automatically inherit admin_role privileges
CREATE ROLE dba_carol WITH LOGIN PASSWORD '...' NOINHERIT;
GRANT admin_role TO dba_carol;

-- dba_carol connects: she has NO DDL or DML privileges yet
-- She must explicitly activate the role before DDL:
SET ROLE admin_role;
CREATE TABLE app.new_table (id serial PRIMARY KEY);

-- When done with privileged work, drop back to base identity:
RESET ROLE;
-- or equivalently: SET ROLE dba_carol;
Attribute Behavior When to use
INHERIT (default) Privileges from all granted roles are active immediately on connect Application service accounts, read-only users, standard developers
NOINHERIT Granted roles are dormant until SET ROLE role_name DBA logins, admin accounts, any role with DDL or superuser-equivalent grants
Security Note

Use NOINHERIT for any login role that has membership in a high-privilege group role. Without it, a compromised DBA session has full admin powers the moment it connects — no escalation step required for an attacker. With NOINHERIT, the attacker must also know to run SET ROLE, and that step is auditable.

5. Role-to-Role Grants vs Direct User Grants

There are two ways to give a user a privilege:

-- Option A: via role hierarchy (preferred)
GRANT readwrite_role TO alice;
-- alice inherits all current and future privileges of readwrite_role

-- Option B: direct grant (avoid in production)
GRANT SELECT ON app.orders TO alice;
-- alice has SELECT on orders only — one table, one user, one grant

Direct grants accumulate silently. After six months, you cannot answer "why does alice have SELECT on that table?" without pg_class archaeology. Role-based grants make the answer obvious: alice has readwrite_role, which grants everything in the app schema. Revoke the role, access is gone — no residual grants to hunt down.

Audit Implication

When you grant a role to a role (GRANT readwrite_role TO alice), the grant is visible in pg_auth_members. When you grant a privilege directly to a user (GRANT SELECT ON ... TO alice), it is recorded in pg_class.relacl and information_schema.role_table_grants separately for each object. Direct grants create a sprawling audit surface that is painful to clean up.

6. Auditing Role Membership

Three tools to inspect what roles exist and who has what:

-- All roles and their attributes (rolcanlogin, rolsuper, rolinherit, etc.)
SELECT rolname, rolcanlogin, rolsuper, rolinherit, rolcreatedb, rolcreaterole
FROM pg_roles
ORDER BY rolcanlogin DESC, rolname;

-- Direct role memberships (one level only)
SELECT
  r.rolname  AS member,
  g.rolname  AS granted_role,
  m.admin_option
FROM   pg_auth_members m
JOIN   pg_roles r ON r.oid = m.member
JOIN   pg_roles g ON g.oid = m.roleid
ORDER BY member, granted_role;

-- psql shortcut: compact view of users and their group memberships
-- \du

The two queries above show only direct memberships. For a deep hierarchy, you need a recursive query. The following resolves all transitive roles for a given user:

-- All roles alice has, including transitive grants through role chains
WITH RECURSIVE role_tree AS (
  -- Anchor: direct roles granted to alice
  SELECT m.member, m.roleid
  FROM   pg_auth_members m
  WHERE  m.member = (SELECT oid FROM pg_roles WHERE rolname = 'alice')

  UNION ALL

  -- Recursive step: roles granted to those roles
  SELECT rt.member, am.roleid
  FROM   pg_auth_members am
  JOIN   role_tree rt ON am.member = rt.roleid
)
SELECT DISTINCT r.rolname AS effective_role
FROM   role_tree rt
JOIN   pg_roles  r  ON r.oid = rt.roleid
ORDER BY effective_role;

Run this query when onboarding or auditing a user. It exposes every privilege bundle they carry, including grants two or three levels deep.

SQL Server Mapping

The equivalent in SQL Server is sys.database_role_members joined to sys.database_principals. SQL Server's IS_MEMBER() function checks a single role. For transitive membership, you also need a recursive CTE against sys.database_role_members. The structure is similar; the catalog names differ. SQL Server also has sys.fn_my_permissions() to enumerate effective permissions on a specific object — PostgreSQL has no single equivalent function, but \z tablename in psql and the has_table_privilege() function cover most needs.

7. Common Mistake 1 — The Mega-Role Anti-Pattern

Top Mistake — Coming from SQL Server

Creating one powerful role and granting it to everyone. The temptation: define a single app_full_access role with SELECT, INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER on all tables, then grant it to every developer, every service account, and every reporting tool. It works — but you lose:

  • Least privilege: your read-only reporting dashboard can now truncate production tables.
  • Auditability: when a row is deleted, you cannot tell if it was the app, a developer, or a report query.
  • Blast radius: a compromised reporting credential has full write access to all application data.

The fix is enforced at design time: draw the 3-tier hierarchy first, assign users to tiers, and never grant a higher tier to a service that only needs a lower one.

8. Common Mistake 2 — Forgetting NOINHERIT for Privileged Roles

Top Mistake — Coming from SQL Server

Leaving privileged DBA logins as INHERIT (the default). In SQL Server, granting a user membership in db_owner is an explicit, intentional escalation — most DBAs do not do it casually. In PostgreSQL, a DBA login that is a member of admin_role with the default INHERIT attribute has full DDL access the moment they connect. No SET ROLE required. A mistyped DROP TABLE runs without friction.

Always create DBA logins with NOINHERIT:

CREATE ROLE dba_alice WITH LOGIN PASSWORD '...' NOINHERIT;
GRANT admin_role TO dba_alice;

Now dba_alice logs in with minimal privileges and must consciously SET ROLE admin_role before performing any DDL. This is the PostgreSQL equivalent of running runas /user:Administrator only when needed.
