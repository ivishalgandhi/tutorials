SSL/TLS — Encrypting Data in Transit

Every query you run, every password challenge exchanged, every row returned travels across a network socket. On a fully isolated single-machine deployment that socket never leaves the kernel. In every other case — application server to database, DBA laptop to database, replica to primary — unencrypted traffic is readable by anything sitting on the path. This lesson covers how PostgreSQL implements TLS, how to configure it correctly, and the precise difference between encrypting a connection and verifying it.

1. Why TLS Matters Even on Internal Networks

The most common objection to TLS on internal connections is: "we're inside the VPC / private subnet, nobody is sniffing." That reasoning has three flaws.

Lateral movement. Once an attacker compromises any host on your internal network — a build server, a monitoring agent, a misconfigured container — they are inside the trust boundary. Without TLS, a compromised host adjacent to the database path can passively capture credentials and query results in cleartext.

Compliance mandates encryption in transit regardless of network topology. PCI-DSS Requirement 4.2.1 requires strong cryptography for all transmission of cardholder data over open and internal networks. HIPAA Security Rule (45 CFR § 164.312(e)(1)) requires transmission security for ePHI. "Internal network" is not an exemption category in either standard.

Cloud and container infrastructure is not equivalent to physical isolation. Multi-tenant cloud networks, software-defined overlays, and shared hypervisors all present threat surfaces that a private VLAN in a dedicated rack does not. Assume the wire is observable.

SQL Server Comparison

SQL Server enables TLS through two mechanisms: the certificate is bound in SQL Server Configuration Manager → Protocols → Certificate, and the Force Encryption flag forces all client connections to use TLS. PostgreSQL takes a file-based approach: certificate paths are declared in postgresql.conf, and whether TLS is required or optional is controlled per connection-type entry in pg_hba.conf. There is no GUI equivalent — everything is text configuration.

2. Server-Side Configuration: postgresql.conf

Four parameters in postgresql.conf control the TLS stack. Find your config file with SHOW config_file;. All file paths are relative to the data directory (SHOW data_directory;) unless you supply an absolute path.

# postgresql.conf — TLS parameters

ssl = on                          # master switch; default off on some builds
ssl_cert_file  = 'server.crt'    # server's public certificate (PEM)
ssl_key_file   = 'server.key'    # server's private key (PEM, must be mode 0600)
ssl_ca_file    = 'root.crt'      # optional: CA cert for verifying client certs
                                  #   required only when using cert auth (mTLS)
Info

Changing ssl, ssl_cert_file, or ssl_key_file requires a server restart, not just a reload. PostgreSQL reads the certificate and key once at startup and holds them in memory. A pg_reload_conf() will not pick up new certificate files.

Two additional parameters worth knowing for production hardening:

ssl_min_protocol_version = 'TLSv1.2'   # reject TLS 1.0 and 1.1 clients
ssl_ciphers = 'HIGH:!aNULL:!MD5'         # OpenSSL cipher string; restricts weak suites

The documentation for all SSL GUCs is at postgresql.org/docs/current/runtime-config-connection.html#RUNTIME-CONFIG-CONNECTION-SSL.

3. Generating a Server Certificate

Self-signed (development and test only)

A self-signed certificate gets TLS working immediately. It encrypts the connection. It does not provide any server identity verification — a client cannot distinguish your server from an impostor without a trusted CA chain. Use self-signed only for local development or throwaway test instances.

# Generate a self-signed certificate valid for 365 days
openssl req -new -x509 -days 365 -nodes \
  -out server.crt -keyout server.key \
  -subj "/CN=my-postgres-server"

# Lock down key permissions — PostgreSQL refuses to start if the key
# is world-readable (modes 0644 or 0777 will cause startup failure)
chmod 600 server.key

# Set ownership to the postgres OS user
chown postgres:postgres server.key server.crt

# Copy both files into the PostgreSQL data directory
PGDATA=$(psql -U postgres -c "SHOW data_directory;" -t | tr -d ' ')
cp server.crt server.key "${PGDATA}"
Danger

Never use a self-signed certificate in a production environment. A self-signed cert encrypts traffic but provides no protection against a man-in-the-middle attack: an attacker can present their own self-signed cert during the TLS handshake and the client has no way to detect the substitution unless the client was pre-loaded with the exact expected certificate. For production, use a certificate signed by your internal CA or a public CA such as Let's Encrypt.

Production: CA-signed certificate

For production deployments, generate a CSR and have it signed by your organisation's internal CA, or use ACME automation (Let's Encrypt, HashiCorp Vault PKI):

# Step 1: generate a key and CSR
openssl req -new -nodes \
  -out server.csr -keyout server.key \
  -subj "/CN=db01.internal.example.com"

# Step 2: submit server.csr to your CA; receive server.crt back
# Step 3: copy both files into PGDATA with correct ownership and mode
chmod 600 server.key
chown postgres:postgres server.key server.crt
SQL Server Comparison

In SQL Server, certificate binding is done through the SQL Server Configuration Manager GUI — you select a certificate from the Windows Certificate Store and SQL Server reads it from there. PostgreSQL has no certificate store integration; files are referenced directly by path in postgresql.conf. The operational implication: certificate rotation means replacing the files on disk and restarting PostgreSQL, not clicking through Configuration Manager.

4. Requiring TLS in pg_hba.conf

Enabling ssl = on in postgresql.conf makes TLS available. Whether it is required for a given connection is controlled by the connection type field in pg_hba.conf. Three type values are relevant:

Type field What it means When to use
host TCP connection, TLS optional. Accepts both encrypted and unencrypted connections. Legacy clients that cannot support TLS. Should not appear in a new production deployment.
hostssl TCP connection, TLS required. Connection is refused if the client does not negotiate TLS. All production application and DBA connections. Use this exclusively for host-based rules.
hostnossl TCP connection, TLS forbidden. Connection is refused if the client offers TLS. Explicitly blocking TLS-capable clients from a specific path. Rarely useful; documents intent rather than adds security.

A production pg_hba.conf combining these correctly:

# pg_hba.conf — enforce TLS for all network connections

# Application connections: TLS required, SCRAM password auth
hostssl  mydb  app_user   10.0.2.0/24   scram-sha-256

# Admin access via local Unix socket: no TLS (sockets are in-kernel)
local    all   postgres              peer

# Catch-all: reject anything not explicitly permitted above
host     all   all        0.0.0.0/0     reject
Security-Critical

The single most common TLS misconfiguration in PostgreSQL is using host instead of hostssl in pg_hba.conf. With host, the server offers TLS but does not enforce it. A misconfigured client — or an attacker performing a downgrade attack — can connect in cleartext. The hostssl keyword is the only way to guarantee that no unencrypted network connection ever reaches the authentication phase.

5. Client Certificate Authentication (Mutual TLS)

Standard TLS authenticates the server to the client. Mutual TLS (mTLS) goes further: the client must also present a certificate signed by a trusted CA, and the CN of that certificate is mapped to a PostgreSQL role. This replaces password authentication entirely for the highest-assurance service connections.

Server-side setup:

# postgresql.conf: point to the CA that signed client certificates
ssl_ca_file = 'root.crt'   # your internal CA's root certificate

# pg_hba.conf: use cert method (no password — certificate IS the credential)
hostssl  mydb  app_user  10.0.2.0/24  cert

The client (application) connection string parameters:

-- psql connection string with client certificate
psql "host=db.internal.example.com \
      dbname=mydb \
      sslcert=client.crt \
      sslkey=client.key \
      sslrootcert=root.crt \
      sslmode=verify-full"

-- libpq environment variable equivalents (useful for application config)
-- PGSSLCERT=client.crt
-- PGSSLKEY=client.key
-- PGSSLROOTCERT=root.crt
-- PGSSLMODE=verify-full

The CN of the client certificate must exactly match the PostgreSQL role name unless you configure a pg_ident.conf mapping. If the role is app_user, the certificate CN must be app_user.

SQL Server Comparison

SQL Server supports certificate-based authentication for logins, but it is rarely deployed in practice outside of Always Encrypted scenarios. PostgreSQL cert authentication is more commonly used — it is a clean way to authenticate service accounts (CI/CD pipelines, microservices) without storing passwords anywhere. Think of it as the PostgreSQL equivalent of Azure Managed Identity or certificate-based service principal authentication.

6. The sslmode Connection Parameter

sslmode is a client-side parameter that controls how much the client validates the server's TLS certificate. There are six levels. Most developers default to prefer or require, neither of which actually validates the server's identity.

sslmode Encrypts? Verifies CA? Verifies hostname? MITM protection
disable No No No None
allow Only if server insists No No None
prefer If available (default libpq) No No None
require Yes No No None — encrypts but does not verify
verify-ca Yes Yes No Partial — blocks unknown CAs but not CN substitution
verify-full Yes Yes Yes Full — the only mode that prevents MITM
Security-Critical

sslmode=require is not secure against man-in-the-middle attacks. It enforces encryption but performs zero certificate validation. An attacker with the ability to intercept your network traffic can present any certificate — including a self-signed one — and your client will complete the TLS handshake, encrypting traffic to the attacker's key rather than your server's key. sslmode=verify-full is the only setting that actually prevents this. It requires that sslrootcert is also set to the CA certificate that signed the server's certificate.

The documentation for all client SSL parameters is at postgresql.org/docs/current/libpq-ssl.html.

Connection string examples

-- Wrong: encrypts but does not verify; MITM is possible
postgresql://app_user:password@db.example.com/mydb?sslmode=require

-- Correct: encrypts AND verifies server identity
postgresql://app_user:password@db.example.com/mydb?sslmode=verify-full&sslrootcert=/etc/ssl/certs/internal-ca.crt

-- For JDBC (Java applications): equivalent settings
-- jdbc:postgresql://db.example.com/mydb?ssl=true&sslmode=verify-full&sslrootcert=/etc/ssl/certs/internal-ca.crt

7. Verifying TLS Is Active

PostgreSQL provides the pg_stat_ssl view to inspect the TLS status of every current connection. Run this from any connected session:

-- Check TLS status of your own current connection
SELECT ssl, version, cipher, bits, client_dn
FROM   pg_stat_ssl
WHERE  pid = pg_backend_pid();

-- Example output when TLS is active:
--  ssl  | version |         cipher          | bits | client_dn
-- ------+---------+-------------------------+------+----------
--  t    | TLSv1.3 | TLS_AES_256_GCM_SHA384  |  256 |

-- If ssl = f, the connection is unencrypted

-- See all current connections and their TLS status (superuser or
-- pg_monitor role required to see other backends)
SELECT pid, ssl, version, cipher, bits, client_dn, compression
FROM   pg_stat_ssl
ORDER BY ssl DESC;

-- client_dn is non-null only when client certificate auth (cert method) is used
-- compression is almost always f — TLS compression has been deprecated

The view documentation is at postgresql.org/docs/current/monitoring-stats.html#MONITORING-PG-STAT-SSL-VIEW.

8. Common Mistakes

Mistake 1: sslmode=require in application connection strings

As shown in the table above, sslmode=require enforces encryption but does not validate server identity. It is better than plaintext, but it is not actually secure against a network-level attacker. The fix is always sslmode=verify-full combined with a valid sslrootcert pointing to the CA that signed the server's certificate.

This is a pervasive mistake because many tutorials, ORMs, and cloud provider documentation examples show sslmode=require as "enabling SSL." It does enable the protocol; it does not enable trust verification.

Mistake 2: World-readable server.key causes cryptic startup failure

PostgreSQL refuses to start if ssl_key_file is readable by anyone other than the owner. The startup error in the PostgreSQL log is:

FATAL:  private key file "server.key" has group or world access
DETAIL: File must have permissions u=rw (0600) or less to be used as
        the server's private key file.

This is correct security behaviour — a world-readable private key is compromised — but the error confuses DBAs who do not immediately connect the startup failure to a file permissions issue. The fix:

chmod 600 /var/lib/postgresql/data/server.key
chown postgres:postgres /var/lib/postgresql/data/server.key
