Schema Security

search_path Hijacking and SECURITY DEFINER Risks

1. What the search_path Is

When you write SELECT * FROM orders, PostgreSQL needs to determine which schema's orders table you mean. It resolves the name by scanning a list of schemas in order: the search_path. The first schema in the list that contains a relation named orders wins.

-- Check the current search_path for the session
SHOW search_path;
-- Default result: "$user", public
-- Means: try a schema named after the current user first, then public

-- See the effective search_path (with "$user" resolved):
SELECT current_schemas(true);
-- e.g. {alice, public} if current user is alice
SQL Server Mapping

SQL Server resolves unqualified names using the user's DEFAULT SCHEMA setting, falling back to dbo. It is a single schema, not a list. PostgreSQL's search_path is a ranked list of schemas, making it more flexible but also more dangerous: an attacker who can influence which schemas are in your search_path can redirect object resolution to their own malicious objects. SQL Server's single-schema resolution is more predictable; PostgreSQL requires active management.

2. The search_path Hijacking Attack — Step by Step

This attack is real, documented, and affects production systems that use SECURITY DEFINER functions without setting a fixed search_path. Here is how it works:

sequenceDiagram participant A as Attacker (low-priv role) participant PG as PostgreSQL participant F as SECURITY DEFINER function
(owned by high-priv role) A->>PG: CREATE SCHEMA attack; A->>PG: CREATE FUNCTION attack.get_config() ...
(malicious override) Note over A,PG: Attacker now has a function
shadowing a legitimate one A->>F: EXECUTE get_salary(1) F->>PG: Searches search_path: attack, public PG-->>F: Finds attack.get_config() first! F-->>A: Runs malicious code under high-priv identity

A SECURITY DEFINER function with a mutable search_path calls a shadowed function, executing attacker-controlled code under the function owner's high privileges.

The four steps of the attack:

  1. Attacker has a low-privilege role with CREATE on some schema, or TEMP privilege (which everyone has by default)
  2. Attacker creates a function with the same name as one called by a SECURITY DEFINER function, placed in a schema earlier in the search_path
  3. When the SECURITY DEFINER function runs and calls the helper function without a schema qualifier, PostgreSQL resolves it to the attacker's version
  4. The attacker's function runs with the privileges of the SECURITY DEFINER function's owner — potentially a superuser
-- Vulnerable function (no SET search_path — attacker can shadow any dependency)
CREATE FUNCTION get_salary(emp_id int)
RETURNS numeric
LANGUAGE plpgsql
SECURITY DEFINER
-- Missing: SET search_path = app, pg_catalog
AS $$
  SELECT salary FROM employees WHERE id = emp_id;
$$;
-- "employees" resolves via the caller's (attacker's) search_path!
SQL Server Mapping

In SQL Server, EXECUTE AS OWNER (the equivalent of SECURITY DEFINER) resolves unqualified names using the owner's default schema, not the caller's. This is more predictable. SQL Server also has EXECUTE AS CALLER (the safe default) and EXECUTE AS 'user'. PostgreSQL's SECURITY INVOKER is the safe default equivalent of EXECUTE AS CALLER. The key difference: SQL Server's ownership chains and schema resolution make name hijacking much harder; PostgreSQL's flexible search_path makes it a first-class attack surface.

3. Preventing search_path Hijacking — Four Defenses

Defense 1: Always use fully-qualified names in production code

-- Unqualified (vulnerable — depends on search_path being correct)
SELECT * FROM orders;

-- Fully qualified (safe — always resolves to app.orders regardless of search_path)
SELECT * FROM app.orders;

Defense 2: Lock search_path per role

-- app_user always resolves unqualified names in app schema only
ALTER ROLE app_user SET search_path = app;

-- DBA role: explicit schemas only, no public fallback
ALTER ROLE dba_alice SET search_path = app, audit;

-- Verify the setting persists:
SELECT rolname, rolconfig
FROM   pg_roles
WHERE  rolname IN ('app_user', 'dba_alice');

Defense 3: Lock search_path globally in postgresql.conf

# postgresql.conf
# Restrict the default: only the user's own schema, no implicit public
search_path = '"$user"'

# Or, if all production objects are in a fixed schema:
search_path = 'app, pg_catalog'

After editing postgresql.conf, reload the server:

SELECT pg_reload_conf();  -- no restart needed for search_path changes

Defense 4: SET search_path in every SECURITY DEFINER function

This is the most important defense — it locks the search_path at function creation time:

-- Safe SECURITY DEFINER function with locked search_path
CREATE FUNCTION app.get_salary(emp_id int)
RETURNS numeric
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = app, pg_catalog, pg_temp
AS $$
BEGIN
  RETURN (SELECT salary FROM app.employees WHERE id = emp_id);
END;
$$;
-- Even if caller's search_path includes a malicious schema,
-- this function always resolves names against app, pg_catalog, pg_temp only.
Why include pg_catalog and pg_temp in SET search_path

pg_catalog holds built-in functions and types (now(), integer, etc.). Without it in the search_path, even system functions may not resolve. pg_temp is listed last (not first) as a deliberate defense — this prevents temporary table shadows (see section 5). PostgreSQL implicitly prepends pg_temp to the search_path when temporary objects exist; listing it explicitly at the end overrides that implicit prepending.

4. SECURITY DEFINER vs SECURITY INVOKER

Every PostgreSQL function runs under one of two privilege contexts:

ModeRuns with privileges ofSQL Server equivalentWhen to use
SECURITY INVOKER The caller (who called the function) EXECUTE AS CALLER Default for almost everything. The caller cannot access anything they could not access directly.
SECURITY DEFINER The creator (who defined the function) EXECUTE AS OWNER When a low-privilege user needs to perform one specific privileged operation without granting them the underlying access.

A legitimate use case for SECURITY DEFINER: you want app_role to be able to call an audit-log function that writes to a table app_role normally cannot write to.

-- audit.events is only writable by audit_owner, not by app_role
CREATE FUNCTION audit.log_event(event_type text, payload jsonb)
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = audit, pg_catalog, pg_temp
AS $$
BEGIN
  INSERT INTO audit.events (event_type, payload, logged_by, logged_at)
  VALUES (event_type, payload, current_user, now());
END;
$$;

-- Grant app_role execute on this specific function only
GRANT EXECUTE ON FUNCTION audit.log_event(text, jsonb) TO app_role;
-- app_role can now call log_event() and it will write to audit.events
-- app_role still cannot SELECT/INSERT/UPDATE/DELETE on audit.events directly
Security Note

Every SECURITY DEFINER function is a potential privilege escalation path. The attack surface is: (1) the function's search_path must be locked, (2) the function must not blindly use caller-supplied input in dynamic SQL without parameterization, (3) the function should be granted only to the specific roles that need it, not to PUBLIC. Audit all SECURITY DEFINER functions regularly with the query in section 7 below.

5. The pg_temp Shadow Attack

Every database session has access to a temporary schema named pg_temp_N (where N is the session's process ID). PostgreSQL provides the alias pg_temp to refer to the current session's temporary schema. By default, pg_temp is implicitly prepended to the search_path — meaning it is searched before any explicitly listed schema.

-- Attacker in their own session:
CREATE TEMP TABLE employees (id int, salary numeric);
INSERT INTO employees VALUES (1, 9999999);

-- If they can invoke a SECURITY DEFINER function that queries "employees"
-- (unqualified) and whose search_path is not locked, pg_temp is searched first:
SELECT * FROM employees;
-- Returns attacker's temp table, not app.employees

The defense is to explicitly place pg_temp at the end of the search_path in your SECURITY DEFINER functions, overriding the implicit prepending behavior:

-- Safe: pg_temp listed last — temporary objects do not shadow real ones
SET search_path = app, pg_catalog, pg_temp

-- Vulnerable: pg_temp first (or absent, which lets PostgreSQL prepend it implicitly)
SET search_path = pg_temp, app, pg_catalog

6. Revoking Public EXECUTE on Functions

By default, when you create a function in PostgreSQL, PUBLIC is granted EXECUTE on it. This means any authenticated role in any database can call any function in the public schema. In a multi-tenant or shared-cluster environment, this is a significant oversharing default.

-- Remove public execute from all existing functions in the public schema
REVOKE EXECUTE ON ALL FUNCTIONS IN SCHEMA public FROM PUBLIC;

-- Then grant execute explicitly to only the roles that need each function
GRANT EXECUTE ON FUNCTION public.my_utility_func() TO app_role;

-- To prevent future functions from inheriting the PUBLIC execute grant,
-- set a default privilege revocation (run as the role that creates functions):
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  REVOKE EXECUTE ON FUNCTIONS FROM PUBLIC;

-- Audit: which functions are currently executable by PUBLIC?
SELECT   n.nspname, p.proname, p.proacl
FROM     pg_proc p
JOIN     pg_namespace n ON n.oid = p.pronamespace
WHERE    n.nspname = 'public'
AND      p.proacl IS NULL  -- NULL acl means PUBLIC has EXECUTE (default)
ORDER BY p.proname;
PostgreSQL 15 and Later

PostgreSQL 15 changed the defaults for new databases: CREATE on the public schema is revoked from PUBLIC by default, and new databases get a cleaner baseline. However, EXECUTE on functions is still granted to PUBLIC by default in all versions. If you are upgrading from pre-PG15, the public schema hardening steps here still apply.

7. Schema Security Hardening Checklist

Run these statements against every new production database, in this order:

-- 1. Revoke default CONNECT access (covered in Lesson 6)
REVOKE CONNECT ON DATABASE mydb FROM PUBLIC;

-- 2. Revoke CREATE on public schema (any role was allowed to create objects here)
REVOKE CREATE ON SCHEMA public FROM PUBLIC;

-- 3. Revoke EXECUTE on all public schema functions
REVOKE EXECUTE ON ALL FUNCTIONS IN SCHEMA public FROM PUBLIC;

-- 4. Lock the default search_path in postgresql.conf
-- search_path = '"$user"'   -- users only see their own schema by default

-- 5. Audit all SECURITY DEFINER functions and verify search_path is set
SELECT
  n.nspname     AS schema,
  p.proname     AS function,
  r.rolname     AS owner,
  p.prosecdef   AS security_definer,
  p.proconfig   AS config  -- should contain search_path entry
FROM   pg_proc p
JOIN   pg_namespace n ON n.oid = p.pronamespace
JOIN   pg_roles     r ON r.oid = p.proowner
WHERE  p.prosecdef = true       -- only SECURITY DEFINER functions
ORDER BY schema, function;

-- Any row with proconfig IS NULL or proconfig not containing 'search_path'
-- is a vulnerability that needs the SET search_path fix applied.
Hardening Step Why It Matters SQL Server Equivalent
REVOKE CREATE ON SCHEMA public FROM PUBLIC Prevents any role from creating shadow objects in public schema Schema ownership prevents non-owners from creating objects
REVOKE EXECUTE ON ALL FUNCTIONS IN SCHEMA public FROM PUBLIC Stops unauthorized function calls across roles DENY EXECUTE ON SCHEMA::dbo TO public

8. Common Mistake 1 — SECURITY DEFINER Without SET search_path

Top Mistake — Coming from SQL Server

Writing SECURITY DEFINER functions without SET search_path. This is the number one PostgreSQL privilege escalation vector, documented in the PostgreSQL security advisory pages and in multiple CVEs against PostgreSQL extensions. In SQL Server, EXECUTE AS OWNER resolves names against the owner's schema, so the equivalent attack surface does not exist in the same way.

The symptom is invisible during development because you are the object owner and your search_path is correct. The vulnerability only manifests when a lower-privilege role calls the function with a malicious search_path in their session.

The fix is one line, added to every SECURITY DEFINER function:

SET search_path = app, pg_catalog, pg_temp

Run the audit query from section 7 against every production database immediately and fix any SECURITY DEFINER functions that lack it.

9. Common Mistake 2 — Trusting the Public Schema

Top Mistake — Coming from SQL Server

Creating production application objects in the public schema and relying on it for production traffic. In SQL Server, the dbo schema is controlled by db_owner — ordinary users cannot create objects there without explicit db_ddladmin membership. In PostgreSQL (pre-PG15), the public schema allows any authenticated role to create objects by default. This means a compromised application role with no special grants can still create functions, tables, and views in public — including objects that shadow system functions.

The fix has two parts:

  1. Move all production objects out of public into an application-owned schema (app, myapp, etc.)
  2. Run REVOKE CREATE ON SCHEMA public FROM PUBLIC immediately after database creation

For PostgreSQL 15 and later, the CREATE revocation from PUBLIC is the default for new databases — but EXECUTE on functions remains publicly accessible. The full hardening checklist in section 7 applies regardless of PostgreSQL version.
