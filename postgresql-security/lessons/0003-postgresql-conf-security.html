postgresql.conf Security Parameters — The 12 GUCs Every DBA Must Know

pg_hba.conf controls who can connect. postgresql.conf controls how the server behaves once they are connected — and several of its parameters are directly security-critical. This lesson covers the twelve Grand Unified Configuration parameters that every security-conscious DBA must understand and actively manage.

1. What a GUC Is

PostgreSQL calls every configuration parameter a GUC — Grand Unified Configuration. The name reflects that a single system governs parameters across a wide range of categories: memory, connections, WAL, query planning, and security. Every GUC has a name, a value, a data type, a default, and a context that determines when a change to that parameter takes effect.

SQL Server Comparison

GUCs are PostgreSQL's equivalent of sp_configure options in SQL Server. Just as sp_configure 'max degree of parallelism', 4 sets a server-level behaviour, ALTER SYSTEM SET max_parallel_workers = 4 sets a GUC at the instance level. The key difference: in SQL Server, most sp_configure changes require RECONFIGURE (and some require RECONFIGURE WITH OVERRIDE). In PostgreSQL, some GUC changes require only a reload, while others require a full server restart — and the system is explicit about which is which.

2. How to Inspect Current Values

-- Show one parameter
SHOW ssl;
SHOW listen_addresses;

-- Show all parameters at once
SHOW all;

-- The pg_settings view is more powerful — includes source file, context, and units
SELECT name, setting, unit, context, source, sourcefile, sourceline
FROM   pg_settings
WHERE  name IN ('ssl', 'listen_addresses', 'password_encryption',
               'log_connections', 'log_statement')
ORDER BY name;

-- 'context' column tells you how to apply a change:
--   postmaster  → requires full server restart
--   sighup      → reload only (SELECT pg_reload_conf())
--   superuser   → can be changed per-session by a superuser
--   user        → can be changed per-session by any user

-- Show only security-relevant parameters that are not at their default
SELECT name, setting, boot_val AS default_val, source
FROM   pg_settings
WHERE  setting != boot_val
AND    name LIKE ANY(ARRAY['%ssl%', '%log%', '%auth%', '%encrypt%'])
ORDER BY name;

3. The 12 Security GUCs

Category A — SSL/TLS (4 Parameters)

These four parameters govern encrypted transport between clients and the PostgreSQL server. Without SSL, all query text, result data, and credentials travel in cleartext — including passwords even when using scram-sha-256, because the password exchange itself is the plaintext that gets encrypted by the TLS layer. The official documentation is at postgresql.org/docs/current/ssl-tcp.html.

ssl

-- Enable TLS support
-- Default: on in PostgreSQL 14+ (was off in earlier versions)
-- Context: postmaster (requires restart)
ALTER SYSTEM SET ssl = 'on';

-- Verify:
SHOW ssl;  -- should return 'on'

Setting ssl = on enables the server to accept TLS connections. It does not require TLS — that enforcement happens in pg_hba.conf using hostssl entries. Think of ssl = on as "TLS is available" and hostssl as "TLS is mandatory for this rule."

ssl_cert_file and ssl_key_file

-- Paths to the server certificate and private key
-- Context: postmaster (requires restart)
ALTER SYSTEM SET ssl_cert_file = '/etc/ssl/certs/pg-server.crt';
ALTER SYSTEM SET ssl_key_file  = '/etc/ssl/private/pg-server.key';

-- Quick self-signed certificate for testing (run as postgres OS user):
-- openssl req -new -x509 -days 365 -nodes \
--   -subj "/CN=db.example.com" \
--   -out /etc/ssl/certs/pg-server.crt \
--   -keyout /etc/ssl/private/pg-server.key
-- chmod 600 /etc/ssl/private/pg-server.key
-- chown postgres:postgres /etc/ssl/private/pg-server.key

ssl_ca_file

-- Path to the CA certificate used to verify CLIENT certificates
-- Only needed if you are using cert-based client authentication
-- Context: postmaster (requires restart)
ALTER SYSTEM SET ssl_ca_file = '/etc/ssl/certs/internal-ca.crt';

password_encryption

-- Controls how passwords are hashed when stored in pg_authid
-- Default in PG14+: scram-sha-256
-- Default in PG13 and earlier: md5
-- Context: sighup (reload only — no restart needed)
ALTER SYSTEM SET password_encryption = 'scram-sha-256';
SELECT pg_reload_conf();

-- IMPORTANT: changing this setting only affects NEW passwords.
-- Existing users retain their old hash until their password is reset.
-- To identify users with legacy md5 hashes:
SELECT rolname, rolpassword
FROM   pg_authid
WHERE  rolpassword LIKE 'md5%'
AND    rolcanlogin = true;

-- Force a password reset for those users:
ALTER ROLE legacy_user PASSWORD 'new_strong_password';
-- Their hash is now stored as scram-sha-256.
Security Note

After setting password_encryption = scram-sha-256, you must also update the pg_hba.conf METHOD from md5 to scram-sha-256 for each relevant entry. The pg_hba.conf METHOD and the stored hash must agree. A mismatch causes authentication failures — md5-hashed credentials cannot satisfy a scram-sha-256 challenge and vice versa.

Category B — Network Exposure (3 Parameters)

listen_addresses

-- Which network interfaces PostgreSQL binds to
-- Default: 'localhost'
-- Context: postmaster (requires restart)

-- Safest — only local socket and loopback TCP:
ALTER SYSTEM SET listen_addresses = 'localhost';

-- Bind to a specific interface (recommended for multi-NIC servers):
ALTER SYSTEM SET listen_addresses = '10.0.1.20';

-- Bind to all interfaces — use only when network-level controls are in place:
ALTER SYSTEM SET listen_addresses = '*';
Danger — listen_addresses = '*'

Setting listen_addresses = '*' means PostgreSQL binds to every network interface on the host, including any interface exposed to the internet. The blast radius: every port-scanning service on the internet immediately sees a PostgreSQL service. Even with pg_hba.conf correctly configured, you have increased attack surface — unexpected interfaces, cloud provider management networks, or a misconfigured firewall rule can expose the port. Bind to the minimum set of interfaces required. If the application lives on the same server, localhost is sufficient.

SQL Server Comparison

listen_addresses maps to the TCP/IP protocol binding in SQL Server Configuration Manager (SQL Server Network Configuration → Protocols for [instance] → TCP/IP → IP Addresses). In SQL Server, you can enable or disable specific IP addresses individually; in PostgreSQL you specify a comma-separated list of addresses or * for all.

port

-- TCP port PostgreSQL listens on. Default: 5432
-- Context: postmaster (requires restart)
ALTER SYSTEM SET port = 5432;

-- Changing the port reduces automated scan noise but provides no
-- real security barrier. A determined attacker will port-scan all 65535 ports.
-- Never rely on a non-standard port as a security control.
-- Document any port change and update your firewall rules accordingly.

max_connections

-- Maximum simultaneous client connections
-- Default: 100
-- Context: postmaster (requires restart)
ALTER SYSTEM SET max_connections = 200;

-- Security relevance: a low max_connections cap limits the blast radius
-- of a connection flood (DoS). When all slots are consumed, new connections
-- are refused. Set it to the application's realistic peak plus a small
-- reserve for DBA emergency access (e.g., use pg_hba.conf to reserve
-- connections via the reserved_connections GUC in PG16+).

-- In PG16+, reserve connections for superusers explicitly:
ALTER SYSTEM SET reserved_connections = 3;  -- PG16+
-- superuser_reserved_connections remains available in all versions:
ALTER SYSTEM SET superuser_reserved_connections = 3;

Category C — Logging (5 Parameters)

PostgreSQL's built-in logging is not a compliance-grade audit trail — that requires the pgaudit extension — but these five parameters provide a meaningful operational security baseline. The logging documentation lives at postgresql.org/docs/current/runtime-config-logging.html.

log_connections and log_disconnections

-- Log every successful connection (log_connections)
-- and every session termination (log_disconnections)
-- Context: sighup (reload — no restart)
ALTER SYSTEM SET log_connections    = 'on';
ALTER SYSTEM SET log_disconnections = 'on';
SELECT pg_reload_conf();

-- Sample log output with log_connections = on:
-- LOG:  connection received: host=10.0.1.50 port=54321
-- LOG:  connection authorized: user=app_user database=appdb SSL enabled
--
-- This is invaluable for spotting unexpected connection sources,
-- unusual connection counts, and authentication timing patterns.
SQL Server Comparison

SQL Server logs successful logins only when the Login Auditing setting (Server Properties → Security) is set to "Both failed and successful logins." PostgreSQL's log_connections is the equivalent — and like SQL Server, it produces one log line per connection, which can be high volume in connection-pooled environments. Use a connection pooler like PgBouncer in such cases; it will reduce the raw connection count PostgreSQL sees.

log_connections and Failed Authentication

-- PostgreSQL does not have a standalone GUC named log_failed_authentications.
-- Failed authentication attempts are logged automatically at LOG severity
-- regardless of log_connections setting.
--
-- You control the minimum message level written to the log file via:
--   log_min_messages (default: warning)
--   client_min_messages (default: notice)
--
-- To ensure failed auth messages appear in the log:
ALTER SYSTEM SET log_min_messages = 'info';
SELECT pg_reload_conf();

-- Sample failed auth log entry:
-- LOG:  connection received: host=192.168.99.1 port=41234
-- FATAL: password authentication failed for user "app_user"
-- DETAIL: Connection matched pg_hba.conf line 5: "hostssl appdb app_user ..."
--
-- The DETAIL line is only shown to superusers and to the client.
-- Non-superuser clients see only the FATAL line without details.

log_statement

-- Controls which SQL statements are written to the log.
-- Context: sighup (reload — no restart)
--
-- Values:
--   none  — log nothing (default)
--   ddl   — log CREATE, ALTER, DROP, TRUNCATE (schema changes only)
--   mod   — log ddl + INSERT, UPDATE, DELETE, COPY (data-modifying statements)
--   all   — log every statement including SELECT

-- Recommended starting point for most production systems:
ALTER SYSTEM SET log_statement = 'ddl';
SELECT pg_reload_conf();

-- For compliance environments, use pgaudit instead of log_statement = 'all':
-- pgaudit produces structured, per-object audit records that can be filtered
-- without the noise of log_statement = 'all' (see the pgaudit warning below).
SQL Server Comparison

log_statement = 'ddl' approximates SQL Server's Schema Changes audit action group. log_statement = 'mod' approximates the DML audit action groups. log_statement = 'all' is closest to SQL Server Audit at the database level with all statement-level actions enabled. However, SQL Server Audit writes to a structured binary audit log; log_statement writes to the text-based PostgreSQL log file. For structured, queryable audit output in PostgreSQL, use the pgaudit extension.

log_line_prefix

-- A printf-style format string prepended to every log line.
-- Context: sighup (reload — no restart)
--
-- Format codes:
--   %m  — timestamp with milliseconds
--   %p  — process ID
--   %u  — database user name
--   %d  — database name
--   %r  — remote host and port
--   %a  — application name
--   %q  — stop here in non-session processes (silences background worker noise)
--   %i  — command tag (e.g., SELECT, UPDATE)
--   %e  — SQLSTATE error code

-- Recommended format: timestamp [pid] user@database — supports log parsing tools
ALTER SYSTEM SET log_line_prefix = '%m [%p] %q%u@%d ';
SELECT pg_reload_conf();

-- Sample output with this prefix:
-- 2024-03-15 14:22:01.823 UTC [18432] app_user@appdb SELECT ...

-- Extended prefix for environments using log aggregation (Splunk, Datadog, etc.):
-- ALTER SYSTEM SET log_line_prefix = '%m [%p] %q%u@%d %r %a ';

Category D — Row Security Guard

row_security

-- Global switch: if 'off', Row Level Security policies are bypassed
-- for all users (including non-superusers). Default: on.
-- Context: sighup (reload — no restart)
ALTER SYSTEM SET row_security = 'on';

-- This GUC is a guard, not the policy itself.
-- RLS policies are defined at the table level with:
--   ALTER TABLE employees ENABLE ROW LEVEL SECURITY;
--   CREATE POLICY staff_isolation ON employees
--     USING (department = current_setting('app.department'));
--
-- Even with row_security = on, a SUPERUSER or table owner bypasses all policies
-- unless the table was created with FORCE ROW LEVEL SECURITY:
--   ALTER TABLE employees FORCE ROW LEVEL SECURITY;

4. SQL Server Mapping Table

PostgreSQL GUC SQL Server Equivalent Notes
ssl_cert_file / ssl_key_file Certificate binding in SQL Server Configuration Manager Both bind a TLS certificate to the database listener; PostgreSQL references files directly, SQL Server uses the Windows certificate store.
listen_addresses TCP/IP protocol binding (SQL Server Configuration Manager → IP Addresses) Both control which network interfaces the database engine listens on.
port TCP port in SQL Server Configuration Manager Default 5432 vs. 1433. Dynamic ports vs. static ports in SQL Server have no PostgreSQL equivalent.
password_encryption No direct equivalent; SQL Server uses its own algorithm SQL Server does not expose the password hashing algorithm as a configurable parameter.
log_connections Login Auditing: Successful logins (Server Properties → Security) Both produce one log entry per successful connection.
log_statement = 'ddl' SQL Server Audit: Schema Changes action group SQL Server Audit writes structured binary records; PostgreSQL writes text log entries.
log_statement = 'all' SQL Server Audit: all statement-level action groups Use pgaudit for structured compliance output instead of log_statement = 'all'.
max_connections No direct equivalent; SQL Server uses a resource pool and connection limits per workload group SQL Server Resource Governor provides more granular per-pool limits.

5. Applying Changes: Reload vs. Restart

This is where SQL Server DBAs frequently reach for a restart out of habit. In PostgreSQL, many security-critical parameters take effect with a simple reload — no downtime required. Knowing which is which saves production incidents.

-- RELOAD ONLY (SELECT pg_reload_conf()) — no restart needed:
--   password_encryption, log_connections, log_disconnections
--   log_statement, log_line_prefix, log_min_messages
--   ssl_cert_file*, ssl_key_file*, ssl_ca_file*
--   row_security, listen_addresses** (see note)

-- RESTART REQUIRED (pg_ctl restart / systemctl restart postgresql):
--   ssl (on/off toggle itself)
--   listen_addresses (changing which interfaces to bind)
--   port
--   max_connections
--   superuser_reserved_connections

-- * ssl_cert_file and ssl_key_file: PostgreSQL rereads certificate files
--   on SIGHUP (reload) for NEW connections. Existing TLS sessions are unaffected.
--   The ssl = on/off toggle itself requires a restart.
--
-- ** listen_addresses requires a restart to take effect.

-- The pg_settings view tells you the context for any GUC:
SELECT name, context
FROM   pg_settings
WHERE  context IN ('postmaster', 'sighup')
AND    name IN ('ssl', 'listen_addresses', 'log_connections',
               'password_encryption', 'log_statement')
ORDER BY context, name;

-- Apply a reload after changing sighup-context parameters:
ALTER SYSTEM SET log_connections    = 'on';
ALTER SYSTEM SET log_disconnections = 'on';
ALTER SYSTEM SET log_statement      = 'ddl';
SELECT pg_reload_conf();  -- applies all three at once
Info — ALTER SYSTEM vs. direct file edit

ALTER SYSTEM SET writes the setting to postgresql.auto.conf, which is read after postgresql.conf and overrides it. This is the preferred approach — it is transactional, logged, and avoids manual file editing mistakes. To revert a setting to its default: ALTER SYSTEM RESET parameter_name. To reset all overrides: ALTER SYSTEM RESET ALL.

6. Mistake 1 — Leaving ssl = off Because "It's an Internal Network"

Top DBA Mistake — Coming from SQL Server

SQL Server environments with Windows Authentication use Kerberos, which provides mutual authentication and session key establishment — effectively encrypting the authentication exchange even on an unencrypted connection. The network traffic between client and server is still cleartext, but the auth tokens are not. This creates a false sense of security: if auth is handled, why encrypt the connection?

In PostgreSQL without TLS: query text, result rows, bind parameters, and application data are all transmitted in cleartext. An attacker with network access — a compromised machine on the same VLAN, a misconfigured cloud security group, a rogue switch — can perform a trivial man-in-the-middle attack or passive wiretap. The "internal network" assumption fails the moment any internal host is compromised, which is the foundational premise of modern lateral movement attacks.

The fix is two lines and a restart:

ALTER SYSTEM SET ssl          = 'on';
ALTER SYSTEM SET ssl_cert_file = '/path/to/server.crt';
ALTER SYSTEM SET ssl_key_file  = '/path/to/server.key';
-- Then restart the PostgreSQL service, and update pg_hba.conf to use hostssl.

7. Mistake 2 — log_statement = 'all' in Production Without pgaudit

DBA Migration Warning — Top Mistake

SQL Server DBAs who need a complete SQL audit trail instinctively reach for maximum logging. In PostgreSQL, log_statement = 'all' writes every SQL statement to the text-based log file. In a busy OLTP database with thousands of queries per second, this produces gigabytes of unstructured text per hour. The consequences: disk fills up, I/O contention slows the database, and finding any specific event in the flood of SELECT statements becomes practically impossible without significant tooling investment.

The correct approach for compliance auditing is the pgaudit extension. pgaudit integrates with PostgreSQL's object access hook infrastructure to produce structured, per-statement audit records that can be filtered by object, by statement class (READ, WRITE, DDL, ROLE), and by role — without logging routine query traffic. log_statement = 'ddl' is the appropriate general-purpose setting; reserve pgaudit for statements where a structured audit trail is a compliance requirement.

-- Install pgaudit (example: Debian/Ubuntu with PG16)
-- sudo apt install postgresql-16-pgaudit

-- Enable it as a shared preload library (restart required)
ALTER SYSTEM SET shared_preload_libraries = 'pgaudit';
-- restart server

-- Then configure which statement classes to audit:
ALTER SYSTEM SET pgaudit.log = 'ddl, write';  -- log schema changes and data modifications
SELECT pg_reload_conf();

-- pgaudit log entry format (structured, parseable):
-- AUDIT: SESSION,1,1,DDL,CREATE TABLE,TABLE,public.accounts,
--        CREATE TABLE public.accounts (id serial PRIMARY KEY, ...),<not logged>
