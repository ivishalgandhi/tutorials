Auditing and Logging — pgaudit, Log Parameters, and SQL Server Audit Mapping

A database without structured auditing is a database you cannot investigate. When a security incident occurs — a data exfiltration, an unexpected schema change, an unauthorized SELECT on a sensitive table — your logs determine whether you can reconstruct what happened, who did it, and when. PostgreSQL ships with flexible logging infrastructure. Getting it right requires understanding the boundary between general-purpose logging and structured audit, and that is exactly where pgaudit lives.

1. The Gap: Built-in Logging vs. Structured Audit

PostgreSQL's built-in log_statement parameter can log SQL text. Set it to 'all' and every statement appears in the server log. That sounds sufficient. It is not, for three reasons:

  • No object-level granularity. You cannot say "log all SELECTs on the payroll table but not on product_catalog." It is all or nothing.
  • No structured context. The log entry is raw SQL text with a timestamp and PID. There is no reliable separation of: which object was accessed, what type of operation it was, whether it succeeded, and what role's privileges were used.
  • Compliance tools cannot parse it. SIEM systems, compliance platforms, and log aggregators expect structured fields. Free-form SQL text in log lines requires fragile regex parsing that breaks on multi-line statements and quoted literals.

pgaudit (PostgreSQL Audit Extension) was built to fill this gap. It hooks into the PostgreSQL executor and emits structured log records with consistent, named fields for every audited operation. The pgaudit documentation is at pgaudit.org and the source is maintained at github.com/pgaudit/pgaudit.

SQL Server Comparison

In SQL Server, structured auditing is provided by SQL Server Audit (available since SQL Server 2008). You create a Server Audit object pointing to a file, Windows Event Log, or Application Log destination, then create Server Audit Specifications (for server-level events) and Database Audit Specifications (for database-level events) that define which action groups to capture. pgaudit maps to this model closely: session audit is analogous to a database audit spec covering all objects, and object audit is analogous to a database audit spec scoped to specific tables or schemas.

2. Installing pgaudit

pgaudit is a shared library extension. It must be compiled against your specific PostgreSQL version and loaded at server startup via shared_preload_libraries. It cannot be loaded dynamically at runtime — adding it requires a server restart.

# Debian / Ubuntu — install the package matching your PostgreSQL major version
apt-get install postgresql-16-pgaudit

# RHEL / Rocky Linux / AlmaLinux
dnf install pgaudit_16

# macOS via Homebrew (bundled with postgresql formula on recent versions)
# brew install pgaudit

Then add it to postgresql.conf and restart:

# postgresql.conf
shared_preload_libraries = 'pgaudit'

# If you already have other libraries loaded, append pgaudit:
# shared_preload_libraries = 'pg_stat_statements, pgaudit'
# OS shell: restart the PostgreSQL service (Debian/Ubuntu package layout)
systemctl restart postgresql@16-main

# Verify pgaudit is loaded after restart:
-- psql
SELECT name, default_version, installed_version
FROM   pg_available_extensions
WHERE  name = 'pgaudit';

-- pgaudit does NOT need CREATE EXTENSION — it is a library, not a schema extension
-- The GUCs (pgaudit.log etc.) are available once the library is loaded
Warning

Unlike most PostgreSQL extensions, pgaudit does not require CREATE EXTENSION pgaudit. Loading it in shared_preload_libraries is sufficient. Running CREATE EXTENSION pgaudit is optional and only needed if you want the object audit role feature. Attempting to use pgaudit GUCs without the library loaded will produce a "unrecognized configuration parameter" error.

3. Two Audit Modes

pgaudit provides two complementary audit modes that can be used independently or together:

Mode Controlled by Granularity Best for
Session audit pgaudit.log GUC By statement class (read, write, ddl, etc.) across all objects Broad compliance coverage: "audit all DDL" or "audit all writes"
Object audit pgaudit.role GUC + role grants Specific tables, views, or sequences, regardless of which role accesses them Targeted surveillance: "audit every read on the payroll table"

When both modes are active, pgaudit emits a log record for each mode that triggers independently. A statement that matches both session and object audit rules produces two log entries: one tagged SESSION and one tagged OBJECT.

4. Session Audit Configuration

Session audit is controlled by the pgaudit.log GUC. It accepts a comma-separated list of statement class keywords. Every statement matching the specified classes is logged for every session, server-wide.

# postgresql.conf — session audit settings

# Log all data-modification and schema-change statements
pgaudit.log = 'write, ddl'

# Reduce noise: do not audit system catalog reads (pg_catalog schema)
pgaudit.log_catalog = off

# Log each relation (table/view) accessed separately per statement
# A JOIN touching 3 tables produces 3 log entries — one per table
pgaudit.log_relation = on

# Log the full statement text only once even if multiple substatements fire
pgaudit.log_statement_once = off   # off = log statement with each record

The available class keywords and what they cover:

Class Statements covered SQL Server audit spec equivalent
read SELECT, COPY FROM SELECT action group in database audit spec
write INSERT, UPDATE, DELETE, TRUNCATE, COPY TO INSERT, UPDATE, DELETE action groups
function Function calls via DO and function execution EXECUTE action group
role GRANT, REVOKE, CREATE/ALTER/DROP ROLE DATABASE_ROLE_MEMBER_CHANGE_GROUP (server spec)
ddl All CREATE, ALTER, DROP statements SCHEMA_OBJECT_CHANGE_GROUP (database spec)
misc DISCARD, FETCH, CHECKPOINT, VACUUM, SET No direct equivalent
misc_set SET statements specifically (subset of misc) No direct equivalent
all Every statement class All action groups combined
none Disable session audit Disabling the audit spec
Key Concept

A sensible minimum for most production environments is pgaudit.log = 'write, ddl'. This captures all data modifications and all schema changes without the log volume overhead of auditing every SELECT. Add role to capture privilege changes. Use all only for short-term investigation or when regulatory requirements explicitly demand SELECT-level audit (PCI-DSS Requirement 10.2 mandates read access to cardholder data).

5. Object Audit Configuration

Object audit uses a dedicated audit role as an indirection mechanism. Instead of specifying which tables to audit directly to pgaudit, you grant object privileges to a special NOLOGIN role, then tell pgaudit to watch that role. Any access to an object that the audit role has been granted access to will be logged, regardless of which actual user runs the statement.

-- Step 1: create a dedicated audit role (cannot login — it is a sentinel)
CREATE ROLE rds_pgaudit NOLOGIN;

-- Step 2: grant privileges to the audit role for objects you want to audit
--   Here: audit all reads on the sensitive_data table
GRANT SELECT ON sensitive_data TO rds_pgaudit;

--   Audit writes on the payroll table
GRANT INSERT, UPDATE, DELETE ON payroll TO rds_pgaudit;

-- Step 3: in postgresql.conf, point pgaudit at the audit role
-- pgaudit.role = 'rds_pgaudit'

-- Reload (no restart needed for pgaudit.role change)
SELECT pg_reload_conf();

Now every time any user executes a SELECT on sensitive_data or a write on payroll, pgaudit emits an OBJECT audit log entry. The user running the query does not need to know they are being audited, and there is no way to circumvent it by connecting as a different role.

SQL Server Comparison

This maps to SCHEMA_OBJECT_ACCESS_GROUP scoped to specific objects in a SQL Server Database Audit Specification. The difference: SQL Server lets you name the object directly in the audit spec (ON OBJECT::dbo.sensitive_data BY PUBLIC). PostgreSQL uses the indirect role-grant mechanism. The role-based approach is more flexible — you can audit a new table by simply granting to the audit role without restarting or modifying configuration files.

6. Reading a pgaudit Log Entry

pgaudit writes to the standard PostgreSQL log (controlled by log_destination, logging_collector, etc.). Audit entries are prefixed with AUDIT: and contain structured comma-separated fields. Understanding the field layout is essential for writing parsers or SIEM rules.

# A pgaudit SESSION log entry (reformatted for readability)
2024-01-15 14:23:01 UTC [12345] app_user@mydb LOG:
  AUDIT: SESSION,1,1,WRITE,INSERT,TABLE,app.orders,
  "INSERT INTO orders VALUES (42, 'data')",

Field-by-field breakdown:

Position Field name Value in example Meaning
1 Audit type SESSION SESSION (session audit) or OBJECT (object audit)
2 Statement ID 1 Sequential number for statements in this session; resets each session
3 Substatement ID 1 For statements containing multiple sub-operations (e.g., rules, triggers)
4 Class WRITE The pgaudit class that triggered this entry (READ, WRITE, DDL, etc.)
5 Command INSERT The specific SQL command
6 Object type TABLE TABLE, INDEX, VIEW, FUNCTION, SCHEMA, etc.
7 Object name app.orders Fully qualified object name (schema.object)
8 Statement INSERT INTO ... The full SQL statement text
9 Parameter <none> Bind parameters when pgaudit.log_parameter = on

Who ran the statement? The role name and database appear in the standard log prefix (controlled by log_line_prefix), not inside the AUDIT record itself. This is why setting log_line_prefix correctly is critical — see Section 7.

7. Essential log_ Parameters to Set Alongside pgaudit

pgaudit generates the structured audit records; the surrounding log infrastructure determines whether those records are useful. These parameters should be set in postgresql.conf for any production instance:

# postgresql.conf — logging parameters for a production audit setup

# Prefix format: timestamp [pid] user@database space
# %m = timestamp with milliseconds
# %p = process ID
# %q = empty string if not in a backend session (silent for background workers)
# %u = user name
# %d = database name
log_line_prefix = '%m [%p] %q%u@%d '

# Log connections and disconnections (equivalent to SQL Server LOGIN/LOGOUT audit group)
log_connections    = on
log_disconnections = on

# Log all queries that take more than 1 second (1000ms)
log_min_duration_statement = 1000

# Log checkpoint activity (helps diagnose write workload spikes)
log_checkpoints = on

# Log lock waits (helps diagnose contention and blocked queries)
log_lock_waits = on

# Log creation of any temporary file — 0 means log all sizes
log_temp_files = 0

# Destination: write structured CSV logs for easier parsing
# log_destination = 'csvlog'   # or 'jsonlog' on PostgreSQL 15+
# logging_collector = on
# log_directory = 'log'
# log_filename = 'postgresql-%Y-%m-%d.log'
Info

PostgreSQL 15 introduced jsonlog as a log destination format. JSON-formatted logs are the easiest to ingest into SIEM tools, log aggregation pipelines, and cloud log management services. If you are on PostgreSQL 15 or later, prefer log_destination = 'jsonlog' over csvlog for new deployments.

8. SQL Server Audit to pgaudit Mapping

If you are migrating an audit policy from SQL Server, use this table to find the pgaudit equivalent for each SQL Server audit action group:

SQL Server Audit pgaudit / PostgreSQL equivalent
Database Audit Spec: INSERT, UPDATE, DELETE pgaudit.log = 'write'
Database Audit Spec: SELECT pgaudit.log = 'read'
Database Audit Spec: EXECUTE pgaudit.log = 'function'
Server Audit Spec: CREATE/ALTER/DROP (SCHEMA_OBJECT_CHANGE_GROUP) pgaudit.log = 'ddl'
Server Audit Spec: GRANT/REVOKE (DATABASE_ROLE_MEMBER_CHANGE_GROUP) pgaudit.log = 'role'
Server Audit Spec: SUCCESSFUL_LOGIN_GROUP log_connections = on
Server Audit Spec: LOGOUT_GROUP log_disconnections = on
Server Audit Spec: FAILED_LOGIN_GROUP log_connections = on (auth failures appear in connection log)
Specific table access (SCHEMA_OBJECT_ACCESS_GROUP scoped) pgaudit object audit: GRANT on pgaudit.role sentinel role
Audit file destination log_destination = 'csvlog' or 'jsonlog' (PG 15+)
Windows Event Log destination log_destination = 'syslog' (Linux) or 'eventlog' (Windows)

9. Per-Role Audit Levels

pgaudit settings can be applied at the role level using ALTER ROLE ... SET. A role-level setting overrides the server-level pgaudit.log for sessions authenticated as that role. This is useful for escalating audit coverage for privileged accounts or accounts with access to sensitive data, without imposing the log volume on everyone.

-- Give a sensitive user full audit coverage regardless of server-level setting
ALTER ROLE sensitive_user SET pgaudit.log = 'all';

-- Verify what is set for a role
SELECT rolname, rolconfig
FROM   pg_roles
WHERE  rolname = 'sensitive_user';

-- Remove the role-level override and revert to server default
ALTER ROLE sensitive_user RESET pgaudit.log;
SQL Server Comparison

SQL Server does not have a direct equivalent to per-login audit granularity within the audit spec engine. To audit a specific login more heavily in SQL Server you would need a separate database audit spec or an Extended Events session filtered by login name. PostgreSQL's ALTER ROLE ... SET mechanism is cleaner — the audit configuration follows the role identity regardless of which database they connect to.

10. Common Mistakes

Mistake 1: Using log_statement = 'all' as a compliance substitute

log_statement = 'all' writes every SQL statement to the log as unstructured text. It appears to be comprehensive. It has three serious problems:

  • No object-level granularity: you cannot tell from the log which table a multi-table JOIN was auditing for compliance purposes.
  • Unstructured output: the log line is raw SQL text prefixed by a timestamp and PID. SIEM ingestion requires regex parsing that breaks on edge cases (quoted identifiers containing the parser's expected delimiters, multi-line statements).
  • Enormous volume: on a busy system, log_statement = 'all' fills log files faster than almost any workload pattern, consumes I/O bandwidth needed by the database, and makes forensic analysis impractical due to volume.

The correct approach for compliance is pgaudit with a targeted log mask ('write, ddl' minimum) combined with log_connections = on.

Mistake 2: Not setting pgaudit.log_catalog = off

When pgaudit.log includes 'read' or 'all', and pgaudit.log_catalog is left at its default (on), pgaudit logs every access to system catalog tables (pg_catalog.pg_class, pg_catalog.pg_attribute, etc.). A single query planner evaluation may read dozens of catalog entries. The result is hundreds of pgaudit log entries per application query, the vast majority irrelevant to any audit requirement.

# Fix: disable catalog logging in postgresql.conf
pgaudit.log_catalog = off

# Or set at runtime (takes effect on next session)
ALTER SYSTEM SET pgaudit.log_catalog = off;
SELECT pg_reload_conf();
