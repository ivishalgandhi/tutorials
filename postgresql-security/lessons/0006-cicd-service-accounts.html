CI/CD Service Account Patterns

Never Run Migrations as Superuser

1. Why This Matters — The Superuser CI/CD Pipeline

The most common CI/CD setup new PostgreSQL teams create: one database user, superuser or the object owner, used for both running migrations and serving application traffic. It is expedient and it is a significant risk.

A pipeline that runs as superuser or as the application object owner can:

  • Accidentally DROP TABLE if a migration script has a bug or targets the wrong environment
  • Bypass Row-Level Security entirely — superusers are exempt by default
  • Create backdoor superuser accounts in a compromised pipeline step
  • Read any data in any schema, including data the application itself cannot see
  • Expose credentials with far broader scope than the pipeline actually needs
SQL Server Mapping

In SQL Server, running SSDT deployments or Flyway migrations as sa is widely recognised as malpractice. The standard approach is a dedicated deployment login with db_ddladmin membership and explicit object-level grants. PostgreSQL needs exactly the same discipline — but PostgreSQL's permission model requires you to build those boundaries yourself rather than borrowing fixed server roles.

The solution is three purpose-built roles with sharply separated responsibilities. Each role does exactly one job and holds exactly the privileges that job requires.

2. Three Purpose-Built Roles Every Production Environment Needs

graph LR ci["ci_migrate\n(LOGIN, NOINHERIT)"] app["app_user\n(LOGIN, INHERIT)"] ro["readonly_user\n(LOGIN, INHERIT)"] mr["migration_role\n(NOLOGIN)\nDDL on schema"] ar["app_role\n(NOLOGIN)\nDML only"] rr["readonly_role\n(NOLOGIN)\nSELECT only"] ci -->|SET ROLE| mr app -->|inherits| ar ro -->|inherits| rr

Three login roles, each a member of exactly one permission group. The CI runner uses NOINHERIT and must SET ROLE to activate migration privileges.

migration_role — Schema migrations only

This role holds DDL privileges on specific schemas. It has no DML access to application data tables. Its only job is running CREATE TABLE, ALTER TABLE, CREATE INDEX, and similar structural changes.

-- The permission group (cannot connect by itself)
CREATE ROLE migration_role NOLOGIN;
GRANT CONNECT ON DATABASE mydb TO migration_role;

-- Give migration_role ownership of the schema so it can run DDL freely
-- This is the cleanest approach: the schema owner has implicit DDL rights
-- on everything created within it by this role
CREATE SCHEMA app AUTHORIZATION migration_role;
-- Or, if the schema already exists:
ALTER SCHEMA app OWNER TO migration_role;

-- The CI/CD runner login — NOINHERIT is critical (see Mistake 1 below)
CREATE ROLE ci_migrate WITH LOGIN PASSWORD 'from-your-secrets-manager' NOINHERIT;
GRANT migration_role TO ci_migrate;

-- In the CI/CD migration script, the first statement is always:
-- SET ROLE migration_role;
-- All subsequent DDL runs under migration_role identity
Why schema ownership, not just GRANT CREATE

The GRANT CREATE ON SCHEMA app TO migration_role statement allows the role to create new objects. But ALTER TABLE, DROP TABLE, and CREATE INDEX CONCURRENTLY on existing objects require ownership of those objects. When migration_role owns the schema and creates all objects, it automatically owns those objects too. This is why ALTER SCHEMA app OWNER TO migration_role is the cleanest approach: all future objects created by migration_role within that schema are owned by migration_role.

app_role — DML only, used by the running application

This role has SELECT, INSERT, UPDATE, DELETE on application tables. No TRUNCATE, no REFERENCES, no TRIGGER, no DDL. The application cannot accidentally drop a table even if it has a bug.

CREATE ROLE app_role NOLOGIN;
GRANT CONNECT ON DATABASE mydb TO app_role;
GRANT USAGE ON SCHEMA app TO app_role;

-- Grant DML on all currently existing tables
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA app TO app_role;

-- Grant sequence access for SERIAL / IDENTITY columns
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA app TO app_role;

-- Critical: also cover tables created by FUTURE migrations
-- This must run in the context of migration_role (the schema owner)
ALTER DEFAULT PRIVILEGES FOR ROLE migration_role IN SCHEMA app
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_role;

ALTER DEFAULT PRIVILEGES FOR ROLE migration_role IN SCHEMA app
  GRANT USAGE, SELECT ON SEQUENCES TO app_role;

-- The application login
CREATE ROLE app_user WITH LOGIN PASSWORD 'from-your-secrets-manager';
GRANT app_role TO app_user;

readonly_role — SELECT only for reporting and dashboards

CREATE ROLE readonly_role NOLOGIN;
GRANT CONNECT ON DATABASE mydb TO readonly_role;
GRANT USAGE ON SCHEMA app TO readonly_role;

-- Existing tables
GRANT SELECT ON ALL TABLES IN SCHEMA app TO readonly_role;

-- Future tables created by migration_role
ALTER DEFAULT PRIVILEGES FOR ROLE migration_role IN SCHEMA app
  GRANT SELECT ON TABLES TO readonly_role;

-- The reporting login (e.g. for a BI tool or read replica connection)
CREATE ROLE report_user WITH LOGIN PASSWORD 'from-your-secrets-manager';
GRANT readonly_role TO report_user;

3. ALTER DEFAULT PRIVILEGES — The Statement Most DBAs Miss

ALTER DEFAULT PRIVILEGES tells PostgreSQL: "whenever this role creates a new object of this type in this schema, automatically apply these grants." Without it, every new table migration creates a table that app_role cannot access, causing runtime errors in the application the moment it tries to touch a newly migrated table.

-- Full example: set up default privileges so migration_role's future tables
-- are automatically accessible to app_role and readonly_role

-- Run these as a superuser or as migration_role itself
ALTER DEFAULT PRIVILEGES FOR ROLE migration_role IN SCHEMA app
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_role;

ALTER DEFAULT PRIVILEGES FOR ROLE migration_role IN SCHEMA app
  GRANT USAGE, SELECT ON SEQUENCES TO app_role;

ALTER DEFAULT PRIVILEGES FOR ROLE migration_role IN SCHEMA app
  GRANT SELECT ON TABLES TO readonly_role;

-- Verify default privileges are set:
SELECT pg_catalog.pg_get_userbyid(d.defaclrole) AS role,
       d.defaclnamespace::regnamespace              AS schema,
       d.defaclobjtype                              AS object_type,
       pg_catalog.array_to_string(d.defaclacl, ', ') AS acl
FROM   pg_catalog.pg_default_acl d
ORDER BY role, schema, object_type;
SQL Server Mapping

SQL Server has no direct equivalent to ALTER DEFAULT PRIVILEGES. In SQL Server, you handle this by granting permissions at the schema level (GRANT SELECT ON SCHEMA::dbo TO role), which covers all current and future objects in that schema. PostgreSQL's schema-level GRANT does not work this way — it grants only on currently existing objects, which is why ALTER DEFAULT PRIVILEGES is essential and surprising to SQL Server DBAs.

4. Password Rotation Without Downtime

Service account passwords must rotate regularly. The trick is to add the new credential before removing the old one, allowing the CI/CD pipeline to be updated between those two operations.

-- Step 1: Create the new CI login with the new password
CREATE ROLE ci_migrate_v2 WITH LOGIN PASSWORD 'new-password-from-vault' NOINHERIT;
GRANT migration_role TO ci_migrate_v2;

-- Step 2: Update CI/CD pipeline configuration to use ci_migrate_v2
-- (deploy the pipeline change, verify it works)

-- Step 3: After confirming the new login works, remove the old one
DROP ROLE ci_migrate;  -- old login gone; migration_role group role is untouched

The key insight: you are rotating the login role (the authentication credential), not the permission group role. The migration_role group role keeps its grants intact throughout the rotation. This is a clean separation that SQL Server's login/user distinction also supports.

SQL Server Mapping

SQL Server equivalent: CREATE LOGIN new_ci WITH PASSWORD = '...', then CREATE USER new_ci FOR LOGIN new_ci, then sp_addrolemember 'deployment_role', 'new_ci', update the pipeline, then DROP LOGIN old_ci. The PostgreSQL approach is identical in structure; it just uses CREATE ROLE ... WITH LOGIN instead of the separate CREATE LOGIN + CREATE USER two-step.

5. Scoping CI/CD Roles in pg_hba.conf

Granting a role the right privileges is necessary but not sufficient. You also need to ensure that role can only connect from the expected source. pg_hba.conf is the host-based authentication configuration file — it controls which roles can connect from which addresses using which authentication methods.

# /etc/postgresql/16/main/pg_hba.conf
#
# TYPE   DATABASE  USER          ADDRESS          METHOD
#
# CI/CD migration runner — only from the specific CI server IP, SSL required
hostssl  mydb      ci_migrate    10.0.1.50/32     scram-sha-256

# Application runtime — from the app subnet, SSL required
hostssl  mydb      app_user      10.0.2.0/24      scram-sha-256

# Reporting tool — from the BI server, SSL required
hostssl  mydb      report_user   10.0.3.10/32     scram-sha-256

# Reject everything else (explicit catch-all)
host     all       all           0.0.0.0/0        reject

After editing pg_hba.conf, reload — no restart needed:

-- From psql as superuser:
SELECT pg_reload_conf();

-- Or from the OS shell:
-- pg_ctl reload -D /var/lib/postgresql/data
pg_hba.conf Rule Evaluation

Rules are evaluated top to bottom; the first matching rule wins. If the matching rule's method is reject, the connection is denied immediately — no further rules are checked. Place the most specific rules at the top and the catch-all reject at the bottom.

6. Revoking CONNECT from PUBLIC

By default, when you create a database in PostgreSQL, the PUBLIC pseudo-role is granted CONNECT on it. This means any valid database role can connect to any database unless you explicitly revoke this. SQL Server DBAs find this surprising — SQL Server requires an explicit login-to-database mapping (CREATE USER) before any connection is possible.

-- Remove the default CONNECT grant from PUBLIC
-- (Run this immediately after CREATE DATABASE)
REVOKE CONNECT ON DATABASE mydb FROM PUBLIC;

-- Now explicitly grant CONNECT only to roles that need it
GRANT CONNECT ON DATABASE mydb TO migration_role;
GRANT CONNECT ON DATABASE mydb TO app_role;
GRANT CONNECT ON DATABASE mydb TO readonly_role;

-- Verify: which roles have CONNECT on mydb?
SELECT datacl
FROM   pg_database
WHERE  datname = 'mydb';

Without revoking from PUBLIC, any new role created in the cluster can attempt a connection to your production database. Combined with a weak password or an overly broad pg_hba.conf rule, this creates an unnecessary attack surface.

Security Critical

REVOKE CONNECT ON DATABASE mydb FROM PUBLIC should be step one in every new production database setup script, immediately after CREATE DATABASE. Pair it with REVOKE ALL ON SCHEMA public FROM PUBLIC and REVOKE CREATE ON SCHEMA public FROM PUBLIC (covered in Lesson 8) to close the three default open doors PostgreSQL ships with.

7. Common Mistake 1 — Using the Object Owner as the Runtime Role

Top Mistake — Coming from SQL Server

Using the same role that created all the schema objects as the application's runtime role. In SQL Server, this is less common because the login/user/role separation makes it awkward. In PostgreSQL, many tutorials show CREATE DATABASE mydb; \c mydb; CREATE TABLE ... all under the same superuser or owner role, and that role becomes the application credential by default.

The problem: in PostgreSQL, an object's owner has implicit ALL PRIVILEGES on that object — including DROP TABLE, TRUNCATE, and full DDL. If your application connects as the schema owner, a bug in the application code (or a compromised dependency) can issue DROP TABLE orders and it will succeed.

The fix: always separate the role that creates objects (migration_role) from the role that reads and writes data (app_role). The application never connects as a role that owns the schema.

8. Common Mistake 2 — Not Setting ALTER DEFAULT PRIVILEGES

Top Mistake — Coming from SQL Server

Forgetting ALTER DEFAULT PRIVILEGES and then being confused when the application fails after a migration. The failure mode is subtle: the migration runs successfully, the new table exists, but the first application request that touches the new table returns "permission denied for table orders_v2" — even though app_role has SELECT, INSERT, UPDATE, DELETE on all tables in the schema.

The reason: GRANT ... ON ALL TABLES IN SCHEMA app applies to tables that exist at the time the statement runs. The new table was created after that statement. ALTER DEFAULT PRIVILEGES is the mechanism that auto-applies grants to future objects.

Checklist after adding a new schema to a database:

  1. Run GRANT ... ON ALL TABLES IN SCHEMA new_schema TO app_role for existing objects
  2. Run ALTER DEFAULT PRIVILEGES FOR ROLE migration_role IN SCHEMA new_schema GRANT ... TO app_role for future objects
  3. Verify with: SELECT * FROM information_schema.role_table_grants WHERE table_schema = 'new_schema'
