The PostgreSQL Security Hardening Checklist

This is the synthesis lesson. No new concepts are introduced here. Everything in this checklist was covered in detail in Lessons 1 through 10. The goal now is to take all of those individual mechanisms — pg_hba.conf rules, role design, schema security, RLS, SSL/TLS, pgaudit — and apply them as a coherent, prioritised hardening process against a real PostgreSQL instance.

Work through each category in order. For every item: run the verification query, record what you find, and address any gaps before moving to the next category. The verification queries in Section 3 will give you a fast snapshot of the current posture of any PostgreSQL instance you administer.

How to Use This Checklist

This checklist is based on the CIS PostgreSQL Benchmark, adapted and extended with SQL Server migration context. Each item includes: what to check, what command to run, and a note for SQL Server DBAs where the concept differs from SQL Server defaults or mental models.

Category 1: Authentication (pg_hba.conf)

Authentication is the front door. Every other security control is meaningless if the authentication configuration allows unauthorised connections.

1.1 — No trust entries except local Unix socket for postgres

Check: SHOW hba_file; then inspect the file, or query SELECT * FROM pg_hba_file_rules WHERE auth_method = 'trust';

Acceptable: local all postgres peer — OS identity trust on the local socket only.

Not acceptable: Any host ... trust entry, or local all all trust.

SQL Server DBA Note

There is no SQL Server equivalent to trust authentication — even Windows Authentication requires a valid domain identity. Trust is completely absent from SQL Server's model. Treat any trust entry on a TCP connection as a critical finding requiring immediate remediation.

1.2 — All host connections use scram-sha-256, not md5

Check: SELECT * FROM pg_hba_file_rules WHERE auth_method = 'md5';

Fix: Replace md5 with scram-sha-256 in pg_hba.conf. Also set password_encryption = 'scram-sha-256' in postgresql.conf and require all users to reset their passwords so stored hashes are re-generated as SCRAM hashes.

1.3 — All host connections use hostssl, not host

Check: SELECT * FROM pg_hba_file_rules WHERE type = 'host';

Fix: Change each host entry to hostssl unless it is a deliberate catch-all reject rule. Plain host allows unencrypted connections from clients that do not negotiate TLS.

1.4 — LDAP or certificate authentication for AD-integrated environments

Check: If your organisation uses Active Directory for identity, application service accounts should authenticate via ldap (AD bind) or cert (certificate), not password-in-connection-string. Review pg_hba.conf for any scram-sha-256 entries that could be replaced with ldap for human users.

SQL Server DBA Note

In SQL Server, Windows Authentication is the strongly preferred method precisely because it delegates identity to Active Directory. PostgreSQL does not integrate with Kerberos by default (that is the gss method), but LDAP bind authentication gives you the same AD-backed identity verification for password flows. Covered in depth in Lesson 2.

1.5 — Catch-all reject rule at the bottom

Check: The last line of pg_hba.conf should be:

host  all  all  0.0.0.0/0  reject

Without this, any connection not matched by earlier rules falls through with no rule matching. PostgreSQL treats no-match as a rejection, but explicit intent is clearer, auditable, and immune to future rule reordering accidents.

Category 2: Network Exposure

2.1 — listen_addresses scoped to specific IPs, not wildcard

Check: SHOW listen_addresses;

Acceptable: A specific IP address ('10.0.1.20') or 'localhost' for single-server deployments where all clients are local.

Requires justification: '*' (all interfaces). In cloud environments with a public interface, listen_addresses = '*' combined with security groups is accepted practice — but document the decision and verify the security group is correctly locked down.

SQL Server DBA Note

The SQL Server equivalent is TCP/IP protocol binding in Configuration Manager, where you can enable or disable listening on each network adapter and specify which IP:port combinations are active. PostgreSQL's listen_addresses is less granular (all ports, specific IPs) but serves the same purpose.

2.2 — PostgreSQL port not directly reachable from the internet

Check: Verify firewall / security group rules. Port 5432 should be reachable only from application server IPs and DBA management hosts. An internet-facing PostgreSQL port is the most common entry point for automated credential-stuffing attacks.

2.3 — SSL on with a CA-signed certificate (not self-signed in production)

Check: SHOW ssl; and verify the certificate in ssl_cert_file is signed by a trusted CA. Self-signed certificates are acceptable only in development environments.

2.4 — sslmode=verify-full in all application connection strings

Check: Review application configuration. Connection strings using sslmode=require, sslmode=prefer, or omitting sslmode entirely are not verifying server identity. See Lesson 9, Section 6.

Category 3: Roles and Privileges

3.1 — No application connects as a superuser

Check:

SELECT rolname FROM pg_roles WHERE rolsuper = true;

Cross-reference with pg_hba.conf. Any superuser listed there with a hostssl rule is a finding: application-tier connections must use dedicated, least-privilege roles.

3.2 — Role hierarchy in place: readonly / app / migration tiers

Check: SELECT * FROM pg_auth_members; — verify a group role hierarchy exists and application roles inherit only what they need. Covered in Lessons 3 and 5.

3.3 — REVOKE CONNECT ON DATABASE FROM PUBLIC

Check:

SELECT datacl FROM pg_database WHERE datname = 'your_database';
-- If datacl is NULL, the PUBLIC grant is still in effect

Fix:

REVOKE CONNECT ON DATABASE your_database FROM PUBLIC;
GRANT CONNECT ON DATABASE your_database TO app_role, readonly_role, dba_role;
SQL Server DBA Note

In SQL Server, a login without an explicit database mapping cannot connect to that database (unless it maps to guest, which should be disabled). PostgreSQL does the opposite: by default, every role has the CONNECT privilege on every database. You must explicitly revoke it and then grant back to only the roles that need it. This is a common shock for SQL Server DBAs who expect to grant access rather than revoke defaults.

3.4 — REVOKE CREATE ON SCHEMA public FROM PUBLIC (pre-PostgreSQL 15)

Check: SELECT has_schema_privilege('public', 'public', 'create');

If this returns true and you are on PostgreSQL 14 or earlier, any connected role can create objects in the public schema. PostgreSQL 15 changed the default to revoke this. For pre-15 instances:

REVOKE CREATE ON SCHEMA public FROM PUBLIC;

3.5 — ALTER DEFAULT PRIVILEGES set for new objects in app schemas

Check:

SELECT defaclrole::regrole, defaclobjtype, defaclacl
FROM   pg_default_acl;

Without default privileges configured, tables created by a migration role are not automatically accessible to the application role. This must be set explicitly per schema per owner role. Covered in Lesson 5.

SQL Server DBA Note

SQL Server DBAs are accustomed to schema-level permissions cascading to new objects automatically: granting SELECT on a schema grants SELECT on all current and future tables in that schema. PostgreSQL does not do this. Each new table starts with only the creator's ownership. ALTER DEFAULT PRIVILEGES is the closest equivalent — it pre-declares the grants that should be applied to each new object as it is created.

3.6 — Column-level privileges for sensitive columns

Check: Identify tables containing PII (SSN, salary, date of birth, payment card data). Verify that the reporting role cannot SELECT those columns:

SELECT col_description(attrelid, attnum), attname
FROM   pg_attribute
WHERE  attrelid = 'employees'::regclass
AND    attnum > 0;

Fix: REVOKE SELECT ON employees FROM readonly_role; then GRANT SELECT (id, department, hire_date) ON employees TO readonly_role; — explicitly omit sensitive columns.

Category 4: Schema and Search Path

4.1 — All production objects live in explicit schemas, not public

Check:

SELECT tablename FROM pg_tables WHERE schemaname = 'public'
AND tablename NOT IN (SELECT tablename FROM pg_tables WHERE schemaname = 'information_schema');

Application tables in the public schema cannot be cleanly isolated with schema-level permissions. Create an application schema (app, finance, etc.) and move objects there.

4.2 — search_path locked per role

Check:

SELECT rolname, rolconfig FROM pg_roles WHERE rolconfig IS NOT NULL;

Fix for application roles:

ALTER ROLE app_user SET search_path = app;
-- pg_temp is intentionally excluded to prevent temp-schema hijacking
SQL Server DBA Note

SQL Server has DEFAULT SCHEMA per user, which controls unqualified name resolution. PostgreSQL's search_path is more powerful but also more dangerous: if it includes public as a writable schema, a low-privilege user could create a function named lower() in public and have it called instead of the built-in. Locking search_path to the application schema eliminates this class of attack.

4.3 — Every SECURITY DEFINER function has SET search_path

Check:

SELECT proname, pronamespace::regnamespace
FROM   pg_proc
WHERE  prosecdef = true
AND    proconfig IS NULL;
-- Any row returned is a SECURITY DEFINER function without a fixed search_path

Fix:

CREATE OR REPLACE FUNCTION my_privileged_func()
RETURNS void
SECURITY DEFINER
SET search_path = app, pg_temp   -- lock before any user-controlled objects
LANGUAGE plpgsql AS $$
BEGIN
  -- function body
END;
$$;
SQL Server DBA Note

SQL Server's EXECUTE AS OWNER (the equivalent of SECURITY DEFINER) is less exploitable because SQL Server's name resolution is schema-bound by default, and the DEFAULT SCHEMA setting for the impersonated owner applies. PostgreSQL's SECURITY DEFINER functions inherit the caller's search_path unless you explicitly freeze it with SET search_path inside the function definition. A missing SET search_path on a SECURITY DEFINER function is a privilege escalation vector.

4.4 — REVOKE EXECUTE ON ALL FUNCTIONS IN SCHEMA public FROM PUBLIC

Check: By default in pre-15 PostgreSQL, all functions in the public schema are executable by any role. Even after cleaning up the public schema, this should be explicitly revoked:

REVOKE EXECUTE ON ALL FUNCTIONS IN SCHEMA public FROM PUBLIC;

Category 5: Row-Level Security

5.1 — Multi-tenant tables have RLS enabled

Check:

SELECT schemaname, tablename, rowsecurity
FROM   pg_tables
WHERE  schemaname NOT IN ('pg_catalog', 'information_schema')
AND    rowsecurity = false
ORDER BY schemaname, tablename;

Any tenant-partitioned table appearing in this result is returning all-tenant data to any role that can SELECT from it.

5.2 — Every RLS policy has both USING and WITH CHECK clauses

Check:

SELECT tablename, policyname, cmd, qual, with_check
FROM   pg_policies
WHERE  with_check IS NULL
AND    cmd IN ('ALL', 'INSERT', 'UPDATE');

A policy with a USING clause but no WITH CHECK allows a user to INSERT or UPDATE a row that their USING clause would not let them read back — potentially writing data into another tenant's partition. Covered in Lesson 7.

SQL Server DBA Note

SQL Server separates this cleanly: the FILTER PREDICATE controls reads and the BLOCK PREDICATE controls writes. PostgreSQL's USING maps to FILTER PREDICATE and WITH CHECK maps to the INSERT/UPDATE BLOCK PREDICATE. The common mistake is writing only the USING clause (read filter) and forgetting that INSERT and UPDATE paths also need a write filter.

5.3 — BYPASSRLS granted only to specific admin roles, not application roles

Check:

SELECT rolname FROM pg_roles WHERE rolbypassrls = true;

Superusers always bypass RLS. Any non-superuser with BYPASSRLS should be a named, documented administrative role — never an application role.

5.4 — RLS tested with SET ROLE and current_setting

Verify by running:

SET ROLE app_user;
SELECT set_config('app.tenant_id', '42', true);
SELECT count(*) FROM orders;   -- should return only tenant 42's rows
RESET ROLE;

Do not rely solely on application-layer testing. Test RLS at the database level by impersonating the application role and verifying the row counts are tenant-scoped.

Category 6: Auditing

6.1 — pgaudit installed and loaded

Check:

SHOW shared_preload_libraries;
-- should contain 'pgaudit'

SELECT name, setting FROM pg_settings WHERE name LIKE 'pgaudit%';
-- if pgaudit is not loaded, these GUCs will not appear

6.2 — pgaudit.log includes at minimum 'write, ddl'

Check: SHOW pgaudit.log;

For PCI-DSS or HIPAA compliance, 'read' must also be included to audit SELECT on cardholder data or ePHI tables (either via session audit with 'all' or via object audit on specific tables).

6.3 — log_connections and log_disconnections are on

Check:

SHOW log_connections;
SHOW log_disconnections;

These capture every authentication event including failures — the equivalent of SQL Server's FAILED_LOGIN_GROUP and SUCCESSFUL_LOGIN_GROUP.

6.4 — log_line_prefix includes timestamp, PID, user, and database

Check: SHOW log_line_prefix;

Required minimum: '%m [%p] %q%u@%d '

Without the user and database in the prefix, pgaudit entries have no context identifying who ran the audited statement.

6.5 — Audit logs shipped off the database server

Check: Verify that PostgreSQL logs are ingested by a SIEM, shipped to a centralised log storage system, or replicated off the server in near-real-time. Logs stored only on the database server can be deleted or tampered with by a compromised superuser — defeating the purpose of the audit trail.

Category 7: Ongoing Maintenance

7.1 — Password rotation procedure documented

PostgreSQL has no built-in password rotation automation. Document and test the rotation procedure. The safe pattern (avoids downtime):

-- Step 1: create the new role with a new password
CREATE ROLE app_user_v2 WITH LOGIN PASSWORD 'new-strong-password'
  NOSUPERUSER NOCREATEDB NOCREATEROLE;

-- Step 2: grant it all the same privileges as the old role
GRANT app_role TO app_user_v2;

-- Step 3: update application connection string to use new role
-- Step 4: verify connections succeed; monitor error logs

-- Step 5: drop the old role once all connections have migrated
DROP ROLE app_user_old;

7.2 — pg_hba.conf reviewed when CI/CD IP ranges change

CI/CD pipelines (GitHub Actions, GitLab CI, Jenkins) often run from ephemeral IP ranges that change without notice. If your pg_hba.conf contains specific CI runner IPs, these become stale. Either use a stable egress IP (NAT gateway), a VPN, or a service-account-based certificate authentication that is not IP-bound.

7.3 — Superuser list reviewed quarterly

Check:

SELECT rolname, rolcreatedb, rolcreaterole, rolcanlogin, rolvaliduntil
FROM   pg_roles
WHERE  rolsuper = true;

Superusers bypass all security controls. The list should be minimal — typically only the postgres system role and one named DBA role. Service accounts, application roles, and reporting roles should never appear here.

7.4 — SSL certificate expiry monitored

Check from the OS:

# Check expiry date of the PostgreSQL server certificate
openssl x509 -in /var/lib/postgresql/data/server.crt -noout -enddate

# Output example:
# notAfter=Jan 15 14:23:01 2025 GMT

Set up a monitoring alert at least 30 days before expiry. PostgreSQL will continue to run with an expired certificate (clients with sslmode=verify-full will refuse to connect), making certificate expiry an availability incident, not just a security finding.

7.5 — pgaudit version compatibility verified after each PostgreSQL upgrade

pgaudit is compiled against a specific PostgreSQL major version. After a major version upgrade (e.g., PG 15 to PG 16), the pgaudit package must also be upgraded. After a minor version upgrade, check the pgaudit release notes to confirm compatibility. Mismatched versions can cause PostgreSQL to fail to start.

The 5 Items SQL Server DBAs Most Commonly Skip

Based on the migration patterns covered across this course, these are the five hardening steps that SQL Server DBAs consistently overlook when hardening a PostgreSQL instance for the first time:

Item Why it gets skipped Why it matters
ALTER DEFAULT PRIVILEGES SQL Server DBAs expect schema-level grants to cover future objects automatically. PostgreSQL does not work this way. Without it, tables created by the migration role are invisible to the application role until someone notices and grants manually.
REVOKE CONNECT ON DATABASE FROM PUBLIC SQL Server DBAs instinctively grant access; they do not think to revoke defaults they did not set. Every role — including newly created roles with no other privileges — can connect to every database until this is revoked.
search_path locking per role SQL Server's DEFAULT SCHEMA is set at role creation and does not cause security issues if left at dbo. PostgreSQL's search_path can be manipulated by objects in a writable public schema. An unlocked search_path enables function/operator hijacking attacks via the public schema.
SET search_path on SECURITY DEFINER functions SQL Server's EXECUTE AS OWNER is less exploitable because name resolution uses the owner's default schema. PostgreSQL's SECURITY DEFINER uses the caller's search_path unless frozen. A SECURITY DEFINER function without SET search_path is a privilege escalation vector: any user who can create objects in a schema on the caller's path can intercept the function's object lookups.
WITH CHECK in RLS policies SQL Server separates FILTER PREDICATE (reads) and BLOCK PREDICATE (writes) explicitly. PostgreSQL uses USING and WITH CHECK in the same CREATE POLICY statement, and WITH CHECK is optional — it is easy to forget. A policy with only USING allows INSERT/UPDATE operations that write to rows the user cannot read back, potentially leaking data across tenant boundaries.

Quick Verification Queries: Security Snapshot for Any Instance

Run these seven queries on any PostgreSQL instance to get an immediate security posture snapshot. They are safe to run on a live production instance — all are read-only queries against system catalog views.

-- 1. Find all superusers
--    Expected: only 'postgres' and your named DBA admin role
--    Finding: any service account or application role is a critical issue
SELECT rolname
FROM   pg_roles
WHERE  rolsuper = true;


-- 2. Find login roles with no password expiry
--    Expected: roles have VALID UNTIL set, or you have a documented rotation schedule
--    Finding: NULL means the password never expires regardless of age
SELECT rolname, rolvaliduntil
FROM   pg_roles
WHERE  rolcanlogin = true
AND    rolvaliduntil IS NULL;


-- 3. Check whether SSL is enabled at the server level
--    Expected: 'on'
--    Finding: 'off' means all connections are unencrypted
SHOW ssl;


-- 4. Check the password hashing algorithm in use
--    Expected: 'scram-sha-256'
--    Finding: 'md5' means new passwords are stored as weak MD5 hashes
SHOW password_encryption;


-- 5. Find application tables with RLS disabled
--    Expected: empty result if all tenant tables have RLS enabled
--    Finding: any tenant-partitioned table in this list returns all-tenant data
SELECT schemaname, tablename
FROM   pg_tables
WHERE  schemaname NOT IN ('pg_catalog', 'information_schema')
AND    rowsecurity = false
ORDER BY schemaname, tablename;


-- 6. Find SECURITY DEFINER functions without a frozen search_path
--    Expected: empty result
--    Finding: any function here is a potential privilege escalation vector
SELECT proname, pronamespace::regnamespace AS schema
FROM   pg_proc
WHERE  prosecdef = true
AND    proconfig IS NULL;


-- 7. Check whether PUBLIC still has CREATE on the public schema
--    Expected: 'f' (false) on PostgreSQL 15+ or after manual revoke
--    Finding: 't' (true) means any connected role can create objects in public
SELECT has_schema_privilege('public', 'public', 'create');
Your Homework

Run each of these seven queries against your PostgreSQL instance right now. Record the results. For every gap you find — a superuser that should not exist, tables with RLS disabled, an md5 password_encryption setting — use the checklist items in this lesson to plan the remediation. This is the security assessment skill that matters most on the job.
