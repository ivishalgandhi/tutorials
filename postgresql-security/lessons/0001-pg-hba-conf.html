pg_hba.conf — How PostgreSQL Decides Who Gets In

Before a password is ever checked, before roles are evaluated, before a single query runs — PostgreSQL reads one file to decide whether to allow a connection at all. That file is pg_hba.conf. Understanding it is not optional. It is the front door, the bouncer, and the access policy all in one.

1. What pg_hba.conf Is — and the SQL Server Analogy

In SQL Server, authentication mode (Windows Authentication vs. SQL Authentication) is a single server-level toggle set in Server Properties → Security. You flip it once and it applies to every connection. Individual logins have their own method baked in at creation time.

SQL Server Mental Model

SQL Server's authentication mode is a binary switch: Mixed Mode or Windows-only. There is no per-database, per-user, per-IP authentication routing. Once the mode is set, every connection goes through the same gate.

PostgreSQL takes a fundamentally different approach. pg_hba.conf (Host-Based Authentication configuration) is a rule engine. Every incoming connection is evaluated against an ordered list of rules. The rule specifies not just what authentication method to use, but when to use it — based on connection type, database name, username, and client IP address simultaneously. The official documentation is at postgresql.org/docs/current/auth-pg-hba-conf.html.

Think of it this way: SQL Server decides how to authenticate globally; pg_hba.conf decides whether and how to authenticate for each specific combination of who, from where, and to what.

2. File Location

The file lives in the PostgreSQL data directory. You can always find it from inside a running instance — no filesystem guessing required:

-- Run this inside psql as any connected user
SHOW hba_file;

-- Typical output on Linux (Debian/Ubuntu with pg 16):
--  /etc/postgresql/16/main/pg_hba.conf

-- Typical output on a compiled-from-source install:
--  /var/lib/postgresql/data/pg_hba.conf

-- Also discoverable via:
SELECT setting FROM pg_settings WHERE name = 'hba_file';
Info

On package-managed installations (apt, yum, Homebrew) the data directory and config directory are often separate. SHOW hba_file always gives you the authoritative path regardless of distribution layout.

3. Rule Structure: Five Fields, One Decision

Each non-comment line in pg_hba.conf follows this exact column order:

TYPE  DATABASE  USER  ADDRESS  METHOD  [OPTIONS]
Field What it means Common values
TYPE How the client is connecting — local Unix socket or TCP/IP, with or without TLS local, host, hostssl, hostnossl, hostgssenc
DATABASE Which database(s) this rule applies to all, a database name, a comma-separated list, @file to read from a file, replication
USER Which role(s) this rule applies to all, a role name, +group_role for group membership, a comma-separated list
ADDRESS Client IP address or CIDR block (omitted for local type) 127.0.0.1/32, 10.0.0.0/8, 0.0.0.0/0, ::1/128
METHOD Authentication method to apply when all prior fields match scram-sha-256, md5, peer, trust, reject, ldap, cert
OPTIONS Method-specific parameters (optional) ldapserver=, clientcert=verify-full, etc.
Key Concept

The ADDRESS field only appears for host, hostssl, and hostnossl entries. For local entries (Unix domain sockets), the field is absent entirely — local connections have no IP address to match against.

4. Rule Evaluation: First Match Wins

This is the single most important behaviour to internalise. PostgreSQL scans the rules from top to bottom and stops at the first rule whose TYPE, DATABASE, USER, and ADDRESS all match the incoming connection. Whatever METHOD that rule specifies is used — unconditionally. No further rules are consulted.

Critical Difference from SQL Server

SQL Server has no equivalent ordered-rule engine for authentication. There is no concept of "the first matching rule wins" because there is only one authentication mode. In pg_hba.conf, rule order matters completely. A broad permissive rule placed above a narrow restrictive rule will silently absorb every connection the narrow rule was meant to handle. This is the most common source of security misconfiguration by DBAs coming from SQL Server.

flowchart TD A([Incoming connection]) --> B{Rule 1 matches?} B -- Yes --> C[Apply Method 1] B -- No --> D{Rule 2 matches?} D -- Yes --> E[Apply Method 2] D -- No --> F{Rule N matches?} F -- Yes --> G[Apply Method N] F -- No --> H([Connection rejected\nno rule matched]) C --> I([Auth succeeds or fails\nbased on method]) E --> I G --> I

Figure 1. pg_hba.conf evaluation: the first matching rule terminates the search. Rules below it are never reached.

The practical implication: put your most specific rules first. Place restrictive reject rules for sensitive users before any broad all rules. Place hostssl rules above plain host rules if you intend to force TLS.

5. Authentication Methods Reference

Method How it works When to use SQL Server analogy
trust No credentials required. Anyone who can reach this socket/IP is allowed in. Local dev only, never production. Automated tests with throwaway databases. Closest to a SQL Server login with a blank password and no expiry, but worse — no password prompt at all.
reject Unconditionally denies the connection with an error. No auth attempted. Blocking specific users or IP ranges explicitly. Placing before a broad all rule. A SQL Server login that is disabled — connection attempt is refused at the server gate.
md5 Challenge-response using MD5 hash of password. Password never sent in cleartext. Legacy clients that cannot support scram-sha-256. Not recommended for new deployments. SQL Authentication with password hashing (SQL Server uses its own algorithm; md5 is weaker).
scram-sha-256 SCRAM-SHA-256 mutual authentication. Verifies both client and server. Resistant to replay attacks and offline dictionary attacks. Default for all password-based TCP/IP connections on modern PostgreSQL (14+). SQL Authentication with the most secure password handling — closer to SQL Server's NTLMv2 strength, but SCRAM is standardised.
peer Asks the OS for the client's Unix username. Matches against the requested PostgreSQL role name. Local socket connections on Linux/macOS for system-owned roles. The postgres OS user connecting as the postgres role. Windows Authentication on SQL Server — OS identity is trusted without a password. Only valid for local socket connections.
ident Queries an RFC 1413 ident server on the client machine for the OS username. Legacy internal networks with a trusted ident daemon. Rarely used in modern deployments. A weaker, network-based version of peer — trusts a remote daemon's assertion of identity.
ldap Authenticates by binding to an LDAP server (e.g., Active Directory) with the user's credentials. Centralised identity with Active Directory. Covered in depth in Lesson 2. Windows Authentication / Active Directory integration in SQL Server — same AD credential store, different protocol path.
radius Delegates authentication to a RADIUS server. Enables MFA integration. Environments with existing RADIUS infrastructure or MFA appliances. No direct SQL Server equivalent — SQL Server MFA relies on Azure AD / AAD integration, not RADIUS.
cert Requires the client to present a TLS client certificate signed by a trusted CA. The CN of the certificate is mapped to a PostgreSQL role. Highest-assurance service-to-service connections. Machine identity, not human identity. SQL Server certificate-based authentication using client certificates — uncommon in practice but structurally identical.

One-line syntax examples for each method (to be embedded directly in pg_hba.conf lines):

# trust — local socket only, throwaway dev database
local   devdb     devuser                   trust

# reject — block a specific user from TCP entirely
host    all       blocked_user  0.0.0.0/0   reject

# md5 — legacy client compatibility
host    appdb     legacyapp     10.0.1.5/32 md5

# scram-sha-256 — modern password auth over TLS
hostssl appdb     appuser       10.0.1.0/24 scram-sha-256

# peer — OS user postgres connecting via Unix socket
local   all       postgres                  peer

# ident — legacy network identity (avoid on new deployments)
host    all       all           10.0.0.0/8  ident

# ldap — Active Directory (see Lesson 2 for full options)
hostssl all       all           10.0.1.0/24 ldap ldapserver=ad.example.com ...

# radius — MFA via RADIUS appliance
hostssl all       all           0.0.0.0/0   radius radiusservers=mfa.example.com radiussecrets=sharedsecret

# cert — mutual TLS, CN must match PostgreSQL role name
hostssl all       all           0.0.0.0/0   cert

6. A Production-Ready pg_hba.conf Example

The following configuration covers a typical three-tier application: a superuser for administrative access via the local socket, an application user from the app server subnet over TLS, a read-only reporting user, and an explicit deny-all backstop. The rules are ordered narrowest to broadest.

# ─────────────────────────────────────────────────────────────────
# pg_hba.conf — production example
# Rule order: most specific FIRST, catch-all LAST
# ─────────────────────────────────────────────────────────────────

# 1. Local Unix socket: postgres OS user connects as postgres role (peer)
#    No password required — OS identity is sufficient.
#    This is how pg_dump, vacuumdb, and pg_upgrade typically connect.
local    all          postgres                              peer

# 2. Local Unix socket: application owner connects locally without password
#    Use only during maintenance; comment out in normal operation.
local    appdb        app_owner                             peer

# 3. App server — hostssl enforces TLS; scram-sha-256 for password auth
#    Replace 10.0.1.50/32 with your actual application server IP.
hostssl  appdb        app_user       10.0.1.50/32          scram-sha-256

# 4. Read-only reporting user — allowed from the analytics subnet only
hostssl  appdb        readonly_user  10.0.2.0/24           scram-sha-256

# 5. DBA team — admin database access from the DBA management VLAN
hostssl  all          dba_role       10.0.100.0/24         scram-sha-256

# 6. Replication user — strictly from the standby server IP
hostssl  replication  replicator     10.0.1.60/32          scram-sha-256

# 7. Localhost TCP (IPv4) — reject everything not matched above
#    Explicit reject is cleaner than relying on the implicit fallthrough.
host     all          all            127.0.0.1/32          reject

# 8. All other TCP connections — reject; no implicit trust
#    This is the backstop. Any connection not matched above is refused.
host     all          all            0.0.0.0/0             reject
Security Note

Notice that every TCP rule uses hostssl, not plain host. This means PostgreSQL will refuse the connection if the client does not negotiate TLS. A plain host rule accepts connections with or without TLS — which means credentials could travel in cleartext over the network even if you intend TLS to be required. Force the intent explicitly.

7. Common Misconfigurations

Misconfiguration A: trust on 0.0.0.0/0

This appears in default installations on some platforms and in hastily-assembled containers. It looks like this:

# DANGEROUS — never do this in production
host  all  all  0.0.0.0/0  trust

The blast radius: any client that can reach the PostgreSQL TCP port — from anywhere on the internet or your network — can connect as any user, including the postgres superuser, with no password. There is no credential to brute-force. No log entry for a failed attempt. No rate limiting. This is a complete bypass of the entire authentication system.

Danger

Shodan and similar internet scanning services continuously sweep for PostgreSQL instances with trust authentication on port 5432. Databases exposed this way are typically found and exfiltrated within hours. If you inherited an instance, run SHOW hba_file and inspect the file immediately.

Misconfiguration B: md5 instead of scram-sha-256

MD5 as a password hash has been broken for practical purposes for over a decade. When PostgreSQL uses md5 authentication, it stores an MD5 hash of the password in pg_authid. If an attacker obtains a dump of pg_authid (which any superuser can read), those MD5 hashes can be attacked offline with commodity GPU hardware in hours. SCRAM-SHA-256 uses a salted, iterated hash that is vastly more resistant to offline dictionary attacks.

SQL Server Comparison

SQL Server has used salted SHA-512 hashing for SQL Authentication passwords since SQL Server 2012. If you have been relying on SQL Server's password hashing strength, be aware that PostgreSQL's md5 mode offers significantly weaker protection. Use scram-sha-256 — it is the PostgreSQL equivalent of SQL Server's modern password handling. See Lesson 3 for how to enforce it at the server level via password_encryption.

Misconfiguration C: The Ordering Trap

This is the most subtle and dangerous mistake. Consider this configuration:

# Intended: app_user uses scram-sha-256, but postgres is rejected from TCP
host  all          all       10.0.0.0/8   scram-sha-256   <-- broad rule first
host  all          postgres  10.0.0.0/8   reject          <-- specific rule second

The reject rule for postgres is never reached. The first rule matches every connection from 10.0.0.0/8 — including the postgres superuser — and allows it with password auth. The second rule is dead code. The correct ordering is:

# Correct: specific reject rule FIRST, broad rule SECOND
host  all          postgres  10.0.0.0/8   reject          <-- specific first
host  all          all       10.0.0.0/8   scram-sha-256   <-- broad second
DBA Migration Warning

SQL Server DBAs instinctively think of DENY as overriding GRANT regardless of order — because SQL Server's permission system works that way. In pg_hba.conf, there is no such override. Whichever rule matches first wins, full stop. Mentally shift from "deny beats grant" to "first match wins, order is everything."

8. Reloading Without a Restart

Unlike SQL Server, where changes to connection-level security settings typically require a service restart, pg_hba.conf changes take effect with a simple reload. Existing connections are not affected — only new connections will see the updated rules.

-- From inside psql (requires superuser or pg_reload_conf privilege)
SELECT pg_reload_conf();

-- Returns: pg_reload_conf
--          ----------------
--          t
-- 't' means the signal was sent successfully, not that the config is valid.

-- From the OS shell (Debian/Ubuntu package layout)
-- sudo pg_ctlcluster 16 main reload

-- From the OS shell (generic)
-- pg_ctl reload -D /var/lib/postgresql/data

-- Verify the reload timestamp in the log:
-- LOG:  received SIGHUP, reloading configuration files
-- LOG:  reloading pg_hba.conf
Warning

pg_reload_conf() returning t means the signal was delivered to the postmaster — it does not mean your pg_hba.conf is syntactically valid. If there is a syntax error, PostgreSQL will log it and continue using the last valid configuration. Always check the PostgreSQL log after a reload, and test a new connection before closing your current session.

To check for configuration errors before reloading, inspect the pg_hba_file_rules view (available in PostgreSQL 10+):

-- Preview what PostgreSQL currently sees in pg_hba.conf
SELECT line_number, type, database, user_name, address, auth_method, error
FROM   pg_hba_file_rules
ORDER BY line_number;

-- Any row with a non-null 'error' column indicates a parse problem.
-- Fix the error before reloading, or PostgreSQL will silently ignore that rule.
