Roles and Privileges — PostgreSQL's Answer to SQL Server Logins, Users, and Roles

If there is one area where the conceptual shift from SQL Server to PostgreSQL is most disorienting, it is the authorization model. SQL Server has three separate security objects — logins, users, and roles — and they exist at different levels of the server hierarchy. PostgreSQL collapses all of them into a single object. Understanding this unification is the foundation for everything else in this lesson.

1. The Most Important Conceptual Shift: One Object to Rule Them All

In SQL Server, the security model is layered:

  • Login — a server-level principal that can authenticate. Lives in master. Can be a SQL Auth login or a Windows principal.
  • User — a database-level principal mapped to a login. Exists inside each individual database. A login must be mapped to a user to access a database.
  • Role — a container for permissions, either fixed server roles or user-defined database roles. Users are members of roles.

In PostgreSQL, there is exactly one object: a role.

  • A role with the LOGIN attribute is a login — it can authenticate and connect.
  • A role without LOGIN is a permission group — it can hold privileges and be granted to other roles.
  • There is no separate "user" object at the database level. Roles exist cluster-wide and can be granted access to specific databases via GRANT CONNECT ON DATABASE.
SQL Server Comparison

The SQL standard CREATE USER syntax works in PostgreSQL as an alias for CREATE ROLE ... WITH LOGIN. Many tutorials use CREATE USER because it is familiar, but it is identical to CREATE ROLE with a LOGIN attribute. There is no technical difference between a "user" and a "role" in PostgreSQL — they are the same catalog object in pg_roles.

2. Role Attributes

A role's behaviour is controlled by its attributes, set at creation time or altered later with ALTER ROLE. The official documentation for CREATE ROLE is at postgresql.org/docs/current/sql-createrole.html.

Attribute What it does SQL Server equivalent
LOGIN / NOLOGIN Whether the role can authenticate and open a client connection. Default: NOLOGIN. The distinction between a SQL Server Login (can authenticate) and a Database Role (cannot authenticate, used only for permission grouping).
SUPERUSER / NOSUPERUSER Bypasses all permission checks in the entire cluster. Can do anything. Default: NOSUPERUSER. Membership in the sysadmin fixed server role. Equivalent power: no access check can deny a superuser.
CREATEDB / NOCREATEDB Can create new databases. Default: NOCREATEDB. Membership in the dbcreator fixed server role.
CREATEROLE / NOCREATEROLE Can create, alter, and drop other roles (with restrictions below their own privilege level in PG16+). Default: NOCREATEROLE. Partial equivalent to securityadmin — but PostgreSQL's CREATEROLE is narrower; it cannot grant privileges the role itself does not hold.
REPLICATION / NOREPLICATION Can initiate streaming replication and use replication slots. Default: NOREPLICATION. Membership in the sysadmin or dbcreator role is typically used for replication in SQL Server; there is no direct named equivalent.
BYPASSRLS / NOBYPASSRLS Bypasses all Row Level Security policies on all tables. Default: NOBYPASSRLS. Superusers implicitly bypass RLS unless the table uses FORCE ROW LEVEL SECURITY. No direct equivalent. SQL Server's RLS predicate functions run for all users including sysadmins unless explicitly skipped via a CONTEXT_INFO check.
INHERIT / NOINHERIT Whether this role automatically inherits privileges from roles it is a member of. Default: INHERIT. No direct equivalent. SQL Server role membership always means privilege inheritance. PostgreSQL NOINHERIT means the role must explicitly SET ROLE to activate inherited privileges.
CONNECTION LIMIT Maximum number of simultaneous connections for this role. -1 means unlimited. Default: -1. No direct equivalent at the login level; SQL Server Resource Governor limits connections per workload group, not per login.
PASSWORD Sets the role's password for authentication. Only meaningful if the role has LOGIN and pg_hba.conf uses a password-based method. The password field of a SQL Authentication login in SQL Server.
VALID UNTIL Password expiry timestamp. After this time, password authentication fails for this role. SQL Server login password expiry policy (enforced via Windows policy or CHECK_EXPIRATION setting).

3. Creating Roles — Exact Syntax

-- A login role: an application service account
-- Every security attribute is explicit — never rely on defaults for security-sensitive roles
CREATE ROLE app_user
  WITH
    LOGIN
    PASSWORD  'initial_password_change_on_first_use'
    NOSUPERUSER
    NOCREATEDB
    NOCREATEROLE
    NOREPLICATION
    NOBYPASSRLS
    INHERIT
    CONNECTION LIMIT 10      -- cap connection count per service account
    VALID UNTIL '2025-12-31';  -- optional: force rotation

-- A permission group (no LOGIN — cannot connect directly)
CREATE ROLE readonly_role
  WITH
    NOLOGIN
    NOSUPERUSER
    NOCREATEDB
    NOCREATEROLE;

-- Grant the group to the login role (role membership)
GRANT readonly_role TO app_user;

-- Verify: list all roles and their attributes
SELECT rolname, rolcanlogin, rolsuper, rolcreatedb, rolcreaterole, rolreplication, rolbypassrls
FROM   pg_roles
ORDER BY rolname;

-- Modify an existing role:
ALTER ROLE app_user CONNECTION LIMIT 20;
ALTER ROLE app_user PASSWORD 'new_strong_password';
ALTER ROLE app_user VALID UNTIL 'infinity';  -- remove expiry

4. Object-Level Privileges — GRANT and REVOKE

PostgreSQL object privileges are granted on specific object types. The set of available privileges differs by object type. The documentation for GRANT is at postgresql.org/docs/current/sql-grant.html.

Tables

-- Available table privileges: SELECT, INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER

-- Grant read access to the readonly_role group
GRANT SELECT                    ON orders TO readonly_role;

-- Grant data modification rights (but not schema changes)
GRANT SELECT, INSERT, UPDATE, DELETE  ON orders TO readwrite_role;

-- Grant full table control (including truncate and FK references)
GRANT ALL PRIVILEGES             ON orders TO app_owner;

-- Grant on all tables in a schema at once
GRANT SELECT ON ALL TABLES IN SCHEMA app TO readonly_role;

-- Revoke: removing a privilege
REVOKE INSERT ON orders FROM readonly_role;

-- Inspect current grants on a table:
SELECT grantee, privilege_type, is_grantable
FROM   information_schema.role_table_grants
WHERE  table_schema = 'app' AND table_name = 'orders'
ORDER BY grantee, privilege_type;

Schemas

-- USAGE: required to access objects inside a schema (see objects, resolve names)
-- CREATE: required to create new objects in a schema
-- Both are needed to create objects and use them; USAGE alone is enough to query.

-- Minimum schema grant for a read-only role:
GRANT USAGE ON SCHEMA app TO readonly_role;

-- Grant for an application role that needs to create tables in the schema:
GRANT USAGE, CREATE ON SCHEMA app TO app_owner;

-- Inspect schema grants:
SELECT schema_name, privilege_type, grantee
FROM   information_schema.role_schema_grants
WHERE  schema_name = 'app';

Sequences

-- Sequences power SERIAL / BIGSERIAL columns.
-- USAGE: call nextval() and currval()
-- SELECT: call currval() and lastval() without USAGE
-- UPDATE: call setval() to reset the sequence counter

-- Application role needs INSERT → needs USAGE on the sequence:
GRANT USAGE ON orders_id_seq TO app_user;

-- Or grant on all sequences in a schema:
GRANT USAGE ON ALL SEQUENCES IN SCHEMA app TO app_user;

Functions

-- EXECUTE: call the function
-- Default: PUBLIC has EXECUTE on all functions (revoke this!)

-- Revoke the public default, then grant explicitly:
REVOKE EXECUTE ON FUNCTION calculate_tax(numeric) FROM PUBLIC;
GRANT  EXECUTE ON FUNCTION calculate_tax(numeric) TO app_user;

-- Grant on all functions in a schema:
GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA app TO app_user;

5. Column-Level Privileges

PostgreSQL supports privilege grants at the individual column level for SELECT, INSERT, UPDATE, and REFERENCES. Column-level grants are additive with table-level grants — a role can have SELECT on all columns via a table grant, and separately have UPDATE on only a subset of columns.

-- Grant SELECT on a specific column only (e.g., HR can read salaries)
GRANT SELECT (employee_id, name, salary) ON employees TO hr_readonly;

-- Revoke the table-level SELECT but keep the column-level grant intact.
-- The role still has access to exactly those three columns.
REVOKE SELECT ON employees FROM hr_readonly;
-- hr_readonly can now SELECT only employee_id, name, salary — not other columns

-- Inspect column-level grants:
SELECT column_name, privilege_type, grantee
FROM   information_schema.role_column_grants
WHERE  table_name = 'employees'
ORDER BY grantee, column_name;

-- Note: column-level INSERT or UPDATE grants require the role to also
-- have table-level INSERT or UPDATE (respectively) to perform the operation.
-- Column grants narrow the scope; they do not standalone replace the table grant.
Info

Column-level SELECT grants are a clean alternative to views when you need to expose a subset of columns from a table to a specific role without creating a separate view object. They compose naturally with table-level grants: if a role has table-level SELECT and column-level SELECT on a subset, the column grants are redundant — the table grant already covers everything. Column grants become meaningful only when table-level SELECT is revoked.

6. ALTER DEFAULT PRIVILEGES — The Concept SQL Server DBAs Most Often Miss

This is the most important privilege concept to understand, and it has no direct SQL Server equivalent. Without it, every new table created in a schema will have no privileges for application roles — causing a silent permission denied error the first time the application tries to access the table.

Here is why it matters. In SQL Server, if a user is a member of db_datareader, they can read all tables in the database — including tables created after the role was granted. SQL Server evaluates the role's permission at query time against whatever objects exist at that moment. There is no per-table bootstrap step required.

In PostgreSQL, privileges are stored per-object. A GRANT on a table says "this role may SELECT on this specific table." When a new table is created, it has no grants at all. The role that needs to SELECT from it must be granted SELECT explicitly on the new table — or you use ALTER DEFAULT PRIVILEGES to declare the grants that should be applied automatically to every future object.

-- The problem without ALTER DEFAULT PRIVILEGES:
-- 1. You grant SELECT on all existing tables to readonly_role
GRANT SELECT ON ALL TABLES IN SCHEMA app TO readonly_role;

-- 2. app_owner creates a new table next week
CREATE TABLE app.new_feature (id serial, data text);

-- 3. readonly_role tries to query it:
--    SELECT * FROM app.new_feature;
--    ERROR: permission denied for table new_feature
-- The GRANT on "ALL TABLES" captured tables at that moment — not future ones.


-- THE FIX: ALTER DEFAULT PRIVILEGES
-- "For any table that app_owner creates in schema app,
--  automatically grant these privileges to readonly_role"

ALTER DEFAULT PRIVILEGES
  FOR ROLE app_owner          -- the role doing the creating
  IN SCHEMA app               -- in this schema
  GRANT SELECT ON TABLES TO readonly_role;

-- For a read/write application role, set defaults for tables, sequences, and functions:
ALTER DEFAULT PRIVILEGES
  FOR ROLE app_owner
  IN SCHEMA app
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_user;

ALTER DEFAULT PRIVILEGES
  FOR ROLE app_owner
  IN SCHEMA app
  GRANT USAGE ON SEQUENCES TO app_user;

ALTER DEFAULT PRIVILEGES
  FOR ROLE app_owner
  IN SCHEMA app
  GRANT EXECUTE ON FUNCTIONS TO app_user;

-- Inspect current default privileges:
SELECT pg_get_userbyid(defaclrole) AS owner_role,
       defaclnamespace::regnamespace AS schema_name,
       defaclobjtype,
       pg_get_acl(defaclacl) AS acl
FROM   pg_default_acl;

-- NOTE: ALTER DEFAULT PRIVILEGES is role-specific.
-- If a different role (say, app_developer) also creates tables in the schema,
-- you need a separate ALTER DEFAULT PRIVILEGES FOR ROLE app_developer statement.
-- Default privileges only apply to objects created by the role listed in FOR ROLE.
DBA Migration Warning — Top Mistake

This is the number one silent failure pattern for PostgreSQL deployments managed by DBAs migrating from SQL Server. The initial GRANT SELECT ON ALL TABLES IN SCHEMA app TO readonly_role works perfectly. Two months later, a developer creates a new table, the application starts returning permission errors, and the team has no idea why — because the error looks exactly like the table does not exist from the application's perspective. Always set up ALTER DEFAULT PRIVILEGES at schema creation time, before any application objects are created. Treat it as a schema-level configuration step, not an afterthought.

7. The public Schema Problem

The public schema exists in every PostgreSQL database and has historically been the default landing zone for objects. Its default privilege configuration changed significantly in PostgreSQL 15, and understanding this change is essential when working across different PostgreSQL versions.

Pre-PostgreSQL 15 Behaviour

-- In PostgreSQL 14 and earlier:
-- The PUBLIC pseudo-role (meaning: every role in the cluster) has:
--   CONNECT on every database (by default)
--   CREATE on the public schema (by default)
--   TEMP on every database (by default)
--
-- This means: any role that can log in can CREATE TABLES in public.
-- An attacker who gains access to ANY login role can inject objects
-- into the public schema — tables, functions, triggers — that other
-- users may interact with.
--
-- FIX for PostgreSQL 14 and earlier (run in each database):
REVOKE CREATE ON SCHEMA public FROM PUBLIC;
REVOKE ALL ON DATABASE mydb FROM PUBLIC;
-- Then grant explicitly to the roles that legitimately need these privileges:
GRANT CONNECT ON DATABASE mydb TO app_user;
GRANT CONNECT ON DATABASE mydb TO readonly_role;

PostgreSQL 15+ Behaviour

-- In PostgreSQL 15+:
-- CREATE on the public schema is no longer granted to PUBLIC by default.
-- A new database has public schema with only USAGE granted to PUBLIC.
-- This is the correct baseline for new deployments.
--
-- However: CONNECT on the database is still granted to PUBLIC by default.
-- You still need to revoke this explicitly if you want explicit control
-- over which roles can connect to which databases:
REVOKE ALL ON DATABASE mydb FROM PUBLIC;
GRANT CONNECT ON DATABASE mydb TO app_user;
GRANT CONNECT ON DATABASE mydb TO readonly_role;

-- Check current public schema grants:
SELECT nspname, nspacl
FROM   pg_namespace
WHERE  nspname = 'public';
-- Pre-15 output: {=UC/postgres} — U=USAGE, C=CREATE granted to PUBLIC (=)
-- Post-15 output: {=U/postgres}  — only USAGE granted to PUBLIC
Security Note

Even in PostgreSQL 15+, the habit of placing application objects in the public schema should be avoided for multi-tenant or multi-application databases. Create a dedicated schema per application (CREATE SCHEMA app AUTHORIZATION app_owner), revoke all on public, and use ALTER DEFAULT PRIVILEGES within the application schema. This isolates applications at the schema level and makes privilege management explicit.

8. SQL Server Mapping Summary

SQL Server concept PostgreSQL equivalent Notes
sysadmin fixed server role SUPERUSER role attribute Both bypass all permission checks. Grant to as few accounts as possible — never to application service accounts.
securityadmin fixed server role CREATEROLE role attribute (partial) PostgreSQL CREATEROLE cannot grant privileges the role itself does not hold, making it somewhat safer than SQL Server's securityadmin.
dbcreator fixed server role CREATEDB role attribute Direct equivalent — both grant the ability to create new databases.
db_datareader database role Custom role + GRANT SELECT ON ALL TABLES IN SCHEMA + ALTER DEFAULT PRIVILEGES PostgreSQL has no built-in db_datareader equivalent. You build it with a NOLOGIN role and explicit grants. Remember: ALTER DEFAULT PRIVILEGES is required for future tables.
db_datawriter database role Custom role + GRANT INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA Same pattern as db_datareader. Combine with USAGE on sequences if roles need INSERT into tables with serial columns.
db_ddladmin database role Role with GRANT CREATE ON SCHEMA + CREATE ROLE in relevant schemas PostgreSQL does not have a built-in ddladmin role. Grant CREATE on schemas explicitly, not SUPERUSER.
Fixed server role bulkadmin COPY privilege on specific tables PostgreSQL's COPY privilege is per-table, not a server-wide capability.
Windows Authentication login Role with LOGIN + pg_hba.conf using ldap or gss method The role still needs to exist in PostgreSQL; authentication is delegated to AD or Kerberos.
Schema ownership (db_owner) Schema-level ownership + GRANT ALL ON SCHEMA CREATE SCHEMA myschema AUTHORIZATION app_owner makes app_owner the schema owner with full control.

9. Mistake 1 — SUPERUSER on Application Service Accounts

DBA Migration Warning — Top Mistake

When an application fails with permission errors during initial setup, the fastest workaround is to grant SUPERUSER to the application service account. In SQL Server, the equivalent is adding the login to sysadmin — experienced DBAs know this is wrong, but it happens under deadline pressure. In PostgreSQL, SUPERUSER is even more powerful: it bypasses Row Level Security, all GRANT checks, all schema isolation, and can access every database in the cluster. A compromised application running as SUPERUSER has complete control over the PostgreSQL instance, including the ability to read pg_authid (all password hashes), create new superusers, and read files on the server filesystem via COPY TO/FROM.

The correct approach is minimum privilege. Here is a recipe for a typical application service account:

-- Step 1: Create the application role with minimum attributes
CREATE ROLE app_user
  WITH LOGIN PASSWORD 'strong_random_password'
  NOSUPERUSER NOCREATEDB NOCREATEROLE NOREPLICATION NOBYPASSRLS;

-- Step 2: Grant CONNECT on the specific database (not all databases)
GRANT CONNECT ON DATABASE appdb TO app_user;

-- Step 3: Grant USAGE on the application schema
GRANT USAGE ON SCHEMA app TO app_user;

-- Step 4: Grant object-level privileges
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA app TO app_user;
GRANT USAGE ON ALL SEQUENCES IN SCHEMA app TO app_user;
GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA app TO app_user;

-- Step 5: Set up default privileges for future objects
--         (do this as the schema-owning role, not as app_user)
ALTER DEFAULT PRIVILEGES
  FOR ROLE app_owner IN SCHEMA app
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_user;

ALTER DEFAULT PRIVILEGES
  FOR ROLE app_owner IN SCHEMA app
  GRANT USAGE ON SEQUENCES TO app_user;

-- Verify: what can app_user do?
SELECT table_schema, table_name, privilege_type
FROM   information_schema.role_table_grants
WHERE  grantee = 'app_user'
ORDER BY table_schema, table_name, privilege_type;

10. Mistake 2 — Forgetting ALTER DEFAULT PRIVILEGES

This has been discussed in Section 6, but it deserves explicit emphasis as a top migration mistake because its failure mode is silent and delayed. The symptoms appear weeks after the initial setup, when new objects are created, and they look like a completely different problem.

DBA Migration Warning

The pattern is always the same: initial deployment works, everything is granted on existing objects, application goes live. Then a developer runs a migration that creates a new table. The application starts returning "permission denied for table X" — but X is a brand-new table that did not exist when the grants were set up. The DBA re-runs the GRANT statement, fixes it, deploys the next migration, same error, repeat. This cycle is the unmistakable signature of a missing ALTER DEFAULT PRIVILEGES configuration.

Make ALTER DEFAULT PRIVILEGES part of your schema creation runbook. Run it immediately after creating the schema, before any objects are created. Document it. Include it in your infrastructure-as-code templates.

A complete schema setup sequence — the template to standardise across all deployments:

-- ── SCHEMA SETUP TEMPLATE ─────────────────────────────────────────────────────
-- Run once per schema, at provisioning time.
-- Replace: mydb, app_schema, app_owner, app_rw_role, app_ro_role

-- 1. Revoke public defaults
REVOKE ALL ON DATABASE mydb FROM PUBLIC;
REVOKE CREATE ON SCHEMA public FROM PUBLIC;

-- 2. Create the schema owned by the application owner role
CREATE SCHEMA app_schema AUTHORIZATION app_owner;

-- 3. Grant connect on the database to application roles
GRANT CONNECT ON DATABASE mydb TO app_owner, app_rw_role, app_ro_role;

-- 4. Grant schema usage
GRANT USAGE ON SCHEMA app_schema TO app_rw_role, app_ro_role;
GRANT USAGE, CREATE ON SCHEMA app_schema TO app_owner;

-- 5. Set default privileges for future tables
ALTER DEFAULT PRIVILEGES FOR ROLE app_owner IN SCHEMA app_schema
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_rw_role;

ALTER DEFAULT PRIVILEGES FOR ROLE app_owner IN SCHEMA app_schema
  GRANT SELECT ON TABLES TO app_ro_role;

-- 6. Set default privileges for future sequences
ALTER DEFAULT PRIVILEGES FOR ROLE app_owner IN SCHEMA app_schema
  GRANT USAGE ON SEQUENCES TO app_rw_role;

-- 7. Set default privileges for future functions
ALTER DEFAULT PRIVILEGES FOR ROLE app_owner IN SCHEMA app_schema
  GRANT EXECUTE ON FUNCTIONS TO app_rw_role, app_ro_role;

-- Now create application objects — all future grants will be applied automatically.
