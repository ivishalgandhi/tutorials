LDAP Authentication — Connecting PostgreSQL to Active Directory

Most enterprise environments already have Active Directory. Every Windows server, every user account, every service principal lives there. SQL Server integrates with AD natively through Windows Authentication — your users never need a separate SQL credential. PostgreSQL can join that same identity ecosystem through LDAP authentication, and this lesson shows you exactly how.

1. Why LDAP Authentication

Without centralised authentication, every PostgreSQL instance maintains its own password store in pg_authid. When a user leaves the company, someone must remember to revoke access in every database on every server. When a password rotation policy runs, it has to target each instance individually. In practice, these manual steps get missed — and orphaned credentials persist for months.

LDAP authentication delegates the credential check to Active Directory. PostgreSQL never stores the user's password. The AD account is the single source of truth: when the AD account is disabled, the PostgreSQL connection fails immediately — no per-instance cleanup required.

SQL Server Comparison

This maps directly to how you think about Windows Authentication in SQL Server. The goal is identical: AD is the identity provider, the database is a relying party. The mechanism differs — SQL Server uses Kerberos ticket exchange (SSPI/GSSAPI) so the password never crosses the wire at all. PostgreSQL's ldap method still transmits the password to the LDAP server for verification, which is why TLS (ldaps://) is non-negotiable. PostgreSQL does support full Kerberos/GSSAPI as a separate authentication method, covered in a later lesson.

2. Two Bind Methods — and Why the Distinction Matters

PostgreSQL's LDAP authentication supports two fundamentally different approaches to verifying a user's identity. Choosing the wrong one is a common configuration error. The official documentation is at postgresql.org/docs/current/auth-ldap.html.

Method 1: Simple Bind

In simple bind mode, PostgreSQL constructs the user's Distinguished Name (DN) directly by concatenating a prefix, the username, and a suffix — and then attempts to bind to the LDAP server using that DN and the user's password.

# Simple bind: DN is assembled as prefix + username + suffix
# For a Microsoft AD domain EXAMPLE\username style:
hostssl  appdb  all  10.0.1.0/24  ldap
         ldapserver=ad.example.com
         ldapport=636
         ldapscheme=ldaps
         ldapprefix="EXAMPLE\"
         ldapsuffix=""

# PostgreSQL will attempt: bind as "EXAMPLE\alice" with alice's password
# The username alice is taken from the PostgreSQL connection's user field.

# For UPN-style (alice@example.com):
hostssl  appdb  all  10.0.1.0/24  ldap
         ldapserver=ad.example.com
         ldapscheme=ldaps
         ldapprefix=""
         ldapsuffix="@example.com"

Simple bind works when the username format is predictable and consistent. Its weakness: it cannot handle cases where the user's LDAP DN does not follow a simple formula — for example, users distributed across multiple OUs, or where sAMAccountName differs from the PostgreSQL role name.

Method 2: Search + Bind (the AD-Recommended Approach)

Search + bind uses a two-phase process. In the first phase, PostgreSQL binds to the LDAP server using a dedicated service account (the bind DN and its password). In the second phase, it searches the directory for the connecting user's actual DN using a configurable attribute. Once the DN is found, PostgreSQL unbinds and re-binds as the user with their password to verify the credential.

sequenceDiagram participant C as PostgreSQL Client participant P as PostgreSQL Server participant L as Active Directory (LDAP) C->>P: Connection request (user: alice, password: secret) Note over P: Phase 1 — service account bind P->>L: Bind as ldapbinddn (service account) L-->>P: Bind successful Note over P: Phase 2 — search for alice's DN P->>L: Search: ldapsearchattribute=sAMAccountName, value=alice L-->>P: Found DN: CN=Alice Smith,OU=Staff,DC=example,DC=com P->>L: Unbind service account Note over P: Phase 3 — bind as alice to verify password P->>L: Bind as CN=Alice Smith,... with password "secret" L-->>P: Bind successful (password correct) P-->>C: Authentication approved

Figure 1. Search+bind flow. PostgreSQL uses a service account to locate the user's DN, then verifies the user's own password in a separate bind.

# Search+bind configuration parameters:
#   ldapbinddn      — the service account DN used for the initial search
#   ldapbindpasswd  — the service account's password (see Section 6 for risks)
#   ldapbasedn      — the LDAP subtree to search within
#   ldapsearchattribute — the attribute whose value equals the PostgreSQL username
#   ldapsearchfilter    — optional additional LDAP filter (e.g., group membership)

hostssl  all  all  10.0.0.0/8  ldap
  ldapurl="ldaps://ad.example.com/OU=Staff,DC=example,DC=com?sAMAccountName?sub"
  ldapbinddn="CN=pg-ldap-svc,OU=ServiceAccounts,DC=example,DC=com"
  ldapbindpasswd="ServiceAccountPassword123"

3. pg_hba.conf LDAP Lines — Exact Syntax

LDAP options can be specified either as individual keyword parameters or compacted into an ldapurl following RFC 4516. The two forms are mutually exclusive — do not mix them. The ldapurl form is preferred for search+bind as it is more compact and explicitly encodes the search scope.

# ── FORM 1: Individual keyword parameters (simple bind) ──────────────────
hostssl  mydb  all  10.0.1.0/24  ldap
  ldapserver=ad.example.com
  ldapport=636
  ldapscheme=ldaps
  ldapprefix="EXAMPLE\"
  ldapsuffix=""

# ── FORM 2: ldapurl (search+bind — recommended for AD) ───────────────────
# ldapurl encodes: scheme://host/basedn?attribute?scope[?filter]
# scope: base | one | sub  (use sub for full subtree search)
hostssl  mydb  all  10.0.1.0/24  ldap
  ldapurl="ldaps://ad.example.com/DC=example,DC=com?sAMAccountName?sub"
  ldapbinddn="CN=pg-svc,OU=ServiceAccounts,DC=example,DC=com"
  ldapbindpasswd="Passw0rd!"

# ── FORM 3: ldapurl with additional group membership filter ──────────────
# Only members of the PostgreSQL-Users AD group can authenticate
hostssl  mydb  all  10.0.1.0/24  ldap
  ldapurl="ldaps://ad.example.com/DC=example,DC=com?sAMAccountName?sub?(memberOf=CN=PostgreSQL-Users,OU=Groups,DC=example,DC=com)"
  ldapbinddn="CN=pg-svc,OU=ServiceAccounts,DC=example,DC=com"
  ldapbindpasswd="Passw0rd!"
Info

Note that ldap in pg_hba.conf METHOD field refers to the authentication method, not the scheme. To use LDAP over TLS (port 636), you specify either ldapscheme=ldaps (keyword form) or ldaps:// in the URL (ldapurl form). The METHOD is always ldap regardless of whether you use plain or TLS.

4. Active Directory Specifics

Active Directory is an LDAP directory with Microsoft-specific extensions. These are the parameters that matter most for a successful AD integration:

Parameter AD-specific guidance Example value
ldapurl scheme Always ldaps:// for production. AD supports LDAP over SSL on port 636 with a valid domain certificate. Plain ldap:// on port 389 sends the bind password in cleartext. ldaps://ad.example.com
ldapbasedn The root of your search. Start at the domain root or the OU that contains all database users. Narrower base = faster search. OU=Staff,DC=example,DC=com
ldapsearchattribute Use sAMAccountName for short Windows logon names (DOMAIN\username). Use userPrincipalName for UPN-style (alice@example.com) — but the PostgreSQL role name must match exactly. sAMAccountName
ldapbinddn A dedicated service account with read-only access to the directory. Do not reuse an admin account. The account needs only Read and List Contents permissions on the searched OUs. CN=pg-ldap-svc,OU=ServiceAccounts,DC=example,DC=com
ldapbindpasswd The service account password. Stored in plaintext in pg_hba.conf. See Section 6 for mitigations. Service account password string
AD certificate AD's LDAPS certificate must be trusted by the PostgreSQL server. Add the AD CA certificate to the system trust store or reference it via LDAPTLS_CACERT environment variable. /etc/ssl/certs/example-root-ca.crt
Key Concept — Role Name Matching

PostgreSQL LDAP authentication verifies the password but does not create roles automatically. A PostgreSQL role with the same name as the AD sAMAccountName must already exist in the database. If alice authenticates via AD but there is no PostgreSQL role named alice, the connection fails with "role does not exist." Provision roles with CREATE ROLE alice WITH LOGIN NOLOGIN in advance, or use a pg_ident.conf username map to translate AD usernames to PostgreSQL role names.

5. Comparing LDAP to SQL Server Windows Authentication

SQL Server Comparison

SQL Server's Windows Authentication uses Kerberos (or NTLM fallback) — a ticket-based protocol where the user's password hash never leaves the domain controller. The client presents a Kerberos ticket to SQL Server; SQL Server validates the ticket with the KDC. The database password field does not exist for Windows-authenticated logins because there is no database password.

PostgreSQL LDAP authentication is fundamentally different: the user types their AD password into the PostgreSQL client, PostgreSQL receives it, and then transmits it to the LDAP server to perform a bind. The password travels across the network (which is why ldaps:// is mandatory). This is a bind-and-verify pattern, not a ticket exchange. PostgreSQL does support true Kerberos/GSSAPI authentication as a separate method — the gss method in pg_hba.conf — which provides the ticket-based model SQL Server DBAs expect from Windows Authentication.

Aspect SQL Server Windows Auth PostgreSQL ldap method PostgreSQL gss method
Protocol Kerberos / NTLM LDAP bind (password verification) Kerberos (GSSAPI)
Password over network Never (ticket only) Yes (TLS-protected) Never (ticket only)
Requires service account SPN on SQL Server account Yes (search+bind) or No (simple bind) SPN (postgres/host keytab)
AD group membership used for access Yes, via AD groups mapped to SQL Server logins Via ldapsearchfilter only No — only role name matching

6. Gotcha 1 — ldapbindpasswd is Stored in Plaintext

This is the top concern for DBAs configuring LDAP for the first time. The service account password you place in ldapbindpasswd is stored as clear text in pg_hba.conf. There is no encryption, no secrets manager integration, no vault reference — the password is literally readable in the file.

DBA Migration Warning — Top Mistake

SQL Server DBAs are accustomed to Windows Authentication service accounts whose credentials are managed at the OS level (Windows service account, Managed Service Account, or gMSA). The credential is never written to a config file. In PostgreSQL LDAP configuration, the bind password is visible to anyone who can read pg_hba.conf. This is a significant difference in the security model.

Mitigations to apply in order of effectiveness:

# 1. Restrict file permissions — only the postgres OS user can read it
chmod 600 /etc/postgresql/16/main/pg_hba.conf
chown postgres:postgres /etc/postgresql/16/main/pg_hba.conf

# 2. Verify the permissions are correct
ls -la /etc/postgresql/16/main/pg_hba.conf
# Expected: -rw------- 1 postgres postgres ... pg_hba.conf

# 3. Use a dedicated service account with minimal AD permissions:
#    - Read: allowed on the search base OU
#    - List Contents: allowed on the search base OU
#    - All other permissions: denied
#    This limits the blast radius if the password is ever exposed.

# 4. Rotate the service account password on a scheduled basis.
#    After rotation: update ldapbindpasswd, reload pg_hba.conf.
#    SELECT pg_reload_conf();

# 5. Consider using GSSAPI/Kerberos instead of LDAP if you need
#    to eliminate the stored credential entirely.
Security Note

Restrict access to the pg_hba.conf file at the OS level so that only the postgres OS account can read it. Any OS user who can read the file can extract the LDAP service account password. In containerised environments, ensure the file is not baked into a public image layer and is mounted via a secrets volume instead.

7. Gotcha 2 — Never Use Plain ldap:// in Production

When you use plain ldap:// (port 389, no TLS), the entire LDAP bind — including the user's Active Directory password — is transmitted in cleartext over the network. Any device on the path between the PostgreSQL server and the domain controller can capture this traffic.

# WRONG — credentials travel in cleartext
host  appdb  all  10.0.0.0/8  ldap
  ldapserver=ad.example.com
  ldapport=389           <-- unencrypted LDAP
  ldapbasedn="DC=example,DC=com"

# CORRECT — always use ldaps:// (port 636) with a valid certificate
hostssl  appdb  all  10.0.0.0/8  ldap
  ldapurl="ldaps://ad.example.com/DC=example,DC=com?sAMAccountName?sub"
  ldapbinddn="CN=pg-svc,OU=ServiceAccounts,DC=example,DC=com"
  ldapbindpasswd="Passw0rd!"

Note also that using hostssl in the TYPE field (instead of plain host) encrypts the connection between the PostgreSQL client and the PostgreSQL server — but it says nothing about the PostgreSQL-to-LDAP segment. You need both: hostssl for the client-to-server leg, and ldaps:// for the server-to-AD leg.

Warning

Many internal networks lack TLS on LDAP traffic because "it's all on the same VLAN." An attacker who achieves any foothold on that VLAN — a compromised workstation, a misconfigured switch port — can capture every AD password that PostgreSQL forwards for authentication. LDAP over TLS is not optional for a production deployment.

8. Complete Active Directory Integration Example

The following is a production-ready pg_hba.conf snippet for a search+bind AD integration. It assumes:

  • AD domain: example.com — DC pair at ad1.example.com and ad2.example.com
  • Users are in OU=Staff,DC=example,DC=com
  • Service account: CN=pg-ldap-svc,OU=ServiceAccounts,DC=example,DC=com
  • Only members of CN=DB-Users,OU=Groups,DC=example,DC=com may connect
  • PostgreSQL client connections use TLS (hostssl)
# ─────────────────────────────────────────────────────────────────────────────
# pg_hba.conf — Active Directory LDAP integration (search+bind)
# ─────────────────────────────────────────────────────────────────────────────

# 1. Local postgres superuser via peer (no AD involvement)
local    all  postgres                           peer

# 2. Application user from app server — local pg password (not AD)
#    Service accounts should not be AD accounts; use scram-sha-256 instead.
hostssl  appdb  app_user  10.0.1.50/32         scram-sha-256

# 3. All other users from the office network — authenticate via AD
#    hostssl : client-to-PG connection must be TLS
#    ldapurl : ldaps:// ensures PG-to-AD traffic is also TLS
#    basedn  : search only within Staff OU (faster, narrower scope)
#    ?sAMAccountName?sub : find users by Windows short logon name
#    memberOf filter : only AD group members may authenticate
hostssl  all  all  10.0.0.0/8  ldap
  ldapurl="ldaps://ad1.example.com/OU=Staff,DC=example,DC=com?sAMAccountName?sub?(memberOf=CN=DB-Users,OU=Groups,DC=example,DC=com)"
  ldapbinddn="CN=pg-ldap-svc,OU=ServiceAccounts,DC=example,DC=com"
  ldapbindpasswd="$tr0ngSvcP@ss!"

# 4. Backstop — reject everything else
host     all  all  0.0.0.0/0                    reject
High Availability Tip

As of PostgreSQL 14, you can specify multiple LDAP servers for failover using the ldapserver keyword parameter (space-separated list). With ldapurl, specify only one URL per rule — for multi-server failover, list them with the keyword form: ldapserver="ad1.example.com ad2.example.com". PostgreSQL will try each server in order until one responds.

9. Testing LDAP Connectivity Before Deploying

Never deploy a new pg_hba.conf LDAP configuration cold. Verify each component independently first. The ldapsearch command-line tool lets you simulate exactly what PostgreSQL will do during authentication.

# Step 1: Verify the AD server is reachable and TLS works
# -H : the LDAP URI (tests TLS handshake with ldaps://)
# -x : simple bind (not GSSAPI)
# -D : the bind DN (service account)
# -W : prompt for the bind password (do not put in shell history)
ldapsearch \
  -H ldaps://ad1.example.com \
  -x \
  -D "CN=pg-ldap-svc,OU=ServiceAccounts,DC=example,DC=com" \
  -W \
  -b "OU=Staff,DC=example,DC=com" \
  "(sAMAccountName=alice)" \
  dn sAMAccountName memberOf

# Expected output: alice's DN, sAMAccountName, and group memberships
# If you see "Can't contact LDAP server" — check firewall, port 636, certificate trust
# If you see "Invalid credentials" — the service account password is wrong
# If you see 0 results — check the basedn and the search filter

# Step 2: Test the user bind directly (simulates the final auth step)
ldapsearch \
  -H ldaps://ad1.example.com \
  -x \
  -D "CN=Alice Smith,OU=Staff,DC=example,DC=com" \
  -W \
  -b "OU=Staff,DC=example,DC=com" \
  "(sAMAccountName=alice)" \
  dn

# If this succeeds, alice's password is correct and the DN is valid.
# If PostgreSQL still refuses the connection after this, the issue
# is most likely a missing PostgreSQL role (CREATE ROLE alice WITH LOGIN).
Diagnostic Sequence

When LDAP auth fails, work through this sequence: (1) confirm the AD server is reachable from the PostgreSQL host on port 636; (2) confirm the TLS certificate is trusted; (3) confirm the service account can search and find the user's DN; (4) confirm the user's own bind works; (5) confirm a PostgreSQL role exists with the same name as the sAMAccountName; (6) confirm the pg_hba.conf rule matches the connection's type, database, user, and address fields.
