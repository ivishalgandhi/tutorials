15 Commands Every DBA Must Memorise

These are the 15 most critical PostgreSQL security commands. Know them cold.
Authentication & Configuration
01
Find pg_hba.conf
SHOW hba_file;
Locate the authentication configuration file on any PostgreSQL instance. The file path varies by OS, package manager, and data directory — this is the only reliable way to find it.
No equivalent — SQL Server auth config is in Server Properties → Security, not a file path you query.
02
Reload auth config without restart
SELECT pg_reload_conf();

-- Verify it loaded:
SELECT * FROM pg_hba_file_rules LIMIT 5;
Apply pg_hba.conf and postgresql.conf changes (sighup-context GUCs) without restarting the server. Inspect pg_hba_file_rules immediately after to confirm your edits parsed correctly.
RECONFIGURE — but SQL Server never reloads auth config without a restart. PostgreSQL's live reload is a significant operational advantage.
03
Enforce SCRAM-SHA-256 encryption
ALTER SYSTEM SET password_encryption
  = 'scram-sha-256';
SELECT pg_reload_conf();

-- Existing md5 passwords must be reset
Switch to the recommended password hashing algorithm. This affects only new or reset passwords — existing md5 hashes remain until users change their passwords.
No direct equivalent. SQL Server's password hashing is internal. This is the closest PostgreSQL operation to enforcing a minimum authentication strength.
04
Create a minimal login role
CREATE ROLE app_user
  WITH LOGIN PASSWORD 'use-a-vault'
  NOSUPERUSER NOCREATEDB NOCREATEROLE
  VALID UNTIL '2026-01-01';
The safe template for any application or CI/CD login role. Never omit the NO* attributes — they default to the least-privilege values only on fresh installs; explicitly stating them is defensive documentation.
CREATE LOGIN ... WITH PASSWORD, DEFAULT_DATABASE. SQL Server has no NOSUPERUSER because sysadmin is a separate fixed role, not an attribute.
Privileges & Access Control
05
Grant database connect
-- First, revoke the dangerous default:
REVOKE CONNECT ON DATABASE mydb
  FROM PUBLIC;

-- Grant only to roles that need it:
GRANT CONNECT ON DATABASE mydb
  TO app_role;
By default, PUBLIC has CONNECT on all databases. This means every role in every database on the cluster can connect. Revoke it and re-grant explicitly.
DENY CONNECT SQL TO login_name. SQL Server DBAs often miss the PostgreSQL PUBLIC default because SQL Server requires an explicit GRANT to connect.
06
Set up schema-level access
-- Step 1: schema entry permission
GRANT USAGE ON SCHEMA app TO app_role;

-- Step 2: table DML
GRANT SELECT, INSERT, UPDATE, DELETE
  ON ALL TABLES IN SCHEMA app
  TO app_role;

-- Step 3: sequence access (for SERIAL/BIGSERIAL)
GRANT USAGE, SELECT
  ON ALL SEQUENCES IN SCHEMA app
  TO app_role;
Three-step pattern: schema USAGE, then table DML, then sequence access. All three are required. Skipping USAGE means the role cannot resolve any object name in the schema, even if table grants exist.
GRANT SELECT, INSERT... ON SCHEMA::dbo TO role — a single statement covers schema objects. PostgreSQL's three-layer model is more granular but more surprising to SQL Server DBAs.
07
Set default privileges for future objects
ALTER DEFAULT PRIVILEGES
  FOR ROLE migration_role
  IN SCHEMA app
  GRANT SELECT, INSERT, UPDATE, DELETE
  ON TABLES TO app_role;
The #1 missing step for SQL Server DBAs. Without this, every table created by migration_role in the future is inaccessible to app_role until manually granted. This applies to future objects only.
SQL Server schema-based roles inherit access to new objects automatically — no equivalent step needed. This PostgreSQL behaviour is the single most common onboarding surprise.
08
Enable row-level security
-- Step 1: enable RLS on the table
ALTER TABLE orders
  ENABLE ROW LEVEL SECURITY;

-- Step 2: create the isolation policy
CREATE POLICY tenant_isolation ON orders
  AS PERMISSIVE FOR ALL TO app_role
  USING (
    tenant_id = current_setting(
      'app.tid', true)::int
  )
  WITH CHECK (
    tenant_id = current_setting(
      'app.tid', true)::int
  );
Two steps are required. Enabling RLS alone hides all rows from non-superusers until a policy is created. The WITH CHECK clause prevents silent cross-tenant writes.
CREATE SECURITY POLICY + ADD FILTER PREDICATE + ADD BLOCK PREDICATE. SQL Server separates read filtering and write blocking into distinct predicate types.
Auditing & Inspection
09
Find all superusers
SELECT rolname, rolsuper, rolcanlogin
FROM  pg_roles
WHERE rolsuper = true
ORDER BY rolname;
Run this on every new instance you inherit. Production should have at most one or two superuser roles, none of which should have rolcanlogin = true in day-to-day use.
SELECT name FROM sys.server_principals WHERE IS_SRVROLEMEMBER('sysadmin', name) = 1
10
Audit role membership
SELECT
  r.rolname  AS role,
  m.rolname  AS member,
  a.admin_option
FROM  pg_auth_members a
JOIN  pg_roles r ON r.oid = a.roleid
JOIN  pg_roles m ON m.oid = a.member
ORDER BY r.rolname, m.rolname;
See exactly which roles are granted to which roles/users and whether they hold ADMIN OPTION (ability to re-grant). The source of truth for your privilege hierarchy.
SELECT * FROM sys.database_role_members combined with sys.server_role_members (two separate queries in SQL Server).
11
Verify SSL on active connections
SELECT pid, ssl, version,
       cipher, bits, client_dn
FROM  pg_stat_ssl
ORDER BY ssl DESC;
Every production connection should show ssl = true. Any row where ssl = false is an unencrypted connection transmitting credentials and data in cleartext. Treat each one as an open finding.
SELECT encrypt_option FROM sys.dm_exec_connections — same purpose, DMV pattern instead of a static view.
12
Lock search_path per role
ALTER ROLE app_user
  SET search_path = app, pg_temp;

ALTER ROLE migration_role
  SET search_path = app, pg_temp;
Prevents search_path hijacking attacks. The setting survives reconnects and overrides the server default. Including pg_temp last prevents temporary table shadowing. Every production role should have this set.
ALTER USER ... WITH DEFAULT_SCHEMA. SQL Server's single-schema resolution makes the equivalent attack much harder — the multi-schema search order is unique to PostgreSQL.
Hardening & Compliance
13
Find unsafe SECURITY DEFINER functions
SELECT
  n.nspname   AS schema,
  p.proname   AS function,
  p.proconfig AS config
FROM  pg_proc p
JOIN  pg_namespace n
        ON n.oid = p.pronamespace
WHERE p.prosecdef = true
  AND p.proconfig IS NULL;
Any row returned here is a potential privilege escalation vector. proconfig IS NULL means no SET search_path — an attacker with CREATE on any schema can shadow objects called by the function. Add SET search_path = schema, pg_temp to each function found.
No direct equivalent. EXECUTE AS OWNER functions don't have the same search_path attack surface — SQL Server's schema resolution doesn't enable this class of hijacking.
14
Install and configure pgaudit
-- postgresql.conf (server restart required):
shared_preload_libraries = 'pgaudit'
pgaudit.log = 'write, ddl'
pgaudit.log_catalog = off

-- Per-role override (reconnect to activate):
ALTER ROLE sensitive_user
  SET pgaudit.log = 'all';
Minimum pgaudit configuration for production compliance. shared_preload_libraries requires a server restart. log_catalog = off reduces noise from PostgreSQL's own internal queries. Per-role overrides are effective after the role reconnects.
CREATE SERVER AUDIT + CREATE DATABASE AUDIT SPECIFICATION. SQL Server Audit is built into the engine and GUI-configurable; pgaudit requires manual installation and configuration file edits.
15
Fix public schema create privilege (pre-PG15)
-- Check if the vulnerability exists:
SELECT has_schema_privilege(
  'public', 'public', 'create'
);

-- If true, close it:
REVOKE CREATE ON SCHEMA public
  FROM PUBLIC;
REVOKE EXECUTE ON ALL FUNCTIONS
  IN SCHEMA public FROM PUBLIC;
Pre-PostgreSQL 15 default: all roles can create objects in the public schema. This is the single most common production misconfiguration. PostgreSQL 15+ revokes this by default; older instances must be fixed manually.
SQL Server requires explicit GRANT to create objects in any schema — PostgreSQL's PUBLIC default is the exact opposite. SQL Server DBAs are frequently surprised this needs fixing.
