Reference PostgreSQL Security Course Index

PostgreSQL Security Glossary

25 terms — alphabetical — each with PostgreSQL syntax and SQL Server equivalent

B G I L M N P R S T W
B
BYPASSRLS #

Role attribute that skips all row-level security policies. Even with RLS enabled on a table, a role with BYPASSRLS sees all rows unconditionally. Superusers have this implicitly and cannot lose it via NOBYPASSRLS.

-- Grant the bypass attribute to a role:
ALTER ROLE dba_auditor BYPASSRLS;

-- Confirm:
SELECT rolname, rolbypassrls
FROM pg_roles
WHERE rolbypassrls = true;
SQL Server Equivalent

No exact equivalent. The sysadmin fixed server role implicitly bypasses all security predicates, but there is no fine-grained, per-role bypass attribute you can grant. You cannot give a non-sysadmin the ability to skip RLS while leaving other restrictions in place.

G
GUC (Grand Unified Configuration) #

PostgreSQL term for any server configuration parameter settable via postgresql.conf, ALTER SYSTEM SET, or SET. Each GUC has a context that determines the minimum action required to apply a change:

  • postmaster — server restart required
  • sighup — SELECT pg_reload_conf() sufficient
  • user — session-level, effective immediately via SET
-- Browse all GUCs and their current values:
SHOW all;
SELECT name, setting, context, short_desc
FROM pg_settings
WHERE name LIKE '%ssl%';
SQL Server Equivalent

sp_configure options + RECONFIGURE. SQL Server also distinguishes advanced options (requiring show advanced options = 1) but has no equivalent to session-level GUC overrides per role or per database.

I
INHERIT / NOINHERIT #

Role attribute controlling whether privileges granted to a role flow automatically to member roles. INHERIT is the default: if alice is a member of readonly_role, she immediately has all of readonly_role's privileges without any extra step. NOINHERIT breaks that automatic flow — alice must issue SET ROLE readonly_role explicitly to activate the privileges. Useful for high-privilege roles that should require conscious activation.

-- Create a group role with privileges:
CREATE ROLE readonly_role NOLOGIN;
GRANT SELECT ON ALL TABLES IN SCHEMA app TO readonly_role;

-- alice inherits automatically (default INHERIT):
GRANT readonly_role TO alice;

-- bob must SET ROLE first (NOINHERIT on bob):
ALTER ROLE bob NOINHERIT;
GRANT readonly_role TO bob;
SQL Server Equivalent

No equivalent — SQL Server fixed and user-defined database roles always inherit automatically. There is no mechanism to require a user to explicitly activate a role's permissions within a session.

L
listen_addresses #

GUC controlling which IP interfaces PostgreSQL binds to when accepting connections. Set in postgresql.conf. Requires a server restart (postmaster context). The most secure setting is 'localhost' — the instance is only reachable from the host itself. Setting '*' opens every network interface and should be combined with strict pg_hba.conf rules.

-- In postgresql.conf — safest for internal-only instances:
listen_addresses = 'localhost'

-- All interfaces (requires tight pg_hba.conf rules):
listen_addresses = '*'

-- Check current value:
SHOW listen_addresses;
SQL Server Equivalent

TCP/IP protocol binding configured in SQL Server Configuration Manager → SQL Server Network Configuration → Protocols for [instance] → TCP/IP → IP Addresses. SQL Server DBAs typically manage this via SSCM rather than a text file.

M
md5 auth #

pg_hba.conf authentication method that stores passwords as an unsalted MD5 hash of password + username. Deprecated since PostgreSQL 14. Because the hash is deterministic and unsalted, it is vulnerable to offline dictionary attacks and rainbow table lookups. Any captured hash can be cracked independently of the server. Replace with scram-sha-256.

-- Identify remaining md5-hashed passwords:
SELECT rolname, rolpassword
FROM pg_authid
WHERE rolpassword LIKE 'md5%';

-- Force new passwords to use scram-sha-256:
ALTER SYSTEM SET password_encryption = 'scram-sha-256';
SELECT pg_reload_conf();
SQL Server Equivalent

SQL Authentication with older compatibility levels — SQL Server's internal hashing has evolved across versions but is opaque to the DBA. Microsoft's parallel recommendation is to prefer Windows Authentication (Kerberos/NTLM) over SQL Auth for the same reasons: password-based hashes are inherently weaker than ticket-based auth.

N
NOLOGIN role #

A role without the LOGIN attribute. Cannot connect to the database directly — attempting to do so returns an error. The canonical pattern for a permission group: grant the group role a set of privileges, then grant membership in that group to one or more login roles. Members inherit permissions without the group role ever needing to authenticate.

-- Create a permission group:
CREATE ROLE readonly_group NOLOGIN;
GRANT SELECT ON ALL TABLES IN SCHEMA app TO readonly_group;

-- Grant the group to a login role:
CREATE ROLE reporting_user WITH LOGIN PASSWORD '...';
GRANT readonly_group TO reporting_user;
SQL Server Equivalent

CREATE ROLE role_name in SQL Server creates a database role with no password that cannot log in — the direct equivalent. Assigning a user to the role (ALTER ROLE role_name ADD MEMBER user_name) mirrors the PostgreSQL GRANT group TO user pattern.

P
peer auth #

pg_hba.conf auth method available only for local Unix socket connections. PostgreSQL asks the operating system which user owns the connecting process (via SO_PEERCRED or equivalent). If the OS username matches the PostgreSQL role name, the connection is accepted with no password. Commonly used for the postgres superuser on the database server itself.

-- Typical pg_hba.conf entry for peer auth:
local   all   postgres   peer

-- Verify how postgres OS user connects:
-- (from the server shell as OS user 'postgres')
psql -U postgres   -- succeeds via peer, no password needed
SQL Server Equivalent

Windows Integrated Authentication on local connections — but SQL Server uses a Kerberos or NTLM ticket rather than directly checking OS process ownership. The result is similar (no explicit password), but the mechanism differs: Windows Authentication validates a domain identity, while peer validates local OS process ownership.

pg_auth_members #

System catalog table (not a view) recording every role-membership relationship: which role is granted to which member, whether the grant includes ADMIN OPTION (ability to re-grant), and who granted it. The authoritative source for the PostgreSQL role hierarchy.

-- Human-readable role membership tree:
SELECT
  r.rolname  AS role,
  m.rolname  AS member,
  a.admin_option
FROM  pg_auth_members a
JOIN  pg_roles r  ON r.oid = a.roleid
JOIN  pg_roles m  ON m.oid = a.member
ORDER BY r.rolname, m.rolname;
SQL Server Equivalent

sys.database_role_members (database-scoped role memberships) + sys.server_role_members (server-scoped role memberships). Unlike PostgreSQL, SQL Server separates server-level and database-level role membership into different catalogs.

pg_hba.conf #

The host-based authentication file — the complete authentication decision engine for PostgreSQL. Every incoming connection is matched against its rules top-to-bottom; the first matching rule wins and its auth method is applied. No subsequent rules are evaluated. Rules specify connection type (local, host, hostssl, hostnossl), database name, role name, client IP/CIDR, and auth method (scram-sha-256, md5, peer, trust, reject, ldap, etc.).

-- Find the file location:
SHOW hba_file;

-- Inspect loaded rules without opening the file:
SELECT type, database, user_name, address, auth_method
FROM  pg_hba_file_rules;

-- Reload after editing (no restart needed):
SELECT pg_reload_conf();
SQL Server Equivalent

No single equivalent file. SQL Server authentication mode (Windows-only vs. Mixed Mode) is set in Server Properties → Security and applies globally. Individual login rules are properties of each login object, not entries in a routing file. There is no equivalent to per-database, per-IP, per-method routing in a single config file.

pgaudit #

PostgreSQL extension providing structured, object-level audit logging suitable for compliance requirements (PCI-DSS, HIPAA, SOC 2). Produces AUDIT: log lines that include the audit class, object type, object name, and full statement text. Operates in two modes:

  • SESSION — logs all statements matching a class (READ, WRITE, DDL, ROLE, FUNCTION, MISC) for the session
  • OBJECT — logs statements that access objects visible to a designated audit role
-- postgresql.conf (restart required):
shared_preload_libraries = 'pgaudit'
pgaudit.log = 'write, ddl'
pgaudit.log_catalog = off   -- reduce noise from system catalogs

-- Per-role override (effective after reconnect):
ALTER ROLE sensitive_user SET pgaudit.log = 'all';
SQL Server Equivalent

SQL Server Audit (introduced in SQL Server 2008) with CREATE SERVER AUDIT, CREATE DATABASE AUDIT SPECIFICATION, and CREATE SERVER AUDIT SPECIFICATION. Extended Events also covers many audit scenarios. SQL Server Audit is built into the engine; pgaudit requires explicit installation and preloading.

pg_roles #

System catalog view of all roles and their attributes. A safe, redacted view of pg_authid (the underlying catalog) — password hashes are masked. Key columns: rolsuper, rolcreatedb, rolcreaterole, rolcanlogin, rolbypassrls, rolinherit, rolconnlimit, rolvaliduntil.

-- All roles with their key security attributes:
SELECT rolname, rolsuper, rolcreaterole, rolcreatedb,
       rolcanlogin, rolbypassrls, rolconnlimit,
       rolvaliduntil
FROM  pg_roles
ORDER BY rolsuper DESC, rolname;
SQL Server Equivalent

sys.server_principals (server-level logins and server roles) + sys.database_principals (database users and database roles). Unlike PostgreSQL's unified role model, SQL Server maintains a strict server/database hierarchy with separate catalogs at each level.

pg_stat_ssl #

System view showing the SSL/TLS status of every currently active connection. One row per connection. Critical columns: ssl (bool — is this connection encrypted?), version (TLS version string), cipher, bits (key length), client_dn (client certificate DN if mutual TLS is used). Any row with ssl = false is a cleartext connection in production — treat it as a finding.

-- Audit all active connections for encryption status:
SELECT pid, ssl, version, cipher, bits, client_dn
FROM  pg_stat_ssl
ORDER BY ssl DESC, pid;
SQL Server Equivalent

sys.dm_exec_connections — the encrypt_option column shows whether encryption is active for each connection (TRUE, FALSE, or NOT SUPPORTED). Equivalent in purpose; SQL Server uses the DMV (Dynamic Management View) pattern rather than a static system view.

PERMISSIVE policy #

An RLS policy type where PostgreSQL applies OR logic across all matching permissive policies on a table: if any permissive policy's USING expression evaluates to true for a row, that row is visible. This is the default policy type when AS PERMISSIVE (or nothing) is specified. Multiple permissive policies effectively expand access — the row only needs to pass one of them.

-- Two permissive policies: OR logic applies
-- Rows visible if owned by current user OR public
CREATE POLICY own_rows ON documents
  AS PERMISSIVE FOR SELECT TO app_role
  USING (owner_id = current_user_id());

CREATE POLICY public_rows ON documents
  AS PERMISSIVE FOR SELECT TO app_role
  USING (is_public = true);
SQL Server Equivalent

FILTER PREDICATE in a SQL Server security policy always uses union/OR logic across rows — all rows passing any predicate evaluation are included. SQL Server does not distinguish between permissive and restrictive predicates for FILTER operations.

R
RESTRICTIVE policy #

An RLS policy type where ALL restrictive policies must pass (AND logic), in addition to at least one permissive policy passing. A restrictive policy can only reduce access, never expand it. Use restrictive policies to enforce hard constraints — such as a time window, a tenant boundary, or a data classification — that no permissive policy can override.

-- Hard constraint: only active tenants, regardless of
-- what permissive policies allow
CREATE POLICY active_tenant_only ON orders
  AS RESTRICTIVE FOR ALL TO app_role
  USING (
    EXISTS (
      SELECT 1 FROM tenants
      WHERE id = tenant_id AND active = true
    )
  );
SQL Server Equivalent

BLOCK PREDICATE with AFTER INSERT / AFTER UPDATE is partially equivalent — it blocks writes regardless of other predicates. SQL Server has no direct equivalent to a restrictive filter on SELECT/DELETE. The AND-across-all-policies semantic is unique to PostgreSQL.

row_security GUC #

Session-level GUC that must be on for RLS policies to be enforced. If set to off, all RLS policies on all tables are bypassed — effectively making every user a BYPASSRLS role for the duration of the session. Should always be on in production. Can be set per-role with ALTER ROLE to ensure RLS is never accidentally bypassed.

-- Should always be on; verify the server default:
SHOW row_security;

-- Lock it on for a specific role:
ALTER ROLE app_user SET row_security = on;

-- (Deliberately dangerous — never in production):
SET row_security = off;
SQL Server Equivalent

No equivalent. SQL Server security predicates are always active once bound to a table via ALTER SECURITY POLICY ... WITH (STATE = ON). There is no session-level toggle that bypasses all policies simultaneously.

S
scram-sha-256 #

The recommended pg_hba.conf password authentication method. Uses SCRAM (Salted Challenge Response Authentication Mechanism) with SHA-256. Key properties: the server never transmits the password or its hash in cleartext; each stored hash uses a unique salt; the protocol is immune to replay attacks because it is challenge-response based. Introduced in PostgreSQL 10, made the default in PostgreSQL 14.

-- Set as the default for new passwords:
ALTER SYSTEM SET password_encryption = 'scram-sha-256';
SELECT pg_reload_conf();

-- pg_hba.conf entry:
hostssl  all  all  0.0.0.0/0  scram-sha-256

-- Verify hashed passwords use SCRAM (not md5):
SELECT rolname, LEFT(rolpassword, 10) AS hash_prefix
FROM  pg_authid
WHERE rolpassword IS NOT NULL;
SQL Server Equivalent

Windows Authentication (Kerberos) is the preferred SQL Server equivalent — also challenge-response, also immune to replay attacks. SQL Authentication with modern SQL Server versions uses strong internal hashing, but Microsoft still recommends Windows Auth for the same reasons PostgreSQL recommends SCRAM over md5.

SECURITY DEFINER #

Function attribute causing the function to execute with the privileges of the function's owner, not the caller. Equivalent to sudo for SQL — any user who can call the function gains the owner's access for the duration of the call. Critical risk: if SET search_path is not specified in the function definition, an attacker who can create objects in any schema can shadow functions called by the SECURITY DEFINER function, escalating to the owner's privileges.

-- Safe pattern: always pin search_path
CREATE OR REPLACE FUNCTION app.get_balance(p_account_id int)
RETURNS numeric
LANGUAGE sql
SECURITY DEFINER
SET search_path = app, pg_temp   -- critical
AS $$
  SELECT balance FROM app.accounts WHERE id = p_account_id;
$$;

-- Find SECURITY DEFINER functions WITHOUT pinned search_path:
SELECT n.nspname, p.proname, p.proconfig
FROM  pg_proc p
JOIN  pg_namespace n ON n.oid = p.pronamespace
WHERE p.prosecdef = true
  AND p.proconfig IS NULL;
SQL Server Equivalent

EXECUTE AS OWNER on stored procedures — the procedure runs with the owner's permissions. SQL Server's schema resolution is less flexible (one default schema per user), so the search_path hijacking attack vector doesn't apply in the same way.

SECURITY INVOKER #

Function attribute (the default) causing the function to execute with the privileges of the caller. The safest option for most use cases — the function cannot do anything the caller cannot already do. No privilege escalation is possible. In PostgreSQL 15+, views also support SECURITY INVOKER as an option.

-- Explicit declaration (same as default):
CREATE FUNCTION utils.sanitize_input(s text)
RETURNS text
LANGUAGE sql
SECURITY INVOKER  -- default, but explicit is clearer
AS $$ SELECT regexp_replace(s, '[^\w\s]', '', 'g'); $$;
SQL Server Equivalent

EXECUTE AS CALLER — the default for SQL Server stored procedures and functions. The execution context is the calling user's identity, exactly as in PostgreSQL's SECURITY INVOKER.

search_path #

Session-level GUC listing the schemas to search for unqualified object names, in priority order. When you write SELECT * FROM orders, PostgreSQL looks for orders in each schema in search_path left to right, using the first match. Default is "$user", public — the role's own schema first, then public. An attacker who can create objects in any schema that appears earlier in the path than the real object can shadow the real object (search_path hijacking).

-- Check current search_path:
SHOW search_path;

-- Set safe path for a session:
SET search_path = app, pg_temp;

-- Pin it permanently for a role (survives reconnect):
ALTER ROLE app_user SET search_path = app, pg_temp;

-- Pin it for a specific database:
ALTER DATABASE mydb SET search_path = app, pg_temp;
SQL Server Equivalent

DEFAULT_SCHEMA per user (ALTER USER u WITH DEFAULT_SCHEMA = dbo). SQL Server resolves only one schema by default, making it far less flexible — but also far less exploitable. The multi-schema search order that enables PostgreSQL's search_path hijacking does not exist in SQL Server.

SET ROLE #

SQL command that switches the current session's active role to a role that has been granted to the session user. All privilege checks use the new role's privileges for the remainder of the session or until RESET ROLE. Combined with NOINHERIT, this pattern requires explicit activation before using elevated privileges — similar to sudo requiring a deliberate invocation.

-- Switch to a granted role:
SET ROLE migration_role;

-- Run privileged DDL here...

-- Return to original session user:
RESET ROLE;

-- Confirm current role:
SELECT current_role, session_user;
SQL Server Equivalent

EXECUTE AS USER = 'username' to switch context, REVERT to return. SQL Server's EXECUTE AS is more commonly used in stored procedure definitions than as a session-level command, whereas PostgreSQL's SET ROLE is a first-class session operation.

superuser #

PostgreSQL role attribute that bypasses all privilege checks — object ownership, GRANTs, RLS policies, BYPASSRLS, and CREATEDB/CREATEROLE restrictions. A superuser can do anything to any database object. Should never be used for application connections, CI/CD service accounts, or monitoring agents. The principle of least privilege requires superuser to be reserved for break-glass administrative access only.

-- Find all superusers — run on every new instance:
SELECT rolname, rolcanlogin, rolconnlimit
FROM  pg_roles
WHERE rolsuper = true
ORDER BY rolname;

-- Remove superuser from a role:
ALTER ROLE legacy_admin NOSUPERUSER;
SQL Server Equivalent

sysadmin fixed server role. Members of sysadmin bypass all permission checks at the server level. The key difference: in SQL Server, sysadmin is a role you join; in PostgreSQL, superuser is an attribute directly on the role — a more granular but equally dangerous privilege.

T
TRUST auth #

pg_hba.conf auth method that accepts any connection for the specified role without any authentication at all. No password. No ticket. No certificate. The operating system itself is the only barrier. Legitimate only for local (Unix socket) entries in tightly controlled, single-tenant environments where the OS access controls are the security boundary. Never use trust on host or hostssl entries — it would allow any network client to connect as any role.

-- The only defensible TRUST usage (local socket, specific DB):
local   myapp   monitoring_agent   trust

-- This is catastrophically dangerous — never do this:
host    all     all     0.0.0.0/0   trust   -- DO NOT USE
SQL Server Equivalent

No equivalent. All SQL Server logins require either a password (SQL Authentication) or a valid Windows/Kerberos ticket (Windows Authentication). There is no mechanism to tell SQL Server to accept a connection with zero verification, even for local connections.

W
USING #

RLS policy clause that filters which existing rows are visible for SELECT, UPDATE, and DELETE operations. PostgreSQL appends the USING expression to the query's WHERE clause automatically. Only rows where the expression evaluates to true are returned or modifiable. Rows that fail the expression are silently hidden — not an error — which is the safe default for multi-tenant applications.

-- USING filters read access; tenant only sees own rows:
CREATE POLICY tenant_select ON orders
  AS PERMISSIVE FOR SELECT TO app_role
  USING (
    tenant_id = current_setting('app.tenant_id', true)::int
  );
SQL Server Equivalent

FILTER PREDICATE in a SQL Server Row-Level Security policy — applied to SELECT, UPDATE, and DELETE operations, with the same silent-hide behaviour. Syntax: ADD FILTER PREDICATE fn(col) ON table.

WITH CHECK #

RLS policy clause that validates rows being inserted or updated. While USING controls visibility of existing rows, WITH CHECK controls what rows can be written. If omitted on an INSERT or UPDATE policy, users can write rows they would not be able to read — enabling silent cross-tenant data injection. Always specify WITH CHECK to match the USING condition on write policies.

-- Both USING (read filter) and WITH CHECK (write guard):
CREATE POLICY tenant_isolation ON orders
  AS PERMISSIVE FOR ALL TO app_role
  USING (
    tenant_id = current_setting('app.tenant_id', true)::int
  )
  WITH CHECK (
    tenant_id = current_setting('app.tenant_id', true)::int
  );

-- If WITH CHECK is omitted for INSERT/UPDATE, a user
-- can insert rows for other tenants even though they
-- cannot SELECT them back.
SQL Server Equivalent

BLOCK PREDICATE with AFTER INSERT and AFTER UPDATE — blocks write operations after the row is about to be written, if the predicate fails. SQL Server separates read filtering (FILTER PREDICATE) and write blocking (BLOCK PREDICATE) into distinct predicate types; PostgreSQL unifies both into a single policy's USING + WITH CHECK clauses.
