When to Point, When to Copy

Go is pass-by-value. Every time you hand a variable to a function, Go makes a copy unless you explicitly ask it to pass the address instead. In this lesson you will build the same small feature twice: once with values, once with pointers. The first version will look correct but silently fail. The second version will work, and you will know exactly why.

The real-world scenario is a task status updater. A background worker needs to mark a task as completed and record when it finished. If the update function receives a copy, the caller's task never changes. That is the bug you are here to fix.

The Three Symbols

Pointers in Go use only three symbols. Learn them in this order:

Symbol Where it appears Meaning
&x In front of a variable Give me the address of x
*T In a type declaration A pointer to a value of type T
*p In front of a pointer variable Give me the value stored at the address p
KEY INSIGHT

*T is a type. *p is a read. &x is a write that asks "where is this stored?". The same star character plays two different roles, so pay attention to whether it is on the left side (type) or the right side (dereference) of an expression.

Attempt 1: Without Pointers

Here is the natural first attempt. The function takes a Task, marks it complete, and sets a timestamp. Run it in your head before you run it on your machine. 

package main

import (
    "fmt"
    "time"
)

type Task struct {
    ID        int
    Title     string
    Status    string
    Completed time.Time
}

func UpdateTask(t Task) {
    t.Status = "completed"
    t.Completed = time.Now()
    fmt.Printf("inside UpdateTask:  %s at %s\n", t.Status, t.Completed.Format("15:04:05"))
}

func main() {
    task := Task{ID: 1, Title: "Send welcome email", Status: "pending"}
    UpdateTask(task)
    fmt.Printf("back in main:      %s\n", task.Status)
}

The output will look like this: 

inside UpdateTask:  completed at 14:32:10
back in main:      pending
COMMON MISTAKE

UpdateTask modified a copy of the task. The original task in main is untouched. This is the single most common source of confusion for new Go programmers.

Attempt 2: With Pointers

To make the function modify the caller's task, change the parameter type to *Task and pass the address of the task with &task. Inside the function, t.Status still works because Go automatically dereferences struct pointers for field access. 

package main

import (
    "fmt"
    "time"
)

type Task struct {
    ID        int
    Title     string
    Status    string
    Completed time.Time
}

func UpdateTask(t *Task) {
    t.Status = "completed"
    t.Completed = time.Now()
    fmt.Printf("inside UpdateTask:  %s at %s\n", t.Status, t.Completed.Format("15:04:05"))
}

func main() {
    task := Task{ID: 1, Title: "Send welcome email", Status: "pending"}
    UpdateTask(&task)
    fmt.Printf("back in main:      %s\n", task.Status)
}

Now the output is consistent: 

inside UpdateTask:  completed at 14:32:10
back in main:      completed
WHY THIS WORKS

&task gives the address of the struct in main. UpdateTask receives that address as t *Task. When it writes t.Status, it is writing to the same memory as task.Status in main.

Where the Pointer Lives in Memory

A pointer is just a variable that holds a memory address. You can print it with %p to see that the address in the caller and the address inside the function are the same. 

func UpdateTask(t *Task) {
    fmt.Printf("address inside:   %p\n", t)
    t.Status = "completed"
}

func main() {
    task := Task{Status: "pending"}
    fmt.Printf("address in main:  %p\n", &task)
    UpdateTask(&task)
}

Both prints will show the same address. That is the connection between the caller and the function.

Pointer Receivers: Methods That Mutate

The same idea applies to methods. A method with a value receiver operates on a copy. A method with a pointer receiver operates on the original value. In real Go code, you will see methods like func (t *Task) Complete() far more often than global functions like UpdateTask. 

func (t *Task) Complete() {
    t.Status = "completed"
    t.Completed = time.Now()
}

func main() {
    task := Task{ID: 1, Title: "Send welcome email", Status: "pending"}
    task.Complete()
    fmt.Println(task.Status) // completed
}
KEY INSIGHT

Go automatically converts between values and pointers for method calls, but only when the address can be taken. You can call task.Complete() on a value, and the compiler will take &task for you. But you cannot call a pointer-receiver method on a value that is not addressable, such as a map element.

When to Use a Pointer: A Decision Table

Use this table as a first draft of your intuition. The full reference card is linked at the end of the lesson.

Choose *T when the function needs to Choose T when the value is
mutate the caller's data small and read-only, or the caller does not need to see changes
avoid copying a large struct a slice, map, or channel (already a reference-like header)
signal that the value might be absent (nil is meaningful) guaranteed to exist
share a single value across many callers a local intermediate result
keep a sync.Mutex or other synchronizer un-copied a basic type (int, string, bool) or a small struct
COMMON MISTAKE

Do not use a pointer just to avoid copying a few bytes. For small structs, the copy is cheap and the pointer adds the risk of aliasing and nil dereference. The Go Wiki advice is: if in doubt, use a pointer, but there are times when a value receiver makes sense, such as for small unchanging structs or values of basic type.
