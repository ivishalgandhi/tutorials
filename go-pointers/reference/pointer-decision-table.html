Pointer Decision Table

A one-page cheat sheet for choosing between T and *T in functions, methods, and return values.

Quick Rules

Use *T when Use T when
The function/method must mutate the caller's value. small and read-only, or the caller does not need to see changes
The struct is large and copying it is expensive. The struct is small and cheap to copy.
The value may be absent (nil is meaningful). The value is guaranteed to exist.
Multiple callers must share the same value. Each caller should have its own independent copy.
The struct contains a sync.Mutex or similar synchronizer. The value is a basic type (int, string, bool) or a small struct.

The Three Symbols

Symbol Context Meaning
&x Expression Address of x
*T Type Pointer to T
*p Expression Value at address p

Function Parameters

Signature Can mutate caller? Typical use
func f(t T) No Read-only helpers, small inputs
func f(t *T) Yes Mutators, large structs, optional values

Method Receivers

Receiver Can mutate? Notes
func (t T) M() No Operates on a copy; safe for small, immutable values.
func (t *T) M() Yes Can be called on a value if the address is addressable; cannot be called on a non-addressable map element.

Common Mistakes

  • Expecting mutation after passing by value. If a function receives T, changes inside it are lost.
  • Dereferencing a nil pointer. Always check if p == nil before reading *p or pointer fields.
  • Pointer receivers on non-addressable values. You cannot call func (t *T) M() on a map element or an interface value whose concrete value is not addressable.
  • Unintended aliasing. Two pointers to the same value can overwrite each other unexpectedly.

Standard-Library Examples

  • flag.String returns *string; you must dereference it with *word.
  • json.Unmarshal(data, &v) takes a pointer so it can fill v.
  • time.Time methods use value receivers because the type is designed to be immutable.
  • bytes.Buffer methods use pointer receivers because the buffer must mutate.
